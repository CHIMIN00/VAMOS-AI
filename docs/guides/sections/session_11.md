# §11. B-Series — 기억/스킬 자산 모듈 (B-1 ~ B-6, 총 6개)

> **비유**: S-Series가 VAMOS의 **자기 성장 시스템**이라면, B-Series는 VAMOS가 **배운 것을 기억하고 정리하는 도서관**입니다. 사람으로 비유하면, 공부한 내용을 노트에 정리하고, 오래된 노트는 버리고, 새로운 학습 계획을 세우고, 더 나은 공부법을 연구하는 **학습 자산 관리 시스템**과 같습니다.

## B-Series 핵심 개념: 4계층 메모리 구조

> B-Series를 이해하려면 먼저 VAMOS의 **4계층 메모리 구조**를 알아야 합니다.

```
┌─────────────────────────────────────────────────┐
│  L3: 절차 메모리 (Procedural)  ← B-2 연결       │
│  "방법(How-to)"을 저장. 무기한 보존              │
├─────────────────────────────────────────────────┤
│  L2: 글로벌 지식 (Semantic)    ← B-3 연결       │
│  검증된 사실/지식. 무기한 보존                    │
├─────────────────────────────────────────────────┤
│  L1: 프로젝트 메모리 (Episodic) ← B-1 연결      │
│  프로젝트별 기록. 90일 보존                       │
├─────────────────────────────────────────────────┤
│  L0: 세션 메모리 (Working)     ← B-4 연결       │
│  현재 대화 컨텍스트. 세션 종료 시 만료            │
└─────────────────────────────────────────────────┘
```

| 계층 | 정의 | B-Series 연결 | 보존 기간 |
|------|------|--------------|----------|
| L0 | 세션 메모리 (현재 대화) | B-4 Working | 세션 종료 시 (최대 30일) |
| L1 | 프로젝트 메모리 (프로젝트별 기록) | B-1 Episodic | 90일 (TTL 기본값) |
| L2 | 글로벌 지식 (검증된 사실) | B-3 Semantic | 무기한 |
| L3 | 절차/템플릿 (실행 가능한 방법론) | B-2 Procedural | 무기한 (deprecated 전환으로 폐기) |

[근거: D2.0-06 §2.1~2.4, LOCK]

## §11.1 B-1 Skill Library (스킬 라이브러리)

### 비유
> 자주 쓰는 요리 레시피를 모아둔 **레시피 북**. 한 번 성공한 요리법을 기록해두면, 다음에 같은 요리를 할 때 처음부터 고민하지 않아도 됩니다.

### 목적
재사용 가능한 스킬(skill, 특정 작업을 수행하는 방법)을 중앙에서 저장하고 관리합니다. 과거에 성공한 작업 패턴을 라이브러리에 등록하여, 비슷한 작업이 다시 올 때 빠르게 꺼내 쓸 수 있게 합니다. L3(절차 메모리) 또는 L2(글로벌 지식)의 procedural(절차적) 데이터에 의존합니다.

### 입력 / 출력

| 구분 | 내용 |
|------|------|
| 입력 | 성공한 작업 결과, 스킬 후보(candidate), 사용자 승인 |
| 출력 | 등록된 스킬 템플릿, 스킬 검색 결과, 스킬 버전 이력 |

### 상세 설명
- 스킬 템플릿은 **성공률/QoD(품질 점수) 추적** → 개선 후보 생성 → 승인 후 새 버전 등록의 흐름을 따릅니다
- 중앙 스킬 카탈로그에서 관리하며, §7.5 프롬프트 라이브러리와 연동됩니다
- 작업 중단 시 체크포인트(checkpoint, 진행 상태 자동 저장)를 L0 세션 메모리에 저장하여 복원 가능

### 관련 모듈
- B-2 (Procedural Memory): 스킬의 절차적 지식 저장소
- S-3 (Template Evolution): 스킬 템플릿 진화 연동
- I-18 (Self-evo Engine): 스킬 개선 제안 수집

### 분류 / 버전 / LOCK

| 항목 | 값 |
|------|-----|
| 분류 | EXP (실험적) |
| V1 / V2 / V3 | OFF / OFF / ON |
| LOCK | false (변경 가능) |
| Owner 문서 | D6 (Storage/Memory) |

## §11.2 B-2 Procedural Memory (절차적 기억)

### 비유
> 자전거 타는 법을 **몸이 기억하는 것**. 한번 배우면 잊지 않고, "어떻게 하는지(How-to)"를 실행 가능한 형태로 저장합니다.

### 목적
"실행 가능한 절차"를 저장하는 기억입니다. 플레이북(playbook, 단계별 작업 지침), 워크플로우(workflow, 작업 흐름), 템플릿(template, 반복 사용 양식), 루틴(routine, 반복 작업 자동화 규칙) 등을 L3 계층에 저장합니다. 답변 품질이 아니라 **행동(실행)을 바꾸므로** 저장/활성에 강한 거버넌스(governance, 관리 규칙)가 필요합니다.

### 입력 / 출력

| 구분 | 내용 |
|------|------|
| 입력 | 절차 후보(draft), PolicyCheck 결과, 사용자 승인 |
| 출력 | 승인된 절차(playbook/workflow/template/routine), 버전 이력 |

### 절차 저장 필수 필드

| 필드 | 설명 |
|------|------|
| procedure_id | 고유 식별자 |
| version | 버전 (semver 또는 증가형) |
| target_scope | 적용 범위 (global / project) |
| trigger_conditions | 언제 적용되는지 |
| steps | 절차 본문 (단계 배열) |
| required_tools | 필요 도구/권한 |
| safety_notes | 위험/제한/금지 사항 |
| activation_state | 상태: draft → approved → active → deprecated |
| provenance | 근거 (출처, 생성 주체/시각) |

### 생성/승격 파이프라인

```
1) Candidate 생성 (draft 상태)
       ↓
2) PolicyCheck (D7) — PASS / BLOCK / NEEDS_APPROVAL / ERROR
       ↓
3) Approval (D7) — PENDING → APPROVED / DENIED
       ↓
4) 저장 (write) — L3로 기록 + 버전 부여
       ↓
5) 활성 (active) — 실제 실행에 사용
```

### 버전별 L3 활성 상태

| 버전 | L3 상태 | 설명 |
|------|---------|------|
| V1 | **OFF** | L3 비활성. 절차/템플릿은 L1에 임시 저장만 허용 |
| V2 | **제한적 활성** | 사용자 승인된 playbook/template만 L3 저장 허용. 자동 활성화 조건: approved + 3회 이상 수동 실행 이력 |
| V3 | **전면 활성** | 자동 생성 routine 포함, DSPy 기반 자동 최적화 연동 |

### 접근 제어 (LOCK)
- L3 **읽기**: L1 이상 접근 권한 필요
- L3 **쓰기**: D7 ApprovalGate **필수**
- 벡터 검색 기본값: `top_k=5` (프로시저는 정밀도 우선)

> **참고**: 벡터 검색 백엔드(D2.0-06)의 기본값은 top_k=10이지만, API 레이어 B1과 프로시저 모듈은 정밀도를 위해 top_k=5를 사용합니다. [근거: v13 DELTA-011]

### 관련 모듈
- B-1 (Skill Library): 스킬 저장소와 연동
- D7 (Safety/Cost/Approval): 저장/활성에 반드시 게이트 통과 필요
- I-18 (Self-evo Engine): 절차 개선 제안

### 분류 / 버전 / LOCK

| 항목 | 값 |
|------|-----|
| 분류 | EXP (실험적) |
| V1 / V2 / V3 | OFF / OFF / ON |
| LOCK | false (변경 가능) |
| Owner 문서 | D6 (Storage/Memory) |

## §11.3 B-3 Memory Decay (기억 감쇠)

### 비유
> 냉장고의 **유통기한 관리 시스템**. 오래된 음식은 자동으로 "아직 먹을 수 있나요?"라고 물어보고, 너무 오래되면 정리합니다. 중요한 음식(pinned)은 유통기한 없이 보존됩니다.

### 목적
TTL(Time-To-Live, 생존 시간) 기반으로 오래된 메모리를 자동 감지하고, 신선도(freshness)가 낮은 메모리를 강등(demotion)하거나 삭제합니다. 메모리가 무한히 쌓이는 것을 방지하고, 항상 최신 정보가 우선 활용되도록 관리합니다.

### 입력 / 출력

| 구분 | 내용 |
|------|------|
| 입력 | 메모리 엔트리(entry), 마지막 접근 시각, TTL 정책 |
| 출력 | 신선도 점수(freshness_score), 강등/삭제 후보 목록, 사용자 확인 요청 |

### TTL 기본값 (LOCK — 변경 불가)

| 계층 | 기본 TTL | 비고 |
|------|----------|------|
| L0 (세션) | 세션 종료 시 만료 | 명시적 저장 요청 시 L1로 승격 |
| L1 (프로젝트) | **90일** | 만료 전 사용자 알림/승인 필요 |
| L2 (글로벌) | **무기한** | 민감정보 제외 원칙 유지 |
| L3 (절차) | 정책 기반 | draft/approved/active/deprecated 상태 + 버전 관리 |

### 신선도 점수 계산 (Freshness Score)

```
freshness_score = 1.0 - (days_since_last_access / max_days)
```

- `freshness < 0.3` → "아직 유효한가?" 확인 트리거 발생
- `freshness < 0.2` → 아카이브 전환 대상 (V2+ 확장)
- 주간 배치(batch, 일괄 처리)로 전체 메모리 신선도 재평가

### 메모리 승격 / 강등 규칙

| 전환 | 조건 | 스코어링 공식 |
|------|------|-------------|
| L0→L1 | 세션 종료 시 자동 (요약 생성) | — |
| L1→L2 | 재참조 3회+ OR 사용자 확인 OR QoD ≥ 0.7 | `access_count×0.4 + recency×0.3 + confidence×0.3` |
| L2→L3 | 크로스 프로젝트 재참조 OR 범용 사실 | — |
| L3→Archive | 180일 미참조 (V2+ 확장) | freshness < 0.2 |
| Archive→삭제 | 1년 미참조 + 사용자 확인 (V2+ 확장) | — |

### 예외 규칙
- `pinned=true`인 메모리는 강등/삭제 **제외**
- 강등 전 반드시 사용자에게 "이 메모리를 보관할까요?" 확인

### 관련 모듈
- 모든 L0~L3 메모리 계층과 연동
- D7 (Safety/Cost/Approval): TTL 정책 적용

### 분류 / 버전 / LOCK

| 항목 | 값 |
|------|-----|
| 분류 | **CORE** (핵심 — B-Series 유일한 CORE 모듈) |
| V1 / V2 / V3 | **ON / ON / ON** (항상 활성) |
| LOCK | false (변경 가능) |
| Owner 문서 | D6 (Storage/Memory) |

## §11.4 B-4 Auto Curriculum Generator (자동 커리큘럼 생성기)

### 비유
> 학생의 성적표를 분석해서 **맞춤형 학습 계획표**를 자동으로 만들어주는 AI 과외 선생님. 약한 과목은 더 많이, 강한 과목은 적게 배정합니다.

### 목적
VAMOS의 성능 데이터와 실패 로그를 분석하여 **학습 경로(curriculum)를 자동 생성**합니다. 어떤 영역이 부족한지 파악하고, 해당 영역을 집중 훈련할 수 있는 학습 계획을 제안합니다. S-7(User-Coop Designer)과 연동하여 메타 학습(학습 방법 자체를 학습)을 지원합니다.

|------|------|
| 입력 | 성능 로그, 실패 기록, QoD 점수 이력, 도메인별 통계 |
| 출력 | 학습 경로 제안, 훈련 우선순위 목록, 개선 예상 효과 |

### 관련 모듈

## §11.5 B-5 RL-like Self Trainer (강화학습형 자기 훈련)

### 비유
> 게임을 반복하면서 **높은 점수를 받은 전략은 더 자주 쓰고, 낮은 점수 전략은 버리는** 게이머. 보상(reward)과 벌칙(penalty)을 통해 스스로 더 나은 방법을 찾아갑니다.

### 목적
강화학습(RL, Reinforcement Learning) 방식으로 VAMOS 스스로가 **보상 기반 자기 개선**을 수행합니다. 성공한 행동에는 높은 보상을, 실패한 행동에는 낮은 보상을 부여하여, 점점 더 나은 응답 전략을 학습합니다. Reflexion(반성) 패턴과 연동하여 "생성 → 평가 → 반성 → 재생성" 루프를 수행합니다.

### 입력 / 출력

| 구분 | 내용 |
|------|------|
| 입력 | 행동 결과, 보상 신호(QoD 점수, 성공/실패), 반성(reflection) 힌트 |
| 출력 | 개선된 전략 후보, 반성 기록 (L1 episodic memory로 저장) |

### Reflexion 패턴 (반성 루프)

```
답변 생성 → 답변 평가 → 개선 힌트 생성 → 재생성 (최대 2회 반복)
```

- 반성 결과는 L1(프로젝트 메모리)에 episodic memory(사건 기억)로 저장하여 재사용
- 적용 범위: V3 (연구/실험 단계)

### 관련 모듈
- S-4 (Error Pattern Miner): 실패 분석 데이터 제공
- S-5 (Router Evolution): 도메인별 최적화
- D7 (Safety/Cost/Approval): 개선 적용 시 승인 필요

### 분류 / 버전 / LOCK

|------|-----|
| 분류 | EXP (실험적) |
| V1 / V2 / V3 | OFF / OFF / ON |
| LOCK | false (변경 가능) |

## §11.6 B-6 DSPy Prompt Optimizer (프롬프트 최적화기)

### 비유
> 같은 질문을 **여러 가지 말투로 물어보고**, 가장 좋은 답변이 나오는 말투를 찾아내는 **질문 최적화 전문가**. 자동으로 최적의 프롬프트(prompt, AI에게 주는 지시문)를 찾아줍니다.

### 목적
DSPy(Declarative Self-improving Python, 선언적 자기 개선 파이썬 프레임워크) 기반으로 **프롬프트를 자동 최적화**합니다. 소량의 예시 데이터만으로 프롬프트와 파라미터(parameter, 설정값)를 자동 튜닝(tuning, 미세 조정)하여 최적의 성능을 이끌어냅니다.

### 입력 / 출력

| 구분 | 내용 |
|------|------|
| 입력 | 기존 프롬프트, 평가 데이터(QoD 점수, 성공률, 비용), 예시 셋 |
| 출력 | 최적화된 프롬프트 (신규 버전), 평가 결과 비교 리포트 |

### DSPy 핵심 개념

| 개념 | 설명 |
|------|------|
| Signature (시그니처) | `retrieve → generate` 파이프라인을 선언적으로 정의 |
| Compiler (컴파일러) | 소량 예시로 프롬프트/파라미터 자동 튜닝 |
| Evaluation (평가) | Accuracy(정답률), Faithfulness(충실도), Relevance(관련성), Toxicity(유해성), Cost-efficiency(비용 효율) |

### 관련 모듈
- I-13 (Multimodal Output Renderer): 프롬프트 컨텍스트 관리 연동
- S-3 (Template Evolution): 템플릿 진화 연동
- §7.5 프롬프트 라이브러리: 최적화 결과 등록

### 분류 / 버전 / LOCK

| 항목 | 값 |
|------|-----|
| 분류 | EXP (실험적) |
| V1 / V2 / V3 | OFF / OFF / ON |
| LOCK | false (변경 가능) |
| Owner 문서 | D5 (Agent Workflow) |

## 11.7.1 프롬프트 버전 관리 체계

### 버전 관리 규칙 (ADD-063)

| 항목 | 규칙 |
|------|------|
| 버전 형식 | **Semantic Versioning** (의미적 버전 관리): `MAJOR.MINOR.PATCH` |


| 지표 | 설명 |

| Relevance (관련성) | 질문 대비 답변 적합도 |
| Toxicity (유해성) | 안전 가이드라인 위반 여부 |
| Cost-efficiency (비용 효율) | 품질 대비 비용 |

| 결과 기록 | QoD 점수, 성공률, 비용을 LogEvent에 기록 |

### 핵심 요약 (3줄)

1. **프롬프트 버전 관리**: Semantic Versioning(MAJOR.MINOR.PATCH)으로 모든 프롬프트를 관리하며, 덮어쓰기 금지, 롤백은 이전 버전 참조 방식만 허용됩니다.

3. **A/B 테스트**: P1 이하만 자동 실행, P2 이상은 승인 필요이며, 정책 위반 시 즉시 중단 + 양쪽 롤백합니다.

## §12.1 C-1 Logic Verifier (논리 검증기)

### 비유
> 토론 대회의 **심판**. "앞에서는 A라고 했는데 뒤에서는 A가 아니라고 하네요? 모순입니다!" 라고 논리적 오류를 잡아내는 역할입니다.

### 목적
VAMOS가 생성한 답변의 **논리적 일관성(logical consistency)**을 검증합니다. 주장과 근거 사이에 모순이 없는지, 전제와 결론이 논리적으로 연결되는지 확인합니다. 제약 문제(constraint, 조건 충돌)가 발견되면 EVX-6(Z3 Solver, 수학적 증명 도구)로 분기하여 정밀 검증합니다.

### 입력 / 출력

| 구분 | 내용 |
|------|------|
| 입력 | 생성된 답변, 사용된 근거(evidence), 논리식(logical expression) |
| 출력 | VerifyResult(passed: bool, errors: List) — 통과 여부 + 오류 목록 |

### 에러코드 / 폴백
- 검증 실패 시: 오류 목록과 함께 재생성 또는 사용자 경고
- 제약 문제 발견 시: EVX-6 Z3 Solver로 분기 (조건부)

### 관련 모듈
- I-24 (Knowledge Graph Engine): 근거 그래프 기반 논리 검증
- D7 (Safety/Cost/Approval): 검증 결과 기록
- EVX-6 (Z3 Solver): 제약 문제 정밀 분석

### 분류 / 버전 / LOCK

| 항목 | 값 |
|------|-----|
| 분류 | **CORE** (핵심) |
| V1 / V2 / V3 | **ON / ON / ON** (항상 활성) |
| LOCK | false (변경 가능) |
| Owner 문서 | D4 (Infra/Core) |

## §12.2 C-2 Math Verifier (수학 검증기)

### 비유
> **계산기로 답을 다시 확인하는 수학 선생님**. AI가 "2+3=6"이라고 답했다면, SymPy(수학 계산 라이브러리)로 다시 계산해서 "틀렸습니다, 정답은 5입니다"라고 교정합니다.

### 목적
VAMOS가 포함한 **수학 계산의 정확성**을 검증합니다. SymPy(Symbolic Python, 기호 수학 라이브러리)를 사용하여 수식을 독립적으로 재계산하고, 결과가 일치하는지 확인합니다.

### 입력 / 출력

| 구분 | 내용 |
|------|------|
| 입력 | 답변에 포함된 수식, 계산 과정, 수치 데이터 |
| 출력 | VerifyResult(passed: bool, errors: List) — 검증 통과 여부 + 오류 상세 |

### 에러코드 / 폴백
- 계산 불일치 시: 올바른 계산 결과와 함께 오류 보고
- SymPy 처리 불가 수식: 경고 + 사용자에게 수동 확인 요청

### 관련 모듈
- I-24 (Knowledge Graph Engine): 수치 데이터 근거 확인
- C-1 (Logic Verifier): 논리 + 수학 복합 검증
- D7 (Safety/Cost/Approval): 검증 결과 기록

## §12.3 C-3 Code Verifier (코드 검증기)

### 비유
> **코드 리뷰어 + 테스트 실행기**. AI가 작성한 코드를 1) 구조 분석(AST)으로 문법 오류를 찾고, 2) 실제로 실행(pytest)해서 결과를 확인하는 **이중 검증 시스템**입니다.

### 목적
VAMOS가 생성한 **코드의 정확성**을 검증합니다. AST(Abstract Syntax Tree, 추상 구문 트리) 분석으로 문법 구조를 확인하고, lint(코드 스타일 검사) + pytest(테스트 실행)으로 실제 동작을 검증합니다. 모든 코드/도구 실행은 **Docker 샌드박스(sandbox, 격리 환경)** 안에서만 이루어집니다.

### 입력 / 출력

| 구분 | 내용 |
|------|------|
| 입력 | 생성된 코드, 테스트 케이스, 실행 환경 설정 |
| 출력 | VerifyResult(passed: bool, errors: List) — 검증 통과 여부 + 오류 상세 |

### 샌드박스 규칙 (LOCK — 변경 불가)

| 항목 | 값 |
|------|-----|
| 실행 환경 | Docker 샌드박스 **필수** |
| 네트워크 | **차단** (외부 접근 불가) |
| 실행 시간 제한 | **30초** 타임아웃 |
| 미검증 코드 | 샌드박스 검증 통과 후에만 채택 |

### 에러코드 / 폴백
- AST 파싱 실패: 문법 오류 상세 보고
- 테스트 실패: 실패 케이스와 예상/실제 결과 비교
- 타임아웃: 30초 초과 시 자동 중단 + 경고

### 관련 모듈
- D4 (Infra/Core): 도구 실행 레이어
- D7 (Safety/Cost/Approval): 검증 결과 기록

### 분류 / 버전 / LOCK

| 항목 | 값 |
|------|-----|
| 분류 | **CORE** (핵심) |
| V1 / V2 / V3 | **ON / ON / ON** (항상 활성) |
| LOCK | false (변경 가능) |
| Owner 문서 | D4 (Infra/Core) |

## §12.4 C-4 Domain Simulator (도메인 시뮬레이터)

### 비유
> **비행 시뮬레이터**처럼, 실제로 행동하기 전에 **가상 환경에서 시나리오를 먼저 돌려보는** 시스템. 금융 투자, 법률 판단, 의료 진단 등 위험한 영역에서 "만약 이렇게 하면 어떤 결과가 나올까?"를 미리 시뮬레이션합니다.

### 목적
금융/법률/의료 등 **고위험 도메인에서 시나리오 시뮬레이션**을 수행합니다. 실제 행동 전에 가상 환경에서 결과를 예측하여 위험을 줄입니다. Simulator Tool/Adapter를 경유하며, 반드시 D7 Approval + CostGate를 통과해야 실행됩니다.

### 입력 / 출력

| 구분 | 내용 |
|------|------|
| 입력 | 시나리오 정의, 도메인 파라미터, 시뮬레이션 조건 |
| 출력 | 시뮬레이션 결과, 위험 분석, 예측 신뢰도 |

### 에러코드 / 폴백
- 도메인 미지원 시: "해당 도메인의 시뮬레이터가 없습니다" 경고
- 승인 미통과 시: 실행 차단 (자동 활성 절대 금지)

### 관련 모듈
- D7 (Safety/Cost/Approval): 실행 승인 필수
- D4 (Infra/Core): Simulator Tool/Adapter

### 분류 / 버전 / LOCK

## §12.5 C-5 Bayesian Belief Engine (베이지안 추론 엔진)

### 비유
> **날씨 예보관**처럼 확률을 계산하는 시스템. "비가 올 확률 70%, 맑을 확률 30%"처럼, 불확실한 상황에서 **새로운 정보가 들어올 때마다 확률을 업데이트**하여 점점 더 정확한 판단을 내립니다.

### 목적
**베이지안 확률 추론(Bayesian probabilistic reasoning)**을 수행합니다. 불확실한 상황에서 사전 확률(prior, 기존 믿음)과 새로운 증거(evidence)를 결합하여 사후 확률(posterior, 업데이트된 믿음)을 계산합니다. 의사결정 지원 모듈인 I-5(Condition & Decision Engine)와 연동합니다.

### 입력 / 출력

| 구분 | 내용 |
|------|------|
| 입력 | 사전 확률 분포, 새로운 증거 데이터, 추론 대상 |
| 출력 | 사후 확률 분포, 신뢰 구간, 추론 근거 |

### 에러코드 / 폴백
- 데이터 부족 시: 신뢰도 경고 + 추가 데이터 요청
- 승인 미통과 시: 실행 차단

### 관련 모듈
- I-5 (Condition & Decision Engine): 확률적 의사결정 지원
- D7 (Safety/Cost/Approval): CostGate 통과 필요

### 분류 / 버전 / LOCK

| 항목 | 값 |
|------|-----|
| 분류 | EXP (실험적) |
| V1 / V2 / V3 | OFF / OFF / **ON** |
| LOCK | false (변경 가능) |
| Owner 문서 | D2 (Orange Core) |

## §12.6 C-6 RL Advisor (강화학습 어드바이저)

### 비유
> 체스 게임의 **AI 코치**. 현재 상황(state)에서 어떤 수(action)를 두면 가장 좋은 결과(reward)가 나올지 **정책 네트워크(policy network)**를 기반으로 조언합니다.

### 목적
강화학습(RL, Reinforcement Learning) 기반으로 **행동 최적화 조언**을 제공합니다. 현재 상태에서 가능한 행동들의 예상 보상을 계산하고, 최적의 행동을 추천합니다. RL 정책 네트워크를 사용하여 적응적(adaptive) 추천을 수행합니다.

### 입력 / 출력

| 구분 | 내용 |
|------|------|
| 입력 | 현재 상태(state), 가능한 행동 목록(actions), 보상 함수(reward function) |
| 출력 | 추천 행동, 예상 보상 점수, 행동별 비교 분석 |

### 에러코드 / 폴백
- 학습 데이터 부족 시: 기본 휴리스틱(heuristic, 경험 규칙) 폴백
- 승인 미통과 시: 실행 차단

### 관련 모듈
- B-5 (RL-like Self Trainer): RL 학습 데이터 공유
- D7 (Safety/Cost/Approval): CostGate 통과 필요

### 분류 / 버전 / LOCK

| 분류 | EXP (실험적) |
| V1 / V2 / V3 | OFF / OFF / **ON** |
| LOCK | false (변경 가능) |

## §12.7 C-7 GNN Score Model (그래프 신경망 스코어 모델)

### 비유
> **인맥 지도 분석가**. 사람들(노드)의 관계(엣지)를 그래프로 그려서, "이 사람은 핵심 인물이다", "이 정보는 신뢰도가 높다"를 **관계 구조를 기반으로 점수**를 매기는 시스템입니다.

### 목적
**지식 그래프(Knowledge Graph)**에서 노드(node, 개체)와 엣지(edge, 관계)의 구조를 분석하여 **관계 기반 점수**를 산출합니다. GNN(Graph Neural Network, 그래프 신경망)을 사용하여 정보의 중요도, 신뢰도, 관련성을 그래프 구조로부터 추론합니다.

### 입력 / 출력

| 구분 | 내용 |
|------|------|
| 입력 | 지식 그래프 데이터, 대상 노드/관계, 점수 산출 기준 |
| 출력 | 관계 점수(relationship score), 중요도 순위, 추론 근거 |

### 기술 의존성
- `torch` (PyTorch): 딥러닝 프레임워크
- `torch-geometric`: 그래프 신경망 라이브러리

### 에러코드 / 폴백
- 그래프 데이터 부족 시: 기본 유사도(similarity) 점수 폴백
- 모델 로드 실패 시: 경고 + 대체 스코어링

### 관련 모듈
- I-24 (Knowledge Graph Engine): 지식 그래프 데이터 소스
- D6 (Storage/Memory): 그래프 저장소 연동
- D2 (Orange Core): 점수 기반 의사결정

### 분류 / 버전 / LOCK

| 항목 | 값 |
|------|-----|
| 분류 | EXP (실험적) |
| V1 / V2 / V3 | OFF / OFF / **ON** |
| LOCK | false (변경 가능) |
| Owner 문서 | D4 (Infra/Core) |

### C-Series 핵심 요약 (3줄)

1. **C-1, C-2, C-3는 CORE 검증기**로 V1부터 항상 활성이며, 논리/수학/코드를 독립적으로 검증하는 VAMOS의 핵심 품질 보증 시스템입니다.

3. **모든 검증 결과는 verify.chain_used에 기록**되며, EXP 모듈은 D7 승인/비용 게이트 통과 없이 절대 자동 ON 금지입니다.
