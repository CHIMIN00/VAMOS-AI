# §41. Self-Evolution — 자기 진화 시스템

> **비유**: VAMOS의 자기 진화(Self-Evolution)는 자동차가 스스로 엔진 오일을 교환하고, 타이어 공기압을 조절하는 것과 비슷합니다. 단, 자동차의 핸들이나 브레이크 같은 핵심 안전장치는 절대 바꾸지 않습니다 — 그건 사람만 할 수 있습니다.

**자기 진화 (Self-Evolution)** 란 AI 시스템이 자신의 성능을 분석하고, 스스로 개선 방안을 찾아 적용하는 기능입니다. VAMOS는 사용 패턴, 실패 사례, 피드백을 분석하여 "더 나은 답변을 하려면 어떻게 해야 할까?"를 스스로 고민하고 개선을 **제안**합니다.

핵심은 "제안만 가능, 적용은 사용자 승인 필수"라는 원칙입니다.

[근거: BASE-1.3 §6, D2.0-07 §15.14]

---

## §41.1 자기 진화란? — 스스로 업그레이드하는 소프트웨어

> **비유**: 게임 캐릭터가 경험치를 쌓아 레벨업하되, 어떤 스킬을 올릴지는 항상 플레이어(사용자)가 결정하는 구조입니다.

자기 진화 시스템은 아래 흐름으로 동작합니다:

```
사용 데이터 수집 → 패턴 분석 → 개선안 생성 → 사용자에게 제안 → 승인 → 적용
                                                    ↗ 거부 → 미적용
```

**Self-evo의 3단계**:

| 단계 | 설명 | AI 단독 가능? |
|------|------|:---:|
| **1. 분석/진단** | 로그·사용 패턴·실패 사례 분석 | ✅ 가능 |
| **2. 제안 생성** | 개선안을 만들어 사용자에게 보여줌 | ✅ 가능 |
| **3. 변경 적용** | 실제로 설정이나 로직을 변경 | ❌ 승인 필수 |

[근거: BASE-1.3 §6.3, D2.0-07 §3.3 S-Module 승인 규칙]

## §41.2 변경 가능한 6가지 영역

> **비유**: AI가 스스로 바꿀 수 있는 것은 "옷을 갈아입는 것"과 같습니다. 외모는 바꿀 수 있지만, 성격(정체성)은 바꿀 수 없습니다.

VAMOS는 아래 6가지 영역에서만 자기 진화가 **파라미터 수준으로 자동 조정 가능**합니다 (구조 변경은 승인 필요):

| # | 변경 가능 영역 | 설명 | 예시 |
|---|--------------|------|------|
| 1 | **프롬프트 템플릿** (Prompt Template) | AI에게 보내는 지시문의 문구 개선 | "더 정확한 답변을 위해 근거 제시 방식 조정" |
| 2 | **라우팅 가중치** (Routing Weight) | 어떤 모델에게 어떤 질문을 보낼지의 비중 조정 | "간단한 질문을 더 저렴한 Mini 모델로 보내는 비율을 높임" |
| 3 | **캐시 정책** (Cache Policy) | 이전 답변을 재활용하는 규칙 조정 | "같은 질문에 대한 캐시 유효 시간을 연장" |
| 4 | **도구 파라미터** (Tool Parameter) | 외부 도구 호출 시 사용하는 설정값 조정 | "웹 검색 결과 개수를 5개→10개로 증가" |
| 5 | **검색 파라미터** (Search Parameter) | RAG 검색의 필터링 기준 조정 | "검색 결과 품질 점수(QoD) 가중치 미세 조정 (±0.05)" |
| 6 | **스킬 라이브러리** (Skill Library) | 도구 조합 패턴 최적화 | "코드 리뷰 작업 시 린터→테스트→리뷰 순서 최적화" |

> ⚠️ **중요**: 위 영역이라도 **로직(알고리즘 구조) 변경**은 파라미터 조정이 아니므로 반드시 사용자 승인이 필요합니다. 예: 라우팅 모델 자체를 교체하는 것은 자동 불가.

[근거: BASE-1.3 §6.1, PLAN 3.0 §4.5.2]

## §41.3 변경 불가 7가지 영역 (LOCK) 🔒

> **비유**: 이 7가지는 건물의 기둥과 같습니다. 기둥을 빼면 건물 전체가 무너지듯, 이 영역은 어떤 상황에서도 AI가 변경할 수 없습니다.

| # | 변경 불가 영역 (LOCK 🔒) | 왜 변경 불가인가? | 근거 |
|---|------------------------|-----------------|------|
| 1 | **비용 상한** (Cost Ceiling) | V1: ₩40,000/월, V2: ₩93,000/월, V3: ₩266,000/월 — 무한 과금 방지 | BASE-1.3 §5 |
| 2 | **승인 규칙** (Approval Flow) | 2단계 승인 구조 (계획→실행) — 위험한 행동의 안전장치 | BASE-1.3 §9.2 |
| 3 | **보안 정책** (Safety Rules) | Non-goal 7개 절대 금지사항 — 실거래, 불법 행위 등 | BASE-1.3 §2 |
| 4 | **RBAC** (역할 기반 접근 제어) | OWNER/ADMIN/OPERATOR/VIEWER 4단계 권한 체계 | D2.0-07 §3.6 |
| 5 | **Gate 우선순위** (Gate Priority) | Policy→Cost→Approval→Evidence→SelfCheck 순서 고정 | D2.0-07 §5 |
| 6 | **9-State 상태 머신** (Pipeline States) | S0→S1→...→S8 파이프라인 상태 전이 순서 | PART2 §7.5.5 |
| 7 | **Decision Lock** (단일 결정 잠금) | 하나의 요청에 하나의 결정만 — 결정 후 변경 불가 | PLAN 3.0 |

> 🔒 **절대 원칙**: Self-evo는 성능을 **강화**하는 방향에서만 허용됩니다. 안전 기준을 **완화**하는 방향의 변경은 어떤 영역에서든 금지입니다.

[근거: BASE-1.3 §6.2, D2.0-07 §15.14.1, PART2 §7.5.5]

## §41.4 롤백 정책 — 되돌리기 안전장치

> **비유**: 게임에서 "실행 취소(Undo)" 버튼과 같습니다. AI가 변경을 적용했는데 문제가 생기면, 즉시 이전 상태로 되돌릴 수 있습니다.

### 롤백 트리거 조건 (자동으로 되돌리는 상황)

| # | 트리거 조건 | 설명 |
|---|-----------|------|
| 1 | **오류율 급증** | Self-evo 적용 후 오류율이 기존 대비 +20% 초과 |
| 2 | **Non-goal 위반 감지** | 적용된 변경이 절대 금지사항에 해당함이 사후 발견 |
| 3 | **사용자 요청** | 사용자가 명시적으로 "되돌려줘"라고 요청 |
| 4 | **안전 필터 연속 실패** | 안전 필터가 연속 3회 이상 실패 |
| 5 | **비인가 정책 변경** | 승인 없이 정책이 변경된 것이 감지됨 |

### 롤백 절차

```
1. 변경 전 스냅샷 (Snapshot, 현재 상태의 사진) 자동 저장
   ↓
2. 변경 적용
   ↓
3. 이상 감지? ──→ (예) → 즉시 이전 스냅샷으로 복원
   ↓                      + 롤백 이벤트 로그 기록
  (아니오)                 + 사용자 알림
   ↓
4. 정상 운영 계속
```

### 스냅샷 보관 정책

| 항목 | V1 | V2+ |
|------|-----|------|
| **저장 방식** | 파일 기반 스냅샷 | DB 트랜잭션 기반 포인트-인-타임 복원 |
| **보관 개수** | 최근 3회 | 최소 10개 |
| **롤백 대상** | 설정, 행동 파라미터 | 설정, 메모리, KG, 에이전트 상태, 정책 |

> ⚠️ **재적용 잠금**: 롤백된 변경을 다시 적용하려면 추가 승인(Approval)이 반드시 필요합니다. 같은 실수를 반복하지 않기 위한 안전장치입니다.

[근거: D2.0-07 §3.4 ADD-074, D2.0-07 STEP7 S7E-071]

## §41.5 Constitutional AI + DPO — 피드백 학습

> **비유**: Constitutional AI는 AI에게 "헌법"을 주는 것이고, DPO는 "사용자의 취향을 배우는 것"입니다. 헌법을 지키면서 사용자가 좋아하는 답변 스타일을 학습합니다.

### Constitutional AI (헌법 기반 자기 제한)

AI가 스스로 "이 답변이 원칙에 맞는가?"를 검토하는 시스템입니다.

**VAMOS AI 헌법 4대 원칙**:

| # | 원칙 | 설명 | 위반 시 조치 |
|---|------|------|-------------|
| 1 | **해악 금지** (Non-maleficence) | 사용자·제3자·사회에 해를 끼치는 행동 거부 | 응답 차단 + 안전 대안 제시 |
| 2 | **정직성** (Honesty) | 불확실한 내용은 불확실하다고 명시, 허위 정보 금지 | 불확실성 표시 + 근거 제시 |
| 3 | **자율성 존중** (Autonomy) | 사용자의 결정권을 침해하거나 조종 금지 | 선택지 제시 방식으로 수정 |
| 4 | **투명성** (Transparency) | AI임을 숨기거나 능력을 과장하지 않음 | 능력 범위 명시 |

**자기 비판(Critique) 루프**:
```
응답 생성 → 헌법 원칙 대조 → 위반 발견? → (예) → 응답 수정 후 재생성
                               ↓
                             (아니오) → 응답 출력
```

> 🔒 **헌법 변경은 RULE 1.3 BASE 변경에 해당**하므로 반드시 OWNER 승인 필수 (변경 불가 영역)

### DPO (Direct Preference Optimization, 직접 선호 최적화)

사용자의 "좋아요/싫어요" 피드백을 통해 응답 품질을 개선하는 기법입니다.

| 항목 | 설명 |
|------|------|
| **피드백 수집** | 사용자의 응답 평가 (좋음/나쁨/수정 요청) → 구조화된 이벤트로 기록 |
| **학습 데이터** | 선호/비선호 쌍 데이터를 누적 |
| **적용 범위** | 로컬 선호도 프로파일 업데이트에만 사용 (외부 전송 금지 🔒) |
| **안전장치** | Non-goal 위반 응답은 학습 데이터에서 자동 제외 |
| **승인 연동** | 피드백 기반 행동 변경이 정책 범위를 벗어나면 승인 필수 |

[근거: D2.0-07 §3.5 ADD-042~043]

## §41.6 Red Teaming / Bias Detection / Watermarking

> **비유**: Red Teaming은 은행 금고의 도둑 시뮬레이션, Bias Detection은 저울의 균형 검사, Watermarking은 지폐의 위조 방지 마크입니다.

### Red Teaming (적대적 테스트)

AI 시스템의 약점을 찾기 위해 **일부러 공격하는 테스트**입니다.

| 테스트 유형 | 설명 | 빈도 |
|-----------|------|------|
| 프롬프트 주입 테스트 | 시스템 프롬프트 탈취 시도 | 배포 전 + 월 1회 |
| 탈옥(Jailbreak) 테스트 | Non-goal 우회 시도 | 배포 전 + 월 1회 |
| 역할 혼동 테스트 | RBAC 권한 우회 시도 | 분기 1회 |
| 비용 조작 테스트 | CostBudget 우회 시도 | 분기 1회 |
| 데이터 유출 테스트 | PII/민감정보 추출 시도 | 배포 전 + 월 1회 |

> 🎯 **목표**: Red Team 방어율 ≥ 90% (분기 1회 측정) **LOCK 🔒 하향 불가**

**버전별 Red Teaming 도구**:
- **V1**: 커스텀 테스트 세트 (수동)
- **V2**: garak 프레임워크 + PyRIT (Microsoft) 자동화
- **V3**: AI 기반 자동 위험도 평가 + 커뮤니티 평판 점수

### Bias Detection (편향 감지)

AI가 특정 집단에 대해 **불공평한 답변**을 하는지 감지하는 시스템입니다.

| 탐지 대상 | 방법 | 지표 |
|----------|------|------|
| 성별/연령/인종/국적/장애 편향 | 대조 테스트 (같은 질문, 다른 인구통계 맥락) | DPD (Demographic Parity Difference) ≤ 0.1 |
| 직업/소득/교육 편향 | 임베딩 공간 편향 분석 | EOD (Equalized Odds Difference) ≤ 0.1 |
| 언어 편향 (한국어/영어 품질 차이) | 출력 통계 비교 (감성/톤/길이) | 분기 1회 측정 |

**편향 발견 시**: Constitutional AI 공정성 원칙에 따라 재생성 + 로깅

### Watermarking (AI 출력물 워터마크)

AI가 생성한 콘텐츠에 **보이지 않는 출처 표시**를 넣는 기술입니다.

| 기법 | 설명 | 적용 버전 |
|------|------|----------|
| 메타데이터 워터마킹 | 출력에 보이지 않는 메타데이터 태그 삽입 | V2+ |
| 텍스트 워터마킹 | 토큰 선택 확률 분포 미세 조정으로 검출 가능한 패턴 삽입 | V3 |
| C2PA 콘텐츠 인증 | 국제 표준(C2PA) 기반 콘텐츠 출처 인증 | V3 |

> ⚠️ **한계**: 50토큰 미만 소량 텍스트에서는 검출률 저하, 재작성/패러프레이즈 시 손실 가능

[근거: D2.0-07 §15.5]

## §41.7 Hallucination Detection — NLI 기반 환각 탐지

> **비유**: AI의 "팩트체크 기자"입니다. AI가 말한 내용이 실제 자료와 맞는지 하나하나 대조하여, "그럴듯한 거짓말(환각)"을 잡아냅니다.

**환각 (Hallucination)** 이란 AI가 사실이 아닌 내용을 마치 사실인 것처럼 생성하는 현상입니다. **NLI (Natural Language Inference, 자연어 추론)** 모델을 사용해 탐지합니다.

### 환각 탐지 아키텍처

```
AI 응답 → [사실 추출기] → 핵심 주장(Claims) 목록
              ↓
         [NLI 모델] ← 소스 문서 / 지식 베이스
              ↓
         각 주장별: Entailment(일치) / Contradiction(모순) / Neutral(무관)
              ↓
         [환각 판정기] → 환각 비율 계산 → 임계값 초과 시 경고/차단
```

### 환각 유형 분류

| 유형 | 설명 | 심각도 | 대응 |
|------|------|--------|------|
| **사실 환각** (Factual) | 존재하지 않는 사실 생성 | HIGH | 차단 + 재생성 |
| **출처 환각** (Source) | 존재하지 않는 출처 인용 | HIGH | 차단 + 출처 제거 |
| **추론 환각** (Reasoning) | 부정확한 논리적 추론 | MEDIUM | 경고 + 검증 요청 |
| **과장 환각** (Exaggeration) | 정도/범위를 과장 | LOW | 경고 표시 |

### 환각 임계값 및 대응

| Contradiction 비율 | 판정 | 대응 |
|-------------------|------|------|
| 0~5% | 정상 | 통과 |
| 5~15% | 주의 | 사후 감사에서 경고 기록 |
| 15~30% | 환각 의심 | 사용자에게 경고 표시 + 검증 요청 안내 |
| 30%+ | 심각 환각 | 응답 차단 + 재생성 |

### 버전별 NLI 모델

| 버전 | NLI 모델 | 추론 속도 | 정확도 |
|------|---------|----------|--------|
| V1 | DeBERTa-v3-base-mnli-fever-anli (경량, 로컬) | ~50ms/claim | ~88% |
| V2+ | DeBERTa-v3-large 또는 LLM-as-judge | - | 향상 |

## §41.8 ★구현 운영 규칙 R1~R11 & 세션 관리 — GAP-13

> **비유**: R1~R11은 공사 현장의 "안전 수칙 11가지"입니다. 건설 노동자가 반드시 안전모를 쓰고 안전대를 매듯, AI가 코드를 생성할 때 반드시 따라야 하는 규칙입니다.

VAMOS 구현 과정에서 AI(Claude Code 등)가 코드를 생성할 때 **모든 프롬프트에 강제 적용**되는 공통 규칙 11가지입니다.

### R1~R11 공통 규칙 레지스트리

| ID | 규칙 이름 | 설명 | 위반 시 조치 |
|----|----------|------|-------------|
| **R1** | Python 버전 | Python ≥ 3.11 필수. f-string, match/case, tomllib 활용 | 빌드 실패 |
| **R2** | Pydantic v2 전용 | `model_config = ConfigDict(...)` 사용, `class Config:` 금지 | CI lint 실패 |
| **R3** | **no-create** (새 파일 생성 금지) | 기존 파일 수정 우선. 새 파일은 명시적 지시 시만 허용 | AI에게 재지시 |
| **R4** | **no-delete** (삭제 금지) | 코드/파일 삭제 금지. `[DEPRECATE] + (대체 경로)` 처리만 허용 | 삭제 대신 비활성화 |
| **R5** | LOCK/FREEZE 보호 | LOCK/FREEZE 값 런타임 변경 금지 | 즉시 거부 |
| **R6** | SOT 창작 금지 | SOT 문서에 없는 내용 창작 금지 | 즉시 중단, 사용자 판단 대기 |
| **R7** | 단방향 의존성 | CORE→COND 단방향만 허용, 역방향 금지 | 빌드 검증 실패 |
| **R8** | trace_id 서버 전용 | trace_id는 서버에서 UUID v4로 생성. 클라이언트 전달 금지 | 보안 위반 |
| **R9** | 에러 정보 보호 | 에러 메시지에 내부 정보(스택 트레이스, DB 스키마 등) 노출 금지 | 보안 위반 |
| **R10** | 비용 상한 준수 | V1: ₩40K, V2: ₩93K, V3: ₩266K **ABSOLUTE LOCK 🔒** | 실행 차단 |
| **R11** | 스키마 단일 참조 | `schema_registry.toml` 단일 참조점 활용 | 중복 정의 방지 |

### 판단 실패 시 절대 규칙 (SDD 명세 계층)

AI가 구현 중 불확실한 상황에 직면했을 때:

| # | 상황 | 조치 |
|---|------|------|
| 1 | 필드 수가 SOT 스키마와 다름 | **즉시 중단** + 불일치 보고. 임의 추가/삭제 금지 |
| 2 | LOCK/FREEZE 값 변경 시도 | **거부**. LOCK 값은 runtime 변경 불가 |
| 3 | SOT 문서에 없는 내용 | **창작 금지**. "스펙 미정의" 보고 후 대기 |
| 4 | 삭제가 필요한 경우 | **삭제 금지**. `[DEPRECATE]` 처리만 |
| 5 | 모듈 간 의존성이 불명확 | RULE 1.3 > PLAN 3.0 > DESIGN LOCK 순서로 확인 |

### 세션 운영 규칙

| # | 시점 | 명령/행동 | 목적 |
|---|------|----------|------|
| 1 | 각 STEP/Phase 완료 후 | `/compact` 실행 | 컨텍스트 요약 → 토큰 비용 절감 |

| 5 | 복수 파일 수정 시 | 순차 실행 (큐 방식) | 파일 간 충돌 방지 |

### 컨텍스트 다이어트 원칙

AI에게 과도한 문서를 제공하면 주의력 분산으로 정확도가 하락합니다:

| 구분 | 정의 | 적용 |
|------|------|------|
| **필수 로드 (MUST)** | 해당 STEP에서 직접 구현하는 기능의 SOT | @file 첨부 |
| **선택 참조 (MAY)** | 관련되지만 직접 구현 대상이 아닌 문서 | 필요 시만 부분 참조 |
| **참조 금지 (NEVER)** | 해당 STEP 범위 밖 문서 | 첨부 금지 — 환각 유발 |

> 📌 AI 한 세션에 첨부하는 SOT 문서는 **3~5개 이내**로 제한

### 핵심 요약 (3줄)

## §41.9 ★Adaptive Thinking — 난이도별 사고 깊이 — GAP-13

> **비유**: 식당 주문처럼, 물 한 잔 달라는 주문에 30분간 요리할 필요 없고, 풀코스 요리 주문에 30초 만에 내놓을 수도 없습니다. 질문의 난이도에 맞게 "사고 시간"을 조절하는 것입니다.

**Adaptive Thinking (적응형 사고)** 은 사용자 요청의 난이도를 자동 판단하여 **사고 깊이(thinking budget)**를 동적으로 할당하는 기능입니다. 단순 질문에 과도한 토큰(= 비용)을 소비하지 않고, 복잡한 추론에는 충분한 사고 시간을 부여합니다.

### 사고 깊이 5단계

| 레벨 | 이름 | 설명 | 사고 토큰 수 | 예시 |
|------|------|------|------------|------|
| **INSTANT** | 즉시 응답 | 인사, 단순 사실, 번역 | 0 (사고 없음) | "안녕하세요", "1+1은?" |
| **LOW** | 짧은 사고 | 요약, 간단한 설명 | ~1,024 | "이 문장 요약해줘" |
| **MEDIUM** | 표준 사고 | 분석, 비교, 코드 작성 | ~4,096 | "이 코드 분석해줘" |
| **HIGH** | 깊은 사고 | 복잡 추론, 다단계 계획 | ~16,384 | "프로젝트 아키텍처 설계해줘" |
| **MAX** | 최대 사고 | 연구 수준 분석, 수학 증명 | ~65,536 | "이 논문의 방법론을 비판적으로 분석해줘" |

### 난이도 판단 기준

| 신호 | INSTANT | LOW | MEDIUM | HIGH | MAX |
|------|---------|-----|--------|------|-----|
| 토큰 수 | <20 | 20~50 | 50~200 | 200~500 | 500+ |
| 도메인 위험도 | P0 | P0 | P0~P1 | P1~P2 | P2 |
| 필요 도구 수 | 없음 | 없음 | 1~2개 | 3+개 | 다수+연쇄 |
| 제약 조건 수 | 0 | 0~1 | 2~3 | 4+ | 복합 |
| sub_intents 수 | 1 | 1 | 2~3 | 4+ | 다수 |

### 비용 연동

- thinking 토큰은 **별도 비용 항목**으로 추적됩니다
- HIGH/MAX 레벨은 **사전 비용 알림** 표시
- 비용 상한 근접 시 Cost Gate가 thinking budget을 **자동 다운시프트**

### 버전별 구현

| 버전 | 구현 수준 |
|------|----------|
| V1 | 규칙 기반 난이도 분류 (간단) |
| V2 | Front Mini + Extended Thinking API 연동 (S7B-001 CRITICAL) |
| V3 | ML 기반 정밀 분류 + 자동 에스컬레이션 |

> 사용자가 수동으로 "깊은 사고" 모드를 트리거할 수도 있습니다 (Think Mode, S7-G-001)

[근거: D2.0-02 §2.1-A S7B-001]

### 핵심 요약 (3줄)
1. Self-Evolution은 6가지 영역에서 파라미터 수준 자동 개선이 가능하지만, 7가지 LOCK 영역(비용/승인/보안/RBAC/Gate/9-State/Decision Lock)은 절대 변경 불가합니다.
2. 롤백은 스냅샷 기반으로 자동 복원되며, Constitutional AI 4대 원칙(해악 금지/정직/자율성 존중/투명성)이 AI의 헌법 역할을 합니다.
3. Adaptive Thinking은 질문 난이도를 5단계(INSTANT~MAX)로 분류하여 사고 토큰을 동적 할당, 비용과 품질의 균형을 자동으로 맞춥니다.

## §42.6 마이그레이션 전략 (V1→V2→V3)

> **비유**: 가게를 이사하면서 물건(데이터)을 하나도 잃어버리지 않고 새 가게로 옮기는 작업입니다.

| 이전 항목 | V1 (이전) | V2 (이후) | 방식 |
|----------|----------|----------|------|

| 설정 | config.v1.toml | config.v2.toml | Config 마이그레이션 스크립트 |

### 10-Step Migration Orchestration

| Step | 작업 | 검증 |
|------|------|------|
| 1 | 마이그레이션 계획 생성 | 영향도 분석 완료 |
| 2 | 전체 백업 | 체크섬 무결성 확인 |
| 3 | 스키마 마이그레이션 (DDL) | 스키마 diff 검증 |
| 4 | 데이터 마이그레이션 (DML) | 행 수 일치 확인 |
| 5 | 인덱스/제약조건 재생성 | 쿼리 성능 테스트 |
| 6 | 참조 무결성 검증 | FK 위반 0건 |
| 7 | 애플리케이션 호환성 테스트 | 전체 테스트 PASS |
| 8 | 성능 벤치마크 | V1 대비 ±10% 이내 |
| 9 | Canary 배포 (10% 트래픽) | 에러율 < 0.1% |
| 10 | 전체 전환 + 모니터링 | 24시간 안정성 확인 |

### 설정 마이그레이션

버전 간 설정 전환 시 핵심 원칙:

| 원칙 | 설명 |
|------|------|
| **LOCK 값 유지** | 비용 상한 등 LOCK 값은 새 버전 설정에 그대로 이전 |
| **하위 호환성** | 이전 버전 설정 키가 새 버전에서도 인식되도록 보장 |
| **기본값 채우기** | 새 버전에서 추가된 설정은 안전한 기본값으로 자동 채움 |
| **검증 후 적용** | 변환된 설정이 새 버전 스키마에 맞는지 검증 후 적용 |
