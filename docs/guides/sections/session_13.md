# §15. 도메인 시스템 — P0 / P1 / P2

> **비유**: VAMOS의 도메인 시스템은 **회사의 부서 조직**과 같습니다. 회사에 영업부·기술부·인사부가 있듯이, VAMOS에도 코딩 전문 부서·리서치 전문 부서·콘텐츠 전문 부서가 있습니다. 그리고 각 부서는 **보안 등급**에 따라 출입 권한이 다릅니다 — 일반 사무실(P0)은 아무나 들어갈 수 있지만, 연구실(P1)은 허가증이 필요하고, 금고(P2)는 매번 별도 인증이 필요합니다.

[근거: BASE-1.3 §4.3, §12, D2.0-03 §1~§3]

---

## §15.1 도메인이란?

### 비유: 전문 부서

회사에서 모든 일을 한 사람이 하지 않듯이, VAMOS도 모든 작업을 하나의 모듈이 처리하지 않습니다. **도메인(Domain)**이란 VAMOS가 처리할 수 있는 **전문 분야**를 뜻합니다.

- **도메인(Domain)** = 분야 단위 (예: Dev, Research, Productivity 등)
- **부도메인(Sub-domain)** = 해당 도메인의 세부 실행 단위 (예: Dev 안에서 "코딩", "디버깅", "시스템 설계" 등)

[근거: BASE-1.3 §4.3]

### 도메인의 핵심 원칙

각 도메인은 **Blue Node(블루 노드)**라는 실행 모듈로 구현됩니다. Blue Node는 특정 분야의 "전문의"와 같습니다.

| 원칙 | 설명 | 비유 |
|------|------|------|
| **CORE 종속** | Blue Node는 ORANGE CORE의 규칙을 반드시 상속하며, 독립 실행 불가 (**LOCK**) | 전문의도 병원 규칙을 따라야 함 |
| **안전 불변** | Non-goal/안전/비용 상한/승인 구조 변경 불가 (**변경 불가**) | 병원의 의료법은 진료과가 바꿀 수 없음 |
| **도구 경유** | 외부 도구 사용 시 반드시 ToolRegistry(도구 관리자)를 경유 | 장비 사용 시 반드시 장비실을 거쳐야 함 |

[근거: BASE-1.3 §4.2, D2.0-03 §1.1~§1.3]

### 도메인 우선순위 분류: P0 / P1 / P2

모든 도메인은 **위험도**에 따라 3등급으로 분류됩니다:

| 등급 | 의미 | 승인 방식 | 비유 |
|------|------|-----------|------|
| **P0** | 핵심 도메인 — 항상 활성 | 승인 없이 사용 가능 | 일반 사무실 (자유 출입) |
| **P1** | 확장 도메인 — 1회 승인 후 활성 | 처음 한 번만 승인하면 이후 자유 | 연구실 (허가증 발급 후 자유) |
| **P2** | 위험 도메인 — 세션별 승인 필수 | 매번 새로 승인 필요 (가장 엄격) | 금고 (매번 별도 인증) |

[근거: BASE-1.3 §12.1~§12.3, D2.0-03 §1.3]

### 핵심 요약 (3줄)
1. **도메인**은 VAMOS의 전문 분야 단위이며, **Blue Node**로 실행된다.
2. 모든 Blue Node는 ORANGE CORE에 종속되며 독립 실행 불가 — 안전/비용/승인 구조는 **변경 불가**.
3. 도메인은 위험도에 따라 **P0(항상 활성) / P1(1회 승인) / P2(매번 승인)**으로 분류된다.

## §15.2 P0 — 항상 활성 (승인 없이 사용 가능)

P0는 VAMOS의 **기본 활성 도메인**입니다. 시스템을 시작하면 별도 승인 없이 바로 사용할 수 있는, 안전한 핵심 기능들입니다.

> **비유**: 회사에서 출근하면 바로 자기 자리에 앉아 일할 수 있는 것처럼, P0 도메인은 "출근만 하면 바로 사용 가능"합니다.

**P0 핵심 규칙** (**변경 불가**):
- Self-evo(자기진화)로 P0 도메인을 생성하거나 변경할 수 없음
- 문서 수정 + 사용자 승인 + 버전 승격이 필요
- 단, P0라도 외부 서비스/API 사용은 약관 준수 체크리스트를 통과해야 함

[근거: BASE-1.3 §12.1, D2.0-03 §3.1]

### P0 도메인 목록

| 도메인 | 분류 | 주요 기능 | V0 | V1 | V2 | V3 |
|--------|------|-----------|----|----|----|----|
| **Dev/System** | 핵심 | 코딩, 디버깅, 시스템 관리 | ON | ON | ON | ON |
| **Research** | 핵심 | 정보 검색, 논문 분석, 보고서 | ON | ON | ON | ON |
| **Productivity** | 핵심 | 문서 작성, 일정 관리, 이메일 | ON | ON | ON | ON |

[근거: BASE-1.3 §12.1, D2.0-03 §3.1, BEGINNER_GUIDE §5.2]

### §15.2.1 Dev/System (개발/시스템) — 코딩, 디버깅, 시스템 관리

> **비유**: Dev/System 도메인은 회사의 **기술부서**입니다. 프로그램을 만들고, 고장 난 코드를 고치고, 시스템을 관리하는 전문가 팀입니다.

| 항목 | 내용 |
|------|------|
| **목적** | 코드 작성, 디버깅, 리뷰, 리팩토링, 시스템 설계 및 관리 |
| **비유** | 회사의 IT/개발 부서 — "프로그램을 만들고, 고치고, 관리하는 팀" |
| **입력** | 사용자의 코딩/디버깅/시스템 관련 요청 + IntentFrame(의도 프레임) |
| **출력** | 코드, 디버깅 결과, 시스템 설계 문서, 코드 리뷰 결과 |
| **상태** | 항상 활성 (P0) — 별도 승인 불필요 |
| **이벤트** | 코드 생성/실행/디버깅 이벤트는 LogEvent로 기록 |
| **에러코드** | 코드 실행 실패 시 `CODE_EXEC_FAILED` → 샌드박스 로그 제공 |
| **폴백** | 1차 모델 실패 시 → 대체 모델 자동 전환 (최대 3단계) |
| **관련 모듈** | E-1(Coding & System Design Helper), E-4(Code Executor), TS_CODE(코드 템플릿) |
| **버전별 활성** | V0: ON, V1: ON, V2: ON, V3: ON |

**사용 가능한 도구 예시**:
- 코드 작성/리팩토링/리뷰 (E-1)
- 샌드박스 코드 실행 (E-4, mcp.code.e2b)
- GitHub 연동 — PR 생성/리뷰/머지, Issue 관리

**권한 수준** (**LOCK**):

| 권한 레벨 | 설명 | Dev Node |
|-----------|------|----------|
| Level 0 (읽기) | 정보 조회 | 허용 |
| Level 1 (생성) | 파일/코드 생성 | 허용 |
| Level 2 (수정) | 파일/설정 수정 | 허용 |
| Level 3 (실행) | 코드 실행, API 호출 | 허용 |
| Level 4 (외부 통신) | 이메일/PR 생성 | Ask (승인 필요) |
| Level 5 (금융) | 주문/결제 | 금지 |

## §15.3 P1 — 1회 승인 후 활성 (처음 한 번만 승인하면 이후 자유)

P1은 **확장 도메인**으로, 효용성·생산성·데이터 확장 기능을 제공합니다. 약간의 위험이 있어 처음 한 번은 승인이 필요하지만, 한 번 승인하면 이후에는 자유롭게 사용할 수 있습니다.

> **비유**: 회사의 연구실에 들어가려면 **허가증**이 필요합니다. 하지만 한 번 발급받으면 다음부터는 그냥 들어갈 수 있습니다.

### P1 승인 구조 (**LOCK**)

P1 승인에는 **2가지 채널**이 있습니다:

| 채널 | 방식 | 설명 |
|------|------|------|
| **A 채널** (CORE 제안형) | CORE가 필요성을 감지 → 후보 프로필 생성 → 승인 요청 | "이 기능 활성화하면 좋겠습니다" |
| **B 채널** (사용자 요청형) | 사용자가 직접 명시 요청 → 후보 프로필 생성 → 승인 요청 | "이 기능 켜줘" |

**어떤 채널이든 "활성화"는 승인 없이는 불가** (**LOCK**)

승인은 BASE-1.3의 **2단계 승인 구조**를 따릅니다:
1. **1단계 — 계획 승인**: "무엇을 할지" 설명 + 승인
2. **2단계 — 실행 승인**: "지금 실행할지" 최종 확인 (위험 작업 필수)

**Self-evo(자기진화) 제한**: P1에서 자동화는 **"후보 프로필 생성"까지만** 허용. 활성화는 반드시 사용자 승인 이후.

[근거: BASE-1.3 §9.2, §12.2, D2.0-03 §3.2]

|--------|------|-----------|----|----|----|----|

### §15.3.2 Data & Quant (데이터/퀀트) — 데이터 분석, 퀀트 모델

> **비유**: Data & Quant 도메인은 회사의 **데이터 분석팀**입니다. 숫자와 데이터를 분석하고, 통계 모델을 만들고, 정량적 인사이트를 도출하는 전문가팀입니다.

| 항목 | 내용 |
|------|------|
| **목적** | 데이터 분석, 통계 모델링, 퀀트(정량 분석) 모델 구축, 데이터 시각화 |
| **비유** | 데이터 분석팀 — "숫자를 분석하고, 패턴을 찾고, 예측 모델을 만드는 팀" |
| **입력** | 사용자의 데이터 분석/퀀트 관련 요청 + IntentFrame |
| **출력** | 분석 결과, 차트/그래프, 통계 보고서, 퀀트 모델 결과 |
| **상태** | P1 — 1회 승인 후 활성 |
| **이벤트** | 데이터 분석/모델 생성 이벤트는 LogEvent로 기록 |
| **에러코드** | 데이터 소스 접근 실패 시 `DATA_SOURCE_FAILED` → 대체 소스 탐색 |
| **폴백** | 외부 API 실패 → 로컬 데이터 기반 분석 → 캐시된 데이터 활용 |
| **관련 모듈** | E-4(Code Executor), I-2(Context Builder/RAG), I-15(Evidence & QoD Manager) |
| **버전별 활성** | V0: OFF, V1: ON(승인 필요), V2: ON, V3: ON |

**권한 수준**:

| 권한 레벨 | Quant Node |
|-----------|-----------|
| Level 0 (읽기) | 허용 |
| Level 1 (생성) | 허용 |
| Level 2 (수정) | 금지 |
| Level 3 (실행) | 허용 |
| Level 4 (외부 통신) | 금지 |
| Level 5 (금융) | 금지 |

[근거: D2.0-03 §K-041, BEGINNER_GUIDE §5.2]

### 핵심 요약 (3줄)
1. **P1 도메인**은 Content와 Data & Quant 총 2개로, **1회 승인 후 자유롭게 사용** 가능.
2. 승인 채널은 A(CORE 제안형)와 B(사용자 요청형) 두 가지이며, **어떤 채널이든 승인 필수** (**LOCK**).
3. Self-evo는 **후보 프로필 생성까지만** 허용되며, 실제 활성화는 반드시 사용자가 승인해야 한다.

## §15.4 P2 — 세션별 승인 필수 (매번 새로 승인 필요, 가장 엄격)

P2는 **위험 도메인**으로, 거래·법률·의료 등 민감한 영역을 다룹니다. 가장 엄격한 승인 구조가 적용되며, **매 세션마다 새로 승인을 받아야** 합니다.

> **비유**: 은행 금고에 들어가려면 매번 신분증 확인 + 지문 인식 + 사유 입력이 필요합니다. 어제 들어갔다고 오늘 자동으로 들어갈 수 있는 게 아닙니다.

### P2 핵심 규칙 (**모두 변경 불가**)

| 규칙 | 설명 | LOCK |
|------|------|------|
| **기본 OFF** | P2 도메인은 항상 기본적으로 꺼져 있음 | **LOCK** |
| **세션 종료 시 자동 OFF** | 세션이 끝나면 자동으로 비활성화됨 | **LOCK** |
| **자동 생성 금지** | 시스템이 자동으로 P2 도메인을 생성할 수 없음 (Non-Goal §2.6) | **LOCK** |
| **명시적 승인 필수** | 사용자가 직접, 명시적으로 승인해야만 활성화 | **LOCK** |
| **세션별 재확인** | 매 세션마다 새로 승인 필요 | **LOCK** |
| **모달 확인창** | 활성화 시 팝업 확인창으로 재확인 (DEC-011) | **LOCK** |

[근거: BASE-1.3 §2.6, §3.3, §12.3, D2.0-03 §3.3]

### §15.4.1 Trading Strategy Analysis (투자 전략 분석)

> **비유**: 투자 전략 분석 도메인은 회사의 **투자 자문 부서**입니다. 시장 데이터를 분석하고, 투자 전략을 평가해주지만, **절대로 직접 주문을 넣지는 않습니다** — 의사 결정 보조만 합니다.

| 항목 | 내용 |
|------|------|
| **목적** | 투자 전략 분석, 시장 데이터 분석, 포트폴리오 평가 (**실거래 절대 금지!**) |
| **비유** | 투자 자문사 — "분석과 조언만 하고, 직접 거래는 절대 안 하는 전문가" |
| **입력** | 사용자의 투자 분석 요청 + IntentFrame + 세션별 승인 토큰 |
| **출력** | 투자 분석 보고서, 전략 평가 결과, 시장 데이터 요약 |
| **상태** | P2 — 세션별 승인 필수, 세션 종료 시 자동 OFF |
| **이벤트** | `oc.p2.activated` / `oc.p2.deactivated` (별도 위험 로그 기록) |
| **에러코드** | `P2_ACTIVATION_DENIED` (승인 거부 시), `TRADING_DATA_FAILED` (데이터 소스 실패) |
| **폴백** | P2 승인 거부 → 즉시 차단 + 승인 요청 안내 |
| **관련 모듈** | E-2(Web Search), I-15(Evidence & QoD Manager), 금융 데이터 MCP 서버들 |
| **버전별 활성** | V0: OFF, V1: OFF(P2 기본 OFF), V2: OFF(승인 시 ON), V3: OFF(승인 시 ON) |

**절대 금지 (Non-Goal)**:
- ❌ 실제 주문/계좌/API 연동 (BASE-1.3 §2.1)
- ❌ 자동 매매 실행
- ❌ 증권사 계좌 직접 접근

**권한 수준**:

| 권한 레벨 | Trading Node |
|-----------|-------------|
| Level 0 (읽기) | 허용 |
| Level 1 (생성) | 허용 |
| Level 2 (수정) | 금지 |
| Level 3 (실행) | 허용 |
| Level 4 (외부 통신) | 금지 |
| Level 5 (금융) | Ask (사용자 확인 필수) |

[근거: BASE-1.3 §2.1, §2.6, D2.0-03 §K-041, BEGINNER_GUIDE §5.2]

### §15.4.2 P2 승인 흐름 상세

P2 도메인이 활성화되는 전체 과정을 단계별로 살펴봅니다:

```
[사용자 요청] "투자 전략 분석해줘"
      ↓
[1단계: 요청 접수]
  Front Mini LLM이 P2 도메인 힌트 감지
  → PC_P2_DOMAIN_HINT 정책 훅 실행
      ↓
[2단계: 사유 입력 요청]
  ORANGE CORE가 사용자에게 P2 활성화 사유 입력 요청
  → "투자 전략 분석 기능을 활성화하려면 사유를 입력해주세요"
      ↓
[3단계: 승인 (2단계 승인 구조)]
  ├─ 1단계 계획 승인: "무엇을 할지" 설명 → 사용자 확인
  └─ 2단계 실행 승인: "지금 실행할지" 최종 확인 (모달 팝업)
      ↓
[4단계: 실행]
  P2 도메인 활성화 → oc.p2.activated 이벤트 기록
  → Trading Strategy Analysis Blue Node 실행
  → 별도 위험 로그(P2 로그) 기록
      ↓
[5단계: 세션 종료 시 자동 비활성화]
  세션 종료 → P2 자동 OFF
  → oc.p2.deactivated 이벤트 기록
  → UI/로그에 "세션 종료로 OFF" 표시
```

**승인 타임아웃**: P2는 5분 이내 미응답 시 자동 reject (거부)

[근거: BASE-1.3 §3.3, §9.2, D2.0-03 §3.3, BASE-1.3 아이디어 확장 P7-APR]

### P2 자동 생성 금지 (Non-Goal §2.6)

> **이것은 VAMOS의 절대 금지 규칙입니다.**

- 사용자 승인 없이 시스템이 P2 도메인을 자동 활성화하는 것은 **절대 금지**.
- Self-evo(자기진화)가 P2 도메인을 자동 생성하는 것도 **절대 금지**.
- **위반 시 대응**: 활성화 즉시 차단 → 사유 기록 → 사용자에게 승인 요청 안내.

[근거: BASE-1.3 §2.6, §6.2]

### 핵심 요약 (3줄)
1. **P2 도메인**은 가장 엄격한 승인이 필요하며, **기본 OFF + 세션 종료 시 자동 OFF**. (**LOCK**)
2. P2 활성화 흐름: 요청 → 사유 입력 → 2단계 승인(계획+실행) → 실행 → 세션 종료 시 자동 비활성화.
3. **P2 자동 생성은 절대 금지** (Non-Goal §2.6) — Self-evo도 P2를 생성/활성화할 수 없다. (**변경 불가**)

## §15.5 Blue Node 생명주기 (7가지 상태)

Blue Node(블루 노드)는 생성부터 폐기까지 **7가지 상태**를 거칩니다. 이를 **생명주기(Lifecycle)**라고 합니다.

> **비유**: 회사 직원의 생애와 비슷합니다 — 입사 지원(Created) → 입사 교육(Initializing) → 대기(Ready) → 업무 수행(Running) → 휴가(Paused) → 퇴사(Terminated) → 인사 기록 보관(Archived)

### 7가지 상태 설명

| 상태 | 한글 | 설명 | 비유 |
|------|------|------|------|
| **Created** | 생성됨 | 후보 프로필이 생성된 상태 (아직 실행 안 됨) | 입사 지원서 접수 |
| **Initializing** | 초기화 중 | 도메인 정책, 템플릿, 메모리 로딩 중 | 입사 교육 진행 중 |
| **Ready** | 준비 완료 | 모든 초기화 완료, 실행 대기 중 | 자리에 앉아 업무 대기 |
| **Running** | 실행 중 | 실제 작업을 수행하고 있는 상태 | 업무 수행 중 |
| **Paused** | 일시 정지 | 사용자 입력 대기 또는 승인 대기로 일시 정지 | 휴가 / 잠시 멈춤 |
| **Terminated** | 종료됨 | 작업 완료 또는 세션 종료로 비활성화 | 퇴사 처리 완료 |
| **Archived** | 보관됨 | 실행 기록이 보관된 상태 (재사용 가능한 경험 저장) | 인사 기록 보관 |

### 상태 전이도 (ASCII 다이어그램)

```
                          승인/요청
                             │
                             ▼
                        ┌─────────┐
                        │ Created │ ──── 후보 프로필 생성 (P1: 자동 가능 / P2: 수동만)
                        └────┬────┘
                             │ 초기화 시작
                             ▼
                      ┌──────────────┐
                      │ Initializing │ ──── 정책/템플릿/메모리 로딩
                      └──────┬───────┘
                             │ 초기화 완료
                             ▼
                        ┌─────────┐
             ┌─────────→│  Ready  │←──────────┐
             │          └────┬────┘           │
             │               │ 실행 시작       │ 재개
             │               ▼                │
             │         ┌──────────┐           │
             │         │ Running  │───────────┘
             │         └────┬─────┘
             │              │
      사용자 요청     ┌─────┴──────┐
     (재시작)        │            │
             │       ▼            ▼
        ┌─────────┐      ┌────────────┐
        │ Paused  │      │ Terminated │ ──── 세션 종료 / 작업 완료
        └─────────┘      └─────┬──────┘     (P2: 세션 종료 시 자동 전이)
                               │
                               │ 기록 보관
                               ▼
                        ┌───────────┐
                        │ Archived  │ ──── 실행 기록 보관
                        └───────────┘
```

### 상태 전이 규칙

| 현재 상태 | 전이 가능 상태 | 조건 |
|-----------|---------------|------|
| Created | Initializing | 승인 완료 (P1: 1회, P2: 세션별) |
| Initializing | Ready | 초기화 성공 |
| Initializing | Terminated | 초기화 실패 |
| Ready | Running | 실행 요청 수신 |
| Running | Paused | 사용자 입력/승인 대기 |
| Running | Terminated | 작업 완료 또는 세션 종료 |
| Paused | Running | 사용자 입력/승인 완료 |
| Paused | Terminated | 타임아웃 또는 세션 종료 |
| Terminated | Archived | 기록 보관 처리 |

**P2 특수 규칙**: 세션 종료 시 Running/Paused/Ready 상태의 P2 노드는 **자동으로 Terminated**로 전이됩니다. (**LOCK**)

[근거: D2.0-03 §3.1~§3.3, §5.2]

### 버전별 활성 노드 상한 (모듈 폭발 방지)

| 항목 | V1 | V2 | V3 | LOCK |
|------|----|----|-----|------|
| **활성 노드 상한** (세션당) | 3개 (Lead+2 Sub) | 10개 | 50개 | **LOCK** |
| **후보 노드 상한** (세션당) | 5개 | 20개 | 100개 | **LOCK** |

> 상한 도달 시: 신규 후보 생성 금지 → 기존 NODE 재사용 또는 "범용 NODE + ToolRegistry"로 대체 실행

[근거: D2.0-03 §3.2.1 (B)]

### 핵심 요약 (3줄)
1. Blue Node는 **Created → Initializing → Ready → Running → Paused → Terminated → Archived** 7가지 상태를 거친다.
2. P2 노드는 세션 종료 시 **자동으로 Terminated** 상태로 전이된다. (**LOCK**)
3. 모듈 폭발 방지를 위해 활성 노드 상한이 존재: V1=3개, V2=10개, V3=50개. (**LOCK**)

## §15.6 NodeCapabilityProfile — 필드 설명

**NodeCapabilityProfile(노드 역량 프로필)**은 각 Blue Node가 "무엇을 할 수 있는지"를 정의하는 **신분증** 같은 것입니다.

> **비유**: 회사에서 각 직원의 **인사 카드**에 소속 부서, 사용 가능한 장비, 보안 등급, 급여 등급이 적혀 있듯이, NodeCapabilityProfile에는 노드의 도메인, 사용 도구, 위험 등급, 비용 등급이 기록됩니다.

### 필드 상세 설명

| 필드명 | 한글 | 설명 | 예시 |
|--------|------|------|------|
| **node_id** | 노드 고유 식별자 | 각 Blue Node를 구별하는 고유 ID | `blue_node_dev_001` |
| **domain** | 도메인 (분야) | 이 노드가 담당하는 전문 분야 | `Dev/System`, `Research`, `Trading` |
| **required_tools** | 필수 도구 목록 | 이 노드가 반드시 필요로 하는 MCP 도구 ID 목록 | `["mcp.code.e2b", "mcp.search.tavily"]` |
| **optional_tools** | 선택 도구 목록 | 있으면 좋지만 없어도 동작 가능한 도구 | `["mcp.vision.clip"]` |
| **risk_class** | 위험 등급 | 이 노드의 위험도 (low/med/high) | `low` (P0), `med` (P1), `high` (P2) |
| **cost_class** | 비용 등급 | 이 노드의 비용 분류 (v0/v1/v2/v3) | `v1` (기본), `v2` (Pro), `v3` (Enterprise) |
| **gates_required** | 필수 게이트 목록 | 실행 전 반드시 통과해야 하는 안전 게이트 | `["policy", "cost", "approval", "evidence", "self_check"]` |
| **llm_config** | LLM 설정 | 이 노드가 사용하는 AI 모델 설정 | 1순위: Claude 4.6, 2순위: GPT-4o, 폴백: 로컬 모델 |
| **max_concurrent** | 최대 동시 실행 수 | 이 노드가 동시에 처리할 수 있는 최대 작업 수 | `max_concurrent_mcp: 10` (config.toml) |

### 예시: Dev/System Node의 NodeCapabilityProfile

```
NodeCapabilityProfile:
  node_id:          "blue_node_dev_system"
  domain:           "Dev/System"
  required_tools:   ["mcp.code.e2b", "mcp.doc.pymupdf"]
  optional_tools:   ["mcp.browser.playwright", "mcp.search.tavily"]
  risk_class:       "low"        ← P0 도메인이므로 저위험
  cost_class:       "v1"         ← 기본 비용 등급
  gates_required:   ["policy", "cost"]
  llm_config:       {primary: "claude-4.6", fallback: "gpt-4o"}
  max_concurrent:   10
```

### 예시: Trading Strategy Node의 NodeCapabilityProfile

```
NodeCapabilityProfile:
  node_id:          "blue_node_trading_strategy"
  domain:           "Trading Strategy Analysis"
  required_tools:   ["mcp.search.tavily", "mcp.doc.pymupdf"]
  optional_tools:   ["mcp.code.e2b"]
  risk_class:       "high"       ← P2 도메인이므로 고위험
  cost_class:       "v2"         ← Pro 등급 이상 권장
  gates_required:   ["policy", "cost", "approval", "evidence", "self_check"]
                                 ← 5개 게이트 모두 통과 필수!
  llm_config:       {primary: "claude-4.6", fallback: "gemini-2.5-pro"}
  max_concurrent:   5
```

[근거: D2.0-03 §2.3 NodeCapabilityProfile 개념 스키마]

### 핵심 요약 (3줄)
1. **NodeCapabilityProfile**은 Blue Node의 "신분증"으로, node_id, domain, tools, risk_class, cost_class 등을 정의한다.
2. P0(저위험) 노드는 2개 게이트(policy, cost)만 통과하면 되지만, P2(고위험) 노드는 **5개 게이트 모두** 통과해야 한다.
3. 모든 도구 호출은 반드시 `tool_id`로 표준화되며, ToolRegistry(도구 관리자)를 경유해야 한다. (**LOCK**)
