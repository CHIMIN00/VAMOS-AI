# §9. S-Series — 자기 진화 모듈 (S-1 ~ S-8, 총 8개)

> **비유**: I-Series가 VAMOS의 **두뇌**(생각하고 판단하는 역할)라면, S-Series는 두뇌가 **스스로를 점검하고 개선하는 능력**입니다. 사람으로 비유하면, 시험을 본 뒤 **오답 노트를 정리하고**, 더 나은 공부법을 찾고, 자기 약점을 분석하여 점점 더 똑똑해지는 **자기 성장 시스템**과 같습니다. 단, VAMOS의 자기 진화는 반드시 **제안만** 하고, 실제 적용은 **사람(사용자)의 승인**을 받아야 합니다.

[근거: CLAUDE.md §6 S-Series, D2.0-01 §2.2.3, §5.7]

---

## S-Series 핵심 원칙 (모든 S-모듈 공통)

> 아래 규칙은 S-1부터 S-8까지 **모든** 자기 진화 모듈에 공통 적용됩니다.

### 1. 제안만 가능, 자동 적용 금지 (LOCK — 변경 불가)

```
S-모듈이 개선 아이디어를 "제안"
        ↓
I-18 Self-evo Engine이 제안을 수집·정리
        ↓
S-8 Governance가 제안을 검토
        ↓
I-19 Approval Manager를 통해 사용자에게 승인 요청
        ↓
사용자가 승인해야만 실제 적용
```

- **자동 적용 절대 금지**: S-모듈이 스스로 변경을 적용하는 것은 불가능합니다. 반드시 사용자 승인을 거쳐야 합니다. [근거: D2.0-01 §3.1 LOCK "Self-evo 제한: 제안만 가능, 자동 적용 금지", RULE 1.3 §8]

### 2. 불변 영역 — 절대 수정 불가 (LOCK)

S-Series가 아무리 "개선"이라 해도 아래 영역은 **절대 건드릴 수 없습니다**:

| # | 불변 영역 | 설명 | 왜 불가능한가? |
|---|----------|------|--------------|
| 1 | **RULE 1.3 정체성** | VAMOS가 "누구인지" 정의 | AI가 스스로 정체성을 바꾸면 위험 |
| 2 | **Non-goal (금지 행위)** | 실거래/불법/단정판단 금지 | 법적·윤리적 위험 |
| 3 | **비용 상한** | V1: 40,000 / V2: 93,000 / V3: 266,000 | 비용 폭주 방지 |
| 4 | **승인 구조** | Approval Gate 구조 | 안전장치 해제 방지 |
| 5 | **P0 도메인 기본 규칙** | 핵심 도메인 판단 원칙 | 판단 근거 왜곡 방지 |

[근거: D2.0-01 §2.2.3 "불변 영역: RULE/정체성/비용상한은 수정 불가", RULE 1.3 §2]

### 3. I-18 Self-evo Engine 연동

S-Series 모듈들은 대부분 **I-18 Self-evo Engine**(자기진화 엔진)과 연동하여 동작합니다. I-18은 S-모듈들의 제안을 수집하고 조율하는 **중앙 허브** 역할입니다.

### 4. UI 노출 정책

| S-모듈 | UI 표면 | UI 뷰 모드 | 비고 |
|--------|---------|-----------|------|
| S-1 | builder | panel | 일반 태스크에서도 최소 검증 수행 |
| S-2 ~ S-7 | builder | panel | 진화 제안 패널 |
| S-8 | builder | **modal** (팝업) | 거버넌스 승인은 팝업으로만 노출 |

## §9.1 S-1 자기 검증 엔진 (Self-check Engine)

### 비유
시험을 본 뒤 **스스로 답안을 검토하는 학생**과 같습니다. "내가 쓴 답이 정말 맞는지?" 다시 한번 확인합니다.

### 목적
VAMOS가 생성한 모든 출력물(답변, 코드, 분석 결과 등)의 **품질을 자동으로 검증**합니다. 출력이 사용자에게 전달되기 전에 "이 답변이 정확한가?", "근거가 충분한가?", "안전한 내용인가?"를 점검하는 **마지막 관문** 역할을 합니다. I-6(Self-check 엔진)과 I-15(Evidence & QoD Manager)와 직접 연동됩니다. [근거: D2.0-01 §5.7, §2.3.1 Reflection 단계, CLAUDE.md §6]

### 입력/출력

| 구분 | 내용 |
|------|------|
| **입력 (Input)** | VAMOS가 생성한 출력물, 근거(Evidence), QoD(품질 점수) |
| **출력 (Output)** | 검증 결과(PASS/FAIL), 품질 점수, 개선 제안, 재시도 트리거 |

### 분류/버전

| 항목 | 값 |
|------|-----|
| 분류 | **CORE** (핵심) |
| V1 | **ON** |
| V2 | **ON** |
| V3 | **ON** |
| LOCK | — |
| I-모듈 연결 | I-6, I-15 |

> **특별한 점**: S-Series 중 **유일하게 V1부터 활성화**되는 모듈입니다. 다른 S-모듈은 모두 V3에서만 켜지지만, 자기 검증은 처음부터 필수입니다.

### 핵심 요약 (3줄)
1. S-1은 VAMOS 출력물의 **품질을 자동 검증**하는 핵심 모듈입니다.
2. S-Series 중 유일하게 **V1부터 활성화**(CORE)되며, 일반 태스크에서도 최소 검증을 수행합니다.
3. I-6(자기검증 엔진), I-15(근거 관리자)와 연동하여 출력 전 마지막 품질 관문 역할을 합니다.

## §9.2 S-2 벤치마크 테스트 (Benchmark QA Suite)

### 비유
공장의 **품질 검사 라인**과 같습니다. 제품(VAMOS의 기능)이 출하되기 전에 정해진 기준에 맞는지 **자동 테스트**를 돌립니다.

### 목적
VAMOS의 전체 기능에 대해 **통합 테스트와 회귀 방지 테스트**를 수행합니다. "이전에 잘 되던 기능이 업데이트 후에도 잘 되는가?"를 자동으로 확인하며, 성능 벤치마크(속도, 정확도 등)를 지속적으로 측정합니다. I-24(Knowledge Graph Engine)과 연동됩니다. [근거: D2.0-01 §5.7, CLAUDE.md §6]

### 입력/출력

| 구분 | 내용 |
|------|------|
| **입력 (Input)** | 테스트 케이스 세트, 벤치마크 기준, 이전 성능 기록 |
| **출력 (Output)** | 테스트 결과(PASS/FAIL), 성능 점수, 회귀(regression) 감지 알림 |

### 분류/버전

| 항목 | 값 |
|------|-----|
| 분류 | EXP (실험적) |
| V1 | OFF |
| V2 | OFF |
| V3 | **ON** |
| LOCK | — |
| I-모듈 연결 | I-24 |

### 핵심 요약 (3줄)
1. S-2는 VAMOS 기능의 **통합 테스트와 성능 벤치마크**를 수행하는 모듈입니다.
2. 이전 버전에서 잘 되던 기능이 깨지지 않았는지 **회귀 방지 테스트**를 자동 실행합니다.
3. V3에서만 활성화되는 실험적(EXP) 모듈이며, I-24(지식 그래프 엔진)와 연동됩니다.

## §9.3 S-3 템플릿 진화 (Template Evolution)

### 비유
요리사가 **레시피를 계속 개선**하는 것과 같습니다. "지난번 레시피보다 소금을 조금 줄이면 더 맛있겠다"처럼, VAMOS가 사용하는 프롬프트/응답 템플릿을 점점 더 좋게 다듬어 갑니다.

### 목적
VAMOS가 LLM에게 보내는 **프롬프트 템플릿**과 사용자에게 보여주는 **응답 템플릿**을 분석하여, 더 나은 결과를 이끌어내는 방향으로 자동 개선을 제안합니다. I-12(Workflow Builder)와 I-18(Self-evo Engine)과 연동됩니다. [근거: D2.0-01 §5.7, CLAUDE.md §6]

### 입력/출력

| 구분 | 내용 |
|------|------|
| **입력 (Input)** | 현재 템플릿, 사용 이력, 성공/실패 기록, 사용자 피드백 |
| **출력 (Output)** | 개선된 템플릿 제안, 변경 사유, A/B 테스트 결과 |

### 분류/버전

| 항목 | 값 |
|------|-----|
| 분류 | EXP (실험적) |
| V1 | OFF |
| V2 | OFF |
| V3 | **ON** |
| LOCK | — |
| I-모듈 연결 | I-12, I-18 |

### 핵심 요약 (3줄)
1. S-3은 프롬프트/응답 **템플릿을 분석하고 개선을 제안**하는 모듈입니다.
2. 사용 이력과 성공/실패 데이터를 바탕으로 더 나은 템플릿을 자동 제안합니다.
3. V3에서만 활성화되며, I-12(워크플로우 빌더)와 I-18(자기진화 엔진)과 연동됩니다.

## §9.4 S-4 에러 패턴 분석기 (Error Pattern Miner)

### 비유
병원의 **역학 조사관**과 같습니다. "어떤 종류의 에러가 자주 발생하는지", "공통 원인은 무엇인지" 패턴을 분석하여, 같은 실수를 반복하지 않도록 합니다.

### 목적
VAMOS가 과거에 발생시킨 **에러와 실패 기록**을 수집·분석하여, 반복되는 패턴을 발견합니다. 이를 바탕으로 실패코드(Failure Code) 체계를 개선하고, 폴백(대안 실행) 전략을 최적화하는 제안을 생성합니다. I-20(Failure/Fallback Manager)과 I-18(Self-evo Engine)과 연동됩니다. [근거: D2.0-01 §5.7, CLAUDE.md §6]

### 입력/출력

| 구분 | 내용 |
|------|------|
| **입력 (Input)** | 에러 로그, 실패코드 기록, 폴백 실행 이력 |
| **출력 (Output)** | 에러 패턴 보고서, 폴백 전략 개선 제안, 반복 에러 경고 |

### 분류/버전

| 항목 | 값 |
|------|-----|
| 분류 | EXP (실험적) |
| V1 | OFF |
| V2 | OFF |
| V3 | **ON** |
| LOCK | — |
| I-모듈 연결 | I-20, I-18 |

### 핵심 요약 (3줄)
1. S-4는 반복되는 **에러 패턴을 분석하고 개선 방안을 제안**하는 모듈입니다.
2. 실패코드 체계와 폴백 전략을 최적화하여 같은 실수를 반복하지 않게 합니다.
3. V3에서만 활성화되며, I-20(실패 관리자)과 I-18(자기진화 엔진)과 연동됩니다.

## §9.5 S-5 라우터 진화 (Router Evolution)

### 비유
택시 기사가 **경험을 쌓으며 최적 경로를 찾아가는 것**과 같습니다. "이 시간대에는 A 도로보다 B 도로가 빠르다"는 것을 학습하듯, VAMOS도 "이 질문 유형에는 A 모델보다 B 모델이 더 정확하다"는 것을 학습합니다.

### 목적
VAMOS가 사용자 질문을 적절한 LLM(대규모 언어 모델)이나 BLUE NODE(실행 모듈)로 보내는 **라우팅(경로 배정) 규칙을 자동 최적화**합니다. 과거 라우팅 결과(성공/실패/비용)를 분석하여 더 나은 모델 선택과 경로 배정을 제안합니다. I-10(Tool Registry/Router)과 I-18(Self-evo Engine)과 연동됩니다.

또한 S-5는 **이중 역할**을 수행합니다: 라우팅 자기진화(Router Evolution) + Cloud Library 전략 진화(Cloud Evolver). [근거: D2.0-01 §5.7, CLAUDE.md §6 "S-5 Router Evolution (V3) → Cloud Evolver (V3) — 단일 모듈, 이중 역할"]

### 입력/출력

| 구분 | 내용 |
|------|------|
| **입력 (Input)** | 라우팅 이력(모델 선택, 비용, 성공률), Cloud Library 전략 데이터 |
| **출력 (Output)** | 라우팅 규칙 개선 제안, 모델 선택 최적화 방안, Cloud 전략 진화 제안 |

### 분류/버전

| 항목 | 값 |
|------|-----|
| 분류 | EXP (실험적) |
| V1 | OFF |
| V2 | OFF |
| V3 | **ON** |
| LOCK | — |
| I-모듈 연결 | I-10, I-18 |

### 핵심 요약 (3줄)
1. S-5는 **라우팅 규칙(어떤 모델/노드에 보낼지)을 자동 최적화**하는 모듈입니다.
2. **이중 역할**: Router Evolution(라우팅 진화) + Cloud Evolver(Cloud Library 전략 진화)를 함께 수행합니다.
3. V3에서만 활성화되며, I-10(툴 레지스트리/라우터)과 I-18(자기진화 엔진)과 연동됩니다.

## §9.6 S-6 검색 진화 (Search Evolution)

### 비유
도서관 사서가 **검색 노하우를 축적**하는 것과 같습니다. "이런 질문에는 학술 DB를 먼저 뒤지는 게 효율적이다", "이 키워드로 검색하면 더 좋은 결과가 나온다"를 경험으로 배워갑니다.

### 목적
VAMOS의 RAG(검색 증강 생성) 파이프라인에서 사용하는 **검색 전략과 검색 소스를 자동 최적화**합니다. 어떤 검색 엔진을, 어떤 파라미터로, 어떤 순서로 사용하면 가장 좋은 결과를 얻는지 분석하여 개선을 제안합니다. I-16(Knowledge Search Engine)과 I-18(Self-evo Engine)과 연동됩니다. [근거: D2.0-01 §5.7, CLAUDE.md §6]

### 입력/출력

| 구분 | 내용 |
|------|------|
| **입력 (Input)** | 검색 쿼리 이력, 검색 결과 품질(QoD), 소스별 성공률 |
| **출력 (Output)** | 검색 전략 개선 제안, 파라미터 튜닝 방안, 소스 우선순위 변경 제안 |

### 분류/버전

| 항목 | 값 |
|------|-----|
| 분류 | EXP (실험적) |
| V1 | OFF |
| V2 | OFF |
| V3 | **ON** |
| LOCK | — |
| I-모듈 연결 | I-16, I-18 |

### 핵심 요약 (3줄)
1. S-6은 **검색 전략과 검색 소스를 자동으로 최적화**하는 모듈입니다.
2. 검색 결과의 품질(QoD)을 분석하여 더 나은 검색 방법을 제안합니다.
3. V3에서만 활성화되며, I-16(지식 검색 엔진)과 I-18(자기진화 엔진)과 연동됩니다.

## §9.7 S-7 사용자 협업 설계기 (User-Coop Designer)

### 비유
단골 고객의 **취향을 기억하는 바리스타**와 같습니다. "이 손님은 아이스 아메리카노를 좋아하고, 시럽은 반만 넣는 걸 선호한다"를 기억하듯, 사용자의 선호도와 작업 방식을 학습합니다.

### 목적
사용자와의 상호작용 이력을 분석하여 **사용자 선호도와 작업 패턴을 학습**합니다. 이를 바탕으로 VAMOS의 응답 방식, UI 배치, 도구 추천 등을 사용자에게 맞춤 최적화하는 제안을 생성합니다. 사용자가 능동적으로 참여하는 **협업형(Cooperative) 자기진화**를 지향합니다. I-19(Approval Manager)와 I-18(Self-evo Engine)과 연동됩니다. [근거: D2.0-01 §5.7, CLAUDE.md §6]

### 입력/출력

| 구분 | 내용 |
|------|------|
| **입력 (Input)** | 사용자 상호작용 이력, 피드백, 선호도 데이터, 작업 패턴 |
| **출력 (Output)** | 맞춤 설정 제안, UI/UX 최적화 방안, 도구 추천 개선 |

### 분류/버전

| 항목 | 값 |
|------|-----|
| 분류 | EXP (실험적) |
| V1 | OFF |
| V2 | OFF |
| V3 | **ON** |
| LOCK | — |
| I-모듈 연결 | I-19, I-18 |

### 핵심 요약 (3줄)
1. S-7은 **사용자 선호도와 작업 패턴을 학습**하여 맞춤 최적화를 제안하는 모듈입니다.
2. 사용자가 직접 참여하는 **협업형(Cooperative) 자기진화** 방식을 사용합니다.
3. V3에서만 활성화되며, I-19(승인 관리자)와 I-18(자기진화 엔진)과 연동됩니다.

## §9.8 S-8 자기 진화 거버넌스 (Self-evo Governance)

### 비유
국회의 **법사위원회**와 같습니다. 다른 S-모듈들이 "이렇게 바꾸면 좋겠습니다"라고 제안(법안 발의)을 하면, S-8이 **"이 변경이 안전한가?", "규칙에 어긋나지 않는가?"를 심사**하고, 최종적으로 사용자에게 승인을 요청합니다.

### 목적
S-1~S-7이 생성한 모든 자기 진화 제안을 **감독·심사·승인 관리**하는 거버넌스(governance, 관리 감독 체계) 모듈입니다. 모든 진화 제안이 불변 영역(정체성, 비용 상한, Non-goal 등)을 침범하지 않는지 확인하고, 승인 흐름을 관리합니다. I-19(Approval Manager), I-8(Policy Engine), I-9(Cost Manager), I-24(Knowledge Graph Engine)와 연동됩니다. [근거: D2.0-01 §5.7, CLAUDE.md §6]

### 입력/출력

| 구분 | 내용 |
|------|------|
| **입력 (Input)** | S-1~S-7의 진화 제안, 불변 영역 규칙, 비용 예산 |
| **출력 (Output)** | 승인/거부 결정, 위반 사항 보고, 감사 로그 |

### 분류/버전

| 항목 | 값 |
|------|-----|
| 분류 | **COND** (조건부) |
| V1 | OFF |
| V2 | OFF |
| V3 | **ON** |
| LOCK | **change_lock=true** (변경 불가) |
| I-모듈 연결 | I-19, I-8, I-9, I-24 |

> **change_lock=true**: S-8은 S-Series에서 유일하게 `change_lock`이 설정된 모듈입니다. 거버넌스(감독) 모듈의 동작 방식 자체가 변경되면 안전장치가 무력화될 수 있기 때문입니다.

> **Modal(팝업) 전용 UI**: S-8의 승인/거버넌스 화면은 일반 패널이 아니라 **팝업(Modal)으로만** 표시됩니다. 이는 사용자가 반드시 주의를 기울여 승인하도록 하기 위함입니다. [근거: D2.0-01 §5.7]

### 핵심 요약 (3줄)
1. S-8은 모든 자기 진화 제안을 **감독하고 승인을 관리**하는 거버넌스 모듈입니다.
2. **change_lock=true** (변경 불가) — 감독 모듈의 동작 방식이 바뀌면 안전장치가 무력화되므로 잠금 처리됩니다.
3. V3에서만 활성화되며, 승인 화면은 반드시 **팝업(Modal)** 으로만 표시됩니다.

## §10.1 A-1 멀티 AI 어댑터 (MultiBrain Adapter)

### 비유
여러 나라의 **통역사를 동시에 고용한 회사**와 같습니다. 한국어 문서가 필요하면 한국어 통역사를, 법률 문서가 필요하면 법률 전문 통역사를 배정합니다. VAMOS도 작업에 따라 **가장 적합한 AI 모델(Brain)을 자동 선택**합니다.

### 목적
여러 LLM(Claude, GPT, Gemini, 로컬 모델 등)을 **교체 가능하게 연결하는 추상화 계층**입니다. ORANGE CORE(두뇌)가 "이 작업에는 어떤 모델이 적합한가?"를 판단하면, A-1이 해당 모델에 표준 형식으로 요청을 보내고 결과를 받아옵니다. VAMOS의 핵심 인프라 모듈로, Brain Adapter Layer의 구현체입니다. [근거: D2.0-01 §5.9, D2.0-04 §2~3 Brain Engine Modularization, Brain Adapter Layer]

### 입력/출력

| 구분 | 내용 |
|------|------|
| **입력 (Input)** | LLM 호출 요청(프롬프트, 파라미터), 라우팅 정책(비용/품질/속도) |
| **출력 (Output)** | LLM 응답(표준 형식 ConnectorResponse), 비용/지연 시간 기록 |

### 분류/버전

| 항목 | 값 |
|------|-----|
| 분류 | **CORE** (핵심) |
| V1 | **ON** |
| V2 | **ON** |
| V3 | **ON** |
| LOCK | — |
| UI 노출 | 비노출 (internal) |

### Brain Adapter 표준 인터페이스

```
[ORANGE CORE — I-5 Decision]
        ↓ 표준 요청
[A-1 MultiBrain Adapter]
        ↓ 모델별 변환
┌───────┬───────┬───────┬───────┐
│Claude │ GPT   │Gemini │로컬   │
│(API)  │(API)  │(API)  │(Ollama)│
└───────┴───────┴───────┴───────┘
```

> **핵심 원칙**: CORE(두뇌)는 Brain Engine(모델)의 내부 구현을 직접 의존하지 않습니다. A-1 Adapter를 통해서만 소통합니다. [근거: D2.0-04 §2 "Brain Engine은 교체 가능해야 한다"]

### 핵심 요약 (3줄)
1. A-1은 여러 LLM을 **교체 가능하게 연결하는 추상화 계층**(Brain Adapter)입니다.
2. V1부터 CORE로 활성화되며, 작업 유형에 따라 **최적 모델을 자동 선택**합니다.
3. ORANGE CORE가 모델 내부 구현에 의존하지 않도록 **표준 인터페이스**를 제공합니다.

## §10.2 A-2 프리셋 모듈화 (Preset Modularization)

### 비유
스마트폰의 **자동화 앱(단축어/Tasker)**과 같습니다. "아침에 일어나면 → 날씨 확인 → 일정 브리핑 → 뉴스 요약"이라는 작업 흐름을 하나의 버튼으로 실행할 수 있도록 **미리 설정(프리셋)**해 두는 것입니다.

### 목적
사용자가 자주 사용하는 **작업 흐름(워크플로우)을 템플릿화**하여, 한 번의 클릭으로 반복 실행할 수 있게 합니다. 프리셋은 사용자가 직접 만들 수도 있고, VAMOS가 사용 패턴을 분석하여 추천할 수도 있습니다. I-13(Multimodal Output Renderer)과 연동됩니다. [근거: D2.0-01 §5.9, D2.0-05 §2]

### 입력/출력

| 구분 | 내용 |
|------|------|
| **입력 (Input)** | 프리셋 정의(작업 순서, 파라미터, 조건), 사용자 커스텀 설정 |
| **출력 (Output)** | 실행 가능한 프리셋 템플릿, 실행 결과, 프리셋 추천 |

### 분류/버전

| 항목 | 값 |
|------|-----|
| 분류 | **CORE** (핵심) |
| V1 | **ON** |
| V2 | **ON** |
| V3 | **ON** |
| LOCK | — |
| UI 노출 | 노출 (builder, panel) |

### 핵심 요약 (3줄)
1. A-2는 자주 사용하는 **작업 흐름을 프리셋(템플릿)으로 저장/실행**하는 모듈입니다.
2. V1부터 CORE로 활성화되며, 사용자가 직접 만들거나 VAMOS가 추천합니다.
3. I-13(Multimodal Output Renderer)과 연동되어 워크플로우 자동화를 지원합니다.

## §10.3 A-3 메타 AI (Meta AI)

### 비유
**감독이 선수 배치를 결정하는 것**과 같습니다. 어떤 경기(작업)에 어떤 선수(모델)를 투입하면 최고 성과를 낼 수 있는지, **경기 전략을 세우는 AI** 입니다.

### 목적
**AI가 AI를 관리**합니다. 여러 LLM 모델 중 어떤 모델을 어떤 상황에 사용하는 것이 최적인지를 **메타 학습(meta-learning)** 방식으로 판단합니다. A-1(MultiBrain Adapter)이 모델을 연결하는 "통역사"라면, A-3은 "어떤 통역사를 배치할지 결정하는 관리자"입니다. [근거: D2.0-01 §5.9, D2.0-04 §2]

### 입력/출력

| 구분 | 내용 |
|------|------|
| **입력 (Input)** | 모델별 성능 이력, 작업 유형 분류, 비용/품질/속도 제약 |
| **출력 (Output)** | 최적 모델 선택 권고, 모델 조합 전략, 성능 예측 |

### 분류/버전

| 항목 | 값 |
|------|-----|
| 분류 | EXP (실험적) |
| V1 | OFF |
| V2 | OFF |
| V3 | **ON** |
| LOCK | — |
| UI 노출 | 비노출 (internal) |

### 핵심 요약 (3줄)
1. A-3은 **AI가 AI를 관리**하는 메타 학습 모듈로, 최적 모델 선택을 자동화합니다.
2. A-1(Adapter)이 모델을 "연결"한다면, A-3은 "어떤 모델을 쓸지 결정"합니다.
3. V3에서만 활성화되는 실험적(EXP) 모듈이며, UI에 노출되지 않는 내부 모듈입니다.

## §10.4 A-4 토론 모드 (Debate Mode)

### 비유
법정의 **변호사와 검사가 토론**하는 것과 같습니다. 한쪽은 "이 답이 맞다"고 주장하고, 다른 쪽은 "아니다, 이 답이 더 낫다"고 반론합니다. 판사(ORANGE CORE)가 양쪽 주장을 듣고 **최종 판결**을 내립니다.

### 목적
여러 AI 에이전트(또는 같은 모델의 여러 인스턴스)가 하나의 주제에 대해 **토론(Debate)** 을 벌여, 더 정확하고 깊이 있는 결론을 도출합니다. 특히 복잡하거나 논쟁적인 질문에서 **단일 모델의 편향을 줄이고**, 다양한 관점을 반영할 수 있습니다. [근거: D2.0-01 §5.9, D2.0-05 §5, D2.0-08 §2]

### 입력/출력

| 구분 | 내용 |
|------|------|
| **입력 (Input)** | 토론 주제, 참여 에이전트 설정, 라운드 수, 판정 기준 |
| **출력 (Output)** | 각 에이전트의 주장, 반론, 최종 합의/판정 결과 |

### 분류/버전

| 항목 | 값 |
|------|-----|
| 분류 | **COND** (조건부) |
| V1 | OFF |
| V2 | **COND** |
| V3 | **ON** |
| LOCK | — |
| UI 노출 | 노출 (builder, panel/modal) |

> **V2에서 COND(조건부)**: V2에서는 사용자가 명시적으로 토론 모드를 요청하고, 승인 흐름을 통과한 경우에만 활성화됩니다. V3에서는 완전 활성화됩니다.

### 핵심 요약 (3줄)
1. A-4는 여러 AI 에이전트가 **토론을 통해 더 정확한 결론을 도출**하는 모듈입니다.
2. V2에서 **조건부(COND)** 활성화, V3에서 완전 활성화됩니다.
3. 복잡한 질문에서 단일 모델의 편향을 줄이고, **다양한 관점**을 반영합니다.

---

## §10.5 A-5 지연 생성 (Lazy Generation)

### 비유
음식점의 **주문 즉시 조리(cook-to-order)** 방식과 같습니다. 미리 모든 음식을 만들어 두지 않고, **손님이 주문할 때만** 해당 음식을 만듭니다. 불필요한 음식(출력)을 미리 만드는 낭비를 줄입니다.

### 목적
VAMOS가 출력을 생성할 때, **사용자가 실제로 필요로 할 때까지 생성을 지연**합니다. 예를 들어, 긴 보고서의 "목차"를 먼저 보여주고, 사용자가 특정 섹션을 클릭했을 때 해당 내용을 생성합니다. 이를 통해 **불필요한 LLM 호출과 비용을 절감**합니다. I-11(Output Composer)과 연동됩니다. [근거: D2.0-01 §5.9, D2.0-04 §5]

### 입력/출력

| 구분 | 내용 |
|------|------|
| **입력 (Input)** | 생성 요청, 지연 가능 구간 분석, 사용자 탐색 패턴 |
| **출력 (Output)** | 지연 생성 계획(어디까지 미리 생성/어디부터 지연), 최종 출력 |

### 분류/버전

| 항목 | 값 |
|------|-----|
| 분류 | EXP (실험적) |
| V1 | OFF |
| V2 | OFF |
| V3 | **ON** |
| LOCK | — |
| UI 노출 | 비노출 (internal) |

### 핵심 요약 (3줄)
1. A-5는 **필요할 때까지 출력 생성을 지연**하여 불필요한 비용을 절감하는 모듈입니다.
2. 긴 보고서의 목차만 먼저 보여주고, 사용자가 요청한 부분만 생성하는 방식입니다.
3. V3에서만 활성화되는 실험적(EXP) 모듈이며, UI에 노출되지 않는 내부 모듈입니다.

---

## §10.6 A-6 연합 모듈 네트워크 (Federated Module Network) — LOCK

### 비유
여러 지점의 **프랜차이즈 체인 본사**와 같습니다. 각 지점(모듈 노드)이 독립적으로 운영되지만, 본사(CORE)의 표준과 규칙을 따릅니다. 각 지점은 자기 영역의 고객을 담당하면서도, 필요하면 다른 지점의 자원을 요청할 수 있습니다.

### 목적
V3 Enterprise 환경에서 VAMOS의 모듈들을 **여러 서버/노드에 분산 배치**하여, 대규모 워크로드를 처리할 수 있게 합니다. CORE 노드와 여러 BLUE NODE들이 네트워크로 연합(Federation)하여 협력합니다. [근거: D2.0-01 §5.9, §0.5 LOCK "A-6=Federated 고정"]

### 입력/출력

| 구분 | 내용 |
|------|------|
| **입력 (Input)** | 모듈 분산 배치 요청, 노드 간 통신 규칙, 워크로드 분배 정책 |
| **출력 (Output)** | 분산 배치 결과, 노드 간 동기화 상태, 장애 복구 보고 |

### 분류/버전

| 항목 | 값 |
|------|-----|
| 분류 | EXP (실험적) |
| V1 | OFF |
| V2 | OFF |
| V3 | **ON** |
| LOCK | **A-6 = Federated 고정 (변경 불가)** |
| UI 노출 | 비노출 (internal) |

> **LOCK 주의사항**: A-6은 반드시 "Federated Module Network(연합 모듈 네트워크)"로만 정의합니다. 과거 문서에서 A-6에 "Remote Executor"가 포함되어 있던 것은 **DEPRECATED**(폐기)되었으며, Remote Executor는 A-7로 이관되었습니다. [근거: D2.0-01 §0.5 LOCK]

### 핵심 요약 (3줄)
1. A-6은 V3에서 VAMOS 모듈을 **여러 서버에 분산 배치**하는 연합(Federated) 네트워크 모듈입니다.
2. **LOCK**: A-6은 Federated로 고정되며, Remote Executor는 A-7로 분리되었습니다 (변경 불가).
3. V3에서만 활성화되는 실험적(EXP) 모듈이며, UI에 노출되지 않는 내부 모듈입니다.

## §10.7 A-7 원격 실행기 (Remote Executor) — LOCK

### 비유
**클라우드 컴퓨팅 서비스(AWS, Google Cloud)**와 같습니다. 내 컴퓨터로는 처리할 수 없는 대규모 연산을 **원격 서버의 강력한 GPU/CPU**에 보내서 실행하고, 결과만 받아옵니다.

### 목적
로컬 환경에서 처리하기 어려운 **대규모 연산이나 GPU 집약 작업**을 원격 서버에 위임하여 실행합니다. K8s(Kubernetes, 컨테이너 관리 시스템) 기반의 분산 실행 환경에서, 작업을 원격으로 배포·실행·결과 수집하는 역할을 합니다. [근거: D2.0-01 §5.9, §0.5 LOCK "Remote Executor는 A-7로만 정의"]

### 입력/출력

| 구분 | 내용 |
|------|------|
| **입력 (Input)** | 실행할 작업(코드, 모델 추론 등), 원격 서버 주소, 자원 요청(GPU/메모리) |
| **출력 (Output)** | 실행 결과, 자원 사용 보고, 실행 시간/비용 기록 |

### 분류/버전

| 항목 | 값 |
|------|-----|
| 분류 | EXP (실험적) |
| V1 | OFF |
| V2 | OFF |
| V3 | **ON** |
| LOCK | **A-7 = Remote Executor 고정 (변경 불가)** |
| UI 노출 | 비노출 (internal) |

> **LOCK 주의사항**: A-7은 A-6(Federated)과 별개의 독립 모듈입니다. A-6이 "모듈 분산"이라면, A-7은 "작업 원격 실행"으로, 역할이 명확히 구분됩니다. [근거: D2.0-01 §0.5 LOCK]

### A-6 vs A-7 비교표

| 구분 | A-6 Federated | A-7 Remote Executor |
|------|--------------|-------------------|
| **역할** | 모듈을 여러 서버에 **분산 배치** | 작업을 원격 서버에서 **실행** |
| **비유** | 프랜차이즈 지점 운영 | 클라우드 연산 위탁 |
| **초점** | 모듈 관리/동기화 | 연산 실행/결과 수집 |
| **LOCK** | Federated 고정 | Remote Executor 고정 |

### 핵심 요약 (3줄)
1. A-7은 대규모 연산을 **원격 GPU/서버에 위임하여 실행**하는 모듈입니다.
2. **LOCK**: A-7은 Remote Executor로 고정되며, A-6(Federated)과 별개의 독립 모듈입니다 (변경 불가).
3. V3에서만 활성화되는 실험적(EXP) 모듈이며, UI에 노출되지 않는 내부 모듈입니다.

## A-Series 핵심 요약 (3줄)

1. A-Series는 VAMOS의 **7개 아키텍처 확장 모듈**로, LLM 라우팅(A-1), 프리셋(A-2)부터 메타 AI(A-3), 토론(A-4), 지연 생성(A-5), 분산(A-6), 원격 실행(A-7)까지 **시스템 구조를 확장**합니다. [근거: D2.0-01 §5.9]
2. A-1, A-2만 V1부터 CORE로 활성화되고, A-4는 V2에서 조건부, 나머지는 **V3에서만** 활성화됩니다. 대부분 **UI에 노출되지 않는 내부 모듈**입니다. [근거: CLAUDE.md §6]
3. **A-6(Federated)과 A-7(Remote Executor)은 LOCK**으로 분리 고정되었으며, 혼동·병합이 금지됩니다. [근거: D2.0-01 §0.5 LOCK]
