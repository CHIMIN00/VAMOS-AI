# §16. 메모리 시스템 — 4계층 기억 구조

> VAMOS의 메모리 시스템은 사람의 기억처럼 **단기 기억부터 장기 기억까지 4단계로 나누어** 정보를 저장하고 관리합니다. 이 섹션에서는 왜 AI에게 기억이 필요한지, 4계층(L0~L3)이 각각 어떤 역할을 하는지, 그리고 기억이 어떻게 승격·삭제·보호되는지를 설명합니다.

---

## §16.1 왜 기억이 필요한가?

### 비유: 사람의 단기기억과 장기기억

일상생활을 떠올려 보세요.

- **단기기억**: 방금 들은 전화번호를 잠깐 기억하지만, 시간이 지나면 잊어버립니다.
- **장기기억**: 자전거 타는 법이나 집 주소처럼 오래 기억하는 것들이 있습니다.

AI도 마찬가지입니다. 대화가 끝나면 모든 것을 잊어버리는 AI는 매번 처음부터 다시 설명해야 합니다. VAMOS는 이 문제를 해결하기 위해 **사람의 기억 구조를 본뜬 4계층 메모리 시스템**을 설계했습니다. [근거: D2.0-06 §2]

| 사람의 기억 | VAMOS 메모리 | 설명 |
|------------|-------------|------|
| 지금 하는 대화 | L0 Session Buffer | 현재 대화 내용을 잠깐 기억 |
| 프로젝트 관련 기억 | L1 Project Memory | 진행 중인 프로젝트의 맥락 기억 |
| 오래된 지식 | L2 Global Knowledge | 검증된 지식을 오래 보존 |
| 몸이 기억하는 절차 | L3 Procedural Memory | 반복 작업 방법을 기억 |

### 핵심 요약 (3줄)
1. AI도 사람처럼 단기/장기 기억이 필요하다 — 매번 처음부터 설명하지 않으려면.
2. VAMOS는 L0(세션)→L1(프로젝트)→L2(글로벌)→L3(절차) 4계층으로 기억을 나눈다.
3. 각 계층은 독립적이면서도 서로 연결되어 기억이 승격·강등될 수 있다.

---

## §16.2 L0 Session Buffer (대화 기억)

### 비유
L0은 **칠판**과 같습니다. 수업(대화) 중에 필요한 내용을 칠판에 적어두고, 수업이 끝나면 지워버립니다.

### 정의
L0 Session Buffer는 **현재 진행 중인 대화의 내용을 임시로 저장**하는 가장 짧은 수명의 메모리입니다. 세션(대화)이 종료되면 자동으로 삭제됩니다. [근거: D2.0-06 §2, S7D-035]

### 상세 사양

| 항목 | 값 |
|------|-----|
| 저장소 | 인메모리 (Python dict) — 프로그램 메모리에 직접 저장 |
| 최대 크기 | **200K 토큰** (컨텍스트 윈도우 크기) |
| 수명 | 세션 동안만 유지 (세션 종료 시 만료) |
| B-Series 연결 | B-4 Working (작업 중 메모리, 휘발성) |
| 용도 | 현재 대화 컨텍스트, 즉시 작업 보조 |

### 슬라이딩 윈도우 (Sliding Window) 방식

L0은 **슬라이딩 윈도우** 방식으로 동작합니다. 이것은 마치 **일정 크기의 창문으로 긴 두루마리를 보는 것**과 같습니다.

- LLM 컨텍스트의 **80%**를 L0 버퍼로 사용
- 대화가 길어져 **70% 초과** 시 → 오래된 내용을 **자동 요약**하여 공간 확보
- 컨텍스트 크기별 처리 전략: [근거: D2.0-06 S7-F-096]
  - **10K 토큰 미만**: 직접 전송 (요약 불필요)
  - **10K~200K 토큰**: RAG 검색으로 관련 부분만 활용
  - **200K 토큰 초과**: 분할 처리

### 버전별 활성 여부

| 버전 | L0 상태 | 비고 |
|------|---------|------|
| V0 | — | 메모리 시스템 미도입 |
| V1 | **활성** | 세션 메타(요약, 힌트, hash) 활성화 |
| V2 | **활성** | V1과 동일 |
| V3 | **활성** | PII 정책 강화 적용 |

### 핵심 요약 (3줄)
1. L0은 현재 대화 내용만 임시 저장하는 "칠판" — 세션 종료 시 자동 삭제.
2. 최대 200K 토큰까지 저장하며, 70% 초과 시 자동 요약으로 공간을 확보한다.
3. 인메모리(프로그램 내부 저장)로 가장 빠르지만, 영구 보존은 불가능하다.

## §16.3 L1 Project Memory (프로젝트 기억)

### 비유
L1은 **프로젝트 노트**와 같습니다. 특정 프로젝트를 진행하면서 기록한 메모, 결정 사항, 선호도를 프로젝트 폴더에 정리해 두는 것입니다. 프로젝트가 끝나면 일정 기간 후 정리합니다.

### 정의
L1 Project Memory는 **프로젝트(project_id) 단위로 장기 저장**되는 메모리입니다. 프로젝트 목표, 제약 조건, 사용자 선호, 진행 맥락 등을 기억합니다. [근거: D2.0-06 §2]

### 상세 사양

| 항목 | 값 |
|------|-----|
| 저장소 | SQLite + Vector DB (Chroma/Qdrant) |
| 최대 용량 | **10,000 항목** / 프로젝트 |
| 수명 (TTL) | **90일** (프로젝트 종료 후 30일 연장 가능) 🔒 **변경 불가** |
| B-Series 연결 | B-1 Episodic (사건/대화/상황 기록) |
| 격리 | project_id별 완전 격리 |
| 용도 | 프로젝트 목표/제약/사용자 선호, 장기 작업 연속성 |

### 명시적 저장 원칙
- L1에 저장할 때는 **사용자 확인이 기본**입니다 (RULE 1.3 §7.3)
- "이것을 기억할까요?" 확인 UI를 통해 사용자가 승인/거부 [근거: D2.0-06 S7D-046]
- 중요한 결과만 저장하여 저장 최소화 원칙 준수

### 버전별 활성 여부

| 버전 | L1 상태 | 비고 |
|------|---------|------|
| V0 | — | 미도입 |
| V1 | **선택적 활성** | 프로젝트 단위 요약/결과 메타 선택 저장 |
| V2 | **기본 활성** | Postgres + Chroma 기반, Evidence/RAG 소스 관리 |
| V3 | **전면 활성** | QoD/TTL/PII 정책 강화 적용 |

### 핵심 요약 (3줄)
1. L1은 프로젝트별로 격리된 "프로젝트 노트" — 90일 보존 후 만료.
2. 저장 시 사용자 확인이 기본이며, 명시적으로 저장을 요청해야 한다.
3. 3회 이상 참조되면 L2(글로벌 지식)로 자동 승격 후보가 된다.

## §16.4 L2 Global Knowledge (장기 지식)

### 비유
L2는 **개인 도서관**과 같습니다. 여러 프로젝트에서 배운 것 중 정말 중요하고 검증된 지식만 골라 책장에 보관하는 것입니다. 한 번 도서관에 들어간 책은 영구 보존됩니다.

### 정의
L2 Global Knowledge는 **사용자 전체에 걸친 장기 지식 베이스**입니다. 특정 프로젝트에 국한되지 않고, 검증·요약된 지식을 영구적으로 보존합니다. [근거: D2.0-06 §2]

### 상세 사양

| 항목 | 값 |
|------|-----|
| 저장소 | SQLite + Vector DB (Chroma/Qdrant) + Knowledge Graph |
| 최대 용량 | **무제한** (로컬 디스크) |
| 수명 (TTL) | **무기한** (Core 지식은 영구 보존) 🔒 **변경 불가** |
| B-Series 연결 | B-3 Semantic (정리된 사실/지식) |
| 용도 | 검증/요약된 지식, 재사용 가능한 사실 |

### 3회 참조 시 자동 승격
L1에서 **3회 이상 참조**된 정보는 L2로 승격 후보가 됩니다. [근거: D2.0-06 S7D-040]

- 승격 조건: 재참조 3회 이상 **OR** 사용자 직접 확인 **OR** QoD(품질 점수) ≥ 0.7
- V1: 자동 승격 시 사용자 알림 후 확인 필요
- V2+: 조건 충족 시 자동 승격

### 신선도 감쇠 (Decay)
영구 보존이지만 **신선도 점수(Freshness Score)**로 관리합니다: [근거: D2.0-06 S7D-005]
- 공식: `freshness = 1.0 - (마지막 접근 후 경과일 / 최대 일수)`
- 신선도 < 0.3 → "아직 유효한가?" 재검증 트리거
- 주간 배치로 전체 L2 메모리 신선도 재평가

### 버전별 활성 여부

| 버전 | L2 상태 | 비고 |
|------|---------|------|
| V0 | — | 미도입 |
| V1 | **비활성** | 장기/대용량 벡터 저장은 미사용 |
| V2 | **제한적 활성** | 정책/비용 상한 내에서만, 저장 시 승인 필수 (S03-MOD-001) |
| V3 | **전면 활성** | QoD/TTL/PII 정책 강화 적용 |

### 핵심 요약 (3줄)
1. L2는 프로젝트를 넘어 영구 보존되는 "개인 도서관" — 검증된 지식만 저장.
2. L1에서 3회 이상 참조되면 자동 승격 후보가 되며, QoD ≥ 0.7이면 직접 승격 가능.
3. 영구 보존이지만 신선도 감쇠(decay)로 오래된 지식은 재검증된다.

## §16.5 L3 Procedural Memory (절차 기억)

### 비유
L3은 **요리 레시피북**과 같습니다. "김치찌개 만드는 법"처럼 **방법(How-to)**을 단계별로 정리해 두면, 다음에 같은 작업을 할 때 레시피를 보고 그대로 따라할 수 있습니다.

### 정의
L3 Procedural Memory는 **실행 가능한 절차(워크플로우/플레이북/템플릿/루틴)**를 저장하는 기억입니다. 답변 품질이 아니라 **"행동(실행)"을 직접 바꾸므로** 저장·활성에 강한 거버넌스(관리 체계)가 필요합니다. [근거: D2.0-06 §2.4]

### 상세 사양

| 항목 | 값 |
|------|-----|
| 저장소 | SQLite + Vector DB + Knowledge Graph |
| 최대 용량 | **무제한** (로컬 디스크) |
| 수명 (TTL) | **무기한** (deprecated 전환으로 폐기) 🔒 **변경 불가** |
| B-Series 연결 | B-2 Procedural (절차/플레이북/루틴/템플릿) |
| 검색 top_k | **5** (정밀도 우선, 백엔드 기본값 10 대비 절반, API 레이어 B1 기본값과 동일) |
| 접근 제어 | 읽기: L1 이상 권한 / 쓰기: D7 ApprovalGate 필수 |

### 저장 형태
| 형태 | 설명 | 예시 |
|------|------|------|
| Playbook (플레이북) | 단계/조건/분기가 있는 절차서 | "버그 수정 플레이북" |
| Workflow (워크플로우) | 파이프라인 형태의 작업 흐름 | Perception→Plan→Execute→Verify→Deliver |
| Template (템플릿) | 프롬프트/도구 호출 템플릿 | "코드 리뷰 프롬프트 템플릿" |
| Routine (루틴) | 반복 작업 자동화 규칙 | "매일 아침 뉴스 요약 루틴" |

### 활성화 상태 (Activation State)

```
draft → approved → active → deprecated
 (초안)   (승인됨)  (사용 중)  (폐기됨)
```

- **draft**: 후보로 생성됨 (아직 사용 불가)
- **approved**: D7 Safety/Approval 게이트를 통과함
- **active**: 실제 실행에 사용 중
- **deprecated**: 위험/오류/정책 위반 발견 시 즉시 비활성화

### 스킬 라이브러리 연동
L3 절차 메모리는 VAMOS의 **스킬 라이브러리**와 연동됩니다:
- L3에 저장된 절차 → Knowledge Graph 노드/엣지 자동 생성 [근거: D2.0-06 S7D-038]
- 승격 조건: L2에서 3회 이상 크로스 프로젝트 참조 또는 사용자 명시 저장

### 버전별 활성 여부

| 버전 | L3 상태 | 비고 |
|------|---------|------|
| V0 | — | 미도입 |
| V1 | **비활성 (OFF)** | 절차/템플릿은 L1에 임시 저장만 허용 |
| V2 | **제한적 활성** | 사용자 승인된 playbook/template만 저장, 자동 활성화: approved + 3회 수동 실행 |
| V3 | **전면 활성** | 자동 생성 routine 포함, DSPy 기반 자동 최적화 연동 |

### 핵심 요약 (3줄)
1. L3은 "실행 방법"을 저장하는 "요리 레시피북" — 행동을 직접 바꾸므로 강한 관리 필요.
2. draft→approved→active→deprecated 4단계 상태로 관리되며, D7 승인 게이트가 필수.
3. V1에서는 비활성(OFF)이고, V2부터 제한적으로 사용 가능하다.

## §16.6 기억 4유형: Working / Episodic / Semantic / Procedural

### 비유
기억의 4유형은 마치 **서랍장의 4개 서랍**과 같습니다. 각 서랍에는 성격이 다른 물건(정보)이 들어갑니다.

### 정의
VAMOS는 **무엇을 기억하느냐**에 따라 기억을 4가지 유형(B-Series)으로 분류합니다. 이 유형은 계층(L0~L3)과 조합되어 실제 저장이 결정됩니다. [근거: D2.0-06 §2]

### 4유형 상세

| B-Series | 유형명 | 정의 | 핵심 계층 | 예시 |
|----------|--------|------|----------|------|
| **B-4** | Working (작업 중 기억) | 현재 대화에서 사용 중인 임시 정보. 휘발성. | L0 | "방금 사용자가 말한 요구사항", "현재 작업 중인 코드 스니펫" |
| **B-1** | Episodic (사건 기억) | 특정 사건, 대화, 상황에 대한 기록 | L1 | "3월 10일 회의에서 결정한 사항", "프로젝트 A에서 겪은 에러" |
| **B-3** | Semantic (의미 기억) | 정리되고 검증된 사실/지식 | L2 | "Python에서 리스트는 []로 만든다", "사용자는 한국어를 선호한다" |
| **B-2** | Procedural (절차 기억) | 실행 가능한 절차, 방법론, 루틴 | L3 | "코드 리뷰 시 SOLID 원칙 체크리스트", "배포 파이프라인 순서" |

### (Lx, By) 매핑 규칙 🔒 **LOCK (변경 불가)**

| 저장 조합 | 대표 예시 | 기본 저장 레벨 |
|-----------|----------|--------------|
| (L0, B-4) | 현재 대화 컨텍스트/단기 상태/작업 중 힌트 | L0 |
| (L1, B-1) | 프로젝트 진행 맥락/결정/요약 | L1 |
| (L2, B-3) | 검증/요약된 지식 (검색 재사용) | L2 |
| (L3, B-2) | 플레이북/루틴/템플릿/절차 | L3 |

### 핵심 요약 (3줄)
1. 기억 4유형: Working(작업 중), Episodic(사건), Semantic(의미/지식), Procedural(절차).
2. 각 유형은 특정 계층(L0~L3)에 주로 저장되며, (Lx, By) 매핑 규칙으로 결정된다.
3. 이 매핑 규칙은 LOCK(잠금) 상태로, 변경할 수 없다.

## §16.7 L0→L1→L2 승격 규칙

### 비유
승격은 **시험 합격**과 같습니다. 여러 번 사용(참조)되어 "이건 중요하다"고 인정받으면 더 높은 계층으로 올라갑니다.

### 승격 흐름도

```
L0 (세션 버퍼)
    │ 세션 종료 시 자동 (요약 생성)
    ▼
L1 (프로젝트 메모리)
    │ 재참조 3회+ OR 사용자 확인 OR QoD ≥ 0.7
    ▼
L2 (글로벌 지식)
    │ 크로스 프로젝트 재참조 OR 범용 사실
    ▼
L3 (절차/템플릿 메모리)
    │ (V2+ 확장) 180일 미참조 시
    ▼
[Archive] (V2+ 확장 — 정본 4계층 외 참조용)
```

[근거: D2.0-06 S7D-040, S7D-041]

### 승격 조건 표

| 전환 | 조건 | 스코어링 공식 |
|------|------|-------------|
| **L0→L1** | 세션 종료 시 자동 (요약 생성 후 저장) | — (자동) |
| **L1→L2** | 재참조 **3회 이상** OR 사용자 확인 OR QoD ≥ 0.7 | `access_count×0.4 + recency×0.3 + confidence×0.3` |
| **L2→L3** | 크로스 프로젝트 재참조 OR 범용 사실 | — |
| L3→Archive (V2+) | **180일** 미참조 | freshness < 0.2 |
| Archive→삭제 (V2+) | **1년** 미참조 + 사용자 확인 | — |

### 강등/삭제 규칙 [근거: D2.0-06 S7D-041]

| 전환 | 조건 | 비고 |
|------|------|------|
| L2→L3 | 90일 미참조 | 절차적 지식으로 전환 |
| L3→Archive (V2+) | 180일 미참조 | 아카이브로 압축 이동 |
| Archive→삭제 (V2+) | 1년 미참조 + 사용자 확인 | 완전 삭제 |
| 예외 | `pinned=true` 메모리 | 강등 제외 |

### QoD (품질 점수) 기반 제어 [근거: D2.0-06 §2.5.2]
- QoD(Quality of Data) 점수: 0.0 ~ 1.0 🔒 **DEC-010 확정, 변경 불가**
- **QoD 가중치 공식** 🔒 **DEC-014 확정, 변경 불가**:
  - `qod_score = relevance×0.30 + accuracy×0.25 + freshness×0.25 + completeness×0.20`
- QoD < 0.4 → L2/L3 저장 금지 (L0/L1 임시 보관 후 재평가 또는 폐기)
- QoD ≥ 0.7 → L2 장기 저장 허용 (정책/승인 조건 충족 시)

### 핵심 요약 (3줄)
1. L0→L1은 세션 종료 시 자동, L1→L2는 3회 참조·사용자 확인·QoD≥0.7 중 하나 충족 시 승격.
2. 승격 스코어링: `access_count×0.4 + recency×0.3 + confidence×0.3`으로 계산.
3. QoD < 0.4이면 L2/L3 저장 금지, pinned=true 메모리는 강등에서 제외된다.

## §16.8 TTL (유효기간) 정책

### 비유
TTL은 **음식의 유통기한**과 같습니다. 각 기억에는 "얼마나 오래 보관할지" 기한이 붙어 있고, 기한이 지나면 자동으로 정리됩니다.

### TTL 기본값 표 🔒 **Design 2.0 기본값 확정, 변경 불가**

| 계층 | 기본 TTL | 만료 후 처리 | 비고 |
|------|----------|------------|------|
| **L0** (세션) | 세션 종료 시 만료 | 자동 삭제 | 명시적 저장 요청 시 L1로 승격 |
| **L1** (프로젝트) | **90일** | 사용자 알림 후 삭제/연장 선택 | 프로젝트 종료 후 30일 연장 가능 |
| **L2** (글로벌) | **무기한** | — (영구 보존) | 민감정보 제외 원칙, QoD 재평가 주기 적용 |
| **L3** (절차) | **무기한** | deprecated 전환으로 폐기 | draft/approved/active/deprecated 상태 관리 |

[근거: D2.0-06 §6.1, §2.5.3]

### 데이터 보존 정책 상세 (S7E-038) [근거: D2.0-07 §5.3 S7E-038]

| 데이터 유형 | 보존 기간 | 만료 후 처리 |
|------------|----------|------------|
| 세션 대화 (L0) | 7일 | 자동 삭제 |
| 단기 메모리 (L1) | 30일 | L2 승격 또는 삭제 |
| 프로젝트 (L2) | 90일 활성 | 아카이브 이동 |
| 영구 메모리 (L3) | 무기한 | 수동 삭제만 |
| 감사 로그 | 1년 | 자동 삭제 |
| 임시 캐시 | 24시간 | 자동 삭제 |

### 백업 정책 (버전별)

| 버전 | 정책 |
|------|------|
| V1 | 수동 백업 (SQLite/JSONL 파일 복사) |
| V2 | 일 단위 자동 백업 (Postgres dump + JSONL 아카이브) |
| V3 | 매니지드 서비스 백업 정책 적용 |

[근거: D2.0-06 §6.2]

### 핵심 요약 (3줄)
1. L0은 세션 종료 시, L1은 90일, L2/L3은 무기한 보존 — 각 계층마다 유통기한이 다르다.
2. L1 만료 전에는 사용자 알림/승인이 필요하다 (자동 삭제 방지).
3. 감사 로그는 최소 1년 보존, 임시 캐시는 24시간 후 자동 삭제된다.

## §16.9 PII 마스킹 & 데이터 보호

### 비유
PII 마스킹은 **편지에서 개인정보를 검은색 마커로 지우는 것**과 같습니다. 이름, 전화번호, 주민번호 같은 민감한 정보를 자동으로 감지하여 가려줍니다.

### PII (개인식별정보) 자동 감지 [근거: D2.0-07 §5.3 S7E-031]

VAMOS는 아래 8가지 PII 패턴을 **자동 탐지·마스킹**합니다:

| PII 유형 | 정규식 패턴 예시 | 마스킹 결과 |
|----------|----------------|-----------|
| 주민번호 | `\d{6}-[1-4]\d{6}` | `******-*******` |
| 전화번호 | `01[0-9]-\d{3,4}-\d{4}` | `010-****-****` |
| 이메일 | `[a-zA-Z0-9._%+-]+@...` | `k***@e****.com` |
| 카드번호 | `\d{4}[\s-]?\d{4}...` | `****-****-****-****` |
| 여권번호 | `[A-Z]{1,2}\d{7,8}` | `M********` |
| 운전면허번호 | `\d{2}-\d{2}-\d{6}-\d{2}` | `**-**-******-**` |
| 계좌번호 | `\d{3,6}-\d{2,6}-\d{2,8}` | `***-***-********` |
| 사업자등록번호 | `\d{3}-\d{2}-\d{5}` | `***-**-*****` |

### 버전별 마스킹 방식

| 버전 | 방식 | 설명 |
|------|------|------|
| V1 | **정규식 패턴 매칭** | 한국/글로벌 PII 패턴 최소 8개 |
| V2 | **Microsoft Presidio** (무료) | ML 기반 문맥 인식 마스킹 |
| V3 | **고급 NER** | 한국어 특화 NER + 문맥 인식 |

### 저장 정책 (Allow / Restrict / Deny) 🔒 **LOCK (변경 불가)** [근거: D2.0-06 §3.2]

| 정책 | 코드 | 처리 |
|------|------|------|
| **Allow** (허용) | PASS | 정책 위반/민감정보 없음 → 저장 가능 |
| **Restrict** (제한) | NEEDS_APPROVAL | 마스킹/요약/TTL 제한/승인 조건 부여 후 저장 |
| **Deny** (거부) | BLOCK | 정책 위반 → 저장 불가, **벡터 삽입 절대 금지** |

### 데이터 분류 체계 (4등급) [근거: D2.0-07 §5.3, S7D-065]

| 등급 | 설명 | 예시 | 저장 | 전송 |
|------|------|------|------|------|
| PUBLIC (공개) | 공개 정보 | 일반 지식, 공개 문서 | 모든 저장소 허용 | TLS |
| INTERNAL (내부) | 내부 정보 | 프로젝트 메모, 일정 | L1/L2 허용, 공유 제한 | TLS |
| CONFIDENTIAL (비밀) | 기밀 정보 | 개인 일기, 재무 데이터 | L1만(TTL 강제), 벡터 마스킹 필수 | E2EE |
| RESTRICTED (극비) | 최고 기밀 | API 키, 비밀번호, 의료정보 | **저장 금지 (Deny)**, 세션 종료 시 삭제 | E2EE+제로지식 |

### 저장 금지 항목
- 모든 레벨(L0~L3)에서 정책 위반(PII/금지 도메인/위험 요청 등) 시 **저장 불가**
- **벡터 삽입(RAG/임베딩) 절대 금지**
- 로그/트레이스에도 원문 저장 금지 (필요 시 요약/힌트만)
- 캐시에 개인정보 저장 금지 [근거: D2.0-06 S7D-054]

### 핵심 요약 (3줄)
1. 8가지 PII(주민번호, 전화번호, 이메일 등)를 자동 감지하여 `***`로 마스킹한다.
2. 데이터는 4등급(PUBLIC~RESTRICTED)으로 분류되며, RESTRICTED는 저장 자체가 금지된다.
3. Deny 판정 시 벡터 삽입(RAG) 절대 금지, 로그에도 원문 저장 불가.

### §16.10.1 GDPR 준수 설계 [근거: D2.0-07 §5.3 S7E-037]

GDPR(General Data Protection Regulation, 유럽 일반 데이터 보호 규정)은 개인정보를 보호하기 위한 국제 법률입니다. VAMOS는 EU GDPR 및 한국 개인정보보호법을 모두 준수하도록 설계되었습니다.

| 권리 | 설명 | VAMOS 구현 |
|------|------|-----------|
| **동의 관리** | 데이터 수집 전 명시적 동의 | 동의 수집 + 철회 기능 |
| **열람권** (Right of Access) | 내 데이터를 볼 수 있는 권리 | 사용자 데이터 조회/다운로드 |
| **삭제권** (Right to Erasure) | "잊힐 권리" — 데이터 완전 삭제 | 모든 데이터 완전 삭제 (`vamos purge --all`) |
| **정정권** | 잘못된 정보 수정 | 메모리 내 정보 수정 기능 |
| **이동권** (Right to Portability) | 데이터를 가져갈 수 있는 권리 | JSON 형식 내보내기 |
| **처리 기록** | 데이터 처리 활동 기록 | 처리 활동 기록 유지 |

### §16.10.2 데이터 암호화 (at-rest, in-transit) [근거: D2.0-07 §5.3 S7E-032, S7E-036]

**"at-rest"**는 저장된 상태의 데이터, **"in-transit"**는 네트워크로 전송 중인 데이터를 의미합니다.

#### 저장 데이터 암호화 (at-rest)

| 대상 | V1 | V2+ |
|------|-----|-----|
| SQLite | **SQLCipher** (AES-256-CBC) | Postgres 암호화 |
| Chroma (벡터 DB) | 디스크 암호화 의존 또는 커스텀 래퍼 | 서버 암호화 |
| JSON 파일 | AES-256 | AES-256 |
| 키 저장 | **OS Keychain** (Windows Credential Manager / macOS Keychain) | HSM 지원 |
| 키 파생 | PBKDF2 (사용자 비밀번호 기반) | — |

#### 전송 데이터 암호화 (in-transit)

| 버전 | 방식 | 설명 |
|------|------|------|
| V1 | **TLS 1.3** | LLM API 호출 시 기본 적용 |
| V2 | **E2E 암호화** | Signal Protocol / Noise Protocol Framework |
| V2+ | **X25519 + AES-256-GCM** | 키 교환 + 대칭 암호화 |
| V2+ 서버 | **Zero-knowledge** | 서버는 암호화된 데이터만 저장 (복호화 불가) |

### §16.10.3 사용자 데이터 삭제 요청 처리 (Right to be Forgotten) [근거: D2.0-07 §5.3 S7E-033, S7D-068]

사용자가 "모든 데이터를 삭제해주세요"라고 요청하면, VAMOS는 **모든 저장소에서 완전 삭제**합니다.

#### 삭제 대상 범위
| 저장소 | 삭제 방식 |
|--------|----------|
| 메모리 (SQLite) | 레코드 삭제 + 테이블 정리 |
| 벡터 DB (Chroma/Qdrant) | 임베딩 + 메타데이터 삭제 |
| Knowledge Graph (NetworkX/Neo4j) | 관련 노드/엣지 삭제 |
| 로그 | 사용자 관련 로그 삭제 |

#### 삭제 방식 (버전별)
| 버전 | 방식 | 설명 |
|------|------|------|
| V1 | **파일 덮어쓰기 (3-pass)** | 3번 덮어써서 복원 불가능하게 처리 |
| V2 | **Crypto-shredding** | 암호화 키를 파기하여 데이터 복호화 불가능하게 처리 |

#### 삭제 증명서
삭제 완료 시 **삭제 증명서**를 발급합니다:
- 삭제 완료 시각
- 삭제 범위 (어떤 저장소에서 삭제했는지)
- 해시 증명 (삭제가 정상 완료되었음을 증명하는 해시값)

#### 원클릭 삭제
```bash
vamos purge --all    # 모든 사용자 데이터 완전 삭제
```

### §16.10.4 감사 로그 (Audit Log) — who accessed what, when [근거: D2.0-06 S7D-007, S7D-071]

감사 로그(Audit Log)는 **"누가, 언제, 무엇에 접근했는지"**를 기록하는 불변(변경 불가) 기록입니다.

#### 로그 필드

| 필드 | 설명 | 예시 |
|------|------|------|
| action | 수행한 동작 | search, insert, delete, update |
| memory_id | 대상 메모리 ID | `mem_abc123` |
| layer | 계층 | L0, L1, L2, L3 |
| query_text_hash | 검색어 해시 (원문 미저장) | `sha256:a1b2c3...` |
| result_count | 검색 결과 수 | 5 |
| latency_ms | 처리 시간 (밀리초) | 45 |
| user_id | 사용자 ID | `user_001` |
| timestamp | 시각 | `2026-03-15T10:30:00Z` |

#### 무결성 보장
- **해시체인**: 각 로그 항목에 이전 항목의 해시를 포함하여 변조를 감지 [근거: D2.0-06 S7D-071]
- 로그 보존: **최소 1년**
- 저장소: V1 SQLite `memory_audit_log` 테이블, V2 PostgreSQL

#### 투명성 제공
- "이 메모리를 참조했습니다" 형태로 사용자에게 표시 (UI 연동) [근거: D2.0-06 S7D-007]
- 가장 자주 참조되는 메모리 Top-N 분석
- 미사용 메모리 식별

### §16.10.5 데이터 보존 정책 (Retention Policy) [근거: D2.0-07 §5.3 S7E-038, D2.0-06 S7D-070]

| 데이터 유형 | 보존 기간 | 만료 후 처리 | 커스텀 가능 |
|------------|----------|------------|-----------|
| 세션 대화 (L0) | 7일 | 자동 삭제 | 프로젝트별 |
| 단기 메모리 (L1) | 30일 | L2 승격 또는 삭제 | 프로젝트별 |
| 프로젝트 (L2) | 90일 활성 | 아카이브 이동 | 프로젝트별 |
| 영구 메모리 (L3) | 무기한 | 수동 삭제만 | — |
| 감사 로그 | 1년 | 자동 삭제 | 법적 요건(7년) 대응 가능 |
| 임시 캐시 | 24시간 | 자동 삭제 | — |

#### 법적 요건 대응
- **보존 필수 데이터**: 법적 요건(예: 7년 보존)이 있는 데이터는 **삭제 방지 플래그** 적용
- 사용자 커스텀: 프로젝트별/계층별 TTL 오버라이드 가능 [근거: D2.0-06 S7D-070]

#### 데이터 최소화 원칙 (S7E-034) [근거: D2.0-07 §5.3 S7E-034]
- LLM API 전송 시 **관련 데이터만 선별** 전송
- 전체 대화가 아닌 **요약만** 전송
- Tool 결과도 **필요 부분만 추출**
- 사용자 설정으로 "어떤 데이터를 LLM에 전송할지" 세분화 제어

### 핵심 요약 (3줄)
1. GAP-8 데이터 프라이버시: GDPR 6대 권리(열람/삭제/정정/이동/동의/처리기록) 모두 지원.
2. 암호화: 저장 시 AES-256(at-rest) + 전송 시 TLS/E2E(in-transit), V2에서 zero-knowledge 서버.
3. 삭제: Crypto-shredding으로 완전 삭제 + 삭제 증명서 발급, 감사 로그는 해시체인으로 무결성 보장.

## 부록: 시중 AI 메모리 비교표

VAMOS의 4계층 메모리 시스템이 다른 AI 서비스와 어떻게 다른지 한눈에 비교합니다. [근거: D2.0-06 S7D Part 1-1]

| AI | 메모리 유형 | 계층 수 | 용량 | 사용자 제어 | 자동 학습 | 크로스 대화 |
|----|-----------|--------|------|-----------|---------|-----------|
| ChatGPT | Saved Memories 등 4종 | 4계층 | 수백 항목 | 추가/삭제/편집 | 자동 추출 | 지원 |
| Claude | Project Knowledge + Memory | 2계층 | 200K 토큰/프로젝트 | 프로젝트 파일 | 제한적 | 프로젝트 내 |
| Gemini | Google 계정 연동 | 1.5계층 | 제한적 | 제한적 | 미지원 | 제한적 |
| Perplexity | 없음 (Stateless) | 0계층 | — | — | — | 미지원 |
| **VAMOS** | **L0→L1→L2→L3** | **4계층** | **무제한 (로컬)** | **완전 제어** | **자동+승인** | **지원** |

> **VAMOS 우위**: 4계층(L0~L3) + 로컬 무제한 저장 + 완전 사용자 제어. 시중 AI 중 가장 정교한 메모리 시스템.

## 전체 핵심 요약 (3줄)

1. **4계층 구조**: L0(세션)→L1(프로젝트)→L2(글로벌)→L3(절차) — 각 계층은 TTL·용도·저장소가 다르며, 3회 참조 시 상위 계층으로 승격된다.
2. **4유형 기억**: Working(B-4)·Episodic(B-1)·Semantic(B-3)·Procedural(B-2) — (Lx, By) 매핑으로 저장 위치가 결정된다 (LOCK).
3. **프라이버시 최우선**: PII 8패턴 자동 마스킹, AES-256 암호화, GDPR 6대 권리 지원, Crypto-shredding 완전 삭제 — GAP-8 데이터 프라이버시 경화 완료.
