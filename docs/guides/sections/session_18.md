# §21. 승인 시스템 — Approval Workflow

> VAMOS에서 AI가 "무엇이든 마음대로" 하지 못하도록 **사람의 허락(승인)**을 받는 체계입니다.
> 회사에서 중요한 일을 할 때 **결재(상급자 서명)**를 받는 것과 같은 원리입니다.
> [근거: D2.0-07 §3 Approval 정책 / D2.0-02 §8.1 5-Gate 체계]

---

## §21.1 언제 승인이 필요한가?

### 비유: 회사 결재 시스템

회사에서 일할 때를 떠올려보세요:

| 회사 상황 | VAMOS 상황 | 승인 필요? |
|-----------|-----------|-----------|
| 복사기 사용 | 읽기 전용 작업 (검색, 조회) | 불필요 (자유롭게 사용) |
| 비품 구매 (소액) | 일반 설정 변경, 오타 수정 | 불필요 (자동 처리) |
| 해외 출장 신청 | 비용 상한 변경, 도메인 확장 | **승인 필요** |
| 회사 규정 변경 | 정책/헌법(Constitution) 수정 | **반드시 OWNER 승인** |

### 정의: 2단계 승인 구조

VAMOS는 RULE 1.3에 따라 **2단계 승인(Two-Stage Approval)** 구조를 사용합니다:

| 단계 | 이름 | 의미 | 비유 |
|------|------|------|------|
| 1단계 | **계획 승인** (Plan Approval) | "무엇을 할지" 설명 후 허락받기 | 기안서 제출 → 상사 검토 |
| 2단계 | **실행 승인** (Execution Approval) | "지금 실행해도 되는지" 최종 확인 | 최종 결재 → 실행 |

> 일반적인 작업은 1단계(계획 승인)만으로 충분하지만, **위험한 작업은 반드시 2단계(실행 승인)까지** 받아야 합니다.
> [근거: D2.0-07 §3.1]

### 자동 승인 가능한 경우 vs 수동 승인 필요한 경우

```
┌──────────────────────────────────────────────────────────────┐
│                    승인 판단 흐름도                             │
│                                                              │
│  작업 요청 → [위험도 판단]                                      │
│               │                                              │
│          ┌────┴────┐                                         │
│       안전함      위험함                                       │
│          │         │                                         │
│     자동 처리   승인 필요                                      │
│     (allow)    (require_approval)                             │
│                    │                                         │
│              ┌─────┴─────┐                                   │
│           일반 위험    고위험                                   │
│              │          │                                    │
│         계획 승인    계획 + 실행 승인                            │
│         (1단계)      (2단계)                                  │
└──────────────────────────────────────────────────────────────┘
```

#### 자동 승인 가능 (승인 불필요)

AI가 **혼자서 처리해도 되는** 안전한 작업들입니다:
- 요약, 재생성, 오타 수정, 포매팅 개선
- 캐시 활용 (이전 결과와 동일한 조건)
- 읽기 전용 도구 사용 (검색, 조회, 파일 읽기)
- Allowlist(허용 목록)에 등록된 동작 (L1 자율 수준 이상)

[근거: D2.0-07 §3.2]

#### 수동 승인 필수 항목

다음은 **반드시 사람의 승인이 필요한** 항목입니다:

| 승인 필수 항목 | 승인 단계 | 승인 권한자 |
|---------------|----------|-----------|
| 도메인 확장 (P1/P2 활성화) | 계획 승인 | OWNER |
| 비용 상한 변경 | 계획 + 실행 승인 | OWNER/ADMIN |
| BASE RULE에 영향을 주는 정책 변경 | 계획 + 실행 승인 | OWNER |
| 위험 기능 (API/서버 명령 실행) | 실행 승인 | OWNER/ADMIN |
| 민감 데이터(PII) 저장 | 계획 승인 | OWNER |
| 자율 운영 수준 변경 (L0↔L3) | 계획 승인 | OWNER |
| Constitution(AI 헌법) 변경 | 계획 + 실행 승인 | OWNER만 |

[근거: D2.0-07 §3.1, RULE 1.3 §9.2]

#### NEVER_AUTO 목록 — 절대 자동 승인 불가 항목 (LOCK)

> 아래 항목은 어떤 자율 운영 수준(L0~L3)에서도 **절대로 자동 승인이 불가능**합니다.
> L3(FULL_AUTO) 상태에서도 이 항목들은 반드시 사람의 승인을 받아야 합니다.

| NEVER_AUTO 항목 | SOT 키 | 이유 | 변경 불가 |
|-----------------|--------|------|----------|
| 안전 규칙 변경 | safety_rules | RULE 1.3 안전 규칙 보호 | **LOCK** |
| 비용 상한 변경 | cost_ceiling | 비용 폭주 방지 | **LOCK** |
| 승인 흐름 변경 | approval_flow | 승인 체계 무력화 방지 | **LOCK** |
| Non-goal 항목 실행 (실거래, 법률/의료 단정 등) | non_goals | RULE 1.3 절대 금지 | **LOCK** |
| 감사 형식 변경 | audit_format | 감사 로그 무결성 보호 | **LOCK** |
| 데이터 보존 정책 변경 | data_retention | 데이터 보존 규정 보호 | **LOCK** |
| 사용자 동의 구조 변경 | user_consent | 사용자 권리 보호 | **LOCK** |
| 자기 권한 상승 (RBAC 역할 변경 포함) | escalate_own_privilege | 보안 위반 방지 | **LOCK** |
| 가드레일(안전 필터) 비활성화 또는 우회 | guardrails | 4-Layer 방어 보호 | **LOCK** |
| 게이트 체계 변경 (5-Gate 무력화) | gate | 5-Gate 안전 관문 보호 | **LOCK** |

[근거: D2.0-07 §15.12.2 LOCK, §3.2.1]

### 핵심 요약 (3줄)
1. VAMOS는 2단계 승인(계획 승인 → 실행 승인) 구조로 AI의 행동을 통제합니다.
2. 안전한 작업(읽기, 요약 등)은 자동 처리되지만, 위험 작업은 반드시 사람 승인이 필요합니다.
3. NEVER_AUTO 목록의 항목은 어떤 자율 수준에서도 절대 자동 승인될 수 없습니다 (LOCK).

## §21.2 P0/P1/P2 도메인별 승인 매트릭스

### 비유: 건물 출입 등급

도메인 등급을 **건물 출입 등급**에 비유하면 이해하기 쉽습니다:

| 건물 비유 | 도메인 등급 | 출입 방식 |
|-----------|-----------|----------|
| 로비/카페 (누구나 출입) | **P0** (일반) | 자유 출입 — 승인 불필요 |
| 사무실 (사원증 필요) | **P1** (주의) | 최초 1회 사원증 발급(승인) → 이후 자유 출입 |
| 금고/서버실 (매번 지문 인식) | **P2** (고위험) | **매번 지문 인식(승인)** + 세션 종료 시 자동 잠금 |

### P0: 승인 불필요 — 항상 허용

**일상적이고 안전한 작업 도메인**입니다. 별도 승인 없이 AI가 자유롭게 수행합니다.

- 예: 일반 질의응답, 텍스트 요약, 번역, 포매팅
- 처리: 자동 허용 (allow)
- [근거: D2.0-07 §2.1]

### P1: 최초 1회 승인 → 이후 자동

**주의가 필요하지만 한 번 허용하면 반복 사용 가능한 도메인**입니다.

- 예: 외부 API 호출, 특정 도구 활성화
- 처리: 최초 사용 시 승인 요청 → 승인 후 Allowlist에 등록 → 이후 자동 허용
- [근거: D2.0-07 §2.1, §3.2]

### P2: 매 세션 승인 필요 — 가장 엄격

**고위험 도메인**으로, 매번 승인이 필요하고 세션이 끝나면 자동으로 비활성화됩니다.

- 예: 금융 데이터 분석, 개인정보 처리, 시스템 설정 변경
- 처리 규칙:
  1. **활성화**: 명시적 승인 + 세션별 재확인 필수
  2. **실행**: 매 요청마다 "P2 도메인 사용 확인" 프롬프트 표시
  3. **HITL 타임아웃**: 5분 내 미응답 시 자동 거부 (deny)
  4. **세션 종료 시**: 자동 OFF (다시 쓰려면 다시 승인)
- [근거: D2.0-07 §2.1, RULE 1.3 §4.3, §3.3]

### 도메인 × 역할 승인 매트릭스

> 누가(역할) 어떤 도메인(P0/P1/P2)에서 승인할 수 있는지를 정리한 표입니다.

| 도메인 | OWNER | ADMIN | OPERATOR | VIEWER | AGENT |
|--------|-------|-------|----------|--------|-------|
| **P0** (일반) | 자동 허용 | 자동 허용 | 자동 허용 | 읽기만 | 자동 허용 |
| **P1** (주의) | 승인 가능 (계획) | 승인 가능 (계획) | 계획 승인까지만 | 불가 | 요청만 가능 |
| **P2** (고위험) | **승인 가능 (계획+실행)** | 불가 (OWNER만) | 불가 | 불가 | 요청만 가능 |

| 승인 유형 | OWNER | ADMIN | OPERATOR | VIEWER | AGENT |
|-----------|-------|-------|----------|--------|-------|
| 계획 승인 부여 | ✅ | ✅ | ✅ | ❌ | ❌ (요청만) |
| 일반 실행 승인 | ✅ | ✅ | ❌ | ❌ | ❌ |
| 고위험 실행 승인 (P2, 비용 상한) | ✅ | ❌ (OWNER만) | ❌ | ❌ | ❌ |

[근거: D2.0-07 §3.6.2, §2.1]

### 핵심 요약 (3줄)
1. P0(일반)은 승인 불필요, P1(주의)은 최초 1회, P2(고위험)는 매 세션 승인이 필요합니다.
2. P2 도메인 승인은 OWNER만 가능하며, 세션 종료 시 자동 비활성화됩니다.
3. AGENT(AI)는 어떤 도메인에서도 직접 승인할 수 없고, 승인 요청만 가능합니다.

## §21.3 승인 타임아웃

### 비유: 결재 기한

회사에서 결재 서류를 올렸는데 **상사가 10분 안에 서명하지 않으면 자동으로 반려**되는 것과 같습니다. 이렇게 하는 이유는 시스템이 "승인 대기" 상태에서 무한히 멈춰있는 것을 방지하기 위해서입니다.

### 타임아웃 규칙

```
┌──────────────────────────────────────────────────────┐
│              승인 타임아웃 흐름                         │
│                                                      │
│  승인 요청 발생 → 사용자에게 알림 → 타이머 시작          │
│                                         │            │
│                              ┌──────────┴──────────┐ │
│                           응답함               미응답  │
│                              │                  │    │
│                        승인/거부 처리      자동 거부    │
│                                          (deny)      │
│                                     + 사유 기록       │
│                                  (APPROVAL_TIMEOUT)  │
└──────────────────────────────────────────────────────┘
```

### 타임아웃 시간 — 상황별 정리

| 상황 | 타임아웃 | 미응답 시 처리 | 사유 코드 |
|------|---------|---------------|----------|
| **일반 승인** (비용 변경, 설정 등) | **10분** (LOCK) | 자동 거부 (deny) | `APPROVAL_TIMEOUT` |
| **P2 도메인 작업** | **5분** | 작업 취소 + P2 비활성화 | `APPROVAL_TIMEOUT` |
| **안전 필터 경계 판정** (점수 0.4~0.6) | **5분** | deny + 사유 로깅 | `APPROVAL_TIMEOUT` |
| **비용 상한 80%+ 호출** | **10분** | Mini 모델로 다운시프트 | `APPROVAL_TIMEOUT` |
| **자율 수준 L2→L3 상승** | **10분** | L2 유지 (변경 안 됨) | `APPROVAL_TIMEOUT` |
| **Non-goal 근접 요청** | **5분** | deny + OWNER 알림 | `APPROVAL_TIMEOUT` |

[근거: D2.0-07 §4.3.2, §15.11 S7E-050]

### 타임아웃 후 처리 상세

1. **자동 거부(deny)**: 요청은 즉시 취소됩니다
2. **사유 기록**: `OC_GATE_APPROVAL_TIMEOUT` 실패 코드가 감사 로그에 기록됩니다
3. **폴백 실행**: `FB_APPROVAL_TIMEOUT` → 안전 종료 (deny + 사유 기록)
4. **사용자 알림**: 타임아웃으로 인해 거부되었음을 UI에 표시합니다
5. **Decision 상태**: HOLD → DENY로 전환됩니다

[근거: D2.0-02 §8.1 ApprovalGate 상세]

### 핵심 요약 (3줄)
1. 일반 승인은 10분, 고위험(P2/안전 경계)은 5분 내 미응답 시 자동 거부됩니다 (LOCK).
2. 타임아웃 거부 시 `APPROVAL_TIMEOUT` 사유가 감사 로그에 기록됩니다.
3. 시스템이 승인 대기 상태에서 무한히 멈추는 것을 방지하기 위한 안전장치입니다.

## §21.4 자율 운영 4단계 (L0~L3)

### 비유: 자동차 자율주행 레벨

자율 운영 수준을 **자동차 자율주행**에 비유하면 직관적입니다:

| 자율주행 비유 | VAMOS 수준 | 설명 |
|-------------|-----------|------|
| 수동 운전 (운전자가 모든 것을 조작) | **L0 FULL_MANUAL** | 모든 행동에 사용자 확인 필요 |
| 크루즈 컨트롤 (속도만 자동) | **L1 SUPERVISED** | 허용 목록 내 동작만 자동, 나머지는 승인 |
| 부분 자율 (직선 도로만 자동) | **L2 SEMI_AUTO** | 읽기/분석 자동, 쓰기/변경은 승인 |
| 완전 자율 (목적지만 입력) | **L3 FULL_AUTO** | 대부분 자동, 정책 위반 시만 차단 |

### 4단계 상세 정의 (LOCK)

| 수준 | 명칭 | 설명 | 승인 요구 | 변경 불가 |
|------|------|------|-----------|----------|
| **L0** | FULL_MANUAL | 모든 행동에 사용자 확인 필요 | 매 동작마다 승인 | **LOCK** |
| **L1** | SUPERVISED | Allowlist(허용 목록) 내 동작만 자동. 나머지는 승인 요청 | Allowlist 외 동작은 계획 승인 | **LOCK** |
| **L2** | SEMI_AUTO | 읽기/생성/분석은 자동. 쓰기/변경은 승인 | 쓰기·변경 시 계획 승인, 고위험은 실행 승인 | **LOCK** |
| **L3** | FULL_AUTO | 비용 상한·Non-goal·P2 금지 외 자동 실행 | 정책 위반·상한 초과 시만 차단 | **LOCK** |

> **기본값: L1 (SUPERVISED)** — VAMOS는 처음 시작할 때 L1 수준으로 동작합니다. [근거: D2.0-07 §3.2.1]

### 각 수준에서 자동/수동 승인 상세 표

| 작업 유형 | L0 (수동) | L1 (감독) | L2 (반자동) | L3 (자동) |
|----------|----------|----------|-----------|----------|
| 텍스트 요약/번역 | 매번 승인 | 자동 | 자동 | 자동 |
| 오타 수정/포매팅 | 매번 승인 | 자동 | 자동 | 자동 |
| 웹 검색/파일 읽기 | 매번 승인 | Allowlist 내 자동 | 자동 | 자동 |
| 파일 쓰기/수정 | 매번 승인 | 계획 승인 | **계획 승인** | 자동 |
| 외부 API 호출 | 매번 승인 | 계획 승인 | **계획 승인** | 자동 |
| 시스템 설정 변경 | 매번 승인 | 계획 승인 | **계획+실행 승인** | 계획 승인 |
| 비용 상한 변경 | 매번 승인 | 계획+실행 승인 | **계획+실행 승인** | **계획+실행 승인** |
| P2 도메인 작업 | 매번 승인 | 계획+실행 승인 | **계획+실행 승인** | **계획+실행 승인** |
| Non-goal 항목 | **차단** | **차단** | **차단** | **차단** |

[근거: D2.0-07 §3.2.1 LOCK]

### 수준 변경 규칙

- **변경 권한**: OWNER의 명시적 승인(Approval: 자율 운영 수준 변경)이 반드시 필요합니다
- **기록**: 현재 자율 운영 수준은 세션 컨텍스트에 기록되며, 감사 로그에 남습니다
- **L3 제한**: L3에서도 P2 도메인, Non-goal, 비용 상한 초과 행동은 자동 실행 **불가** (RULE 1.3 절대 금지)
- **점진적 배포**: 새 기능은 L0→L1→L2 순으로 단계적으로 확대하는 것이 원칙입니다

[근거: D2.0-07 §3.2.1, §15.12.2]

### 핵심 요약 (3줄)
1. VAMOS는 L0(완전수동)~L3(완전자동) 4단계 자율 운영 수준을 제공합니다 (LOCK).
2. 기본값은 L1(SUPERVISED)이며, 수준 변경은 반드시 OWNER 승인이 필요합니다.
3. L3에서도 Non-goal, P2, 비용 상한 초과는 절대 자동 실행 불가합니다.

## §21.5 S-Module / E-Module 승인 규칙

### 비유: 내부 직원 vs 외부 협력업체

VAMOS의 모듈을 **회사 조직**에 비유하면:

| 회사 비유 | VAMOS 모듈 | 설명 |
|-----------|-----------|------|
| **내부 연구소** (자체 개선팀) | **S-Module** (Self-evo, 자기진화) | AI가 스스로를 개선하는 내부 기능 |
| **외부 협력업체** (아웃소싱) | **E-Module** (External, 외부 도구) | 외부 API, 파일 시스템, 브라우저 등 외부 기능 |

> 내부 연구소(S-Module)가 회사 시스템을 변경하려면 항상 대표(OWNER) 승인이 필요하고,
> 외부 업체(E-Module)는 최초 계약(도구 등록) 시 승인받으면 이후 정해진 범위 내에서 일할 수 있습니다.

### S-Module (Self-evo) 승인 규칙 표 (LOCK)

> S-Module은 AI가 **자기 자신을 분석하고 개선하는** 기능입니다.
> 분석/제안은 자유롭지만, **실제 적용(변경)은 항상 OWNER 승인이 필요**합니다.

| 행동 유형 | 기본 처리 | 승인 단계 | 비고 |
|----------|----------|----------|------|
| 분석/진단/제안 (읽기/생성) | **allow** (자동 허용) | 없음 | 제안은 가능, 적용은 별도 |
| 변경 적용 (패치/설정 변경) | **restrict** (제한) | **계획 승인** | 적용 전 변경 요약 + 영향/리스크 표시 |
| 자동 적용/자동 배포 (무인) | **deny** (기본 금지) | (예외 시) **실행 승인** | 기본 금지. 허용 시에도 실행 승인 필수 |

[근거: D2.0-07 §3.3 LOCK]

#### Self-evo 진화 범위별 승인

| 진화 범위 | 자동 허용 | 승인 필요 | 절대 금지 |
|----------|----------|----------|----------|
| 파라미터 조정 (온도, top-p 등) | ✅ (L2+) | — | — |
| 프롬프트 템플릿 수정 | — | ✅ 계획 승인 | — |
| Tool 사용 순서 변경 | — | ✅ 계획 승인 | — |
| 새 Tool 추가/제거 | — | ✅ 실행 승인 | — |
| 정책 규칙 변경 | — | — | ✅ (OWNER만 가능) |
| 안전 필터 변경 | — | — | ✅ (절대 금지) |
| RBAC 역할 변경 | — | — | ✅ (OWNER만 가능) |
| Non-goal 목록 변경 | — | — | ✅ (절대 금지, RULE 1.3) |

[근거: D2.0-07 §15.14.1]

### E-Module (External) 승인 규칙 표 (LOCK)

> E-Module은 **외부 도구/API를 호출하는** 기능입니다.
> 읽기 전용은 자동, 쓰기/변경은 승인 필요, 고위험은 기본 금지입니다.

| 행동 유형 | 기본 처리 | 승인 단계 | 비고 |
|----------|----------|----------|------|
| 읽기 전용 도구 (조회/검색/파일 read) | **allow** (자동 허용) | 없음 | 비용 상한 내 자동 허용 |
| 쓰기/변경 (파일 write, 시스템 설정 변경) | **restrict** (제한) | **계획 승인** | 파괴적/영구 변경은 실행 승인 추가 |
| 고위험 실행 (거래 실행/삭제/외부 송신) | **deny** (기본 금지) | **실행 승인** | 승인 없이는 실행 금지 |

[근거: D2.0-07 §3.3 LOCK]

### 위험 E-Module — 항상 승인 필수

> 특히 위험한 외부 도구는 **어떤 자율 수준에서도 항상 승인이 필요**합니다.

| 위험 도구 유형 | 예시 | 승인 | 이유 |
|---------------|------|------|------|
| **코드 실행기** | shell_exec, code_executor | 항상 실행 승인 | 시스템에 직접 영향 |
| **브라우저/네트워크** | browser, network_request | 항상 실행 승인 | 외부 접근 위험 |
| **파괴적 도구** | db_drop, file_delete | **BLOCKED** (실행 불가) | 복구 불가능한 손상 |
| **결제/거래** | payment, transaction | 항상 실행 승인 | 금전적 영향 |

> `destructiveHint=true`로 표시된 도구는 **항상 사용자 승인 필수**입니다.

[근거: D2.0-07 S7E-025, S7E-079, S7E-015]

### Tool 실행 권한 4단계

```
┌─────────────────────────────────────────────┐
│        Tool 실행 권한 (Permission Levels)     │
├─────────────────────────────────────────────┤
│  AUTO      → 자동 실행 가능 (web_search 등)  │
│  CONFIRM   → 사용자 확인 필요 (file_write 등) │
│  RESTRICTED → 관리자만 실행 (shell_exec 등)   │
│  BLOCKED   → 실행 불가 (db_drop 등)          │
└─────────────────────────────────────────────┘
```

[근거: D2.0-07 S7E-025]

### 승인 연동 아키텍처 — 5-Gate 체계

> 승인은 VAMOS의 **5-Gate 체계(5중 관문)** 중 하나인 **ApprovalGate**에서 처리됩니다.

| Gate (관문) | 역할 | 판정 값 |
|------------|------|---------|
| 1. PolicyGate | 정책/안전 위반 차단 | block / require_approval / mask / allow |
| 2. CostGate | 비용 상한 제어 | normal / downshift / split / stop |
| 3. **ApprovalGate** | **사용자 승인 필요 여부 판정** | **approved / denied / pending** |
| 4. EvidenceGate | 근거 품질(QoD) 검증 | sufficient / insufficient |
| 5. SelfCheckGate | 출력 품질·안전 자동 검증 | pass / fail / retry |

[근거: D2.0-02 §8.1 LOCK]

#### ApprovalGate 동작 흐름

```
PolicyGate에서 require_approval 판정
           │
           ▼
   Decision = HOLD (일시 중단)
           │
           ▼
   사용자에게 승인 요청 (UI 표시)
           │
     ┌─────┴─────┐
  approved     denied / timeout
     │              │
     ▼              ▼
 Decision 재개   DENY + 사유 기록
 (작업 계속)    (작업 취소)
```

- **트리거 조건**: PolicyGate에서 `require_approval` 판정 시, L2 메모리 저장 시, 고비용 실행 계획 확정 시
- **이벤트**: `oc.gate.approval.requested` → `oc.gate.approval.resolved`
- **실패 코드**: `OC_GATE_APPROVAL_TIMEOUT` / `OC_GATE_APPROVAL_DENIED`
- **폴백**: `FB_APPROVAL_TIMEOUT` → 안전 종료 (deny + 사유 기록)

[근거: D2.0-02 §8.1 ApprovalGate 상세]

### 실제 예시: 승인 흐름 시나리오

#### 시나리오 1: 파일 쓰기 (E-Module, 일반)
```
사용자: "보고서를 report.txt에 저장해줘"
  → E-Module 쓰기 요청 감지
  → PolicyGate: require_approval (쓰기 작업)
  → ApprovalGate: pending → 사용자에게 확인 요청
  → 사용자: "승인" 클릭
  → ApprovalGate: approved
  → 파일 저장 실행
```

#### 시나리오 2: 비용 상한 초과 (타임아웃)
```
AI가 고비용 모델 호출 시도 (예산 85% 사용 중)
  → CostGate: downshift 경고 (80%+)
  → ApprovalGate: pending → 사용자에게 확인 요청
  → 10분 경과 — 사용자 미응답
  → ApprovalGate: APPROVAL_TIMEOUT → 자동 거부
  → Mini 모델로 자동 다운시프트
```

#### 시나리오 3: Self-evo 변경 적용 (S-Module)
```
AI가 응답 품질 개선을 위해 프롬프트 템플릿 수정 제안
  → S-Module 분석/제안: allow (자동 허용)
  → "이 템플릿 수정을 적용할까요?" — 변경 요약 + 영향/리스크 표시
  → ApprovalGate: pending → OWNER에게 계획 승인 요청
  → OWNER: "승인"
  → 프롬프트 템플릿 수정 적용
  → 스냅샷 자동 저장 (롤백 대비)
```

### 핵심 요약 (3줄)
1. S-Module(자기진화)은 분석/제안은 자유, 변경 적용은 항상 OWNER 승인 필요합니다 (LOCK).
2. E-Module(외부 도구)은 읽기=자동, 쓰기=계획 승인, 고위험=실행 승인이 기본 규칙입니다 (LOCK).
3. 모든 승인은 5-Gate 체계의 ApprovalGate를 통해 처리되며, 감사 로그에 기록됩니다.

## Approval 객체 스키마 (참고)

| `requester` | 필수 | 요청자 (AGENT 또는 모듈 ID) |
