# §8. E-Series — 외부 도구 모듈 (E-1 ~ E-16, 총 16개)

> **비유**: I-Series가 VAMOS의 **두뇌**(생각하고, 판단하고, 기억하는 역할)라면, E-Series는 두뇌가 사용하는 **도구 상자**입니다. 사람으로 비유하면, 계산기, 인터넷 검색, 카메라, 번역기, 캘린더 같은 **외부 도구들**을 가방에 넣고 다니는 것과 같습니다. 두뇌(ORANGE CORE)가 "지금 이 도구를 써야겠다"고 **판단**한 후에만 도구가 열립니다.

[근거: CLAUDE.md §6 E-Series, D2.0-03 §4.4, D2.0-04 §7]

---

## E-Series 핵심 원칙 (모든 E-모듈 공통)

> 아래 규칙은 E-1부터 E-16까지 **모든** 외부 도구 모듈에 공통 적용됩니다.

### 1. 호출 경로 (LOCK — 변경 불가)

```
BLUE NODE(도메인 실행 모듈)가 "이 도구가 필요합니다" 제안
        ↓
ORANGE CORE(I-5 Decision)가 승인 여부 판단
        ↓
07 Gate 통과 (PolicyCheck → CostBudget → Approval)
        ↓
04 ToolRegistry가 적절한 adapter(연결 장치) 선택
        ↓
MCP(Model Context Protocol) 표준으로 도구 실행
        ↓
결과를 Decision 레코드에 기록
```

- **직접 호출 금지**: BLUE NODE가 도구를 직접 부르는 것은 불가능합니다. 반드시 ORANGE CORE의 판단 → Gate 통과 → ToolRegistry 경유 순서를 지켜야 합니다. [근거: D2.0-03 §2.3 LOCK]
- **MCP 래핑 필수**: 모든 외부 도구는 `@mcp.tool` 데코레이터로 감싸서 ToolRegistry에 등록해야 합니다. [근거: D2.0-03 §6.4.1 LOCK]
- **MCP 전송 계층**: Streamable HTTP (LOCK — DEC-017) [근거: D2.0-03 §4.4]

### 2. 보안 게이트 (07 Gate 연동)

| Gate | 역할 | 실패 시 |
|------|------|--------|
| **PolicyCheck** | "이 도구를 써도 되는가?" 정책 확인 | deny(거부) + 감사 로그 |
| **CostBudget** | "비용 한도 내인가?" 확인 | 차단 또는 다운시프트(저비용 대안으로 전환) |
| **Approval** | 고위험 도구는 사용자 승인 필요 | 승인 대기 또는 거부 |

[근거: D2.0-04 §7.2 ToolRegistry LOCK, D2.0-03 §4.4]

### 3. 외부 서비스 사용 전 체크리스트 (RULE 1.3 §3.1)

모든 E-모듈이 외부 API를 사용하기 전에 반드시 확인해야 하는 항목:
- 공식 API인가?
- 상업적 이용이 허용되는가?
- Rate Limit(호출 횟수 제한)은 얼마인가?
- 약관에서 금지하는 행위가 포함되어 있지 않은가?

### 4. ToolRegistry 등록 스키마

```
tool_id      : "domain.function@semver"  (예: "dev.code_exec@1.0.0")
category     : 8종 중 하나 (browser.render, api.http, code.exec 등)
risk_class   : low / med / high
cost_class   : v0 / v1 / v2 / v3
required_gates : [policy, cost, approval 중 필요한 것]
timeout_s    : 기본 30초
max_retries  : 기본 2회
```

[근거: D2.0-04 §7.2 ToolRegistryEntry]

## §8.1 E-1 코딩 및 시스템 설계 도우미 (Coding & System Design Helper)

### 비유
프로그래머 옆에 앉아서 **코드를 함께 짜주는 페어 프로그래밍 파트너**와 같습니다. 코드를 새로 만들어 주기도 하고, 기존 코드를 더 깔끔하게 정리해 주기도 하고, 버그(오류)를 찾아 고쳐 주기도 합니다.

| **출력 (Output)** | 생성된 코드, 리팩터링된 코드, 버그 수정 결과, 설계 문서 |

## §8.4 E-4 코드 실행기 (Code Executor) — ★ Docker Sandboxing 상세

### 비유
밀봉된 **실험실 안의 화학 실험**과 같습니다. 코드를 실행하는 것은 화학 실험처럼 위험할 수 있으므로, 완전히 격리된 안전한 공간(샌드박스) 안에서만 실행합니다. 실험실 밖(본체 컴퓨터)에는 영향을 줄 수 없습니다.

### 목적
사용자가 요청한 코드나 E-1이 생성한 코드를 **안전하게 실행**합니다. Docker 컨테이너(가상의 격리된 컴퓨터)를 이용하여, 코드가 본체 시스템에 영향을 주지 못하도록 엄격하게 격리합니다. 이것은 VAMOS의 가장 중요한 보안 조치 중 하나입니다. [근거: CLAUDE.md §7.2, D2.0-04 §7]

### 입력/출력

| 구분 | 내용 |
|------|------|
| **입력 (Input)** | 실행할 코드(Python, JavaScript 등), 실행 환경 설정 |
| **출력 (Output)** | 실행 결과(stdout/stderr), 생성된 파일, 실행 시간, 에러 정보 |

### 분류/버전

| 항목 | 값 |
|------|-----|
| 분류 | CORE |
| V1 | ON |
| V2 | ON |
| V3 | ON |

### ★ Docker Sandboxing 상세 (LOCK — 변경 불가)

> **샌드박스(Sandbox)란?** 모래놀이 상자처럼, 코드가 오직 그 상자 안에서만 동작하고 밖으로 나갈 수 없는 격리된 실행 환경입니다.

```
┌─────────────────────────────────────────────┐
│           Docker 샌드박스 컨테이너            │
│                                             │
│  ┌─────────────────────────────────────┐    │
│  │         사용자 코드 실행             │    │
│  └─────────────────────────────────────┘    │
│                                             │
│  🔒 네트워크 차단 (--network=none)          │
│  🔒 파일시스템 읽기전용 (readonly rootfs)    │
│  🔒 타임아웃 30초 (LOCK)                    │
│  🔒 메모리 제한 256MB                       │
│  🔒 CPU 제한 적용                           │
└─────────────────────────────────────────────┘
        ↕ (격리됨 — 본체 접근 불가)
┌─────────────────────────────────────────────┐
│           본체 컴퓨터 (Host)                 │
│    안전하게 보호됨                           │
└─────────────────────────────────────────────┘
```

### 4가지 핵심 보안 규칙 (LOCK — 변경 불가)

| # | 보안 규칙 | 설명 | 왜 필요한가? |
|---|----------|------|-------------|
| 1 | **네트워크 차단** (`--network=none`) | 코드가 인터넷에 접속하는 것을 **완전 차단** | 악성 코드가 외부로 데이터를 유출하거나, 외부에서 해킹 도구를 다운로드하는 것을 방지 |
| 2 | **파일시스템 읽기전용** (readonly rootfs) | 코드가 컨테이너의 파일을 **수정할 수 없음** | 시스템 파일 변조, 악성 파일 생성 방지 |
| 3 | **타임아웃 30초** (LOCK) | 코드 실행이 **30초를 초과하면 자동 종료** | 무한 루프나 과도한 연산으로 시스템이 멈추는 것을 방지 |
| 4 | **메모리 제한 256MB** | 코드가 사용할 수 있는 메모리를 **256MB로 제한** | 메모리 폭주(Memory Leak)로 다른 서비스가 죽는 것을 방지 |

[근거: CLAUDE.md §7.2 "코드 실행 격리: Docker 샌드박스 필수 (네트워크 차단, 30초)"]

### SANDBOX_* 에러코드

코드 실행 중 문제가 발생하면 아래 에러코드가 반환됩니다:

| 에러코드 | 발생 조건 | 의미 | 대응 방법 |
|---------|----------|------|----------|
| `SANDBOX_TIMEOUT` | 실행 시간 > 30초 | 코드가 시간 제한을 초과 | 코드 최적화 또는 작업 분할 필요 |
| `SANDBOX_OOM` | 메모리 사용 > 256MB | 메모리 부족 (Out of Memory) | 데이터 크기 줄이기 또는 청크 단위 처리 |
| `SANDBOX_NETWORK_VIOLATION` | 네트워크 접속 시도 | 차단된 네트워크 접근 시도 감지 | 네트워크 필요 없는 코드로 수정 |

### 특별 보안 규칙

- 코드 실행은 반드시 **사용자 승인 기반** (RULE 1.3 §2.7: 위험 기능 자동 실행 금지)
- ToolRegistry의 `required_gates`에 `[policy, cost, approval]` 모두 포함
- 실행 결과는 Decision 레코드에 연결하여 감사 추적 가능 [근거: D2.0-04 §7.2]

### 핵심 요약 (3줄)
1. E-4는 코드를 **Docker 샌드박스**(격리된 안전 공간) 안에서 실행하는 도구입니다.
2. 네트워크 차단, 파일시스템 읽기전용, **타임아웃 30초(LOCK)**, 메모리 256MB 제한이 적용됩니다.
3. 실행 실패 시 `SANDBOX_TIMEOUT`, `SANDBOX_OOM`, `SANDBOX_NETWORK_VIOLATION` 에러코드가 반환됩니다.

## §8.5 E-5 이미지 분석기 (Image Analyzer)

### 비유
사진을 보고 "이 사진에는 강아지가 있고, 배경은 공원이네요"라고 설명해 주는 **눈이 달린 AI 비서**와 같습니다. 사진·그림·차트 등 이미지를 보고 내용을 분석·설명합니다.

### 목적

## §8.6 E-6 논리 풀이기 (Z3 Solver)

### 비유
수학 시험에서 **자동으로 검증해 주는 채점관**과 같습니다. "이 조건들을 동시에 만족하는 답이 존재하는가?"를 논리적으로 증명하거나 반증합니다.

### 목적
Z3 SMT Solver를 이용하여 논리식(logical formula)을 검증합니다. 코드의 조건 검증, 수학적 증명, 제약 조건 만족 여부(SAT — Satisfiability) 확인 등에 사용됩니다. EVX-6(Z3 Solver Routing) 검증 체인과 연동됩니다. [근거: CLAUDE.md §6, §6 EVX-1~EVX-6]

### 입력/출력

| 구분 | 내용 |
|------|------|
| **입력 (Input)** | 논리식, 제약 조건, 검증 대상 코드/수식 |
| **출력 (Output)** | SAT(만족)/UNSAT(불만족) 결과, 반례(counter-example), 증명 |

### 분류/버전

| 항목 | 값 |
|------|-----|
| 분류 | CORE |
| V1 | ON |
| V2 | ON |
| V3 | ON |

### 핵심 요약 (3줄)
1. E-6은 Z3 SMT Solver를 이용해 **논리적 검증**을 수행하는 도구입니다.
2. "이 조건이 성립하는가?"를 자동으로 판별(SAT/UNSAT)하여 정확성을 보장합니다.
3. V1부터 활성화되며, 코드 검증(C-1~C-3)과 EVX-6 체인에 연동됩니다.

## §8.10 E-10 외부 API 게이트웨이 (External API Gateway)

### 비유
건물의 **정문 경비원**과 같습니다. 외부에서 오는 모든 요청을 검문하고(인증), 너무 많은 사람이 한꺼번에 들어오지 못하게 조절하며(Rate Limiting), 출입 기록을 남깁니다(로깅).

### 목적
외부 REST API/GraphQL API를 호출할 때의 통합 관리 계층입니다. 인증(HMAC-SHA256), 호출 횟수 제한(Rate Limiting), 비용 추적, 에러 핸들링을 중앙에서 관리합니다. [근거: CLAUDE.md §6, D2.0-04 §4.4 category: api.http]

### 입력/출력

| 구분 | 내용 |
|------|------|
| **입력 (Input)** | API 호출 요청(URL, 파라미터, 인증 정보) |
| **출력 (Output)** | API 응답 데이터, 상태 코드, 비용 기록 |

### 분류/버전

| 항목 | 값 |
|------|-----|
| 분류 | EXP (실험적) |
| V1 | OFF |
| V2 | OFF |
| V3 | **ON** |

### 보안 규칙

| 규칙 | 설명 |
|------|------|
| **HMAC-SHA256 인증** | 모든 외부 API 호출에 서명(Signature) 검증 적용 |
| **Rate Limiting** | 외부 API 호출: 10회/분, 고비용 모델: 3회/분 (LOCK) |
| **Burst Protection** | 10초 내 10회 초과 시 1분 쿨다운 |
| **API 키 격리** | NODE 내부 하드코딩 금지, 07 Gate를 통해 런타임 주입 |

[근거: D2.0-04 §6, D2.0-03 §4.4 보안 위협 방어 가이드]

### 핵심 요약 (3줄)
1. E-10은 외부 API 호출을 **중앙에서 관리·보호**하는 게이트웨이 도구입니다.
2. HMAC 인증, Rate Limiting, 비용 추적 기능으로 안전한 외부 연동을 보장합니다.
3. V3에서만 활성화되는 실험적(EXP) 모듈입니다.

| V2 | **COND** |
| V3 | **ON** |

### 특별 보안 규칙

- **이메일 발송은 반드시 사용자 승인** (RULE 1.3 §2.7: 위험 기능 자동 실행 금지)
- **PII(개인정보) 보호**: 이메일 내용에 민감 정보가 포함되면 마스킹 처리
- **Non-goal 준수**: 민감 개인정보 장기 저장 금지 (RULE 1.3 §2.4)

### 핵심 요약 (3줄)
1. E-14는 IMAP/SMTP를 통해 **이메일을 수신·발신·분석**하는 도구입니다.
2. 이메일 발송은 **반드시 사용자 승인**이 필요하며, 자동 발송은 금지됩니다.
3. V2에서 조건부 활성화, V3에서 완전 활성화됩니다.

## §8.15 E-15 클라우드 수집기 / 파일 시스템 (Cloud Collector / File System)

### 비유
**도서관 사서가 여러 서점(클라우드)을 돌아다니며 필요한 책을 수집**해 오는 것과 같습니다. V1에서는 로컬 파일만 다루지만, V2 이후에는 클라우드의 자료까지 수집합니다.

### 목적
이 모듈은 **버전에 따라 역할이 확장**되는 특별한 모듈입니다:
- **V1**: 로컬 파일 시스템 I/O (파일 읽기/쓰기)
- **V2+**: Cloud Library(클라우드 라이브러리) 수집 기능 추가 — 외부 클라우드 서비스에서 데이터를 수집하고 VAMOS 내부로 가져옵니다.

[근거: CLAUDE.md §6 "E-15 File System (V1) → Cloud Collector (V2+) — 단일 모듈, 버전별 역할 확장"]

### 입력/출력

| 구분 | V1 (File System) | V2+ (Cloud Collector) |
|------|-------------------|----------------------|
| **입력** | 파일 경로, 읽기/쓰기 요청 | 클라우드 서비스 인증, 수집 대상 지정 |
| **출력** | 파일 내용, 쓰기 결과 | 수집된 데이터, 메타데이터, 품질 점수 |

### 분류/버전

| 항목 | 값 |
|------|-----|
| 분류 | RE-ADD (재도입) |
| V1 | OFF |
| V2 | **COND** |
| V3 | **ON** |

### 알려진 이슈

> **V1-002**: E-15의 명칭 충돌("File System / Cloud Collector" 겸용) 처리가 필요합니다. 현재는 "단일 모듈, 버전별 역할 확장"으로 정리되었습니다. [근거: CLAUDE.md §9 V1-002]

### 핵심 요약 (3줄)
1. E-15는 V1에서는 **로컬 파일 I/O**, V2+에서는 **클라우드 데이터 수집** 기능을 수행합니다.
2. 하나의 모듈이 버전에 따라 역할이 확장되는 **특별한 구조**입니다.
3. V2에서 조건부 활성화, V3에서 완전 활성화됩니다.

## §8.16 E-16 클라우드 저장소 동기화 (Cloud Storage Sync)

### 비유
여러 개의 USB에 **같은 파일을 자동으로 복사해 주는 동기화 프로그램**과 같습니다. VAMOS의 데이터를 Google Drive, S3, Dropbox 등 클라우드 저장소와 자동 동기화합니다.

### 목적
VAMOS 내부 데이터(메모리, 프로젝트 파일, 백업 등)를 외부 클라우드 저장소(S3, GCS, Google Drive, Dropbox, OneDrive 등)와 동기화합니다. 데이터 백업, 멀티 디바이스 접근, 팀 공유 등에 활용됩니다. [근거: CLAUDE.md §6, D2.0-03 §K-035]

### 입력/출력

| 구분 | 내용 |
|------|------|
| **입력 (Input)** | 동기화 대상(파일/폴더), 클라우드 서비스 인증, 동기화 방향(업/다운/양방향) |
| **출력 (Output)** | 동기화 결과(성공/실패), 충돌 보고, 변경 이력 |

### 분류/버전

| 항목 | 값 |
|------|-----|
| 분류 | RE-ADD (재도입) |
| V1 | OFF |
| V2 | **COND** |
| V3 | **ON** |

### 지원 클라우드 서비스

| 서비스 | 용도 | 적합 버전 |
|--------|------|----------|
| Google Drive | 파일 업/다운로드, Docs/Sheets 편집 | V2+ |
| Dropbox / OneDrive | 범용 파일 동기화 | V2+ |
| AWS S3 / GCS | 대용량 데이터 저장 | V2+ (서버 환경) |
| Cloudflare R2 | 비용 효율적 오브젝트 저장 | V3 |

[근거: D2.0-03 §K-035, D2.0-04 §4.4 category: storage.obj]

### 특별 보안 규칙

- **PII(개인정보) 체크**: 클라우드 업로드 전 PII 포함 여부 확인 필수
- **PolicyCheck**: `required_gates`에 `[policy(PII), cost]` 포함 (D2.0-04 §4.4)
- **암호화**: 전송 중 TLS 1.3 필수, 저장 시 AES-256 권장

### 핵심 요약 (3줄)
1. E-16은 VAMOS 데이터를 **S3, Google Drive 등 클라우드 저장소와 동기화**하는 도구입니다.
2. 업로드 전 **PII(개인정보) 포함 여부를 자동 확인**하는 보안 규칙이 적용됩니다.
3. V2에서 조건부 활성화, V3에서 완전 활성화됩니다.

## E-Series 핵심 요약 (3줄)

1. E-Series는 VAMOS의 **16개 외부 도구 모듈**로, 두뇌(ORANGE CORE)가 판단한 후에만 사용되며 반드시 **MCP → ToolRegistry → 07 Gate 경로**를 경유합니다. [근거: D2.0-03 §2.3, §4.4, §6.4.1]
2. V1에서 **6개 핵심 도구**(E-1~E-6)가 활성화되고, V2에서 **4개 조건부 도구**(E-13~E-16), V3에서 **나머지 6개 실험적 도구**(E-7~E-12)가 추가됩니다. [근거: CLAUDE.md §6]
3. 특히 E-4(Code Executor)의 **Docker Sandboxing**(네트워크 차단, 읽기전용 파일시스템, 타임아웃 30초, 메모리 256MB)은 LOCK이며 변경 불가합니다. [근거: CLAUDE.md §7.2]
