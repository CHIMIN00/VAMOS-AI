# §26. Cloud Library — 정보 수집 & 관리 시스템

> **비유**: 여러분이 대학 도서관 사서라고 상상해보세요. 전 세계에서 쏟아지는 책, 논문, 뉴스, SNS 게시글을 모아서, 진짜인지 검증하고, 분류하고, 서가에 정리하고, 오래된 정보는 업데이트하는 일을 합니다. VAMOS의 **Cloud Library**는 이 모든 과정을 자동으로 수행하는 **통합 지식 시스템**입니다 — 인터넷에서 지식을 수집하고, 검증하고, 분류하고, VAMOS AI를 지속적으로 업그레이드합니다.

[근거: VAMOS_CLOUD_LIBRARY_SPEC §1.1, §1.2]

## §26.1 10-Layer 아키텍처

### 비유: 도서관의 10단계 업무 흐름

도서관에서 새 책을 들이는 과정을 생각해보세요:
1. 주문 접수 (INPUT) → 2. 신간 탐색 (DISCOVERY) → 3. 책 평가 (EVALUATION) → 4. 구매/배송 (COLLECTION) → 5. 창고 보관 (DATA LAKE) → 6. 내용 요약/색인 (EXTRACTION) → 7. 분석/비교 (ANALYSIS) → 8. 품질 검증 (VALIDATION) → 9. 서가 배치/버전 관리 (VERSION CONTROL) → 10. 열람 서비스 (OUTPUT)

Cloud Library도 정확히 이 10단계로 동작합니다.

### 전체 구조도

```
Layer 1  INPUT           →  외부 입력 수신 (URL, 키워드, 웹훅)
Layer 2  DISCOVERY       →  자율 사이트 발견 (7-Stage Pipeline)
Layer 3  EVALUATION      →  사이트 평가 (0-100점)
Layer 4  COLLECTION      →  데이터 수집 (YouTube, Instagram, Web 등)
Layer 5  DATA LAKE       →  원본 데이터 저장 + 중복 제거
Layer 6  EXTRACTION      →  키워드 추출 + VAMOS 4계층 매핑
─────────────────────────── E-15 / S-5 경계선 ───────────────────────
Layer 7  ANALYSIS        →  분석 및 충돌 해결
Layer 8  VALIDATION      →  5단계 품질 검증 (G0-G4)
Layer 9  VERSION CONTROL →  버전 관리 (Semantic Versioning)
Layer 10 OUTPUT          →  결과 출력 및 진화
```

> **E-15/S-5 분할**: Layer 1~6은 **E-15 Collector** (BLUE NODE, 수집 담당), Layer 7~10은 **S-5 Evolver** (ORANGE CORE, 진화 담당)가 담당합니다. 이 분할은 **LOCK (변경 불가)**입니다.

[근거: VAMOS_CLOUD_LIBRARY_SPEC §2.1, §16 L2]

### 각 레이어 상세

| Layer | 이름 | 역할 (비유) | 입력 | 출력 | 담당 모듈 |
|-------|------|-----------|------|------|----------|
| **L1** | INPUT (입력) | 주문 접수 | URL, 키워드, 웹훅 payload | 정규화된 수집 요청 (CollectionRequest) | E-15 |
| **L2** | DISCOVERY (발견) | 신간 탐색 | 시드 키워드, 기존 키워드 풀 | 평가 대상 사이트 후보 목록 | E-15 |
| **L3** | EVALUATION (평가) | 책 평가 | 사이트 후보 URL + 콘텐츠 | SiteEvaluation (total_score 0-100) | E-15 |
| **L4** | COLLECTION (수집) | 구매/배송 | 승인된 사이트 목록 + 스케줄 | 정규화된 Raw Data (JSON/Markdown) | E-15 |
| **L5** | DATA LAKE (데이터 레이크) | 창고 보관 | 수집된 Raw Data | 정규화/인덱싱 데이터 + 중복 제거 | E-15 |
| **L6** | EXTRACTION (추출) | 내용 요약/색인 | 정규화 데이터 + 템플릿 | 분류된 키워드 + VAMOS 매핑 정보 | E-15 |
| **L7** | ANALYSIS (분석) | 분석/비교 | 추출 키워드 + 기존 키워드 DB | 분석 결과 + 충돌 해결 로그 + 트렌드 | S-5 |
| **L8** | VALIDATION (검증) | 품질 검증 | 분석 완료 데이터 | Gate 검증 결과 (pass/fail + 점수) | S-5 |
| **L9** | VERSION CONTROL (버전) | 서가 배치 | Gate 통과 데이터 + 변경 목록 | 새 버전 번호 + CHANGELOG + 체크포인트 | S-5 |
| **L10** | OUTPUT (출력) | 열람 서비스 | 버전 관리 완료 데이터 | 업데이트된 VAMOS 문서, 대시보드, API | S-5 |

[근거: VAMOS_CLOUD_LIBRARY_SPEC §2.2~§2.11]

### 전체 데이터 흐름

```
[인터넷] → [E-15 Collector (L1~L6)] → [Raw Data]
                                          │
                                          ├→ [I-2 RAG DB] → [Main LLM 응답 품질 향상]
                                          │
                                          └→ [S-5 Evolver (L7~L10)]
                                                 │
                                                 ├→ [분석/패턴 발견] → [S-4 메타학습]
                                                 ├→ [품질 검증 (Gate)] → [I-6 Self-check]
                                                 ├→ [버전 관리] → [VAMOS 문서 자동 업데이트]
                                                 ├→ [인사이트] → [I-24 Knowledge Graph Engine]
                                                 └→ [진화 제안] → [S-7 설계 개선] → [Human 승인]
```

[근거: VAMOS_CLOUD_LIBRARY_SPEC §6.7]

### 핵심 요약 (3줄)
1. Cloud Library는 인터넷 지식을 수집→발견→평가→수집→저장→추출→분석→검증→버전관리→출력하는 10계층(10-Layer) 아키텍처입니다.
2. Layer 1~6은 E-15 Collector(수집), Layer 7~10은 S-5 Evolver(진화)가 담당하며, 이 분할은 LOCK(변경 불가)입니다.
3. 수집된 데이터는 I-2 RAG에 삽입되어 LLM 응답 품질을 높이고, S-5를 통해 VAMOS 자체를 지속적으로 진화시킵니다.

> **참고**: 이 Cloud Library의 10계층(10-Layer) 파이프라인은 **4-Layer LLM Guardrails(MASTER_SPEC §9.1)**와 별개의 체계입니다. 10계층은 지식 수집·처리 파이프라인, 4-Layer는 LLM 안전 필터입니다. [근거: v13 DELTA-009~010]

## §26.2 G0~G4 5-Gate 검증 시스템

### 비유: 공항 보안 검색대

공항에서 탑승하려면 여러 검색대를 통과해야 합니다:
- 1차: 여권/신분 확인 (형식 검증)
- 2차: 수하물 엑스레이 (내용 검증)
- 3차: 금속 탐지기 (일관성 검증)
- 4차: 보안 요원 최종 확인 (보안 검증)
- 5차: 탑승구 최종 체크 (최종 검증)

Cloud Library의 **G0~G4 5-Gate**도 동일합니다 — 데이터가 VAMOS에 반영되기 전에 5단계 검증을 통과해야 합니다.

[근거: VAMOS_CLOUD_LIBRARY_SPEC §8.1~§8.5]

### 5-Gate 요약

| Gate | 이름 | 비유 | 통과 기준 | 실패 시 처리 |
|------|------|------|----------|------------|
| **G0** | FORMAT GATE (형식 검증) | 여권/신분 확인 | **100%** 충족 | 자동 수정 시도 → 불가 시 거부 |
| **G1** | CONTENT GATE (내용 검증) | 수하물 엑스레이 | **95%** 이상 | 누락 항목 보완 요청 |
| **G2** | CONSISTENCY GATE (일관성 검증) | 금속 탐지기 | **90%** 이상 | 충돌 항목 표시 + 해결 제안 |
| **G3** | SECURITY GATE (보안 검증) | 보안 요원 확인 | **100%** 충족 | 즉시 거부 + 알림 |
| **G4** | FINAL GATE (최종 검증) | 탑승구 체크 | **100%** 완료 | 배포 차단 |

### G0: FORMAT GATE (형식 검증) — "서류가 올바른 형식인가?"

| 검증 항목 | 설명 |
|-----------|------|
| 템플릿 섹션 완전성 | Section 1~10 존재 여부 |
| 필수 필드 존재 | 문서 ID, 작성일, 상태 |
| 마크다운 문법 유효성 | 문법 오류 검사 |
| 테이블 형식 일관성 | 테이블 구조 검증 |
| 코드 블록 닫힘 확인 | 열린 코드 블록 검사 |

> 통과 기준: **100%** — 형식 오류는 타협 없음

### G1: CONTENT GATE (내용 검증) — "내용이 충분한가?"

| 검증 항목 | 설명 |
|-----------|------|
| 키워드 최소 개수 | **10개** 이상 추출 |
| 우선순위 분류 존재 | CRITICAL/HIGH/MEDIUM/LOW 분류 |
| 검색 쿼리 예시 포함 | 키워드별 검색 쿼리 |
| VAMOS 연관 매핑 | 4계층(RULE/PLAN/DESIGN/SCHEMA) 매핑 |
| 출처 명시 | 추측 표현 불가 ("아마도", "일반적으로" 사용 금지) |

> 통과 기준: **95%** 이상

### G2: CONSISTENCY GATE (일관성 검증) — "기존 데이터와 모순 없는가?"

| 검증 항목 | 설명 |
|-----------|------|
| 기존 키워드 중복 확인 | 글로벌 인덱스 대조 |
| 동의어 그룹 일관성 | LLM = Large Language Model 등 통일 |
| 우선순위 논리성 | CRITICAL이 LOW보다 중요한지 확인 |
| VAMOS 계층 매핑 정확성 | 올바른 계층에 매핑되었는지 확인 |
| 이전 버전 호환성 | 기존 버전과 호환 여부 |

> 통과 기준: **90%** 이상

### G3: SECURITY GATE (보안 검증) — "안전한 데이터인가?"

| 검증 항목 | 설명 |
|-----------|------|
| 악성 URL 필터링 | 블랙리스트 대조 |
| 민감 정보 누출 방지 | API 키, 비밀번호 패턴 탐지 |
| 저작권 침해 필터 | 저작권 침해 콘텐츠 차단 |
| 스팸/광고 필터 | 스팸, 광고성 콘텐츠 차단 |
| 비윤리적 콘텐츠 필터 | 윤리 기준 위반 콘텐츠 차단 |

> 통과 기준: **100%** — 보안은 타협 불가

### G4: FINAL GATE (최종 검증) — "배포 준비 완료인가?"

| 검증 항목 | 설명 |
|-----------|------|
| G0~G3 모두 통과 확인 | 전 Gate 통과 필수 |
| 변경 영향 범위 분석 | 영향받는 VAMOS 문서 파악 |
| 롤백 가능 여부 | 체크포인트 존재 확인 |
| Human 승인 필요 여부 | 정책에 따라 결정 (조건부) |
| 배포 준비 상태 | 모든 전제조건 충족 |

> 통과 기준: 체크리스트 **100%** 완료 → 통과 시 버전 업데이트 + 배포

[근거: VAMOS_CLOUD_LIBRARY_SPEC §8.1~§8.6]

### Gate 검증 흐름

```
수집 데이터 → G0(형식) → G1(내용) → G2(일관성) → G3(보안) → G4(최종)
                                                                 │
                                                          ┌──────┴──────┐
                                                          ▼             ▼
                                                     PASS: 배포     FAIL: 거부
                                                  (버전 업데이트)   (피드백 루프)
```

> Gate 실패 시: S-5가 거부 사유를 분석하여 E-15에 재수집 요청. 최대 3회 재시도 후에도 실패 시 Human Review 대기열에 등록.

[근거: VAMOS_CLOUD_LIBRARY_SPEC §8.7, §11.4]

### 핵심 요약 (3줄)
1. Cloud Library는 G0(형식)→G1(내용)→G2(일관성)→G3(보안)→G4(최종)의 5단계 Gate를 통해 데이터 품질을 검증합니다.
2. G0(형식)과 G3(보안)은 100% 충족 필수이며, G1(내용)은 95%, G2(일관성)은 90% 이상이어야 통과합니다.
3. Gate 실패 시 자동 재수집 피드백 루프가 작동하며, 3회 실패 시 수동 검토 큐로 이관됩니다.

## §26.3 평가 점수 & 소스 신뢰도

### 비유: 식당 평가 시스템

맛집 앱에서 식당을 평가할 때 "맛, 서비스, 위생, 접근성"에 점수를 매기듯, Cloud Library도 사이트를 4가지 기준으로 평가합니다.

[근거: VAMOS_CLOUD_LIBRARY_SPEC §5.1]

### 사이트 평가 점수 (총 100점) — **LOCK (변경 불가)** 🔒

| 평가 항목 | 배점 | 세부 기준 | 비유 |
|-----------|------|----------|------|
| **신뢰도 (Trust)** | **25점** | 도메인 연령(5), HTTPS(5), 글로벌 순위(5), 업데이트 빈도(5), 저자 정보(5) | "이 식당이 오래되고 인증받았나?" |
| **관련성 (Relevance)** | **30점** | VAMOS 키워드 매칭률 (키워드당 2점, 최대 30점) | "내가 찾는 메뉴가 있나?" |
| **품질 (Quality)** | **25점** | 콘텐츠 길이(10), 코드 예제(5), 인용(5), 원본성(5) | "음식 맛이 좋은가?" |
| **접근성 (Access)** | **20점** | robots.txt 허용(5), RSS(5), API(5), 페이월 없음(5) | "예약 없이 갈 수 있나?" |

> **변경 불가 (LOCK)**: 이 배점 비율 (Trust 25 : Relevance 30 : Quality 25 : Access 20)은 확정된 설계 결정입니다. 관련성(Relevance)이 최우선입니다.

### 승인 임계값 — **LOCK (변경 불가)** 🔒

| 점수 범위 | 모드 | 동작 | 비유 |
|-----------|------|------|------|
| **>= 80점** | AUTO (자동 승인) | 자동 추가, 인간 개입 없음 | "즉시 추천 식당" |
| **60~79점** | NOTIFY (알림 승인) | 알림 발송 후 자동 추가 | "괜찮은 식당, 참고만" |
| **30~59점** | MANUAL (수동 승인) | 사용자 승인 필요, 대기열 등록 | "애매한 식당, 직접 확인 필요" |
| **< 30점** | BLOCK (자동 차단) | 자동 거부, 로그 기록 | "비추천 식당" |

### 소스 신뢰도 가중치 (7단계) — **LOCK (변경 불가)** 🔒

소스 간 충돌 시, 어떤 소스를 더 신뢰할 것인가를 결정하는 가중치입니다.

| 순위 | 소스 유형 | 가중치 | 예시 | 비유 |
|------|-----------|--------|------|------|
| 1 | 공식 발표 | **1.0** | OpenAI 공식 블로그, Anthropic | 정부 공식 발표 |
| 2 | 논문/연구 | **0.9** | arXiv, 학술지 | 학술 연구 결과 |
| 3 | 기술 문서 | **0.85** | 공식 docs, API 문서 | 공식 매뉴얼 |
| 4 | 기술 블로그 | **0.7** | Netflix Tech Blog 등 | 전문가 칼럼 |
| 5 | 뉴스 | **0.6** | TechCrunch, The Verge | 뉴스 기사 |
| 6 | 개인 블로그 | **0.5** | 전문가 개인 블로그 | 개인 후기 |
| 7 | SNS | **0.3** | Twitter/X, Reddit | 소문/의견 |

> **변경 불가 (LOCK)**: 이 7단계 가중치 순서와 값은 확정된 설계 결정입니다.

[근거: VAMOS_CLOUD_LIBRARY_SPEC §5.2, §16 L5~L7]

### 충돌 해결 알고리즘

동일 주제에 대해 소스 간 상반된 정보가 발견될 때:

| 단계 | 동작 | 예시 |
|------|------|------|
| **Step 1** | 소스 신뢰도 가중치 적용 | 공식 발표(1.0) vs SNS(0.3) → 공식 발표 우선 |
| **Step 2** | 다수결 원칙 | 3개 이상 일치 → 자동 채택 |
| **Step 3** | 해결 불가 시 | `CONFLICT_FLAG=true` + Human Review 등록 |

[근거: VAMOS_CLOUD_LIBRARY_SPEC §10.1]

### 핵심 요약 (3줄)
1. 사이트 평가는 신뢰도(25점)+관련성(30점)+품질(25점)+접근성(20점)=100점 만점으로, 80점 이상이면 자동 승인, 30점 미만이면 자동 차단됩니다 (LOCK).
2. 소스 신뢰도는 공식 발표(1.0)부터 SNS(0.3)까지 7단계로 분류되며, 충돌 시 가중치+다수결로 해결합니다 (LOCK).
3. 해결 불가능한 충돌은 CONFLICT_FLAG를 설정하고 Human Review 대기열에 자동 등록됩니다.

## §26.4 LOCK 결정사항

> **LOCK이란?**: 확정(LOCK)된 설계 결정은 **변경 불가**합니다. 시스템의 안정성과 일관성을 보장하기 위해 핵심 설계 값들을 고정한 것입니다.

[근거: VAMOS_CLOUD_LIBRARY_SPEC §16]

### 아키텍처 LOCK (4건)

| # | LOCK 항목 | 확정 값 |
|---|----------|---------|
| L1 | **10-Layer 아키텍처 유지** | Input→Discovery→Evaluation→Collection→Data Lake→Extraction→Analysis→Validation→Version Control→Output |
| L2 | **E-15/S-5 하이브리드 분할** | E-15(BLUE NODE) = L1-L6 수집, S-5(ORANGE CORE) = L7-L10 진화 |
| L3 | **5단계 Gate 시스템** | G0 Format(100%), G1 Content(95%), G2 Consistency(90%), G3 Security(100%), G4 Final(100%) |
| L4 | **Semantic Versioning** | MAJOR.MINOR.PATCH (MAJOR=호환성 변경, MINOR=키워드 10+개/섹션 추가, PATCH=오타/소수 변경) |

### 스코어링 LOCK (3건)

| # | LOCK 항목 | 확정 값 |
|---|----------|---------|
| L5 | **평가 점수 배분** | Trust 25, Relevance 30, Quality 25, Access 20 (총 100점) |
| L6 | **승인 임계값** | AUTO>=80, NOTIFY 60-79, MANUAL 30-59, BLOCK<30 |
| L7 | **소스 가중치 7단계** | 공식(1.0) > 논문(0.9) > 기술문서(0.85) > 기술블로그(0.7) > 뉴스(0.6) > 개인블로그(0.5) > SNS(0.3) |

### 진화 정책 LOCK (3건)

| # | LOCK 항목 | 확정 값 |
|---|----------|---------|
| L8 | **S1-S5 진화 단계** | Pattern Recognition→Rule Generation→Validation→Deployment→Feedback Loop |
| L9 | **Human-in-the-Loop 필수 조건** | CRITICAL 변경, 키워드 20+개 추가, 새 카테고리, 템플릿 구조 변경 |
| L10 | **일일 버전 증가 상한** | 최대 5회, 연속 실패 3회 시 자동 중단 |

### 하이브리드 LOCK (3건)

| # | LOCK 항목 | 확정 값 |
|---|----------|---------|
| L11 | **코드블럭 적용 순서** | CB1(V1-V2)→CB3(V2)→CB4(V3) |
| L12 | **비현실적 기능 대체 필수** | C8/C9/C10/E1-E5 모두 ALT 버전 사용 |
| L13 | **E-15 오케스트레이터 역할** | 기존 E-1(YouTube), E-2(Instagram) 등을 서브 수집기로 재활용 |

### 버전별 기술 스택 진화

| 시기 | 수집 | 저장 | 분석 | 검증 | UI |
|------|------|------|------|------|-----|
| **V0** | yt-dlp, WebFetch | Markdown, JSON | 템플릿 기반 | 수동 | CLI |
| **V1** | + GitHub API | + SQLite, JSONL | + 규칙 매칭 | + G0-G1 자동 | CLI + 파일 |
| **V2** | + RSS, 검색 API | + Postgres, ChromaDB | + LLM API | + G0-G4 자동 | Web Dashboard |
| **V3** | + WebSocket, Kafka | + Neo4j, Pinecone | + Fine-tuned LLM | + 이상 탐지 ML | REST API, SDK |

[근거: VAMOS_CLOUD_LIBRARY_SPEC §13.5, §16]

### 핵심 요약 (3줄)
1. Cloud Library의 LOCK 결정사항은 아키텍처 4건, 스코어링 3건, 진화 정책 3건, 하이브리드 3건으로 총 13건입니다.
2. 모든 LOCK은 변경 불가하며, 시스템 안정성과 일관성을 보장하기 위한 핵심 설계 결정입니다.
3. 기술 스택은 V0(로컬 CLI)에서 V3(K8s+vLLM+REST API)까지 점진적으로 진화하되, 10-Layer와 5-Gate 구조는 모든 버전에서 유지됩니다.

# §27. RT-BNP — 실시간 속보 파이프라인

> **비유**: 스마트폰의 **뉴스 속보 알림**을 생각해보세요. "긴급: 주식시장 거래 중단", "속보: 전쟁 발발" 같은 알림이 즉시 울립니다. 일반 뉴스는 아침에 모아서 읽지만, 속보는 **30초 이내**에 도착해야 의미가 있습니다. VAMOS의 **RT-BNP** (Real-Time Breaking News Pipeline, 실시간 속보 파이프라인)이 바로 이 역할 — 속보를 감지하고, 검증하고, VAMOS AI와 투자 전략 엔진에 **초고속으로 전달**합니다.

[근거: VAMOS_구현가이드_PART2 §6.10.1]

## §27.1 RT-BNP란?

### 비유: 뉴스 속보 알림 시스템

일반 뉴스 읽기와 속보 알림의 차이를 생각해보세요:

| 구분 | 일반 뉴스 (Cloud Library 배치) | 속보 (RT-BNP) |

| **대상** | 기술 지식, 논문, 블로그 | 시장 충격, 전쟁, 긴급 발표 |
| **비유** | 아침에 모아 읽는 신문 | 즉시 울리는 긴급 알림 |

### 핵심 정보

| 항목 | 내용 |
|------|------|
| **정식 명칭** | Real-Time Breaking News Pipeline (실시간 속보 파이프라인) |
| **약어** | RT-BNP |
| **목적** | 속보/급변 이벤트를 30초 이내 감지 → VAMOS AI + AI Investing에 실시간 전파 |
| **관련 모듈** | E-15 Collector, Cloud Library, I-2 RAG, AI Investing Strategy Engine |
| **정본** | VAMOS_CLOUD_LIBRARY_SPEC §7 확장 |

### 아키텍처 개요

```
[News Sources] → [RT Collector (L1.5)] → [Breaking Detector]
     → [Fast Gate (CL-G0+G3)] → [Kafka: cl.breaking.*]
     → [VAMOS EventBus]
     → [I-2 RAG 즉시 삽입] + [AI Investing Strategy Engine]
     → [UI 알림] + [사용자 승인 (P2)]
```

[근거: VAMOS_구현가이드_PART2 §6.10.1]

1. RT-BNP는 속보를 30초 이내에 감지하여 VAMOS AI와 AI Investing에 실시간 전파하는 파이프라인입니다.
2. 일반 Cloud Library 배치 수집(시간~일 단위)과 달리, 속도 우선의 간소화 검증(Fast Gate)을 사용합니다.
3. 속보 전파 후 30분 이내에 정규 G0~G4 Gate 재검증을 수행하여 정확성을 보장합니다.

## §27.2 뉴스 소스 Tier 분류

### 비유: 정보 전달 속도별 등급

소방서에서 화재 신고를 받는 방식에 비유할 수 있습니다:
- **T1**: 화재 감지 센서 자동 알림 (실시간) — 가장 빠름
- **T2**: 119 전화 신고 (30초 내) — 빠름
- **T3**: 이웃 주민 제보 (몇 분 내) — 보통
- **T4**: SNS 목격담 (수분~수십분) — 느림

| Tier | 지연 목표 | 소스 예시 | 수집 방식 | 적용 버전 |
|------|----------|----------|----------|----------|
| **T1** | <10초 | Bloomberg, Reuters, Finnhub WebSocket | WebSocket/SSE 스트리밍 | **V3** |
| **T2** | <60초 | NewsAPI, Alpha Vantage News, Finnhub REST | 30초 폴링 | **V2** |
| **T3** | <300초 | 주요 뉴스 RSS (Reuters, AP, 연합뉴스) | 60초 RSS 폴링 | **V1** |
| **T4** | <600초 | Twitter/X API, Reddit API | 120초 폴링 | **V2** |

> V1은 무료 RSS만 사용(T3), V2부터 유료 API(T2+T4), V3에서 프리미엄 WebSocket(T1)까지 확장됩니다.

[근거: VAMOS_구현가이드_PART2 §6.10.1]

### RT-BNP 전용 소스 신뢰도 가중치

RT-BNP에서는 기존 Cloud Library의 소스 가중치를 **뉴스/이벤트 도메인에 맞게 보정**합니다:

| 소스 유형 | 기존 가중치 (LOCK) | RT-BNP 보정 가중치 | 비고 |
|----------|-------------------|------------------|------|
| 공식 발표 (정부/중앙은행) | 1.0 | **1.0** | 최우선, 변경 없음 |
| 통신사 (Reuters, AP, 연합) | 0.85 | **0.95** | 속보 전문 +상향 |
| 금융 데이터 (Bloomberg, Finnhub) | 0.85 | **0.95** | 시장 데이터 전문 +상향 |
| 주요 언론 | 0.6 | **0.75** | 속보 시 상향 |
| SNS/소셜 | 0.3 | **0.4** | 다수 소스 교차 확인 시에만 |

> **기존 LOCK과의 관계**: 기존 소스 신뢰도 LOCK(L7)은 기술 지식 수집용이며, RT-BNP 전용 가중치는 뉴스/이벤트 도메인에만 적용됩니다.

[근거: VAMOS_구현가이드_PART2 §6.10.1]

### 핵심 요약 (3줄)
1. 뉴스 소스는 T1(실시간 WebSocket)부터 T4(SNS 폴링)까지 4단계로 분류되며, 버전별로 활성화 범위가 다릅니다.
2. RT-BNP는 속보 도메인 특화로 통신사·금융 데이터 가중치를 0.85→0.95로 상향 보정합니다.
3. V1은 무료 RSS(T3)만 사용하고, V2에서 REST API(T2+T4), V3에서 WebSocket 스트리밍(T1)까지 확장됩니다.

## §27.3 Breaking Event 분류 체계 (Level 1~5)

### 비유: 재난 경보 등급

기상청이 태풍 경보를 "주의보 → 경보 → 특보"로 나누듯, RT-BNP도 속보를 **영향도에 따라 4등급**으로 분류합니다.

| 등급 | 기준 | 예시 | 액션 |
|------|------|------|------|
| **BREAKING-P0** | 시장 즉시 영향 | 전쟁 발발, 금리 긴급 변경, 거래소 중단 | **즉시 전파** + Circuit Breaker 평가 |
| **BREAKING-P1** | 24시간 내 영향 | 대규모 기업 실적, 제재 발표, 주요 파산 | **5분 내 전파** + 전략 재평가 |
| **BREAKING-P2** | 섹터/테마 영향 | 규제 변경, 기술 돌파, 산업 동향 | 정규 큐 **우선 삽입** |
| **NORMAL** | 일반 정보 | 일반 뉴스, 분석 기사 | 기존 배치 파이프라인 (Cloud Library) |

> **BREAKING-P0**는 가장 긴급한 등급으로, VAMOS AI와 AI Investing 모두에 **30초 이내** 전파되어야 합니다.

[근거: VAMOS_구현가이드_PART2 §6.10.1]

### 핵심 요약 (3줄)
1. 속보는 BREAKING-P0(즉시 영향)부터 NORMAL(일반)까지 4등급으로 분류됩니다.
2. P0는 30초 내 즉시 전파, P1은 5분 내 전파, P2는 정규 큐 우선 삽입으로 처리됩니다.
3. NORMAL 등급은 기존 Cloud Library 배치 파이프라인으로 처리되어, 속보 전용 자원을 절약합니다.

## §27.4 Breaking Detector 엔진

### 비유: 속보 판정 위원회

방송국에서 "이것이 속보인가?"를 판단할 때 여러 전문가가 의견을 모으듯, Breaking Detector도 **4가지 구성요소**가 협력하여 속보를 판정합니다.

| 구성요소 | 역할 | 비유 | 기술 | 적용 버전 |
|---------|------|------|------|----------|
| **Keyword Trigger** | 사전 정의 키워드 매칭 | "전쟁", "crash" 같은 키워드 감지 | 규칙 기반 | **V1** |
| **Velocity Detector** | 동일 주제 다수 소스 동시 보도 감지 | "여러 채널에서 동시에 같은 뉴스!" | 빈도 분석 | **V2** |
| **NLP Classifier** | 자연어 처리 기반 속보 분류 | AI가 문맥을 파악하여 분류 | FinBERT + 커스텀 모델 | **V2+** |
| **Impact Scorer** | 시장 영향도 0-100 점수 산출 | "이 속보가 시장에 얼마나 충격적인가?" | 규칙+ML 하이브리드 | **V2+** |

### 판정 흐름

```
뉴스 수신 → Keyword Trigger (V1: 1차 필터)
              → Velocity Detector (V2: 다수 소스 동시 보도?)
              → NLP Classifier (V2+: 문맥 분석)
              → Impact Scorer (V2+: 영향도 점수)
              → 등급 결정 (BREAKING-P0/P1/P2/NORMAL)
```

[근거: VAMOS_구현가이드_PART2 §6.10.1]

### 핵심 요약 (3줄)
1. Breaking Detector는 Keyword Trigger, Velocity Detector, NLP Classifier, Impact Scorer의 4가지 요소로 속보를 판정합니다.
2. V1에서는 키워드 규칙 기반으로 시작하고, V2부터 빈도 분석과 FinBERT 기반 ML 분류가 추가됩니다.
3. 최종적으로 Impact Scorer가 시장 영향도 점수(0-100)를 산출하여 BREAKING-P0/P1/P2/NORMAL 등급을 결정합니다.

## §27.5 Fast Gate (속보 전용 간소화 검증)

### 비유: 응급환자 전용 진료

일반 환자는 접수→대기→진료→검사→처방의 전체 과정을 거치지만, **응급환자는 즉시 응급실로** 갑니다. 일부 검사는 나중에 합니다. Fast Gate도 마찬가지 — 속도가 생명인 속보에서는 **핵심 검증만 하고 나머지는 사후 검증**합니다.

| Cloud Library Gate | Fast Gate 적용 여부 | 이유 |
|-------------------|-------------------|------|
| **CL-G0 (형식)** | **적용** | URL 유효성/기본 형식은 필수 |
| **CL-G1 (내용)** | **간소화** | T1/T2 사전 등록 소스는 자동 통과 |
| **CL-G2 (일관성)** | **스킵** | 속보는 키워드 매칭이 아닌 Impact 기준 |
| **CL-G3 (보안)** | **적용** | 악성 URL/허위 정보 필터는 필수 |
| **CL-G4 (최종)** | **스킵** | 속도 우선, 사후 검증으로 대체 |

### Fast Gate vs 정규 Gate 비교

```
[정규 Gate (일반 수집)]
G0(형식) → G1(내용) → G2(일관성) → G3(보안) → G4(최종)
                     5단계 전부 통과 필요 → 분~시간 소요

[Fast Gate (속보 전용)]
G0(형식) → G3(보안) → 즉시 전파!
                     2단계만 → 초 단위 소요
                     → 30분 내 정규 G0~G4 사후 검증
```

### 사후 검증 규칙

| 항목 | 규칙 |
|------|------|
| 사후 검증 시한 | 속보 전파 후 **30분 이내** 정규 G0~G4 재검증 |
| 검증 실패 시 | **RETRACTION 이벤트 발행** + 사용자 알림 + 이전 이벤트 무효화 |
| 동일 속보 중복 억제 | **5분 윈도우** 내 동일 주제 병합 |

> **Fast Gate ↔ VAMOS 5-Gate 관계**: Fast Gate(CL-G0~G4)는 Cloud Library 전용 Gate이며, VAMOS 파이프라인 5-Gate(Policy→Cost→Approval→Evidence→SelfCheck)와는 별도 체계입니다. Fast Gate 통과 후 속보가 VAMOS_EVENT로 전파되면, 해당 이벤트 기반 파이프라인 실행 시 VAMOS 5-Gate가 다시 적용됩니다.

[근거: VAMOS_구현가이드_PART2 §6.10.1]

### 핵심 요약 (3줄)
1. Fast Gate는 속보 전용 간소화 검증으로, G0(형식)+G3(보안)만 적용하여 초 단위로 통과시킵니다.
2. 속도를 위해 G1(내용), G2(일관성), G4(최종)은 스킵하되, 30분 이내 정규 Gate 재검증을 반드시 수행합니다.
3. 사후 검증 실패 시 RETRACTION(철회) 이벤트가 발행되어 이전 속보를 무효화합니다.

# §28. DCL — Domain Context Layer (도메인 컨텍스트 레이어)

> **비유**: 여러분이 아침에 커피를 마시며 **"오늘의 브리핑"**을 듣는 상황을 생각해보세요. 비서가 "오늘 증시가 2% 하락했고, 새로운 AI 모델이 출시되었고, 중동 정세가 긴박합니다"라고 알려줍니다. 이것이 바로 **DCL** — VAMOS AI에게 "지금 세상이 어떤 상황인지"를 선택적으로 알려주는 **배경 인식 시스템**입니다. 모든 뉴스를 다 읽는 게 아니라, **필요한 도메인의 핵심 정보만** 골라서 AI에게 주입합니다.

[근거: VAMOS_구현가이드_PART2 §6.10.2]

## §28.1 선택적 배경 인식 설계 원칙

### 핵심 질문: 왜 "전체 배경"이 아닌 "선택적 배경"인가?

| 접근 방식 | 비용 | 정보 품질 | 환각 위험 (거짓 정보 생성) | LOCK 호환 |
|----------|------|----------|----------------------|----------|
| 전체 인터넷 배경 크롤링 | V1 예산 **3~4배 초과** | 노이즈 (쓸모없는 정보) **90%** | **높음** | **위반** |
| **선택적 도메인 배경 (DCL)** | **V1 예산 내** | **관련 정보만** | **낮음** | **호환** |

> **DCL의 핵심 원칙**: "모든 정보를 가져오는 것"보다 **"필요한 정보를 빠르게 가져오는 것"**

### 비유: 뷔페 vs 코스 요리

- **전체 인터넷 배경** = 뷔페: 모든 음식이 있지만, 90%는 내가 먹지 않을 것. 비용도 비싸고, 뭘 먹어야 할지 혼란스러움
- **선택적 배경 (DCL)** = 코스 요리: 셰프가 골라준 핵심 메뉴만. 비용 효율적이고, 품질 보장

[근거: VAMOS_구현가이드_PART2 §6.10.2]

### 핵심 요약 (3줄)
1. DCL은 "전체 인터넷 크롤링" 대신 "필요한 도메인만 선택적으로" 배경을 제공하는 설계 원칙을 따릅니다.
2. 이 접근은 V1 예산 내에서 운영 가능하며, 노이즈와 환각 위험을 최소화합니다.
3. 기존 LOCK 결정사항(비용 상한 등)과 완전히 호환되는 실용적 설계입니다.

## §28.2 VAMOS AI 정보 환경 6계층

### 비유: 건물의 6층 구조

VAMOS AI가 정보를 얻는 경로를 건물 층으로 비유할 수 있습니다:

```
[0층] Domain Context Layer (DCL)  ← 신규: "오늘의 세상 상황" (선택적 배경)
[1층] 사용자 입력               ← 100% 신뢰, 최우선 ("사용자가 직접 말한 것")
[2층] 내부 메모리 L0-L3          ← 95% 신뢰 ("AI의 기억")
[3층] I-2 RAG 검색              ← 80% 신뢰 ("인덱싱된 지식 검색")
[4층] Cloud Library 보조 보강    ← 70% 신뢰 ("배치 수집된 기술 지식")
[5층] 외부 API/MCP 도구          ← 60% 신뢰 ("실시간 외부 도구 호출")
```

### 6계층 상세

| 계층 | 이름 | 신뢰도 | 특징 | DCL과의 관계 |
|------|------|--------|------|-------------|
| **Layer 0** | **DCL (도메인 컨텍스트)** | 가변 (소스에 따라) | 도메인별 선택적 실시간 배경 | **DCL 자체** |
| **Layer 1** | 사용자 입력 | **100%** | 사용자가 직접 제공한 정보 | DCL보다 항상 우선 |
| **Layer 2** | 내부 메모리 (L0-L3) | **95%** | AI의 장기/단기 기억 | DCL이 메모리를 보충 |
| **Layer 3** | I-2 RAG 검색 | **80%** | 인덱싱된 지식 벡터 검색 | DCL 데이터가 RAG에 삽입됨 |
| **Layer 4** | Cloud Library 배치 | **70%** | E-15/S-5 배치 수집 (논문/블로그) | DCL과 상호 보완 |
| **Layer 5** | 외부 API/MCP 도구 | **60%** | 실시간 외부 도구 호출 | DCL이 API 호출 필요성 감소 |

> **Layer 0 (DCL)과 Layer 4 (Cloud Library 배치)의 관계**:
> - **DCL**: 실시간 도메인 배경 (속보, 트렌드, 시장 상황) — "지금 무슨 일이 일어나고 있는가?"
> - **Cloud Library 배치**: 심층 기술 지식 (논문, 문서, 블로그) — "이 주제에 대해 깊이 알려면?"
> - DCL에서 부족한 정보는 Cloud Library 배치가 보충

[근거: VAMOS_구현가이드_PART2 §6.10.2]

### 핵심 요약 (3줄)
1. VAMOS AI의 정보 환경은 6계층으로 구성되며, DCL은 최상위 Layer 0으로 "세상 상황 배경"을 제공합니다.
2. 사용자 입력(100%)이 항상 최우선이며, DCL은 다른 계층을 보완하는 역할입니다.
3. DCL(실시간 배경)과 Cloud Library 배치(심층 지식)는 상호 보완 관계로, 시간축과 깊이 축을 나누어 담당합니다.

## §28.4 DCL → I-2 RAG 연동 흐름

### 비유: 비서의 브리핑이 서류함에 정리되는 과정

비서(DCL)가 "오늘의 세상 상황"을 정리하면:
1. 핵심 내용은 **서류함 (I-2 RAG)**에 영구 보관 → 나중에 검색 가능
2. 오늘의 핵심 요약은 **책상 위 메모 (I-3 L0 Context)**에 올려놓음 → 바로 참고 가능

### 연동 흐름도

```
[DCL 채널]
  ├─ DCL-FIN → RT-BNP → Kafka → VAMOS_EVENT
  ├─ DCL-TECH → RSS Collector → 키워드 추출
  └─ DCL-GEO → RSS Collector → Breaking Detector
        ↓
  [DCL Aggregator] (데이터 수집·통합)
        ↓
  ┌─────┴─────┐
  │           │
  [I-2 RAG]  [I-3 L0 Context]
  (벡터 삽입)  (세션 배경 주입)
        ↓
  [Main LLM: 배경 인식된 응답 생성]
```

### 두 가지 주입 경로

| 경로 | 대상 | 비유 | 효과 | 지속성 |
|------|------|------|------|--------|
| **I-2 RAG 삽입** | 벡터 DB | 서류함에 영구 보관 | 향후 관련 질문 시 검색 가능 | 영구 (TTL까지) |
| **I-3 L0 Context 주입** | 현재 세션 메모리 | 책상 위 메모 | 사용자가 묻지 않아도 배경 인지 | 세션 동안만 |

> **I-3 L0 Context 주입**: DCL은 RAG 외에도 현재 세션의 L0 메모리에 "현재 세상 상황 요약"을 주입하여, 사용자가 묻지 않아도 AI가 배경을 인지합니다.

[근거: VAMOS_구현가이드_PART2 §6.10.2]

### 핵심 요약 (3줄)
1. DCL 데이터는 I-2 RAG(벡터 DB 영구 삽입)과 I-3 L0 Context(세션 배경 주입) 두 경로로 VAMOS AI에 전달됩니다.
2. RAG 삽입은 향후 검색을 위한 영구 저장이고, L0 Context 주입은 현재 세션에서 즉시 활용되는 배경입니다.
3. 이를 통해 VAMOS AI는 사용자가 묻지 않아도 "지금 세상이 어떤 상황인지" 인식한 응답을 생성할 수 있습니다.
