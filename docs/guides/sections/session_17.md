# §19. RBAC — 역할 기반 접근 제어 (Role-Based Access Control)

> VAMOS에서는 "누가 무엇을 할 수 있는가"를 **역할(Role)**로 나누어 관리합니다.
> 이 섹션에서는 4가지 역할, 권한 매트릭스, 자율도 수준을 알아봅니다.
> [근거: D2.0-07 §3.6 LOCK, §3.2.1 LOCK / BASE-1.3 §9]

---

## §19.1 4가지 역할 — 회사 직급에 비유하기

### 비유: 회사 직급 체계

VAMOS의 역할 시스템을 **회사 직급**에 비유하면 쉽게 이해할 수 있습니다:

| 회사 직급 | VAMOS 역할 | 하는 일 |
|-----------|-----------|---------|
| 대표이사 (CEO) | **OWNER** (소유자) | 모든 권한, 최종 결정권, 회사의 모든 것을 바꿀 수 있음 |
| 부장/이사 (Director) | **ADMIN** (관리자) | 부서 관리, 정책 조정 가능하지만 회사 헌법은 못 바꿈 |
| 대리/과장 (Manager) | **OPERATOR** (운영자) | 일상 업무 수행, 허용된 범위 내에서만 일함 |
| 인턴/방문자 (Intern) | **VIEWER** (열람자) | 읽기만 가능, 어떤 것도 실행하거나 변경할 수 없음 |

### 각 역할 상세 설명

#### OWNER (소유자) — "대표이사"

시스템의 **최고 권한자**입니다. 사용자 본인이 곧 OWNER입니다.

- **할 수 있는 것**: 모든 정책 변경, 비용 상한 변경, P2 도메인(위험 도메인) 승인, Constitution(AI 헌법) 편집, 자율도 수준 변경
- **핵심**: OWNER만 할 수 있는 일이 있습니다 — P2 승인, Constitution 편집, 자율도 L3 설정
- [근거: D2.0-07 §3.6.1]

#### ADMIN (관리자) — "부장/이사"

OWNER가 **권한 일부를 위임받은 관리자**입니다. V2(멀티유저 환경)부터 활성화됩니다.

- **할 수 있는 것**: 정책 조회, 일반 설정 변경, 비용 상한 변경, Agent 실행 전체 허용
- **할 수 없는 것**: Constitution(AI 헌법) 편집, P2 도메인 승인 — 이것은 **OWNER만 가능**
- [근거: D2.0-07 §3.6.1, §3.6.2]

#### OPERATOR (운영자) — "대리/과장"

**일상적인 작업을 수행하는 일반 사용자**입니다.

- **할 수 있는 것**: 일반 Task(작업) 실행, 허용목록(Allowlist) 내 Tool 호출, 비P2 도메인 작업, 개인 설정만 변경
- **할 수 없는 것**: 비용 상한 변경, 시스템 설정 변경, P2 도메인 작업
- [근거: D2.0-07 §3.6.1]

#### VIEWER (열람자) — "인턴/방문자"

**읽기만 가능한 가장 제한된 역할**입니다.

- **할 수 있는 것**: 로그/이벤트 조회, 대시보드 열람 — **오직 읽기만**
- **할 수 없는 것**: 어떤 실행도 불가, 승인 행위 불가, 설정 변경 불가
- [근거: D2.0-07 §3.6.1]

#### AGENT (시스템 주체) — 특별 분류

> **참고**: AGENT는 RBAC의 "역할"이 아니라 **시스템 주체(system principal)**로 재분류됩니다.
> AI 에이전트는 PolicyCheck/CostCheck 결과 내 자동 허용 범위만 실행할 수 있으며, **승인 요청만 가능**(직접 승인은 불가)합니다.
> [근거: D2.0-07 §3.6.1]

### 핵심 요약 (3줄)
1. VAMOS는 OWNER/ADMIN/OPERATOR/VIEWER 4가지 역할로 접근을 제어합니다.
2. OWNER만 P2 도메인 승인, Constitution 편집, 자율도 L3 설정이 가능합니다.
3. AGENT(AI)는 역할이 아닌 시스템 주체로 분류되며, 스스로 승인할 수 없습니다.

## §19.2 역할별 권한 매트릭스

### 비유: 열쇠 묶음

각 역할은 서로 다른 **열쇠 묶음**을 갖고 있다고 생각하세요. OWNER는 마스터키(모든 문 열림), VIEWER는 로비 열쇠만(보는 것만 가능) 갖고 있는 셈입니다.

### 권한 매트릭스 표

| 권한 항목 | OWNER | ADMIN | OPERATOR | VIEWER |
|-----------|:-----:|:-----:|:--------:|:------:|
| Memory 읽기 | ✅ 전체 | ✅ 전체 | ✅ 전체 | ✅ 읽기전용 |
| Memory 쓰기 | ✅ 전체 | ✅ 전체 | ⚠️ 제한 | ❌ 불가 |
| Agent 실행 | ✅ 전체 | ✅ 전체 | ⚠️ 허용목록 | ❌ 불가 |
| Tool 호출 | ✅ 전체 | ✅ 전체 | ⚠️ 허용목록 | ⚠️ 읽기전용 |
| 설정 변경 | ✅ 가능 | ✅ 가능 | ⚠️ 개인만 | ❌ 불가 |
| Constitution(AI 헌법) 편집 | ✅ **가능** | ❌ 불가 | ❌ 불가 | ❌ 불가 |
| 비용 상한 변경 | ✅ 가능 | ✅ 가능 | ❌ 불가 | ❌ 불가 |
| P2 도메인 승인 | ✅ **가능** | ❌ 불가 | ❌ 불가 | ❌ 불가 |
| Self-evo 변경 적용 | ✅ 가능 | ✅ 가능 | ❌ 불가 | ❌ 불가 |
| RBAC 역할 변경 | ✅ **OWNER만** | ❌ 불가 | ❌ 불가 | ❌ 불가 |

> [근거: D2.0-07 S7E-023 CRITICAL, §3.6.1~§3.6.3]

### 역할별 승인 요구 수준

| 역할 | 계획 승인 | 일반 실행 승인 | 고위험 실행 승인 (P2/비용 상한) |
|------|:---------:|:-------------:|:-----------------------------:|
| OWNER | ✅ 직접 가능 | ✅ 직접 가능 | ✅ 직접 가능 |
| ADMIN | ✅ 가능 | ✅ 가능 | ❌ 불가 (OWNER만) |
| OPERATOR | ✅ 가능 | ❌ 불가 (OWNER/ADMIN만) | ❌ 불가 |
| VIEWER | ❌ 불가 | ❌ 불가 | ❌ 불가 |
| AGENT | 승인 **요청**만 가능 | 승인 **요청**만 가능 | 승인 **요청**만 가능 |

> [근거: D2.0-07 §3.6.2]

### 권한 초과 시 처리

역할 권한을 초과하는 요청을 하면 어떻게 될까요?

```
[OPERATOR가 비용 상한 변경 시도]
    ↓
즉시 거부 (deny)
    ↓
failure_code: RBAC_PERMISSION_DENIED 기록
    ↓
OWNER에게 자동 알림 (감사 로그에 기록)
```

- AGENT가 허용 범위 밖 실행을 시도하면, 자동으로 OWNER에게 **승인 요청을 상신**합니다.
- [근거: D2.0-07 §3.6.3]

### 핵심 요약 (3줄)
1. OWNER는 모든 권한을 가지며, P2 승인과 Constitution 편집은 **OWNER만** 가능합니다.
2. ADMIN은 대부분의 관리 권한이 있지만 고위험 승인(P2/비용 상한)은 OWNER에게 위임됩니다.
3. 권한 초과 요청 시 즉시 deny + `RBAC_PERMISSION_DENIED` 코드가 기록됩니다.

## §19.3 자율도 수준: L0 ~ L3

### 비유: 자동차 자율주행 단계

자동차의 자율주행 수준이 0단계(수동)에서 5단계(완전 자율)까지 있듯, VAMOS의 AI 에이전트도 **4단계 자율도 수준**이 있습니다.

| 자동차 비유 | VAMOS 자율도 | 설명 |
|------------|-------------|------|
| 수동 변속 (모든 것을 운전자가) | **L0** (FULL_MANUAL) | 모든 동작에 사용자 승인 필요 |
| 크루즈 컨트롤 (일부 자동) | **L1** (SUPERVISED) | 허용된 동작만 자동, 나머지는 승인 |
| 고속도로 자율주행 (많은 것이 자동) | **L2** (SEMI_AUTO) | 읽기/생성은 자동, 쓰기/변경은 승인 |
| 완전 자율주행 (거의 모든 것이 자동) | **L3** (FULL_AUTO) | 대부분 자동, 정책 위반 시만 차단 |

### 자율도 수준 상세 표 (LOCK — 변경 불가)

| 수준 | 명칭 | 설명 | 승인 요구 |
|------|------|------|-----------|
| **L0** | FULL_MANUAL (완전 수동) | 모든 행동에 사용자 확인 필요 | 매 동작마다 승인 |
| **L1** | SUPERVISED (감독) | 허용목록(Allowlist) 내 동작만 자동. 나머지는 승인 요청 | 허용목록 외 동작은 계획 승인 |
| **L2** | SEMI_AUTO (반자동) | 읽기/생성/분석은 자동. 쓰기/변경은 승인 | 쓰기·변경 시 계획 승인, 고위험은 실행 승인 |
| **L3** | FULL_AUTO (완전 자동) | 비용 상한·Non-goal·P2 금지 외 자동 실행 | 정책 위반·상한 초과 시만 차단 |

> **기본값: L1 (SUPERVISED)** — VAMOS는 처음부터 L1으로 시작합니다.
> [근거: D2.0-07 §3.2.1 LOCK]

### 자율도 변경 규칙

```
[자율도 수준 변경 규칙]

1. 수준 변경은 OWNER의 명시적 승인 필요 (다른 역할은 불가)
2. 기본값은 L1 (SUPERVISED) — 허용 목록 내 동작만 자동
3. 현재 자율 운영 수준은 세션 컨텍스트에 기록 + 감사 로그에 남김
4. L3에서도 절대 자동 실행 불가한 것:
   - P2 도메인 작업
   - Non-goal 항목 실행
   - 비용 상한 초과 행동
   ↑ 이 3가지는 RULE 1.3의 절대 금지 사항 (LOCK)
```

### L3 (최고 자율)에서도 절대 불가능한 행동

L3(FULL_AUTO)는 가장 높은 자율 수준이지만, **아래 행동은 어떤 수준에서도 절대 자동 실행되지 않습니다**:

| 절대 금지 항목 | 이유 |
|---------------|------|
| Non-goal 항목 실행 | 실거래, 해킹, 의료 판단 등 — BASE 규칙 절대 금지 |
| RBAC 역할 변경 | AI가 스스로 권한을 높이는 것 불가 |
| 비용 상한 변경 | AI가 스스로 예산을 늘리는 것 불가 |
| P2 도메인 자동 실행 | 위험 도메인은 항상 OWNER 승인 필수 |
| 안전 필터 비활성화 | Guardrails를 스스로 끄는 것 불가 |

> [근거: D2.0-07 §3.2.1 LOCK, BASE-1.3 §6.2]

### 버전별 자율도 활성화

| 버전 | 기본 자율도 | 최대 자율도 | 비고 |
|------|-----------|-----------|------|
| V0 (설계) | — | — | 설계 단계 |
| V1 | L0~L1 | L1 | 기본 감독 모드 |
| V2 | L0~L2 | L2 | 반자동 지원 |
| V3 | L0~L3 | L3 | 완전 자동 가능 (OWNER만 설정) |

### 핵심 요약 (3줄)
1. 자율도는 L0(완전 수동)~L3(완전 자동)까지 4단계이며, **기본값은 L1(SUPERVISED)**입니다.
2. 자율도 변경은 **OWNER만** 가능하고, L3에서도 Non-goal/P2/비용 상한은 자동 실행 불가입니다.
3. 자율도 수준은 세션 컨텍스트에 기록되며, 감사 로그에 남겨 투명성을 보장합니다.

# §20. 비용 관리 — Cost Control

> VAMOS는 AI 사용 비용이 통제 없이 불어나는 것을 방지하기 위해 **엄격한 비용 관리 시스템**을 갖추고 있습니다.
> 비용 상한은 **절대 변경 불가(ABSOLUTE LOCK)**이며, 코드에서도 하드코딩됩니다.
> [근거: D2.0-07 §4 전체 / BASE-1.3 §5, §6.2]

---

## §20.1 버전별 비용 상한 (ABSOLUTE LOCK — 절대 변경 불가)

### 비유: 교통카드 충전 한도

교통카드에 월 충전 한도가 정해져 있어 그 이상은 충전할 수 없듯, VAMOS도 **버전별로 매월 사용할 수 있는 최대 비용이 정해져 있습니다**. 이 금액은 코드에 직접 새겨져(하드코딩) 있어 AI가 스스로 바꿀 수 없습니다.

### 비용 상한 표 (ABSOLUTE LOCK)

| 버전 | 일일 상한 | 월 상한 | 모델 사용 전략 | 대상 사용자 |
|------|-----------|---------|---------------|------------|
| **V1** (기본) | ₩1,300 ($1) | **₩40,000 ($30)** | Mini 모델 90%+ 사용 | 일반 사용자, 가벼운 작업 |
| **V2** (Pro) | ₩3,100 ($2.3) | **₩93,000 ($70)** | Mini 60~70% / Main 30~40% | 전문가, 코딩/리서치 |
| **V3** (Enterprise) | ₩8,900 ($6.7) | **₩266,000 ($200)** | Main 중심, 플래그십 적극 활용 | 파워 유저, 복잡한 작업 |

> **ABSOLUTE LOCK**: 위 금액은 코드에서 하드코딩되며, **Self-evo(AI 자기 진화)로도 절대 변경 불가**합니다.
> 상한 초과 모델 호출은 **승인 없이 자동 차단**됩니다.
> [근거: D2.0-07 §4.1 LOCK / BASE-1.3 §5, §6.2]

### "변경 불가"의 의미

```
[비용 상한이 변경 불가인 이유]

1. BASE RULE에서 "비용 상한(V1/V2/V3) 금액"은 Self-evo 절대 불가 영역
2. 코드에 하드코딩 → AI가 코드를 수정해도 변경 불가
3. OWNER가 "예산 상향"을 요청할 수는 있음 (§20.4 참조)
   → 단, 이것은 "상한 자체"를 바꾸는 것이 아니라
     "승인 절차를 거쳐 임시로 확장"하는 것
4. Self-evo로 비용 최적화는 가능 (더 적게 쓰는 것)
   → 단, 비용 상한 금액 자체를 올리는 것은 불가
```

> [근거: BASE-1.3 §6.2 — Self-evo 절대 불가 영역: "비용 상한(V1/V2/V3) 금액"]

### 핵심 요약 (3줄)
1. V1: 월 ₩40,000 / V2: 월 ₩93,000 / V3: 월 ₩266,000 — **절대 변경 불가(LOCK)**
2. 이 금액은 코드에 하드코딩되어 있어 AI(Self-evo)도 변경할 수 없습니다.
3. 상한 초과 시 **승인 없이 자동 차단**되며, 우회 방법은 존재하지 않습니다.

## §20.2 Downshift 메커니즘 (자동 모델 다운그레이드)

### 비유: 자동차 연료 절약 모드

자동차가 연료가 얼마 안 남으면 자동으로 **에코 모드(연비 우선)**로 전환하듯, VAMOS도 예산이 일정 비율에 도달하면 자동으로 **저비용 모델로 전환**합니다.

### Downshift 3단계 (LOCK)

| 단계 | 예산 사용률 | 동작 | 비유 |
|------|-----------|------|------|
| 1단계: 정보 알림 | **50%** 도달 | 대시보드에 정보성 알림 표시 (행동 변화 없음) | 연료 반 남음 — 안내만 |
| 2단계: 경고 + 강제 다운시프트 | **80%** 도달 | ⚠️ 경고 + **force_mini** (Mini 모델 우선 강제) | 에코 모드 자동 전환 |
| 3단계: 완전 차단 | **100%** 도달 | ❌ **차단(deny)** — 모든 유료 모델 호출 중단 | 연료 바닥 — 주행 정지 |

> **Gate 임계값(LOCK — SOT)**: 80% 경고(force_mini) → 100% 차단(deny)
> 95%에서도 긴급 안내 알림(빨간색)이 표시되며 Mini-only 전환을 안내합니다.
> [근거: D2.0-07 §4.2 LOCK, §15.6.2]

### Downshift 동작 흐름

```
[매 API 호출 시]
    ↓
비용 카운터 확인: 현재 사용량 / 상한
    ↓
┌──────────────────────────────────────────┐
│ 50% 미만   → 정상 처리 (제한 없음)       │
│ 50% 도달   → 정보 알림 (대시보드 표시만)  │
│ 80% 도달   → ⚠️ 경고 + force_mini       │
│              (Mini 우선 강제 라우팅)       │
│              Main/Flagship은 승인 시만    │
│ 95% 도달   → 🔴 긴급 안내 + Mini-only    │
│ 100% 도달  → ❌ 차단(deny)               │
│              예산 상향 승인 요청 제시      │
└──────────────────────────────────────────┘
```

### 80% 근접 구간의 규칙

- Mini 우선 강제: 가능한 요청은 **Mini 모델로 자동 라우팅**
- Main/Flagship 모델은 **사용자 승인(Approval)이 있을 때만** 허용
- 근접 구간이라도 **Non-goal/P2/정책 deny는 그대로 deny** (보안은 타협 없음)
- 적용 범위: daily_limit(일일 상한) / monthly_limit(월 상한) 중 **먼저 도달하는 쪽** 기준
- [근거: D2.0-07 §4.2]

### 핵심 요약 (3줄)
1. 예산 50%에서 알림, **80%에서 강제 다운시프트(Mini 모델 전환)**, 100%에서 완전 차단합니다.
2. 80%~100% 구간에서는 Main/Flagship 모델 사용에 **명시적 승인이 필요**합니다.
3. 다운시프트는 비용 차단을 피하기 위한 "비용 안정화 장치"이며, 보안 정책은 그대로 유지됩니다.

## §20.3 비용 최적화 전략

### 비유: 가정의 전기 절약법

전기 절약을 위해 LED 전구(에너지 효율), 타이머 콘센트(불필요한 사용 제거), 대기 전력 차단(중복 사용 방지) 등 여러 방법을 병행하듯, VAMOS도 **3가지 전략**으로 비용을 최적화합니다.

### 전략 1: 캐싱 (Caching)

같은 질문을 반복하면 처음 답을 **저장해두고 재사용**하여 모델 호출 비용을 0으로 만드는 전략입니다.

| 캐싱 유형 | 설명 | 절감 효과 |
|-----------|------|----------|
| **응답 캐시** (Response Cache) | 동일 입력(해시 기준)에 대한 이전 응답을 TTL(유효시간) 내 재사용 | 동일 요청 비용 → **0** |
| **컨텍스트 캐시** (Context Cache, 프롬프트 캐싱) | 반복 사용되는 시스템 프롬프트·문서를 미리 고정(프리픽스 캐시) | 반복 시스템 프롬프트 비용 대폭 절감 |
| **시맨틱 캐시** (Semantic Cache) | 정확히 같지 않지만 **의미가 유사한 질문**에도 기존 답변 재사용 | 유사 질문 비용 절감 |

- **캐시 무효화**: 정책 변경 또는 TTL(유효시간) 만료 시 자동으로 캐시를 삭제합니다.
- [근거: D2.0-07 §4.2.1 (1)]

### 전략 2: 경량 모델 우선 전환 (Smart Routing)

**요청의 복잡도를 분석**하여, 단순한 요청은 저렴한 Mini 모델로, 복잡한 요청만 비싼 Main 모델로 보내는 전략입니다.

```
[요청 접수]
    ↓
복잡도 분류기 (Complexity Classifier) 분석
    ↓
┌─────────────────────────────────────────────┐
│ 단순 요청 (간단한 질문, 포맷 변환 등)        │
│  → Mini 모델 자동 라우팅 (저비용)            │
├─────────────────────────────────────────────┤
│ 복잡 요청 (코드 생성, 심층 분석 등)          │
│  → Main/Flagship 모델 (고비용, 승인 시만)    │
└─────────────────────────────────────────────┘
```

- 복잡도 기준: 토큰 수, 추론 깊이, 도메인 위험도(P0/P1/P2) 조합
- [근거: D2.0-07 §4.2.1 (2)]

### 전략 3: 불필요 호출 제거 (배치 처리 포함)

**중복되거나 불필요한 API 호출을 제거**하여 비용을 줄이는 전략입니다.

| 방법 | 설명 | 효과 |
|------|------|------|
| **중복 호출 감지** | 같은 세션 내 중복 요청을 자동 병합 또는 캐시 응답 반환 | 중복 비용 제거 |
| **배치 처리** (Batch Processing) | 즉시 응답이 필요하지 않은 요청을 모아서 한 번에 처리 | 비용 절감 (배치 할인) |
| **스트리밍 최적화** | 불필요한 중간 결과 스트리밍 비활성화 | 토큰 낭비 방지 |

- [근거: D2.0-07 §4.2.1 (3)]

### 핵심 요약 (3줄)
1. **캐싱**: 동일/유사 질문은 저장된 답변을 재사용하여 모델 호출 비용을 줄입니다.
2. **경량 모델 우선**: 복잡도 분석으로 단순 요청은 Mini, 복잡 요청만 Main 모델로 라우팅합니다.
3. **불필요 호출 제거**: 중복 감지, 배치 처리, 스트리밍 최적화로 낭비를 방지합니다.

## §20.4 예산 초과 시 행동 흐름

### 비유: 신용카드 한도 초과

신용카드 한도가 초과되면 결제가 거부되고, 한도 상향을 위해 카드사에 승인을 받아야 하듯, VAMOS에서도 비용 상한에 도달하면 **차단 → 알림 → 선택지 제시 → 승인** 절차를 거칩니다.

### 예산 초과 시 흐름도

```
[API 호출 요청]
    │
    ▼
[비용 카운터 확인]
    │
    ├── 상한 미만 ──────────────→ ✅ 정상 처리
    │
    ├── 80% 도달 ──────────────→ ⚠️ 경고 + force_mini
    │                               (Mini 모델로 강제 전환)
    │
    └── 100% 도달(상한 초과) ──→ ❌ 즉시 차단(deny)
                                     │
                                     ▼
                              [사용자에게 알림]
                                     │
                                     ▼
                              [선택지 제시]
                               ┌─────┴─────┐
                               │           │
                          (A) 예산 상향   (B) 현재 유지
                               │           │
                               ▼           ▼
                          [Approval:     [차단 유지 or
                           비용 상한     Mini-only로
                           변경 승인]    축소 운영]
                               │
                               ▼
                          승인 완료 → 재개
                          거부 → 차단 유지
```

> [근거: D2.0-07 §4.3]

### 예산 상향 절차

1. **초과 감지** → 즉시 차단(deny) + 사용자 알림
2. **선택지 제시**:
   - **(A) 예산 상향 요청**: Approval(비용 상한 변경) → 승인되면 monthly_limit 상향(또는 임시 확장) 후 재개
   - **(B) 현재 유지**: 차단 상태 유지, 또는 Mini-only로 범위 축소(가능한 작업만 수행)
3. **승인 필수**: 예산 상향은 "비용 상한 변경"에 해당하므로 **반드시 승인 절차** 필요 (RULE 1.3 §9.2)
4. **V3**: 추가 예산 부과 가능하되, 승인 없이 자동 상향되지 않음

### 승인 타임아웃

- 승인 요청 후 **10분** 내 응답 없으면 자동 거부(deny) 처리
- 고위험(P2 도메인/안전 필터 경계 판정)은 **5분** 타임아웃 적용
- [근거: D2.0-07 §4.3.2]

### 핵심 요약 (3줄)
1. 예산 100% 도달 시 **즉시 차단** → 사용자에게 예산 상향 또는 Mini-only 전환을 제안합니다.
2. 예산 상향은 반드시 **승인 절차(Approval)**를 거쳐야 하며, 자동 상향은 불가합니다.
3. 승인 요청 10분 내 미응답 시 자동 거부 처리됩니다.

## §20.5 규제 준수 & 컴플라이언스 — GAP-16

### 비유: 회사의 법무/감사팀

모든 회사에는 법을 지키는지 감시하는 **법무팀**과, 기록이 제대로 되어 있는지 확인하는 **감사팀**이 있습니다. VAMOS도 마찬가지로 국제 규제를 준수하고, 모든 활동을 감사할 수 있도록 설계되어 있습니다.

### GDPR (일반 데이터 보호 규정) 준수

**GDPR**(General Data Protection Regulation)은 유럽연합(EU)의 개인정보 보호 법률로, 세계에서 가장 엄격한 개인정보 보호 규정입니다. VAMOS는 한국 개인정보보호법과 함께 GDPR도 준수합니다.

| GDPR 권리 | 설명 (쉬운 말) | VAMOS 대응 | 버전 |
|-----------|--------------|-----------|------|
| 열람권 (Right of Access) | "내 데이터 보여줘" | 사용자 데이터 조회/다운로드 기능 | V2+ |
| 삭제권 (Right to Erasure) | "내 데이터 지워줘" (잊힐 권리) | `vamos purge --all` 원클릭 삭제 | **V1: MUST** |
| 이동권 (Right to Portability) | "내 데이터 가져갈게" | JSON 표준 포맷 내보내기 | V2+ |
| 정정권 | "잘못된 정보 고쳐줘" | 메모리 내 잘못된 정보 수정 | V2+ |
| 처리 제한권 | "이 데이터 사용 중지해줘" | 특정 데이터 처리 중지 기능 | V2+ |
| 동의 관리 | "동의 철회할게" | 명시적 동의 수집 + 철회 기능 | V2+ |

> [근거: D2.0-07 S7E-037, §15.4.2]

### SOC2 Type II 대응

**SOC2**(Service Organization Control 2)는 서비스 기업이 고객 데이터를 안전하게 관리하는지 평가하는 국제 표준입니다. Type II는 실제 운영 기간 동안의 이행을 평가합니다.

| SOC2 원칙 | 의미 (쉬운 말) | VAMOS 대응 | 상태 |
|-----------|--------------|-----------|------|
| 보안 (Security) | "데이터를 안전하게 지키는가?" | 4-Layer Guardrails + RBAC | **V1: MUST** |
| 가용성 (Availability) | "서비스가 안정적으로 돌아가는가?" | 장애 복구 + Fallback | V2: NICE |
| 처리 무결성 (Processing Integrity) | "데이터를 정확하게 처리하는가?" | PolicyCheck + 감사 로그 | **V1: MUST** |
| 기밀성 (Confidentiality) | "비밀 데이터를 외부에 안 내보내는가?" | PII 마스킹 + 접근 제어(RBAC) | **V1: MUST** |
| 개인정보 (Privacy) | "개인정보를 적법하게 처리하는가?" | 데이터 최소화 + 동의 관리 | V2: NICE |

> [근거: D2.0-07 §15.4.1]

### 데이터 주권 (Data Sovereignty) 정책

**데이터 주권**이란, "사용자의 데이터는 100% 사용자 소유"라는 원칙입니다. VAMOS는 사용자의 데이터를 **함부로 가져가거나 활용하지 않습니다**.

| 원칙 | 내용 | 버전 |
|------|------|------|
| **100% 사용자 소유** | 데이터는 사용자 것. VAMOS는 접근 도구일 뿐 | **V1: MUST** |
| **로컬 저장 우선** | V1에서 모든 데이터는 로컬 저장 (클라우드 전송 없음, LLM API 제외) | **V1** |
| **데이터 이동 가능** | 전체 데이터를 JSON/SQLite로 내보내기(export) 가능 | **V1** |
| **완전 삭제 보장** | "모든 데이터 삭제" 원클릭 기능 (`vamos purge --all`) | **V1** |
| **LLM 전송 투명성** | LLM API에 데이터 전송 시 사용자 동의 + 전송 내용 투명 표시 | **V1** |

> [근거: D2.0-07 S7E-033 CRITICAL]

### 감사 로그 보존 정책

| 데이터 유형 | 보존 기간 | 만료 후 처리 |
|------------|----------|------------|
| 세션 대화 (L0) | 7일 | 자동 삭제 |
| 단기 메모리 (L1) | 30일 | L2 승격 또는 삭제 |
| 프로젝트 (L2) | 90일 활성 | 아카이브 이동 |
| 영구 메모리 (L3) | 무기한 | 수동 삭제만 |
| **감사 로그** | **1년 (규제 요건)** | 자동 삭제 |
| 임시 캐시 | 24시간 | 자동 삭제 |

> [근거: D2.0-07 S7E-038, §15.4.2]

### 감사 로그 무결성 보장

감사 로그는 **변조 방지** 기능이 내장되어 있습니다:

- **Append-only**: 감사 로그는 추가만 가능, **수정/삭제 불가**
- **해시 체인(Hash Chain)**: 각 로그에 이전 로그의 해시를 포함 → 하나라도 변조되면 즉시 탐지
- **타임스탬프 무결성**: NTP 동기화된 서버 시간 사용 → 시간 조작 방지
- [근거: D2.0-07 §15.4.3]

### 규제 준수 자동 검증 체계

| 검증 항목 | 주기 | 결과 형태 |
|-----------|------|----------|
| RBAC 정합성 검사 | 주 1회 자동 | 준수/미준수/불확실 3단계 |
| 로그 무결성 스캔 | 주 1회 자동 | 해시 체인 검증 결과 |
| PII 잔존 스캔 | 주 1회 자동 | 미마스킹 데이터 탐지 |
| 보존 기간 만료 데이터 삭제 확인 | 주 1회 자동 | 미삭제 데이터 알림 |
| 감사 보고서 자동 생성 | 월간 자동 | PDF + JSON 형식 |

- **대응 규제**: GDPR, 한국 개인정보보호법, EU AI Act, 자본시장법
- **정책 변경 시** 즉시 검증 실행
- [근거: D2.0-07 §15.4.4, S7E-058]

### 국가별 규제 적응 (V3)

| 국가/지역 | 주요 규제 | VAMOS 대응 |
|-----------|----------|-----------|
| 한국 | AI 기본법(안), 개인정보보호법, 자본시장법 | 규제 프로파일 자동 적용 |
| EU | AI Act, GDPR, MiFID II | 고위험 AI 시스템 분류 + 투명성 의무 |
| 미국 | NIST AI RMF, SEC 규정, 주별 AI 규제 | NIST 프레임워크 매핑 |

> V3에서 국가별 규제 설정이 자동 적용되며, Regulatory Change Alert가 규제 변경을 감지합니다.
> [근거: D2.0-07 S7E-060]

### 핵심 요약 (3줄)
1. VAMOS는 **GDPR**(삭제권, 열람권 등)과 **SOC2 Type II** 원칙을 준수하도록 설계되었습니다.
2. **데이터 주권**: 모든 데이터는 100% 사용자 소유이며, V1에서 로컬 저장 우선 + 원클릭 삭제를 지원합니다.
3. **감사 로그**는 1년 보존, append-only(추가만 가능), 해시 체인으로 **변조 방지**됩니다.
