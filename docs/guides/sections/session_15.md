# §17. RAG 시스템 — 지식 검색 & 활용

> VAMOS의 RAG 시스템은 AI가 **자기가 모르는 것도 도서관에서 책을 찾아 답하듯** 외부 지식을 검색하여 답변하는 핵심 기능입니다. 이 섹션에서는 RAG가 무엇인지, 6단계 파이프라인이 어떻게 작동하는지, 그리고 검색 품질을 높이기 위한 다양한 기술을 설명합니다.

---

## §17.1 RAG란?

### 비유: 도서관에서 책 찾기

시험 문제를 받았다고 상상해 보세요.

1. **질문 확인**: "한국의 수도는?" 이라는 질문을 읽습니다.
2. **도서관 검색**: 관련 책(백과사전, 지리책)을 찾아서 꺼냅니다.
3. **답변 작성**: 찾은 책의 내용을 바탕으로 "서울입니다"라고 답합니다.

RAG(Retrieval-Augmented Generation, 검색 증강 생성)는 정확히 이 과정을 AI가 수행하는 것입니다. [근거: D2.0-06 §1.1]

### 정의

RAG는 **"검색(Retrieval) + 생성(Generation)"을 결합**한 AI 기술입니다.

- **Retrieval (검색)**: 질문과 관련된 문서/정보를 데이터베이스에서 찾아오기
- **Augmented (증강)**: 찾아온 정보를 AI에게 추가 자료로 제공하기
- **Generation (생성)**: AI가 제공받은 자료를 바탕으로 답변 만들기

### 왜 LLM만으로는 부족한가?

LLM(대규모 언어 모델, Large Language Model)은 학습 당시의 지식만 갖고 있습니다. 다음과 같은 한계가 있습니다:

| 한계 | 설명 | RAG의 해결 방법 |
|------|------|----------------|
| 지식 한계 | 학습 이후의 새 정보를 모름 | 최신 문서를 검색하여 제공 |
| 환각 (Hallucination) | 모르는 것도 그럴듯하게 지어냄 | 검색된 근거 기반으로만 답변 |
| 전문 지식 부족 | 조직 내부 문서를 모름 | 내부 문서를 인덱싱하여 검색 |
| 맥락 부족 | 프로젝트별 맥락을 모름 | 프로젝트별 지식을 분리 관리 |

> VAMOS에서는 입력 크기에 따라 전략이 달라집니다: 10K 토큰 미만은 직접 전송, 10K~200K 토큰은 **RAG 검색**, 200K+ 토큰은 분할 처리합니다. [근거: D2.0-06 §4]

### 핵심 요약 (3줄)
1. RAG는 "검색 + 생성"을 결합하여 AI가 외부 지식을 활용해 답변하는 기술이다.
2. LLM만으로는 최신 정보, 내부 문서, 전문 지식에 접근할 수 없어 RAG가 필요하다.
3. VAMOS는 프로젝트별(project_id) 독립적인 RAG 검색을 지원한다.

## §17.2 6-Stage RAG Pipeline (6단계 RAG 파이프라인)

### 비유: 도서관 운영 시스템

RAG 파이프라인은 **도서관을 처음부터 만들어서 운영하는 전체 과정**과 같습니다:

1. 책을 구매한다 (수집)
2. 챕터별로 나눈다 (분할)
3. 색인 카드를 만든다 (벡터화)
4. 서가에 정리한다 (색인)
5. 질문에 맞는 책을 찾는다 (검색)
6. 찾은 내용으로 답변한다 (합성)

### 전체 구조도

```
[문서 입력] → 수집(Collect) → 분할(Chunk) → 벡터화(Embed)
                                                    ↓
[사용자 질문] → 검색(Retrieve) ← 색인(Store) ← [벡터 DB]
                    ↓
               합성(Generate) → [최종 답변]
```

### 6단계 상세 (LOCK — 변경 불가)

| 단계 | 명칭 | 설명 |
|------|------|------|
| 1 | 수집 (Collect) | 문서/청크 소스 수집 (파일, URL, DB 등) |
| 2 | 쪼개기 (Chunk) | 문서를 검색 단위(청크)로 분할 (크기/오버랩 설정 포함) |
| 3 | 벡터화 (Embed) | 청크를 임베딩 모델로 벡터 변환 |
| 4 | 저장 (Store) | VectorStore 어댑터를 통해 벡터+메타데이터 저장 |
| 5 | 검색 (Retrieve) | 쿼리 벡터와 BM25 앙상블로 상위 K개 청크 검색 + Rerank |
| 6 | 생성 (Generate) | 검색 결과를 컨텍스트로 LLM에 주입 후 응답 생성 |

> **운영 한계 (LOCK)**: 문서 **15개**/프로젝트, 청크 **30개**/문서. 상향은 비용 게이트(07) 승인 필요. [근거: D2.0-06 §1.1, §4.1]

> **재시도 정책 (LOCK)**: 최대 **3회** — 1차 실패 → BM25 단독 폴백 → 2차 실패 → 캐시 검색 폴백 → 3차 실패 → LLM 단독 응답 + 경고. [근거: D2.0-06 §4.1 S03-MOD-002]

각 단계가 실패하면 레지스트리(§8)에 failure_code를 기록하고 파이프라인을 중단합니다. [근거: D2.0-06 §1.1]

### §17.2.1 수집 (Collect) — 문서 입력, 소스 등록

### 비유
도서관에 **새 책이 도착**하는 단계입니다. 다양한 형태의 책(PDF, 워드 문서, 웹페이지 등)이 들어옵니다.

### 상세

- **지원 형식**: PDF, DOCX, TXT, MD, HTML, CSV, XLSX, JSON [근거: D2.0-06 §4 S7D-055]
- **텍스트 추출 도구**: `pymupdf`(PDF), `python-docx`(워드), `beautifulsoup4`(HTML)
- **자동 감시**: `~/.vamos/docs/` 폴더에 파일을 넣으면 `watchdog`(파일 감시 도구)이 자동으로 인덱싱을 시작합니다.

| 항목 | 값 |
|------|-----|
| 입력 | 문서 파일 (다양한 형식) |
| 출력 | 추출된 텍스트 + 메타데이터 (파일명, 언어, 수정일 등) |
| 운영 한계 | 프로젝트당 문서 **15개** (LOCK) |

### 핵심 요약 (3줄)
1. 수집 단계는 다양한 형식의 문서를 텍스트로 변환하는 첫 번째 단계이다.
2. PDF, 워드, HTML 등 8가지 형식을 지원한다.
3. 지정 폴더에 파일을 넣으면 자동으로 수집이 시작된다.

### §17.2.2 분할 (Chunking) — 문서를 적절한 크기로 분할

### 비유
긴 책을 **요약 카드 여러 장**으로 나누는 것입니다. 너무 크면 검색이 부정확하고, 너무 작으면 맥락이 사라집니다.

### 상세

문서를 **청크(Chunk, 검색 단위 조각)**로 분할합니다.

| 설정 항목 | 한국어 기준 권장값 | 설명 |
|-----------|-------------------|------|
| chunk_size (청크 크기) | 300~500 토큰 | 한국어는 영어 대비 토큰 효율 ~70% |
| overlap (겹침 구간) | 50~100 토큰 | 청크 사이 문맥 연속성 보장 |
| 최대 청크 수 | 30개/문서 (LOCK) | 운영 한계 |

[근거: D2.0-06 §4.3.4]

#### 고급 분할 기법

| 기법 | 설명 |
|------|------|
| Semantic Chunking (의미 분할) | 임베딩 유사도가 급격히 떨어지는 지점에서 분할 |
| Contextual Chunking (맥락 분할) | 각 청크에 원본 문서의 제목/섹션 맥락을 프리픽스로 추가 |

> **Contextual Retrieval**: Anthropic 발표에 따르면 청크에 맥락 프리픽스를 추가하면 검색 정확도가 **67% 향상**됩니다. VAMOS는 이를 기본 적용합니다. [근거: D2.0-06 §4.3.4 Contextual Retrieval 벤치마크]

#### 한국어 특수 처리

- **형태소 분석기**: V1은 Mecab-ko, V2는 Kiwi (정확도 우선) [근거: D2.0-06 §4.3.1]
- 한-영 혼용 청크는 언어 경계에서 분할하지 않고 **문장 단위 우선**으로 처리
- BM25 검색 시 불용어(은/는/이/가 등) 제거, 벡터 임베딩 시에는 원문 유지 (의미 손실 방지) [근거: D2.0-06 §4.3.2]

### 핵심 요약 (3줄)
1. 청크(Chunk)는 문서를 검색 가능한 작은 조각으로 나눈 것이다.
2. 한국어 기준 300~500 토큰, 50~100 토큰 겹침이 권장된다.
3. 맥락 프리픽스를 추가하면 검색 정확도가 67% 향상된다.

### §17.2.3 벡터화 (Embedding) — 텍스트를 숫자 벡터로 변환

### 비유
도서관의 **색인 카드를 만드는 과정**입니다. 사서가 책의 내용을 읽고 "이 책은 '역사' 분야이고 '한국'에 관한 것"이라는 태그를 붙이듯, 임베딩은 텍스트의 **의미를 숫자 목록(벡터)으로 변환**합니다.

### 정의

임베딩(Embedding)은 텍스트를 **고차원 숫자 벡터**(예: [0.12, -0.34, 0.56, ...])로 변환하는 과정입니다. 의미가 비슷한 텍스트는 벡터 공간에서 가까운 위치에 놓입니다.

```
"한국의 수도" → [0.12, -0.34, 0.56, ..., 0.78]  (1024개의 숫자)
"서울 위치"   → [0.11, -0.33, 0.55, ..., 0.77]  ← 의미가 비슷하니 숫자도 비슷!
"오늘 날씨"   → [0.89, 0.23, -0.67, ..., -0.12] ← 의미가 다르니 숫자도 다름
```

### 상세

| 항목 | 값 |
|------|-----|
| 입력 | 텍스트 청크 (분할 단계의 출력) |
| 출력 | 숫자 벡터 (1024차원 또는 256차원) |
| 기본 모델 | **BGE-M3** (LOCK — DEC-005 UPDATED) |
| 배치 처리 | 128개 청크 동시 처리, GPU 가속(CUDA) 자동 활용 |
| 캐싱 | `sha256(text)` 키로 중복 청크 재임베딩 방지 |

[근거: D2.0-06 §1.1, S7D-027, S7D-028]

> 임베딩 모델 상세 비교는 **§17.6**에서 다룹니다.

### 핵심 요약 (3줄)
1. 임베딩은 텍스트의 의미를 숫자 벡터로 변환하는 과정이다.
2. 의미가 비슷한 텍스트는 벡터 공간에서 가까운 위치에 놓인다.
3. VAMOS V1 기본 모델은 BGE-M3 (무료, 로컬, 1024차원)이다.

### §17.2.4 색인 (Indexing) — 벡터 DB에 저장

### 비유
색인 카드를 만들었으니 이제 **서가에 정리하는 단계**입니다. 나중에 빠르게 찾을 수 있도록 체계적으로 분류합니다.

### 상세

벡터와 메타데이터를 **벡터 데이터베이스(Vector DB)**에 저장합니다.

| 항목 | 값 |
|------|-----|
| 입력 | 벡터 + 메타데이터 (file_path, language, last_modified, chunk_index) |
| 출력 | 인덱싱된 레코드 (검색 가능 상태) |
| 인덱스 방식 | HNSW (Hierarchical Navigable Small World — 고속 근사 최근접 이웃 검색) |
| 거리 측정 | cosine (코사인 유사도) |
| 컬렉션 분리 | `vamos_memory`, `vamos_docs`, `vamos_cache` |

[근거: D2.0-06 §4 S7D-009, S7D-011]

- **증분 인덱싱**: 변경된 청크만 재임베딩하여 효율적으로 업데이트 [근거: D2.0-06 S7D-058]
- **도메인별 분리**: V1에서는 도메인별 컬렉션 3~5개로 분리, V2에서는 메타데이터 필터 방식으로 통합 가능 [근거: D2.0-06 S7D-011]

> Vector DB 비교는 **§17.7**에서 다룹니다.

### 핵심 요약 (3줄)
1. 색인은 벡터와 메타데이터를 벡터 DB에 저장하여 검색 가능하게 만드는 단계이다.
2. HNSW 인덱스와 코사인 유사도를 사용하여 빠른 검색을 지원한다.
3. 변경된 청크만 업데이트하는 증분 인덱싱으로 효율을 높인다.

---

### §17.2.5 검색 (Retrieval) — 유사 벡터 검색 + BM25 키워드 검색

### 비유
도서관에서 **두 가지 방법으로 동시에 책을 찾는 것**입니다. 사서가 "의미가 비슷한 책"(벡터 검색)과 "키워드가 일치하는 책"(BM25 검색)을 각각 찾아서 합칩니다.

### 상세

| 항목 | 값 |
|------|-----|
| 입력 | 사용자 질문 (텍스트) |
| 출력 | 관련 청크 상위 K개 (기본 top_k = **10**, API 레이어 B1은 **5**, config.toml에서 조정 가능) |
| 검색 방법 | BM25 + Vector 앙상블 (Hybrid Search) |
| 검색 순서 | 현재 프로젝트 → 글로벌 → 아카이브 |
| 최대 결과 | 레이어당 5개, 총 25개 후보 → reranker → top 3~5 |

[근거: D2.0-06 §1.1, §4, S7D-012]

> **Hybrid Search 상세**는 **§17.3**에서 다룹니다.

### 핵심 요약 (3줄)
1. 검색은 사용자 질문을 벡터로 변환한 뒤 유사한 청크를 찾는 단계이다.
2. 벡터 검색과 BM25 키워드 검색을 결합한 하이브리드 방식을 사용한다.
3. 검색 순서는 현재 프로젝트 → 글로벌 → 아카이브 순이다.

---

### §17.2.6 합성 (Synthesis) — 검색 결과 + LLM으로 최종 답변 생성

### 비유
도서관에서 찾은 책들을 펼쳐놓고 **보고서를 작성하는 단계**입니다. 찾아온 참고 자료를 바탕으로 질문에 대한 최종 답변을 만듭니다.

### 상세

| 항목 | 값 |
|------|-----|
| 입력 | 사용자 질문 + 검색된 청크(컨텍스트) |
| 출력 | LLM이 생성한 최종 답변 |
| 주입 방식 | 검색 결과를 프롬프트의 컨텍스트 영역에 삽입 |

- 검색 결과가 답변을 뒷받침하는지 **품질 자동 평가**를 수행합니다 (RAGAS 프레임워크).
- 평가 지표: Faithfulness(충실도), Answer Relevancy(관련성), Context Precision/Recall [근거: D2.0-06 §4.y.1]

### 핵심 요약 (3줄)
1. 합성은 검색된 청크를 LLM에 제공하여 최종 답변을 생성하는 마지막 단계이다.
2. 검색 결과를 프롬프트의 컨텍스트 영역에 삽입하여 LLM이 근거 기반으로 답변한다.
3. RAGAS 프레임워크로 답변 품질을 자동 평가한다.

---

## §17.3 Hybrid Search (하이브리드 검색) — BM25 + Vector + Reranker

### 비유: 세 명의 사서

도서관에 **세 명의 사서**가 있다고 상상해 보세요:

1. **키워드 사서 (BM25)**: "이 단어가 들어간 책" — 정확한 키워드 일치로 검색
2. **의미 사서 (Vector)**: "이런 뜻의 책" — 의미적 유사도로 검색
3. **심판 사서 (Reranker)**: 두 사서가 찾아온 책을 다시 읽고 **최종 순위**를 매김

### 3가지 검색 방법 비교 표

| 검색 방법 | 원리 | 장점 | 단점 | 예시 |
|-----------|------|------|------|------|
| BM25 (키워드 검색) | 단어 빈도·희귀도 기반 점수 산출 | 정확한 키워드 매칭에 강함 | 동의어/유사 표현을 놓침 | "파이썬 설치" 검색 → "python install" 못 찾음 |
| Vector (벡터 검색) | 임베딩 벡터 간 코사인 유사도 | 의미적 유사도 파악 | 정확한 키워드 매칭이 약함 | "파이썬 설치" → "Python 설치 방법" 찾음 |
| Reranker (재순위화) | Cross-Encoder로 (질문, 문서) 쌍 정밀 스코어링 | 가장 정확한 순위 결정 | 계산 비용이 높아 후보 축소 후 적용 | top-20 후보 → 정밀 평가 → top-3~5 |

[근거: D2.0-06 S7D-012, S7D-018]

### 하이브리드 검색 작동 방식

```
사용자 질문
    ├── BM25 검색 → 키워드 기반 후보 목록
    ├── Vector 검색 → 의미 기반 후보 목록
    ↓
앙상블 점수 = alpha × BM25점수 + (1-alpha) × Vector점수
    (alpha = 0.3 — 벡터 검색에 더 높은 가중치)
    ↓
RRF (Reciprocal Rank Fusion) 결합: k=60, top-10~20 후보
    ↓
Reranker (Cross-Encoder) → 최종 top-3~5 선정
```

[근거: D2.0-06 S7D-012]

### Reranker 상세

| 항목 | 값 |
|------|-----|
| 모델 | `cross-encoder/ms-marco-MiniLM-L-6-v2` (로컬 실행 가능) |
| 처리 속도 | ~50ms / 20건 |
| 정확도 향상 | 단순 벡터 검색 대비 **10~15% 향상** (MTEB 벤치마크) |
| 입력 | 1차 검색 후보 top-20 |
| 출력 | 최종 top-3~5 |

[근거: D2.0-06 S7D-018]

### 검색 정확도 비교 (Contextual Retrieval 벤치마크)

| 방식 | 검색 정확도 (상대값) | 비용 영향 |
|------|---------------------|-----------|
| 기본 RAG (Vector만) | 1.0x (기준) | 최저 |
| + BM25 하이브리드 | ~1.3x | 낮음 |
| + Contextual Prefix | ~1.67x (+67%) | 중간 (전처리 1회) |
| + Reranker 추가 | ~1.8x | 높음 |

[근거: D2.0-06 §4.3.4 Contextual Retrieval 벤치마크]

### 핵심 요약 (3줄)
1. 하이브리드 검색은 BM25(키워드) + Vector(의미) + Reranker(재순위화) 3가지를 결합한다.
2. 앙상블 가중치는 alpha=0.3(BM25 30%, Vector 70%)이며, RRF로 결과를 통합한다.
3. Reranker를 추가하면 기본 벡터 검색 대비 약 80% 더 정확한 검색이 가능하다.

## §17.4 Semantic Cache (의미 캐시)

### 비유: 자주 묻는 질문 게시판

도서관에 **"자주 묻는 질문(FAQ) 게시판"**이 있다고 상상해 보세요. "한국의 수도는?" 이라는 질문이 이미 게시판에 있으면, 다시 책을 찾을 필요 없이 **게시판의 답변을 바로 보여줍니다**. 심지어 "대한민국 수도가 어디야?"처럼 **비슷한 질문**도 같은 답변으로 연결합니다.

### 정의

Semantic Cache(의미 캐시)는 **이전에 답변한 질문과 의미적으로 유사한 새 질문**이 들어오면, RAG 검색과 LLM 호출 없이 **캐시된 답변을 즉시 반환**하는 기능입니다. [근거: D2.0-06 §4.7 ADD-012]

### 작동 방식

```
새 질문 → 임베딩 변환 → 캐시 검색 (코사인 유사도)
                              ├── ≥ 0.95 → 캐시 히트! → 즉시 응답 반환
                              └── < 0.95 → 캐시 미스 → 일반 RAG 파이프라인 실행
                                                         → 결과를 캐시에 저장
```

### 상세 사양

| 항목 | 값 |
|------|-----|
| 캐시 히트 임계값 | cosine_similarity ≥ **0.95** (기본, 운영 시 조정 가능) |
| 캐시 저장소 | V1: 인메모리(LRU), V2+: Redis 또는 별도 Chroma 컬렉션 |
| 캐시 TTL (유효 기간) | 기본 **24시간** (소스 변동이 잦으면 6시간) |
| 캐시 레코드 | `{ query_hash, query_embedding, response, source_refs, created_at, ttl, hit_count }` |

[근거: D2.0-06 §4.7.1, S7D-048]

### 캐시 무효화 정책

캐시가 오래되거나 원본 데이터가 바뀌면 자동으로 무효화됩니다:

| 무효화 조건 | 설명 |
|------------|------|
| TTL 만료 | 기본 24시간 경과 시 자동 삭제 |
| 소스 변경 | 인덱싱된 문서가 갱신/삭제되면 관련 캐시 즉시 무효화 |
| QoD 하락 | 소스의 QoD가 0.4 이하로 하락하면 관련 캐시 무효화 |
| 수동 flush | 관리자가 프로젝트/전역 단위로 강제 초기화 |

[근거: D2.0-06 §4.7.2 MOD-017]

### 비용 절감 효과

Semantic Cache를 사용하면:
- **RAG 검색 호출 절감**: 유사 질문이 반복될 때 벡터 DB 검색을 건너뜀
- **LLM 호출 절감**: 생성 단계(6단계)를 완전히 건너뜀
- **응답 속도 향상**: 캐시 히트 시 거의 즉시 응답

> 히트율이 10% 미만이면 캐시 전략 재검토 알림이 발생합니다. [근거: D2.0-06 §4.7.3]

### 핵심 요약 (3줄)
1. Semantic Cache는 의미적으로 유사한 질문에 대해 이전 답변을 재사용하는 기능이다.
2. 코사인 유사도 0.95 이상이면 캐시 히트로 판정하여 RAG+LLM 호출을 생략한다.
3. 소스 변경, QoD 하락, TTL 만료 시 자동으로 캐시가 무효화된다.

## §17.5 QoD — Quality of Data (데이터 품질 점수)

### 비유: 도서관 책의 신뢰도 별점

도서관의 모든 책에 **별점(0~5점)**이 붙어 있다고 상상해 보세요. 별점이 높은 책은 신뢰할 수 있고, 별점이 낮은 책은 참고만 합니다. QoD는 바로 이 **"데이터의 별점"**입니다.

### 정의

QoD(Quality of Data, 데이터 품질 점수)는 저장된 데이터의 **신뢰도를 0.0~1.0 범위로 수치화**한 것입니다. [근거: D2.0-06 §7.3, DEC-010 LOCK]

### 점수 산출 공식 (DEC-014 — LOCK, 변경 불가)

```
qod_score = relevance × 0.30 + accuracy × 0.25 + freshness × 0.25 + completeness × 0.20
```

| 요소 | 가중치 | 설명 |
|------|--------|------|
| relevance (관련성) | **30%** | 질문과 데이터의 관련도 |
| accuracy (정확성) | **25%** | 사실과 일치하는 정도 |
| freshness (최신성) | **25%** | 데이터가 얼마나 최신인지 |
| completeness (완전성) | **20%** | 정보가 빠짐없이 담겨 있는지 |

[근거: D2.0-06 §7.3 DEC-014]

### 임계값과 동작

| QoD 점수 | 등급 | 동작 |
|----------|------|------|
| 0.0 ~ 0.39 | 품질 미달 | L2/L3 저장 **금지**, L0/L1 임시 보관 후 재평가 또는 폐기 |
| 0.4 ~ 0.69 | 보통 | L1 저장 가능, L2 저장은 조건부 |
| 0.7 ~ 1.0 | 양호 | L2 장기 저장 허용 (정책/승인 조건 충족 시) |

[근거: D2.0-06 §2.5.2]

### 출처별 기본 QoD 가중치

외부에서 수집된 콘텐츠는 출처 유형에 따라 기본 가중치가 다릅니다:

| 출처 유형 | 기본 QoD 가중치 | 비고 |
|-----------|-----------------|------|
| 공식 문서 (API Doc, RFC 등) | 0.85 | 검증된 1차 소스 |
| 내부 KB / 팀 문서 | 0.70 | 프로젝트 컨텍스트 유효 |
| 커뮤니티/포럼 (StackOverflow 등) | 0.50 | 검증 필요, 단독 인용 지양 |
| 미분류 / 출처 불명 | 0.30 | L2 저장 전 수동 검토 필수 |

최종 스코어: `base_score × source_weight` [근거: D2.0-06 §4.z]

> QoD < 0.4인 콘텐츠는 **L2 벡터 삽입을 금지**하고 `QoD_BELOW_THRESHOLD` 이벤트를 기록합니다. [근거: D2.0-06 §4.z]

### 핵심 요약 (3줄)
1. QoD는 데이터의 신뢰도를 0.0~1.0으로 수치화한 점수이다 (LOCK).
2. 관련성(30%) + 정확성(25%) + 최신성(25%) + 완전성(20%)의 가중 합산으로 산출한다.
3. QoD 0.4 미만은 장기 저장 금지, 0.7 이상은 L2 장기 저장 허용이다.

## §17.6 Embedding 모델: BGE-M3 vs text-embedding-3-small

임베딩 모델은 **"텍스트를 숫자로 번역하는 번역기"**입니다. 번역기마다 정확도, 비용, 지원 언어가 다릅니다.

[근거: D2.0-06 §7.3 DEC-005, D2.1-A1 §A1-2A]

### 핵심 요약 (3줄)
1. V1 기본 임베딩 모델은 BGE-M3 — 무료, 로컬 실행, 다국어 최강이다 (LOCK).

3. V2+에서는 OpenAI text-embedding-3-small과 BGE-M3를 병행 사용한다.

## §17.7 Vector DB: Chroma vs Qdrant

### 비유: 서고 비교

벡터 DB는 **도서관의 서고(서가 시스템)**입니다. Chroma는 작은 동네 도서관의 서고(간단, 빠른 설치), Qdrant는 대형 시립 도서관의 서고(대용량, 고성능)입니다.

### V1(Chroma 로컬) vs V2+(Qdrant 서버) 비교 표

| 항목 | Chroma (V1) | Qdrant (V2+) |
|------|-------------|-------------|
| **유형** | 임베디드 (로컬) | 서버 모드 (셀프 호스팅) |
| **설치** | pip install, 서버 불필요 | Docker 또는 바이너리 설치 |
| **저장 방식** | 로컬 파일 (`~/.vamos/chroma/`) | 서버 기반 (원격 접속 가능) |
| **확장성** | 소규모 RAG 충분 | 고성능 벡터 검색/필터링 |
| **동시성** | 단일 프로세스 | 멀티 클라이언트 지원 |
| **비용** | **무료** | 무료 (OSS) / 유료 (Cloud) |
| **폴백** | — | pgvector (PostgreSQL 확장) |
| **적합 규모** | V0/V1 (개인/소규모) | V2/V3 (서버/엔터프라이즈) |

[근거: D2.0-06 §2.2, D2.1-A1 §A1-3]

### 버전별 Vector DB 선택 (LOCK — 변경 불가)

| 버전 | 선택 | 비용 | 비고 |
|------|------|------|------|
| **V1** | Chroma 로컬 임베디드 | ₩0 | 설치 단순, 빠른 반복 |
| **V2+** | Qdrant 우선 / pgvector 폴백 | ~$10~40/월 | 확장성, 동시성, 내결함성 |
| **V3** | Qdrant Cloud 또는 동급 매니지드 | ~$50+/월 | 백업/스케일링 편의 |

[근거: D2.0-06 §2.2 LOCK, D2.1-A1 §A1-3 LOCK]

### VectorStore 어댑터 인터페이스 (LOCK — 변경 불가)

벡터 DB를 바꿔도 코드를 수정하지 않도록 **추상화 계층(어댑터 패턴)**을 사용합니다:

| 메서드 | 설명 |
|--------|------|
| `upsert(records)` | 벡터 레코드 삽입/갱신 |
| `search(query_vector, top_k=10, filters)` | 유사 벡터 검색 (API 레이어 B1 기본값: top_k=5) |
| `delete(ids)` | 레코드 삭제 |
| `get_by_id(id)` | ID로 단건 조회 |

- V1: `ChromaAdapter`, V2: `QdrantAdapter` / `PgvectorAdapter`(폴백)
- 어댑터 교체는 **설정(config) 변경만으로** 가능 [근거: D2.0-06 §2.2-A LOCK]

### Chroma V1 설정 상세

| 항목 | 값 |
|------|-----|
| 버전 | Chroma 0.5+ (임베디드 모드) |
| 저장 경로 | `~/.vamos/chroma/` |
| 컬렉션 | `vamos_memory`, `vamos_docs`, `vamos_cache` |
| 거리 함수 | cosine |
| 인덱스 | HNSW |
| 백업 | 일 1회 `~/.vamos/backups/chroma_YYYYMMDD/` |

[근거: D2.0-06 S7D-009, S7D-016]

### 핵심 요약 (3줄)
1. V1은 Chroma(로컬 무료), V2+는 Qdrant(서버 고성능)를 사용한다 (LOCK).
2. VectorStore 어댑터 인터페이스로 벡터 DB를 설정 변경만으로 교체할 수 있다.
3. Qdrant 사용 불가 시 pgvector가 폴백으로 작동한다.

## §17.8 GraphRAG (지식 그래프 기반 RAG)

### 비유: 마인드맵 검색

일반 벡터 검색이 **"비슷한 카드 찾기"**라면, GraphRAG는 **"마인드맵을 따라가며 연결된 개념을 탐색하기"**입니다.

예를 들어 "파이썬은 어떤 프레임워크를 사용하나?"라는 질문에:
- **벡터 검색**: "파이썬 프레임워크"와 비슷한 청크를 찾음
- **GraphRAG**: "파이썬" 노드 → `uses` 엣지 → "Django", "FastAPI", "Flask" 노드를 따라가며 **관계 기반으로 탐색**

### 정의

GraphRAG는 **벡터 검색(유사도 기반) + 지식 그래프 순회(관계 기반)**를 결합한 고급 RAG 방식입니다. 단순 유사도 검색이 놓치는 **엔티티 간 관계와 다중 홉(multi-hop) 추론**을 지원합니다. [근거: D2.0-06 §4.x, DEC-004 LOCK]

### 지식 그래프 구조

```
[노드(Node)] ── 엣지(Edge) ──→ [노드(Node)]
  Person         knows           Concept
  Concept        uses            Tool
  Project        related_to      Preference
```

| 구성 요소 | 설명 | 예시 |
|-----------|------|------|
| 노드 (Node) | 개념/엔티티 | Person, Concept, Project, Tool, Preference, Event |
| 엣지 (Edge) | 관계 | knows, uses, prefers, works_on, related_to |
| 가중치 (Weight) | 관계의 강도 (0~1) | 0.9 = 강한 관계, 0.1 = 약한 관계 |

[근거: D2.0-06 S7D-019, S7D-020]

### 버전별 GraphRAG 구현

| 버전 | 구현 방식 | 규모 | 비고 |
|------|-----------|------|------|
| **V1** | NetworkX (인메모리) + JSON 직렬화 | ~10,000 노드, ~50,000 엣지 | 경량 GraphRAG, `~/.vamos/kg/graph.json` |
| **V2** | Neo4j Community | 중대규모 | 본격 GraphRAG, Cypher 질의, 엔티티-관계 순회 |
| **V3** | Neo4j Aura (매니지드) | 대규모 | 완전 관리형, 고가용성 |

[근거: D2.0-06 S7D-019, D2.1-A1 §A1-6C LOCK]

### GraphRAG 작동 방식

```
사용자 질문 → 엔티티 추출 (NER)
                    ↓
            지식 그래프 순회 (최대 3홉)
                    ↓
    그래프 결과 + 벡터 검색 결과 → 하이브리드 결합
                    ↓
              LLM 답변 생성
```

### 4중 인덱스 검색 (V3 목표)

VAMOS의 최종 목표는 4가지 인덱스를 동시에 활용하는 것입니다:

| 인덱스 | 가중치 | 역할 |
|--------|--------|------|
| Vector (벡터) | 0.35 | 의미 유사도 검색 |
| BM25 (키워드) | 0.25 | 정확한 키워드 매칭 |
| Graph (그래프) | 0.25 | 엔티티-관계 탐색 |
| Summary (RAPTOR 요약) | 0.15 | 문서 요약 기반 전역 검색 |

[근거: D2.0-06 S7D-062]

### RAG 프레임워크 3단계 진화 (LOCK)

| 단계 | 버전 | 구성 | 정확도 목표 |
|------|------|------|-----------|
| Basic | V1 | Vector 단독 검색 + 단순 프롬프트 주입 | 64%+ |
| Hybrid+Rerank | V2 | BM25+Vector 앙상블 + Cross-encoder Rerank | 83%+ |
| Self-RAG+Graph | V3 | Self-RAG 자체 검증 + GraphRAG 다중 홉 | 90%+ |

- 각 단계는 이전 단계를 포함 (누적 확장)
- V1→V2 전환 조건: L2 활성화 + RAGAS 평가 기준 미달 시
- V2→V3 전환 조건: Neo4j 인프라 + Self-RAG 검증 모듈 구현 완료 시

[근거: D2.0-06 §10.6.3 S09-B36]

### 핵심 요약 (3줄)
1. GraphRAG는 벡터 검색 + 지식 그래프 순회를 결합하여 관계 기반 탐색을 지원한다 (DEC-004 LOCK).
2. V1은 NetworkX+JSON(경량), V2는 Neo4j Community(본격), V3는 Neo4j Aura(매니지드)이다.
3. 최종 목표는 Vector+BM25+Graph+Summary 4중 인덱스로 90%+ 정확도를 달성하는 것이다.
