### 0.1 문서 목적
- ORANGE CORE의 역할/경계와 표준 런타임 파이프라인(5단계 + 상태 전이)을 확정한다.
- 내부 기능 I-1~I-25를 “설계 문서 형태”로 정리한다.
- 단일결정 원칙(Decision Kernel)과 Approval/Safety/Cost 연결 지점을 확정한다.
- I-1~I-5는 구현 직전 수준(템플릿 + 이벤트/실패/폴백 포인트 포함)으로 확정한다.
- I-6~I-21은 최소 “명세 + 인터페이스 + 이벤트/실패 포인트”까지 채운다.

### 0.2 이 문서의 범위(경계)
- 포함:
  - ORANGE CORE(판단/제어/감사 가능성) 설계
  - I-1~I-25 인덱스 및 상세 설계(7장)
  - ORANGE 네임스페이스의 event_type/failure_code/fallback 설계 포인트(값 목록은 정본 문서에 둠)
- 제외(별도 문서 참조):
  - BLUE NODE 세부(03)
  - Infra-Core/Tool Adapter(04)
  - 에이전트 워크플로우 표준(05)
  - 저장/메모리 구조 및 스키마(06)
  - Safety/Cost/Approval 정책 스키마/규정(07)
  - UI/UX 화면 설계(08)

### 0.3 용어(최소)
- Front Mini LLM: 입력 1차 해석/기초 안전 필터/도메인 힌트(실행 엔진이 아니라 전처리)
- ORANGE CORE: 정책/룰/비용/라우팅/결정/감사 중심 “판단자/제어자”
- BLUE NODE: 도메인 실행 엔진(분석/코딩/리서치 등)
- OTHER BRAINS/ENGINES: 검색/RAG/DB/크롤러/코드 실행/솔버 등(Infra 계층, 04에서 계약)
- LogEvent(event_type): 모든 이벤트 기록 단위(고유 문자열)
- FailureCode(failure_code): 모든 실패는 코드화되어 레지스트리에 등록
- Fallback(fallback_id): 모든 실패는 fallback 또는 deny로 귀결
- ErrorHandlingStandard: 모든 모듈은 통일된 에러 처리 인터페이스를 준수한다
  - 반환 타입: `Result<T, VamosError>` (예외 throw 금지, 에러 코드 반환 통일)
  - `VamosError` 필드: `failure_code: FailureCode`, `message: str`, `fallback_id: str | null`, `trace_id: str`
  - 예외(Exception)가 발생하더라도 모듈 경계에서 반드시 `VamosError`로 변환 후 상위 전달
- Decision(단일결정): 한 시점·한 컨텍스트·한 결론을 불변 객체로 고정

### 0.4 섹션별 필수 포함 체크리스트
- (1) CORE vs Brain 분리 원칙: CORE는 판단/제어, Brain/Engine은 실행/출력 생성(04와 계약)
- (2) 표준 파이프라인 고정: Perception → Reasoning → Action → Memory(Store/Commit) → Reflection + 출력 3단(요약/근거/Self-check)
- (3) I-1~I-5는 “표준 템플릿 + 이벤트/실패/폴백 포인트(등록/연결)”까지 최소 확정
- (4) Self-check 루프: soft(자동 1회) 이후 hard(사용자 승인) 조건 명시(안전/비용/도메인 충돌 포함)
- (5) Safety/Cost/Approval은 07의 정책 스키마를 참조하고, CORE에서의 호출 타이밍(게이트 위치)만 확정
- (6) event_type 고유, 실패는 레지스트리 등록 후 fallback 또는 deny로 귀결

## 1. ORANGE CORE 역할/경계(“판단자/제어자”)
- 역할:
  - 입력(Perception) 해석 결과를 바탕으로, 정책/승인/비용/근거를 평가해 단일 Decision을 잠금 생성한다.

  - ORANGE는 실행 엔진이 아니다(직접 크롤링/실행/분석 결과 생성 금지).

- 핵심 산출물:
  - Decision 객체(단일결정): gates 결과 + 선택된 node/tool/brain + 근거 요약 + self-check 요약 + audit 링크

# 1. ORANGE CORE 역할/경계 (“판단자/제어자”)

## 1.1 CORE vs Brain 분리 원칙(확정)
- CORE는 “판단/제어/승인/감사”를 수행한다.
- Brain(Blue Node/Other Brains)은 “실행/생성/검색/분석”을 수행한다.
- CORE는 결과물을 **그대로 믿지 않고**, Self-check/Policy/Cost/Evidence 게이트를 통해 “결정(Decision)”으로 확정한다.

## 1.2 CORE의 책임(확정)
- 입력 해석의 표준화(I-1) + 근거 수집(I-2) + 결정 생성(I-5)
- 비용/리소스 상한 준수(I-8) + 승인/정책 연결(07 참조, CORE 타이밍 확정)
- 로그/트레이스/실패/폴백의 표준화(I-9)
- UI로 노출되는 상태/근거/비용/승인 상태의 오케스트레이션(I-10)



### 1.2-A 모듈 간 공통 통신 인터페이스 — Runnable 프로토콜 (확정)
- **배경**: 기존 설계에서 모듈 간 통신 방식이 제각각이었음(직접 호출, 콜백, 이벤트 혼재).
- **확정**: 모든 VAMOS 모듈(CORE, Blue Node, Tool, Engine)은 **Runnable 프로토콜**을 준수한다.
- **Runnable 인터페이스 최소 계약**:
  - `run(input: RunnableInput) -> RunnableOutput`
  - `get_metadata() -> ModuleMetadata` (id, version, capabilities)
  - `health_check() -> HealthStatus`
- **효과**: USB 규격처럼 모든 모듈이 동일한 소켓으로 연결 — 교체/확장 시 인터페이스 재작성 불필요.
- **적용 범위**: I-1~I-25 전체 모듈, Blue Node, Infra Tool Adapter(04 참조).

## 1.3 CORE의 비책임(확정)
- 도메인 실행 로직(전략/분석/코딩/리서치)의 “내용 생성”은 BLUE NODE가 담당(03/05).
- 저장 계층의 물리 구현/스키마는 06이 담당. CORE는 “저장 결정과 호출 타이밍”만 확정.



### 1.3-A 코드 실행 격리 환경 — Docker 샌드박싱 (확정)
- **배경**: AI가 생성한 코드를 격리 없이 실행할 경우 호스트 시스템 오염/파괴 위험 존재.
- **확정**: VAMOS 내 모든 AI 생성 코드 실행은 **Docker 샌드박스 컨테이너** 내에서만 수행한다.
- **샌드박스 설계 원칙**:
  - 네트워크: 기본 차단(허용 목록만 열림)
  - 파일시스템: 임시 볼륨만 마운트, 호스트 경로 직접 접근 금지
  - 실행 시간 제한: 기본 30초(I-8 Cost Gate 연동)
  - 메모리/CPU 상한: 설정 파일로 관리
- **실패 코드**: `SANDBOX_TIMEOUT`, `SANDBOX_OOM`, `SANDBOX_POLICY_DENIED` — 레지스트리(6.2) 등록 대상.
- **폴백**: 타임아웃/정책 위반 시 `deny` + LogEvent 기록 후 사용자에게 오류 반환.
- **참조**: Infra-Core/Tool Adapter 04에서 컨테이너 생명주기 관리.


### [ADD-069] LangChain 제거 후 경량 대안 평가 결과
- **배경**: VAMOS 설계 과정에서 LangChain 의존성을 제거하기로 결정하였으며, 대체 경량 오케스트레이션 수단에 대한 평가가 필요하였다.
- **평가 대상 및 결과**:
  | 대안 | 평가 항목 | 결론 |
  |------|----------|------|
  | 순수 Python 직접 구현 | 의존성 없음, 완전한 제어 가능, 유지보수 부담 있음 | **채택** (VAMOS 핵심 파이프라인) |
  | LlamaIndex (core only) | RAG 파이프라인 특화, 필요 시 I-2와 연계 가능 | **부분 채택** (I-2 근거 수집 한정) |
  | Haystack | 검색/문서 파이프라인 특화, 과도한 추상화 | **미채택** |
  | DSPy | 프롬프트 최적화 특화, 런타임 오버헤드 있음 | **LATER 검토** (I-12 Self-evo 단계) |
- **확정**: ORANGE CORE 파이프라인(I-1~I-5)은 순수 Python + 최소 표준 라이브러리로 구현하며, LangChain 의존성은 완전 제거한다.

## 1.4 Front Mini의 위치와 책임 (확정)

- **위치**: ORANGE CORE의 **최전방 게이트**로 정의한다 (CORE 내부 모듈).

### 1.4-A MCP·LangChain 통합 전략 (확정)
- **배경**: MCP(Model Context Protocol)와 LangChain 중 어느 것을 주축으로 쓸지 미확정 상태였음.
- **확정**: **MCP 우선, LangChain 패턴 참조만** 전략을 채택한다.
- **세부 규칙**:
  1. 툴/컨텍스트 호출은 **MCP 규격**을 정본으로 사용한다.
  2. LangChain은 `import` 하지 않는다 — 아키텍처 패턴(Chain, Router, Memory 추상화 개념)만 참조.
  3. LangChain 의존성이 필요한 경우 반드시 Adapter 계층(04)을 경유해야 하며, CORE 직접 의존 금지.
- **이유**: MCP는 표준화된 컨텍스트 프로토콜로 멀티-LLM 확장에 유리; LangChain import는 버전 종속성 위험이 높음.
- **참조**: Tool Adapter 인터페이스 설계는 04 문서에서 계약.

## 2.1 표준 5단계(확정)
1) Perception: 입력 수신/전처리/형식 파악(텍스트/파일/이미지 등)
2) Reasoning: 의도/제약/도메인/필요 근거/계획 수립
3) Action: 실행(Blue Node/Tools 호출), 산출물 생성
4) Memory: 저장 후보 생성/정책/승인/저장 실행
5) Reflection: Self-check/정합성/품질/안전/비용 검증 및 루프 결정


### 2.1-A [S7B-001] Adaptive Thinking — 난이도별 사고 깊이 자동 조절 (CRITICAL, V2)

> **출처**: STEP7-B 대화프로세스 #1 | 참조 AI: Claude (Adaptive Thinking), GPT o3 (Reasoning Effort), Gemini (Thinking Level), DeepSeek (think 태그)
> **파이프라인 위치**: S1(Intent Parsed) → S2(Evidence Ready) 사이 Reasoning 단계

- **목적**: 사용자 요청의 난이도를 자동 판단하여 사고 깊이(thinking budget)를 동적으로 할당한다. 단순 질문에 과도한 토큰을 소비하지 않고, 복잡한 추론에는 충분한 사고 시간을 부여한다.
- **난이도 분류기 (Complexity Classifier)**:

```python
class ThinkingLevel(str, Enum):
    """사고 깊이 레벨"""
    INSTANT = "instant"     # 즉시 응답 — 인사, 단순 사실, 번역
    LOW = "low"             # 짧은 사고 — 요약, 간단한 설명
    MEDIUM = "medium"       # 표준 사고 — 분석, 비교, 코드 작성
    HIGH = "high"           # 깊은 사고 — 복잡 추론, 다단계 계획
    MAX = "max"             # 최대 사고 — 연구 수준 분석, 수학 증명

class AdaptiveThinkingConfig(BaseModel):
    """Adaptive Thinking 설정"""
    classifier_model: str = "front_mini"    # 난이도 판단에 사용할 모델
    thinking_budget: dict = {
        "instant": 0,           # thinking 토큰 없음
        "low": 1024,            # ~1K 토큰
        "medium": 4096,         # ~4K 토큰
        "high": 16384,          # ~16K 토큰
        "max": 65536            # ~64K 토큰
    }
    auto_escalate: bool = True  # 사고 중 복잡도 상향 자동 허용
    user_override: bool = True  # 사용자가 수동으로 레벨 지정 가능

class ThinkingResult(BaseModel):
    """사고 결과"""
    level: ThinkingLevel
    thinking_tokens_used: int
    thinking_content: Optional[str] = None  # 사용자 표시 가능한 사고 과정
    confidence: float                        # 분류 신뢰도 (0~1)
    escalated: bool = False                  # 자동 상향 발생 여부
```

- **난이도 판단 기준**:
  | 신호 | instant | low | medium | high | max |
  |------|---------|-----|--------|------|-----|
  | 토큰 수 | <20 | 20-50 | 50-200 | 200-500 | 500+ |
  | 도메인 위험도 | P0 | P0 | P0-P1 | P1-P2 | P2 |
  | 도구 필요 | 없음 | 없음 | 1-2개 | 3+개 | 다수+연쇄 |
  | 제약 조건 수 | 0 | 0-1 | 2-3 | 4+ | 복합 |
  | sub_intents 수 | 1 | 1 | 2-3 | 4+ | 다수 |

- **I-1 IntentFrame 확장 필드**: `complexity: ThinkingLevel` 추가
- **I-5 Decision 반영**: `thinking_budget_tokens: int` → Cost Gate에서 비용 예측에 포함
- **I-8 Cost 연동**: thinking 토큰은 별도 비용 항목으로 추적 (high/max 레벨은 사전 비용 알림)
- **이벤트**: `oc.i1.thinking.level_classified`, `oc.i1.thinking.escalated`
- **실패 코드**: `OC_I1_THINKING_BUDGET_EXCEEDED` — 예산 초과 시 Cost Gate downshift
- **구현 시점**: V2 (Front Mini + Extended Thinking API 연동 필요)
- **STEP6 연관**: ADD-052 (Chat Engine 재설계)에 Thinking Level 결정 단계 추가

### [ADD-052] Chat Engine 재설계 방향
- 기존 Chat Engine 아키텍처는 단일 LLM 호출 구조로 설계되어, ORANGE CORE의 파이프라인 5단계와 통합되지 않은 초기 형태였다.
- **재설계 원칙**:
  1. Chat Engine은 ORANGE CORE의 Perception → Reasoning → Action 흐름에 종속되는 **세션 오케스트레이터**로 재정의한다.
  2. 다중 턴 컨텍스트 관리는 Chat Engine이 담당하되, 각 턴의 의도 해석은 반드시 I-1(IntentFrame)을 거친다.
  3. 스트리밍 응답은 S4(EXECUTING) 단계에서 Blue Node 출력을 직접 중계하며, Chat Engine은 버퍼링/청크 조립만 수행한다.
  4. 세션 상태(S0~S8)는 Chat Engine이 보유하며, Decision 객체와 연결된 trace_id로 감사 가능하게 유지한다.
- **경계**: Chat Engine은 결정 생성(I-5)에 개입하지 않는다.

### 2.2-A 동시성 처리 전략 (확정)
- **배경**: 병렬 Blue Node 실행, 멀티 툴 호출 등에서 동시성 제어 전략이 미정이었음.
- **확정**: **큐(Queue) + 세마포어(Semaphore) 패턴**을 동시성 제어 표준으로 채택한다.
- **세부 설계**:
  - 작업 큐: 모든 Blue Node/Tool 호출 요청은 내부 작업 큐에 적재 후 처리
  - 세마포어: 동시 실행 상한은 모델별/툴별로 설정 파일에서 관리
    - 예: `MAX_CONCURRENT_BLUE_NODES=3`, `MAX_CONCURRENT_TOOLS=5`
  - 우선순위: 비용이 낮은 작업(mini 모델) > 비용이 높은 작업(main 모델) 순으로 스케줄링
- **실패 처리**: 큐 포화 시 `QUEUE_OVERFLOW` FailureCode 발생 → fallback: 요청 지연 반환 또는 deny.
- **로깅**: 큐 대기 시간/처리 시간 LogEvent 기록(성능 모니터링 연계).

### 2.2-B [S7B-005] Hook 라이프사이클 시스템 — CRITICAL, V1

> **출처**: STEP7-B 대화프로세스 #5 | 참조 AI: Claude (6종 Hook — SessionStart, PreToolUse, PostToolUse, PreCompact, PostCompaction, Stop)
> **파이프라인 위치**: 전체 파이프라인 횡단 (S0~S8 모든 단계에서 Hook 포인트 제공)

- **목적**: 파이프라인 각 단계의 전후에 사용자/시스템이 커스텀 로직을 삽입할 수 있는 Hook 시스템을 제공한다. 도구 실행 전 검증, 세션 초기화, 출력 후처리 등에 활용한다.
- **Hook 타입 정의**:

```python
class HookType(str, Enum):
    """파이프라인 Hook 타입"""
    SESSION_START = "session_start"         # 세션 시작 시
    SESSION_END = "session_end"             # 세션 종료 시
    PRE_ANALYSIS = "pre_analysis"           # I-1 의도 분석 전
    POST_ANALYSIS = "post_analysis"         # I-1 의도 분석 후
    PRE_TOOL_USE = "pre_tool_use"           # 도구 실행 전 (검증/차단 가능)
    POST_TOOL_USE = "post_tool_use"         # 도구 실행 후 (결과 가공 가능)
    PRE_OUTPUT = "pre_output"               # 출력 생성 전
    POST_OUTPUT = "post_output"             # 출력 생성 후
    PRE_COMPACT = "pre_compact"             # 컨텍스트 압축 전
    POST_COMPACT = "post_compact"           # 컨텍스트 압축 후
    STOP = "stop"                           # 응답 완료 직전

class HookRegistration(BaseModel):
    """Hook 등록 정보"""
    hook_id: str
    hook_type: HookType
    handler: str                            # 핸들러 함수/스크립트 경로
    priority: int = 100                     # 0=최우선, 999=최후순
    enabled: bool = True
    timeout_ms: int = 10000                 # 실행 타임아웃 (기본 10초)
    sandbox: bool = True                    # 샌드박스 실행 여부
    conditions: Optional[dict] = None       # 조건부 실행 (tool_name, domain 등)

class HookResult(BaseModel):
    """Hook 실행 결과"""
    hook_id: str
    hook_type: HookType
    status: Literal["pass", "block", "modify", "error"]
    modified_data: Optional[dict] = None    # 수정된 데이터 (modify 시)
    block_reason: Optional[str] = None      # 차단 사유 (block 시)
    execution_ms: float
```

- **Hook Registry**: `.vamos/hooks.json`에 Hook 설정 관리
  ```json
  {
    "hooks": [
      {
        "hook_type": "pre_tool_use",
        "handler": "hooks/validate_tool.py",
        "priority": 0,
        "conditions": {"tool_name": ["bash", "write"]},
        "timeout_ms": 5000
      },
      {
        "hook_type": "session_start",
        "handler": "hooks/load_project_context.py",
        "priority": 10
      }
    ]
  }
  ```
- **보안**: Hook 실행은 타임아웃 10초 + 샌드박스 환경에서 수행. 무한 루프/악성 코드 방지
- **이벤트**: `oc.hook.registered`, `oc.hook.executed`, `oc.hook.blocked`, `oc.hook.timeout`
- **실패 코드**: `OC_HOOK_TIMEOUT`, `OC_HOOK_EXECUTION_FAIL` — Hook 실패 시 기본 동작으로 진행 (Hook은 비필수)
- **STEP6 연관**: S06-B21 (.vamosrules) 확장, S7-A-004 (Hooks 라이프사이클) 통합

### 2.2-C [S7B-008] 컨텍스트 자동 압축 (Compaction) — HIGH, V2

> **출처**: STEP7-B 대화프로세스 #8 | 참조 AI: Claude (PreCompact/PostCompaction Hook + 자동 압축)
> **파이프라인 위치**: S4(Executing) 또는 S7(Memory Committed) 이후, 컨텍스트 윈도우 한계 접근 시

- **목적**: 긴 대화에서 컨텍스트 윈도우 한계에 접근하면, 이전 메시지를 자동 요약/압축하여 대화를 지속할 수 있게 한다.
- **구현 설계**:

```python
class CompactionConfig(BaseModel):
    """컨텍스트 압축 설정"""
    trigger_threshold: float = 0.8          # 컨텍스트 80% 사용 시 트리거
    target_ratio: float = 0.5               # 압축 후 목표: 50% 이하
    preserve_recent_turns: int = 5          # 최근 N턴은 원문 유지
    preserve_system_prompt: bool = True     # 시스템 프롬프트 보존
    summary_model: str = "front_mini"       # 요약에 사용할 모델
    max_summary_tokens: int = 2048          # 요약 최대 토큰

class CompactionResult(BaseModel):
    """압축 결과"""
    compaction_id: str
    original_tokens: int
    compressed_tokens: int
    compression_ratio: float
    preserved_turns: int
    summary: str                            # 압축된 대화 요약
    information_loss_estimate: float        # 정보 손실 추정 (0~1)
    timestamp: datetime
```

- **압축 전략**:
  1. **턴 기반 압축**: 오래된 턴부터 요약으로 대체 (최근 N턴 보존)
  2. **중요도 기반 압축**: Decision/Tool 결과가 포함된 턴은 우선 보존
  3. **점진적 압축**: 한 번에 전체가 아닌, 가장 오래된 블록부터 단계적 압축
- **Hook 연동**: `pre_compact` → 압축 전 사용자 정의 로직 실행, `post_compact` → 압축 후 컨텍스트 재초기화
- **정보 손실 방지**: 압축 전 핵심 정보(결정 사항, 사용자 선호, 도구 결과)를 별도 저장 → L0 메모리에 기록
- **이벤트**: `oc.context.compact.triggered`, `oc.context.compact.completed`
- **실패 코드**: `OC_CONTEXT_COMPACT_FAIL` — 압축 실패 시 대화 초기화 제안
- **비용**: 압축 1회당 ~$0.01 (Front Mini 모델 사용)

### 2.2-D [S7B-018] 스트리밍 출력 구현 — HIGH, V1

> **출처**: STEP7-B 대화프로세스 #18 | 참조 AI: 전체 AI (모든 시중 AI가 스트리밍 지원)
> **파이프라인 위치**: S4(Executing) → S5(Output Ready) 단계

- **목적**: LLM 응답을 토큰 단위로 실시간 스트리밍하여 사용자 체감 응답 시간을 단축한다.
- **구현 설계**:

```python
class StreamConfig(BaseModel):
    """스트리밍 출력 설정"""
    enabled: bool = True
    protocol: Literal["sse", "websocket"] = "sse"   # Server-Sent Events 기본
    chunk_size: int = 1                     # 토큰 단위 (1=실시간, N=배치)
    buffer_size: int = 0                    # 버퍼링 없음 (즉시 전송)
    heartbeat_interval_ms: int = 15000      # 연결 유지 하트비트

class StreamEvent(BaseModel):
    """스트리밍 이벤트"""
    event_type: Literal["token", "tool_start", "tool_end", "thinking", "done", "error"]
    data: str
    trace_id: str
    timestamp: datetime
    metadata: Optional[dict] = None         # 토큰 카운트, 비용 등
```

- **스트리밍 프로토콜**:
  - **SSE (Server-Sent Events)**: HTTP 기반, 단방향, FastAPI `StreamingResponse` 활용
  - **WebSocket**: 양방향, 실시간 상호작용 필요 시 (V2+)
- **스트리밍 이벤트 흐름**:
  ```
  [thinking] → 사고 과정 표시 (opt-in)
  [token] → 응답 토큰 실시간 전송
  [tool_start] → 도구 실행 시작 알림
  [tool_end] → 도구 실행 완료 + 결과 요약
  [done] → 응답 완료 + 최종 메타데이터
  ```
- **Chat Engine 연동**: ADD-052 (Chat Engine 재설계)에서 스트리밍 버퍼링/청크 조립 담당
- **이벤트**: `oc.stream.started`, `oc.stream.completed`, `oc.stream.error`
- **실패 코드**: `OC_STREAM_DISCONNECT` — 연결 끊김 시 재연결 또는 폴링 모드 전환

### [ADD-036b] Pydantic 스키마 전수 검증 표준
- **적용 범위**: I-1(IntentFrame), I-2(EvidencePack), I-5(Decision), I-4(OutputSpec) 등 모든 ORANGE CORE 내부 데이터 객체.
- **전수 검증 규칙**:
  1. 모든 내부 객체는 `BaseModel`(Pydantic v2) 기반 스키마로 정의한다.
  2. 파이프라인 각 단계(S0→S8) 경계에서 `.model_validate()` 호출을 의무화한다.
  3. 검증 실패 시 `ValidationError`를 포착하여 즉시 `FailureCode: OC_ERR_SCHEMA_INVALID`로 변환하고 Fallback으로 분기한다.
  4. 외부 입력(API 요청, Tool 응답) 역시 진입 시점에 스키마 검증을 수행한다.
- **금지**: `dict` 또는 `Any` 타입으로 검증 없이 단계 간 전달하는 것을 금지한다.

### [ADD-037b] Function Call(FC) 에러 처리 표준
- **대상**: ORANGE CORE가 Blue Node/Tool을 호출할 때 발생하는 모든 FC 관련 오류.
- **FC 에러 분류 및 처리**:
  | 에러 유형 | FailureCode | 처리 방법 |
  |----------|------------|----------|
  | FC 스키마 불일치 | `OC_ERR_FC_SCHEMA` | 재시도 1회 → deny |
  | FC 응답 누락/타임아웃 | `OC_ERR_FC_TIMEOUT` | Fallback: 다운그레이드 모드 |
  | FC 응답 파싱 실패 | `OC_ERR_FC_PARSE` | 재시도 1회 → deny |
  | FC 정책 위반 | `OC_ERR_FC_POLICY` | 즉시 deny (재시도 없음) |
  | FC 비용 한도 초과 | `OC_ERR_FC_COST` | Cost Gate → downshift or stop |
- **공통 원칙**:
  1. 모든 FC 호출은 `try/except` 블록으로 감싸고, 에러는 반드시 위 분류 코드로 변환한다.
  2. 재시도 횟수는 최대 1회로 제한하며, 재시도 후에도 실패 시 Fallback 또는 deny로 귀결한다.
  3. 모든 FC 에러는 `LogEvent(event_type: OC_FC_ERROR)`로 레지스트리에 기록한다.

## 3.3 충돌 해결 규칙(확정)
- 정책 충돌: policy_gate가 우선(deny > restrict > allow, 07 정본)
- 비용 충돌: cost_gate가 라우팅/모델/실행 계획을 제한
- 근거 충돌: evidence_gate 미달이면 conclusion은 HOLD/ESCALATE로 제한
- 포맷 충돌: output_spec 위반은 Self-check FAIL로 처리 후 재생성 or 명시적 사유와 함께 종료

## 3.4 외부 Agent/Workflow 프레임워크 적용 원칙(P2 반영 / FREEZE 확정)
- 정의: **LangGraph**는 워크플로우 오케스트레이션 전용으로 **LOCK 확정**(DEC-002)되었다. AutoGen / CrewAI / LlamaIndex / ReAct 등 기타 프레임워크는 “후보”일 뿐이며, VAMOS의 정본(SOT)을 대체하지 않는다.
- 고정: 최종 Decision(단일결정)과 Gate 결과(PolicyCheck/Approval/CostBudget)는 항상 ORANGE CORE(02) → SAFETY/COST/APPROVAL(07) 경로로만 확정된다.
- 금지: 외부 프레임워크가 “승인/정책/비용”을 에이전트 내부 if문으로 임의 처리하여 Gate를 우회하는 구조는 허용하지 않는다. (우회 위험)
- 역할 분리:
  - 외부 프레임워크: 노드/스텝 실행, 상태 전이, 체크포인트/리플레이 같은 “실행 오케스트레이션”만 담당
  - VAMOS CORE(02/07): 결론 확정, 승인 요구/차단, 비용 상한/다운시프트, 정책 판정, 감사 로그 스키마를 담당
- 연결 참조:
  - 실행 워크플로우/루프/재현성(Checkpoint/Replay/Fork) 규칙: 05 참조
  - 승인/정책/비용 Gate 스키마 및 이벤트: 07 참조
  - UI에서 Gate/Trace 표시: 08 참조

# 5. 공통 설계 템플릿(모든 I-모듈 공통 / FREEZE 확정)

각 모듈은 아래 항목을 반드시 포함한다.

1) 목적(WHY)
2) 입력(INPUTS) — 타입/출처/필수·선택
3) 출력(OUTPUTS) — 타입/후속 모듈 의존
4) 의존성(DEPENDENCIES) — 다른 I-모듈/문서/외부 리소스
5) 상태(STATE) — 내부 상태/전이
6) 정책(POLICY HOOKS) — 07과 연결되는 지점
7) 로그(LOG EVENTS) — LogEvent(event_type) 목록(고유 문자열)
8) 실패(FAILURE POINTS) — FailureCode 목록
9) 폴백(FALLBACKS) — Fallback Strategy 연결(또는 deny)
10) 테스트(TEST) — 최소 검증 항목(단, I-6~I-21은 “핵심 2~3개”만)

## 5.1 표준 출력 포맷(답변/근거/Self-check) — ORANGE CORE 최소 규격(LOCK)

- 목적: “근거 없는 단정” 방지 + UI/로그/검증이 같은 형태로 붙도록 출력 구조를 고정한다.
- 원칙:
  - EvidenceGate(I-2/I-19) 미달 시, 결론(conclusion)은 HOLD/ESCALATE/REQUEST_EVIDENCE 범위로 제한한다.
  - Self-check(I-6)는 결과 채택/재시도/종료를 결정하는 “관문”으로만 사용한다(자동 개선 적용 금지).

### 5.1.1 (A) ResponseEnvelope (최상위 출력)
- answer:
  - summary: string (짧은 결론/요약)
  - details: string (필요 시만)
  - next_actions: [string] (사용자에게 제안할 다음 행동)
- evidence:
  - coverage: number (0~1, 근거 충족률)
  - items: [EvidenceItem]
  - qod: number (0~1, QoD 종합)
- self_check:
  - score: number (0~1)
  - verdict: PASS | WARN | FAIL
  - reasons: [string]
  - retry_allowed: boolean
- decision_ref:
  - decision_id: string
  - gates: {policy: PASS|FAIL, cost: PASS|FAIL|DOWNSHIFT, evidence: PASS|FAIL}
- audit:
  - event_ids: [string]
  - failure_codes: [string]
  - fallback_ids: [string]

### 5.1.2 (B) EvidenceItem (근거 단위)
- source_type: RAG | TOOL | MEMORY | WEB | USER
- source_id: string
- claim: string
- support: string (짧은 인용/요약)
- freshness: string (timestamp/기간)
- qod: number (0~1)

### 5.1.3 (C) UI 노출 최소 필드(08과 연결)
- UI는 ResponseEnvelope에서 아래만 반드시 읽을 수 있어야 한다:
  - answer.summary
  - evidence.coverage, evidence.qod
  - self_check.verdict, self_check.score
  - decision_ref.gates
  - audit.event_ids(요약), audit.failure_codes(요약)

## 6.3 Fallback Strategy Registry (fallback_id)

각 FallbackStrategy 필드(표준)
- fallback_id: string(unique)
- goal: string (안전/비용/정확도/속도 등)
- preconditions: string | list[string]
- steps: list[string] (1~N)
- degrade_level: V0 | V1 | V2 | V3
- stop_condition: string
- log_event_type: string (연결 event_type)

- FB_INTENT_HEURISTIC_PARSE
  - goal: 요구사항 최소 파싱 성공(정확도/속도)
  - preconditions: OC_I1_PARSE_FAIL
  - steps:
    1) 입력에서 핵심 슬롯(목표/대상/산출물 형식)만 추출
    2) 불확실 슬롯은 "unknown"으로 두고 다음 단계(I-2/I-5)에서 보수 처리
  - degrade_level: "V1"
  - stop_condition: intent_frame_min_ready
  - log_event_type: oc.i1.parse.failed

- FB_ASK_CLARIFICATION
  - goal: 모호성 해소(정확도)
  - preconditions: OC_I1_AMBIGUOUS_UNRESOLVED
  - steps:
    1) 모호한 슬롯 1~3개만 질문으로 환원
    2) 사용자 응답 전까지 Decision은 HOLD
  - degrade_level: "V1"
  - stop_condition: 사용자 응답 수신 또는 타임아웃
  - log_event_type: oc.i1.intent.ambiguous

- FB_RAG_RETRY_EXPAND
  - goal: 근거 부족 해결(정확도)
  - preconditions:
    - OC_I2_EVIDENCE_QOD_LOW

  - steps:
    1) 검색 범위를 확장(내부→외부, 최신성 완화 등)
    2) QoD 기준 미달이면 단정 금지 + 요약 수준으로 다운시프트
  - degrade_level: "V1"

  - log_event_type: oc.i2.evidence.insufficient

- FB_RAG_SWITCH_SOURCE

    - OC_I2_TIMEOUT
  - steps:
    1) 동일 쿼리를 다른 소스 세트로 재시도
    2) 타임아웃이면 짧은 응답 모드로 축소
  - degrade_level: "V1"

  - log_event_type: oc.i2.fetch.blocked

  - steps:
    1) 승인 요청 메시지 생성
    2) 승인 전까지 Decision은 HOLD
  - degrade_level: "V1"
  - stop_condition: 승인/거절/타임아웃
  - log_event_type: oc.i5.approval.required

    ANXIETY = “anxiety”
    CALM = “calm”
    CONFUSION = “confusion”
    STRESS = “stress”
    FATIGUE = “fatigue”

    “””개별 감정 점수”””
    category: EmotionCategory
    intensity: float                        # 0.0~1.0 (강도)

class EmotionFrame(BaseModel):

### 7.10-B [S7B-003] 적응형 응답 톤 조절 — CRITICAL, V1

> **출처**: STEP7-B 대화프로세스 #3 | 참조 AI: GPT-4.5, Gemini
> **파이프라인 위치**: I-4 (Output Structuring) + I-13 (Template Engine)

- **목적**: 감정 감지 결과를 기반으로 응답의 톤, 길이, 상세도, 공감 수준을 자동 조절한다.
- **구현 설계**:

```python
class ToneAdapter:
    “””감정 상태에 따른 출력 톤 자동 조절”””

    EMOTION_TONE_MAP: dict = {
        “anxiety”:      {“tone”: “reassuring”,    “detail”: “concise”,  “empathy”: 0.8},
        “frustration”:  {“tone”: “encouraging”,   “detail”: “concise”,  “empathy”: 0.9},
        “excitement”:   {“tone”: “neutral”,       “detail”: “detailed”, “empathy”: 0.5},
        “sadness”:      {“tone”: “warm”,          “detail”: “standard”, “empathy”: 0.9},
        “confusion”:    {“tone”: “professional”,  “detail”: “detailed”, “empathy”: 0.6},
        “stress”:       {“tone”: “reassuring”,    “detail”: “concise”,  “empathy”: 0.8},
        “calm”:         {“tone”: “professional”,  “detail”: “standard”, “empathy”: 0.4},
        “joy”:          {“tone”: “warm”,          “detail”: “standard”, “empathy”: 0.5},
    }

    def adapt(self, emotion_frame: EmotionFrame) -> ResponseAdaptation:
        “””감정에 따른 응답 적응 결정”””
        dominant = emotion_frame.dominant_emotion.value
        config = self.EMOTION_TONE_MAP.get(dominant, {“tone”: “neutral”, “detail”: “standard”, “empathy”: 0.4})
        return ResponseAdaptation(**config, suggested_approach=self._get_approach(dominant))

    def _get_approach(self, emotion: str) -> str:
        approaches = {
            “stress”:      “간결하고 따뜻하게, 핵심만 전달 + 대안 제시”,
            “frustration”: “공감 표현 후 구체적 대안 제시 + 격려”,
            “confusion”:   “단계별 설명 + 예시 포함 + 추가 질문 유도”,
            “excitement”:  “구체적 정보 제공 + 다음 단계 안내”,
            “anxiety”:     “안심시키는 톤 + 확실한 정보만 제공”,
        }
        return approaches.get(emotion, “표준 응답”)
```

- **시스템 프롬프트 동적 삽입**: 감정에 따라 시스템 프롬프트에 톤 지시를 동적으로 추가
  ```
  [tone_instruction]: 사용자가 현재 불안한 상태입니다. 간결하고 안심시키는 톤으로 응답하세요.
  핵심 정보만 전달하고, 불확실한 내용은 피하세요.
  ```
- **사용자 설정**: `tone_adaptation: enabled|disabled` — 사용자가 감정 적응을 비활성화 가능
- **I-4 연동**: StructuredOutput 생성 시 ResponseAdaptation 적용
- **이벤트**: `oc.i4.tone.adapted`
- **비용**: 무료 (프롬프트 엔지니어링으로 구현, 추가 API 호출 없음)

## 7.53-1 Self-check 임계값 정책 (결정 완료/LOCK, 정본)

> 정본 근거: PLAN 3.0의 “Self-check는 고정된 관문” 원칙을 유지한다.
> 단, 런타임 적용에서는 위험도(P0/P1/P2) 기준으로 임계값을 “가변” 적용한다.
> 본 섹션은 05 §4.1에서 “정본 근거”로 참조된다.

### 1) (LOCK) 가변 임계값(위험도 기반)
- P0: self_check_score >= 70  → PASS
- P1: self_check_score >= 75  → PASS
- P2: self_check_score >= 80  → PASS

> 해석:
> - “관문(Self-check)은 고정” = 반드시 통과해야 다음 단계로 진행 가능
> - “임계값은 가변” = 위험도에 따라 PASS 기준점만 달라진다

### 2) (LOCK) FAIL 처리 규칙(자동 Soft loop + 수렴)
- 1차 FAIL → 자동 1회 Soft loop 후 재평가(재계산)
- 2회 연속 FAIL → Decision은 “게이트 결과 우선”으로 수렴:
    - 승인 필요 상황이면: fallback_id = FB_REQUIRE_APPROVAL
    - 승인 없이 처리 가능한 축소 출력이면: fallback_id = FB_OUTPUT_MINIMAL
    - 정책 위반/Non-goal이면: fallback_id = FB_DENY_WITH_REASON
- (중요) fallback_id 문자열은 02 §6.3 Fallback Strategy Registry 정본만 사용한다.

### 3) (LOCK) P2(고위험) 특례 — 승인/축소 우선
- P2에서 FAIL(또는 QoD/Policy/Cost에서 restrict/deny)일 경우,
  - Soft loop를 강행하지 않고 07의 Approval/Policy/Cost Gate 결론을 우선한다.
  - 승인 없이는 “확정 결론/대규모 실행/확장 도구 호출”로 넘어가지 않는다.

### 4) (LOCK) 기록/감사
- PASS/FAIL/Soft loop/수렴(fallback/deny/restrict) 결과는 LogEvent로 기록한다.
- 단일결정 원칙(한 시점·한 컨텍스트·한 결론)을 위반하지 않도록,
  - 결론이 lock된 이후에는 downshift(축소)만 허용한다.

### 7.59-A [S7B-028] Prompt Caching (시스템 프롬프트 캐싱) — CRITICAL, V1

> **출처**: STEP7-B 대화프로세스 #28 | 참조 AI: Claude, GPT, Gemini
> **파이프라인 위치**: Stage 1 (Input) + I-8 (Cost) 연동

- **목적**: 반복 요청 시 시스템 프롬프트/정적 컨텍스트를 캐싱하여 API 호출 비용을 30~90% 절감한다.
- **구현 설계**:
  - `PromptCacheManager` 클래스: 캐시 적중률(hit rate) 추적 + 비용 절감 리포트 생성
  - 캐시 키 생성: `cache_key = hash(system_prompt + static_context + model_id)`
  - 제공사별 캐싱 전략:
    - **Anthropic**: `cache_control: {"type": "ephemeral"}` 헤더 — TTL 5분, 90% 비용 절감
    - **OpenAI**: Automatic Caching — API 자동 캐시, 50% 비용 절감
    - **Google**: Context Caching API — 명시적 캐시 생성/관리, TTL 설정 가능

```python
class PromptCacheManager(BaseModel):
    """시스템 프롬프트 캐시 관리자"""
    cache_id: str
    model_id: str
    cache_key: str                          # hash(system_prompt + static_context)
    provider: Literal["anthropic", "openai", "google"]
    ttl_seconds: int = 300                  # 기본 5분
    hit_count: int = 0
    miss_count: int = 0
    estimated_savings_usd: float = 0.0
    created_at: datetime
    last_hit_at: Optional[datetime] = None

    def hit_rate(self) -> float:
        total = self.hit_count + self.miss_count
        return self.hit_count / total if total > 0 else 0.0

class CacheConfig(BaseModel):
    """캐시 설정"""
    enabled: bool = True
    max_cache_entries: int = 100
    eviction_policy: Literal["lru", "ttl", "lfu"] = "lru"
    cost_tracking_enabled: bool = True
```

- **비용 절감 예측**:

  |---------|-----------|---------|--------|
  | 동일 시스템 프롬프트 100회 호출 | $5.00 | $0.50~$2.50 | 50~90% |

### 8.1 5-Gate 체계 정의 (LOCK)

> **[OC-06]** 기존 3-Gate(PolicyGate/CostGate/EvidenceGate)에 ApprovalGate, SelfCheckGate 2개를 추가하여 **5-Gate 체계**를 확정한다.

| Gate | 역할 | 판정 값 | 연결 정본 |
|------|------|---------|----------|
| PolicyGate | 정책/안전 위반 차단 | block / require_approval / mask / allow | 07 §6.4 |
| CostGate | 비용 상한 제어 | normal / downshift / split / stop | 07 §4.1, I-8 |
| EvidenceGate | 근거 품질(QoD) 검증 | sufficient / insufficient | I-2, I-19 |
| **ApprovalGate** | 사용자 승인 필요 여부 판정 | approved / denied / pending | 07 §3, I-19 |
| **SelfCheckGate** | 출력 품질·안전·정합성 자동 검증 | pass / fail / retry | I-6, 07 |

#### ApprovalGate 상세
- **트리거 조건**: PolicyGate에서 `require_approval` 판정 시, L2 메모리 저장 시, 고비용 실행 계획 확정 시
- **판정 흐름**: Decision HOLD → 사용자 승인 요청(UI/08) → approved → Decision 재개 / denied → DENY + 사유 기록
- **이벤트**: `oc.gate.approval.requested`, `oc.gate.approval.resolved`
- **실패 코드**: `OC_GATE_APPROVAL_TIMEOUT`, `OC_GATE_APPROVAL_DENIED`
- **폴백**: FB_APPROVAL_TIMEOUT → 안전 종료(deny + 사유 기록)
- **연결 정본**: 승인 정책 상세는 07 §3(Approval 규정) 참조

#### SelfCheckGate 상세
- **트리거 조건**: 출력 생성 완료 후 자동 1회(soft loop) 실행
- **검증 항목**: 출력 포맷 정합성, 안전 가이드라인 준수, 근거-결론 일치, 자기모순 탐지
- **판정 흐름**: pass → 출력 확정 / fail → soft retry 1회 / retry 후 fail → hard loop(사용자 승인 필요, ApprovalGate 연동)
- **이벤트**: `oc.gate.selfcheck.started`, `oc.gate.selfcheck.passed`, `oc.gate.selfcheck.failed`
- **실패 코드**: `OC_GATE_SELFCHECK_FORMAT_FAIL`, `OC_GATE_SELFCHECK_SAFETY_FAIL`, `OC_GATE_SELFCHECK_EVIDENCE_MISMATCH`
- **폴백**: FB_SELFCHECK_RETRY → 재생성 1회 / FB_SELFCHECK_HARD → ApprovalGate 이관
- **연결 정본**: Self-check 정책 상세는 I-6(Self-check Engine), 승인 규칙은 07 §3 참조

# 9. 결정 완료(LOCK)

1) L2 저장 기본 정책(승인 필요 여부)
- LOCK: 옵션 A — L2 저장은 기본 “승인 필요”
- 해석:
  - L2(장기/글로벌) 저장은 기본적으로 비활성(또는 보류)이며,
  - 승인(Approval) 없이는 L2 커밋/영구 저장을 수행하지 않는다.
- 연계 정본:
  - 저장/보존/TTL/마스킹/인덱싱은 06(Storage/Memory) 문서를 정본으로 따른다.

2) Self-check 자동 루프 횟수(soft retry)
- LOCK: 옵션 A — 자동 1회(soft)만 허용, 이후는 승인 필요(hard)
- 해석:
  - 자동 재시도는 “품질 보정” 수준에서 1회까지만 허용한다.
  - 추가 반복/외부 재실행(hard loop)은 승인 없이는 금지한다.
- 연계 정본:
  - 승인/재실행 통제는 07(Safety/Cost/Approval)의 Approval 규칙을 따른다.

3) Evidence(QoD) 하한선
- LOCK: 옵션 A — QoD<threshold면 “결론”은 HOLD/ESCALATE만 허용
- 허용(사용성 보완):
  - 불확실성 명시, 추가 근거 요청, 재검색/재수집 제안, 다음 액션 제시는 허용한다.
  - 단, 결론을 단정/확정 형태로 내리지 않는다(HOLD 유지).
- threshold 수치(숫자) 잠금:
  - QoD 스키마/운영 설정에서 근거 기반으로만 확정(문서에서 발명 금지).

## I-2. RAG/검색 (RAG & Evidence Retrieval)

## I-5. 라우팅/결정/정책 엔진 (Routing & Decision Kernel)
### 1) 목적(WHY)
- IntentFrame + EvidencePack + Policy/Approval/Cost 조건을 종합하여 Decision을 생성하고 locked=true로 고정한다.
- 선택된 Blue Node 및 실행 모드(mini/main/tool)를 확정하며, 게이트 결과에 따라 deny/hold/fallback을 결정한다.

### 2) 입력(INPUTS)
- IntentFrame (required)
- EvidencePack (required; 없거나 insufficient면 HOLD/루프)
- policy_rules / approval_rules (required; 07 참조)
- cost_status (optional; I-8 참조)
- project_status (optional; I-7 참조)
- system_caps (optional; 사용 가능한 모델/도구/노드)

### 3) 출력(OUTPUTS)
- Decision (locked=true)
  - decision_id, trace_id, timestamp
  - intent_frame_ref, evidence_pack_ref
  - policy_gate: allow|mask|require_approval|block
  - approval_required: true|false (+ scope)
  - cost_gate: normal|downshift|split|stop
  - routing: selected_blue_node_id, execution_mode
  - memory_plan: save_candidate, target_layer, requires_user_approval
  - output_spec: format_constraints/requirements
  - conclusion: ACCEPT|REJECT|HOLD|ESCALATE
  - locked: true

### 4) 의존성(DEPENDENCIES)
- 07(정책/승인/금지/마스킹), I-8(비용), I-9(로그)
- 03/05(도메인/노드 라우팅 규정), I-2/I-19(근거/QoD)

### 5) 상태(STATE)
- I5_S0_READY → I5_S1_GATES_EVAL → I5_S2_ROUTE_SELECTED → (I5_S3_DECISION_LOCKED|I5_S4_WAIT_APPROVAL|I5_S5_DENY)

### 6) 정책(POLICY HOOKS) — 게이트 우선순위(확정)
- Policy Gate(최우선): block > require_approval > mask > allow
- Cost Gate: 예산 초과 위험이면 downshift/split/stop
- Evidence Gate: insufficient/QoD 미달이면 HOLD/ESCALATE + 재검색 루프 권장

### 7) LogEvent(event_type)
- oc.i5.gates.evaluated
- oc.i5.route.selected
- oc.i5.decision.locked
- oc.i5.approval.required
- oc.i5.cost.downshifted
- oc.i5.policy.blocked
- oc.i5.decision.failed

### 8) 실패(FAILURE POINTS) / FailureCode
- OC_I5_POLICY_BLOCK
- OC_I5_APPROVAL_REQUIRED
- OC_I5_COST_OVER_BUDGET
- OC_I5_EVIDENCE_INSUFFICIENT
- OC_I5_ROUTE_NOT_FOUND

### 9) 폴백(FALLBACKS)
- FB_REQUIRE_APPROVAL: 승인 요청 후 대기(Decision=HOLD)
- FB_COST_DOWNSHIFT: 모델/도구/범위 축소
- FB_ROUTE_SAFE_NODE: 더 안전/저비용 노드로 라우팅
- FB_DENY_WITH_REASON: 정책 block 또는 라우팅 불가 시 종료

### 10) 테스트(TEST)
- T1: 동일 입력/동일 상태에서 Decision 재현 가능(결정 안정성)
- T2: policy/cost/evidence 게이트 우선순위가 일관되는가
- T3: Decision lock 이후 결론이 변경되지 않는가

### 11) 최소 인터페이스(Interface)
- evaluate_gates(intent_frame, evidence_pack, policy_ctx, cost_ctx) -> gate_result
- select_route(gate_result, system_caps) -> routing
- lock_decision(intent_frame_ref, evidence_pack_ref, gate_result, routing) -> Decision

## 10.6 S-Module(S-1~S-8) ↔ I-Module “경유 동작” 명시 (NEW)

- 원칙: S-모듈은 “독립적으로 시스템을 바꾸는 존재”가 아니라,
  I-모듈(특히 I-6/I-14/I-19/I-21/I-15/I-9 등)에 쌓인 로그·근거·템플릿·스냅샷·품질지표를 입력으로 받아
  “후보 생성 → 거버넌스(S-8) → 승인 → 문서/자산 반영 → 회귀 테스트(S-2)” 흐름으로만 작동한다.

- 설계상 연결(고정 문장):
  1) S-1(Self-check)는 I-6(Self-check/QoD)의 실행/산출을 통해 “문제 신호”를 만든다.
  2) S-3~S-7이 만든 후보는 S-8로 전달되고, S-8은 위험·비용·정책을 1차 확인한 뒤,
     “승인된 항목만” 문서/설정/템플릿/정책 파일 갱신 + 스냅샷(I-15) + 로그(I-9) 기록을 거친다.
  3) 적용 이후 S-2가 "개선 전/후 성능 비교" 회귀 테스트를 수행한다.

### 11.1 Decision 감사/Failover/Registry 정합 (S03-D20, 8건)

#### 11.1.1 Decision Audit Trail API (S03-ADD-001)

- 목적: Decision 생성/잠금/조회의 감사 추적 API를 정의한다.
- API 인터페이스:
  - `create_decision(context, intent, evidence)` → `Decision` (불변 객체)
  - `lock_decision(decision_id, gates_result)` → `LockedDecision`
  - `query_decisions(filters, time_range)` → `List[Decision]`
  - `get_decision_chain(trace_id)` → `List[Decision]` (전체 이력)
- 모든 Decision 생성/잠금은 LogEvent에 기록 (§6.1)
- 감사 요구사항: 누가, 언제, 왜 결정을 만들었는지 재현 가능해야 함

#### 11.1.2 Multi-Brain Failover (S03-ADD-006)

- 목적: 주 Brain(GPT-4o) 장애 시 대체 Brain으로 자동 전환하는 Failover 체인.
- Failover 순서(LOCK):
  1) GPT-4o (주) → 2) Claude Sonnet (1차 대체) → 3) 로컬 Ollama (2차 대체)
- 전환 조건: 연속 3회 타임아웃 또는 HTTP 5xx 응답
- 전환 시 trace_id 유지, Failover 이벤트 LogEvent 기록
- Failover 발동 시 자동 다운시프트(I-8) 연동: 비용/성능 조건 재평가

### 11.10 DI Kernel / TEE 루프 (S06-ATS2, 20건)

#### 11.10.1 DI Kernel (의존성 주입) 설계 — CRITICAL

- 목적: ORANGE CORE의 모든 서비스를 의존성 주입(DI)으로 관리한다.
- DI 컨테이너에 등록되는 서비스:

| 서비스 | 인터페이스 | 구현체 예시 |
|--------|----------|-----------|
| BrainAdapter | `IBrain` | GPT4oBrain, ClaudeBrain, OllamaBrain |
| RAGRetriever | `IRetriever` | ChromaRetriever |
| CostTracker | `ICostTracker` | BudgetAwareCostTracker |
| GateClient | `IGateClient` | PolicyCheckClient |
| MCPClient | `IMCPClient` | StdioMCPClient |
| Logger | `ILogger` | StructuredLogger |
| MemoryStore | `IMemoryStore` | SessionMemory, ProjectMemory |

- 환경별 주입: 프로덕션(실제 서비스), 테스트(Mock 서비스), 스테이징(제한 서비스)
- 수명 관리: Singleton(BrainAdapter, Logger) / Scoped(CostTracker, per-request) / Transient(임시 객체)

#### 11.10.2 TEE 루프 CORE 연동

- CORE에서의 TEE(Think-Execute-Evaluate) 루프 호출 지점:
  - Think: I-1(Intent) + I-5(Router) → 다음 액션 결정
  - Execute: I-5(Router) → Brain/Node 호출 → 결과 수집
  - Evaluate: I-6(Self-check) → QoD 평가 → 목표 달성 여부
- CORE는 TEE 루프의 "제어자" 역할만 수행, 실행은 05(Agent Workflow)에 위임
- 연결 참조: 05(AGENT) §12.5.1 TEE 루프

### 11.11 Trigger/Action / Evaluations 품질 게이트 (S07-ATS3, 15건)

#### 11.11.1 Trigger/Action 이벤트 스키마 — CRITICAL

- CORE에서의 Trigger/Action 처리:
  - Trigger 수신 → I-1(Intent Classifier)에서 해석 → I-5(Router)에서 워크플로우 선택
  - Trigger 유형별 Gate 요구사항:
    - schedule: G1(Policy) 자동 통과 (사전 승인된 경우)
    - event: G1 + G2(Cost) 확인
    - webhook: G0(Input Validation) + G1 + G2
    - manual: 사용자 의도 확인 (I-1)
  - Trigger 이력은 §6.1 LogEvent에 기록 (event_type: `trigger.received`, `trigger.processed`)

#### 11.11.2 Evaluations 품질 게이트 — CRITICAL

- 목적: 에이전트/모델 출력의 품질을 자동 평가하는 게이트.
- 품질 평가 항목:
  - **Accuracy**: 정답 대비 정확도 (벤치마크 기반, §11.9)
  - **Faithfulness**: 근거 충실도 (EvidenceBundle 대비 답변 정합성)
  - **Relevance**: 질문 대비 답변 적합도
  - **Completeness**: 요청 대비 답변 완성도
  - **Safety**: 안전 가이드라인 위반 여부 (07 Gate)
- 평가 결과는 QoD 점수(0~1)로 통합 → I-6(Self-check) 임계값과 비교
- 연결 참조: 05(AGENT) §12.11.2 DSPy Evaluation

### 11.12 IntentFrame / 평가 프레임워크 / LM-Eval (S09-NS, 20건)

#### 11.12.1 IntentFrame Signature — CRITICAL

- 목적: 사용자 의도를 구조화된 프레임으로 분류하는 I-1(Intent Classifier) 확장.
- IntentFrame 필드:

```python
class IntentFrame:
    query: str                    # 원본 질문
    domain: Literal["trading", "coding", "research", "general"]
    intent_type: Literal["analysis", "action", "query", "chat"]
    complexity: Literal["simple", "moderate", "complex"]
    risk_level: Literal["P0", "P1", "P2"]
    required_tools: List[str]     # 예상 필요 도구
    confidence: float             # 분류 신뢰도 (0~1)
    sub_intents: List[str]        # 복합 의도 분해
```

- IntentFrame은 I-1에서 생성 → I-5(Router)에서 소비 → Decision에 기록
- 신뢰도 < 0.5 시: 사용자에게 명확화 요청 (HITL)

#### 11.12.2 평가 프레임워크 표준

- 목적: VAMOS 전체 시스템의 평가 체계를 정의한다.
- 평가 레벨:
  - L1 단위 평가: 개별 모델/프롬프트 성능 (벤치마크, §11.9)
  - L2 에이전트 평가: 에이전트 태스크 완수율 (AgentBench)
  - L3 시스템 평가: 전체 파이프라인 E2E 성능 (사용자 만족도)
  - L4 안전 평가: 안전/편향/할루시네이션 (SafetyBench, 07)
- 평가 결과는 06(STORAGE)에 저장, 트렌드 추적

#### 11.12.3 LM-Eval Harness 연동 — CRITICAL

- 목적: EleutherAI LM-Eval Harness를 VAMOS 벤치마크 파이프라인에 통합.
- 자동 실행: 모델 변경 시 LM-Eval 자동 실행 → 결과 대시보드 반영
- 커스텀 태스크: VAMOS 투자 분석 특화 벤치마크 태스크 추가
- 연결 참조: 04(INFRA) 테스트 인프라, §11.9 벤치마크 기준

#### 11.13.1 I-모듈 25개(I-1~I-25, 단일 레이어) 상세 정의

#### 11.13.3 Gate 자동화 스키마

  - 자동화 규칙: P0 태스크의 G1/G2는 사전 등록 조건 충족 시 자동 통과
