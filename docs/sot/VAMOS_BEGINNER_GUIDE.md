# 1장. VAMOS가 뭔가요?

## 1.1 한 줄 정의

**VAMOS**는 사용자의 지능적 결정을 돕는 **개인 맞춤형 AI 보조 지능**입니다.

쉽게 말하면, 여러분이 "이 주제에 대해 리서치해줘", "이 코드 디버깅해줘", "이 데이터 분석해줘"라고 요청하면, VAMOS가 여러 AI 도구와 지식을 조합하여 **정확하고 근거 있는 답변**을 만들어주는 시스템입니다. 단순한 챗봇이 아니라, 여러 AI 모델과 도구를 **지휘**하는 **지휘자(오케스트라 지휘자)** 같은 역할입니다.

궁극적으로는 **AGI(Artificial General Intelligence, 범용 인공지능)** 수준의 구조를 목표로 합니다. 이는 특정 분야만 잘하는 AI가 아니라, 다양한 분야를 범용적으로 처리할 수 있는 구조를 뜻합니다.

## 1.2 6가지 핵심 철학 (VAMOS가 지키는 원칙)

VAMOS는 다음 6가지 원칙을 항상 지킵니다:

| # | 원칙 | 쉬운 설명 |
|---|------|----------|
| 1 | **사용자 중심** | 항상 사용자에게 맞춰져 있음. "내 입맛대로" 동작 |
| 2 | **정확성/근거 기반** | "대충 맞는 것 같아요"가 아니라, 출처와 근거를 반드시 제시 |
| 3 | **최신성 확보** | 오래된 정보가 아닌, 최신 정보를 찾아서 반영 |
| 4 | **장기 기억 유지** | 지난 대화를 기억하고, 프로젝트별로 맥락 유지 |
| 5 | **다중 의도 처리** | "이것도 하고 저것도 해줘"를 동시에 이해하고 처리 |
| 6 | **모듈화 구조** | 레고 블록처럼 기능을 조립/분리 가능 |

## 1.3 14가지 목표

VAMOS가 달성하려는 목표는 크게 4개 그룹, 총 14개입니다:

### A그룹: 더 똑똑하게 (지능/정확도)
1. **정확도 개선** — 거짓 정보(환각, Hallucination)를 최소화
2. **최신 지식** — RAG(검색 증강 생성)로 최신 정보 반영
3. **장기 기억** — 이전 대화와 프로젝트 맥락을 기억
4. **대화 분석** — 여러 의도를 한 번에 파악
5. **메타인지** — "내가 지금 잘하고 있나?" 스스로 점검

### B그룹: 더 유연하게 (구조/아키텍처)
6. **모듈 구조** — 기능별로 독립된 블록으로 나눔
7. **도메인 확장** — 코딩, 리서치, 트레이딩 등 분야별 확장 가능
8. **다중 AI 활용** — 하나의 AI가 아닌, 여러 AI를 조합 활용
9. **복합 작업** — 여러 분야를 넘나드는 복잡한 작업 처리

### C그룹: 더 생산적으로 (실용)
10. **코딩 능력** — 코드 작성/디버깅/리팩토링 지원
11. **리서치/분석** — 문서 검색, 데이터 분석 자동화
12. **생산성/수입** — 일상 업무 효율화, 부수입 창출 지원

### D그룹: 더 안전하게 (안전/진화)
13. **자기진화** — 스스로 문제를 진단하고 개선안 제안
14. **안전/비용** — 비용 한도 내에서 안전하게 운영

## 1.4 절대 하지 않는 것 (Non-goal, 7가지 금지사항)

VAMOS는 아무리 사용자가 요청해도 **절대** 하지 않는 것들이 있습니다:

| # | 금지 사항 | 이유 |
|---|----------|------|
| 1 | **실제 주문/계좌 연동** | 실거래 금지. 분석 보조만 가능 |
| 2 | **불법/해킹** | 즉시 차단. 법적 경고 |
| 3 | **의료/법률 단정** | "이 약 드세요"처럼 단정 금지. "전문가 상담 필요" 안내만 |
| 4 | **개인정보 장기 저장** | 주민번호 같은 민감 정보는 절대 장기 보관 안 함 |
| 5 | **저작권 위반** | 불법 다운로드 등 거부 |
| 6 | **위험 도메인 자동 생성** | P2(고위험) 도메인은 반드시 사용자 승인 후만 |
| 7 | **위험 기능 자동 실행** | 사람의 확인 없이 위험한 작업 자동 실행 불가 |

## 1.5 성공의 우선순위

VAMOS가 여러 가치를 저울질할 때 이 순서를 따릅니다:

```
정확성/근거 > 이해도/품질 > 안전/규제 > 속도/효율 > 비용
```

즉, **비용을 아끼려고 정확성을 희생하지 않습니다**. 반대로, 정확성을 위해 비용이 더 들 수 있습니다.

# 2장. 큰 그림: VAMOS의 전체 구조

## 2.1 비유로 이해하기

VAMOS를 **병원**에 비유하면 이해가 쉽습니다:

```
[접수처]       → Front Mini LLM (프론트 미니 LLM)
                  "무슨 증상이세요?" — 요청을 처음 받고 분류

[원장실]       → ORANGE CORE (오렌지 코어)
                  "이 환자는 내과로, 이 검사부터" — 판단/지시/관리

[진료과]       → BLUE NODES (블루 노드)
                  "내과", "외과", "정형외과" — 분야별 전문 진료

[검사실/약국]  → OTHER BRAINS (아더 브레인즈)
                  "혈액검사", "MRI", "처방" — 실제 도구 실행

[진료 결과지]  → Main/Hologram LLM (메인/홀로그램 LLM)
                  최종 결과를 정리하여 환자에게 전달
```

## 2.2 실제 처리 흐름 (4계층)

사용자가 무언가를 요청하면, 이런 순서로 처리됩니다:

```
사용자 입력
  ↓
[1층] Front Mini LLM ─── 의도 파악, 도메인 판별, 안전 필터링
  ↓
[2층] ORANGE CORE ────── 정책 확인, 비용 계산, 라우팅 결정, 승인 관리
  ↓
[3층] BLUE NODE ──────── 해당 분야 전문 처리 (코딩/리서치/트레이딩 등)
  ↓
[4층] OTHER BRAINS ───── 실제 도구 실행 (웹검색, 코드실행, DB조회 등)
  ↓
최종 결과 → 사용자에게 전달
```

### 각 계층의 역할 상세

| 계층 | 이름 | 하는 일 | 비유 |
|------|------|---------|------|
| **1층** | Front Mini LLM | 사용자 말을 듣고 "뭘 하려는 건지" 파악. 위험한 요청은 여기서 차단 | 접수처 직원 |
| **2층** | ORANGE CORE | 정책 확인, 비용 계산, "어떤 팀이 처리할지" 결정, 승인 관리 | 원장/관리자 |
| **3층** | BLUE NODE | 분야별 전문 처리. 코딩이면 코딩 전문 노드, 리서치면 리서치 전문 노드 | 전문의/진료과 |
| **4층** | OTHER BRAINS | 검색엔진, 코드 실행기, 데이터베이스 등 실제 도구들 | 검사장비/약국 |

## 2.3 기능 분류 체계 (I/E/S/A 시리즈)

VAMOS의 모든 기능은 8개 시리즈로 분류됩니다:

### I-시리즈 (Internal, 내부 기능) — I-1 ~ I-25

"VAMOS 내부에서 스스로 하는 일"입니다. 사용자에게는 보이지 않지만 핵심입니다.

| ID | 이름 | 쉬운 설명 | 비유 |
|----|------|----------|------|
| **I-1** | Intent Detector | 사용자 말의 의도를 파악 | "뭘 원하시는 거죠?" |
| **I-2** | Context Builder (RAG) | 관련 지식/정보를 검색하여 수집 | "관련 자료 찾아볼게요" |
| **I-3** | Memory System | 기억 저장/관리 (세션, 프로젝트, 장기) | "이전 대화 기억나요" |
| **I-4** | Multimodal Interpreter | 텍스트뿐 아니라 이미지/파일도 이해 | "사진도 볼 수 있어요" |
| **I-5** | Decision Engine | 최종 판단 (어떤 팀이, 어떤 도구로, 어떻게 처리할지) | "최종 결정합니다" |
| **I-6** | Self-check Engine | 결과가 맞는지 스스로 검증 | "다시 한번 확인할게요" |
| **I-7** | Project Manager | 프로젝트/세션 관리 | "어떤 프로젝트 작업 중이죠?" |
| **I-8** | Policy Engine | 정책/규칙 확인 | "이건 규칙에 맞나요?" |
| **I-9** | Cost Manager | 비용 추적 및 관리 | "예산 얼마 남았나요?" |
| **I-10** | Tool Registry | 사용 가능한 도구 목록 관리 | "어떤 장비가 있나요?" |
| **I-11** | Output Composer | 결과를 보기 좋게 정리 | "보고서로 만들어드릴게요" |
| **I-12** | Workflow Builder | 복잡한 작업의 절차 설계 | "순서를 정해볼게요" |
| **I-13** | Multimodal Renderer | 텍스트+이미지+차트 등 다양한 형태로 출력 | "그래프도 그려드릴게요" |
| **I-14** | Summarizer | 긴 내용을 요약하여 메모리에 저장 | "핵심만 정리할게요" |
| **I-15** | Evidence Manager | 근거의 품질(QoD) 관리 | "이 출처 신뢰할 수 있나요?" |
| **I-16** | Knowledge Search | 지식 검색 엔진 | "DB에서 찾아볼게요" |
| **I-17** | Blue Node Manager | 블루 노드들의 관리자 | "어느 팀이 한가하지?" |
| **I-18** | Self-evo Engine | 자기진화 엔진 (V3에서 활성) | "어떻게 하면 더 잘할 수 있을까?" |
| **I-19** | Approval Manager | 승인 절차 관리 | "사용자 확인 받았나요?" |
| **I-20** | Failure Manager | 오류 발생 시 대응 관리 | "실패했을 때 어떻게 하지?" |
| **I-21** | Source Evolution | 지식 소스 진화/최적화 (V3) | "소스를 더 정확하게 진화!" |
| **I-22** | Task/Project Manager | 작업/프로젝트 관리 (V2) | "프로젝트 전체를 관리" |
| **I-23** | Doc/Code Structuring | 문서/코드 구조화 (V2) | "체계적으로 정리해드릴게요" |
| **I-24** | Knowledge Graph | 지식 그래프 엔진 (V3에서 활성) | "연관 관계를 그물처럼 연결" |
| **I-25** | SDAR Engine | 오류 자가진단 & 자동복구 (V2부터 활성) | "아프면 스스로 치료" |

"VAMOS가 외부 세계와 소통하는 도구"입니다. 실제로 검색하고, 코드를 실행하고, 파일을 읽는 등의 작업을 합니다.

| **E-3** | Document Parser | 문서(PDF/Word 등) 읽기 | O |
| **E-4** | Code Executor | 코드 실제 실행 (샌드박스) | O |
| **E-5** | Image Analyzer | 이미지 분석 | O |
| **E-6** | Z3 Solver | 수학적 논리 검증 도구 | O |
| **E-7** | Speech-to-Text | 음성 → 텍스트 변환 | V3 |
| **E-8** | Text-to-Speech | 텍스트 → 음성 변환 | V3 |
| **E-9** | Video Analyzer | 영상 분석 | V3 |
| **E-10** | API Gateway | 외부 서비스 연동 | V3 |

| **E-16** | Cloud Storage | 클라우드 저장소 동기화 | V2 조건부 |

### S-시리즈 (Self-evolution, 자기진화) — S-1 ~ S-8

"VAMOS가 스스로 진화하는 기능"입니다. 대부분 V3(최종 버전)에서 활성화됩니다.

| ID | 이름 | 쉬운 설명 |
|----|------|----------|
| **S-1** | Self-check Engine | 스스로 결과를 검증 (유일하게 V1부터 활성) |
| **S-2** | Benchmark QA | 성능 벤치마크 테스트 |
| **S-3** | Template Evolution | 작업 템플릿 자동 개선 |
| **S-4** | Error Pattern Miner | 반복되는 오류 패턴 발견 |
| **S-5** | Router Evolution | 라우팅(경로 선택) 최적화 |
| **S-6** | Search Evolution | 검색 품질 자동 개선 |
| **S-7** | User-Coop Designer | 사용자 협력 패턴 학습 |
| **S-8** | Self-evo Governance | 자기진화의 관리자 (안전 확인) |

### A-시리즈 (Architecture, 아키텍처 확장) — A-1 ~ A-7

| ID | 이름 | 쉬운 설명 |
|----|------|----------|
| **A-1** | MultiBrain Adapter | 여러 AI 모델을 하나로 연결하는 어댑터 |
| **A-2** | Preset Modularization | 설정을 모듈화하여 쉽게 교체 |

| **A-4** | Debate Mode | AI끼리 토론하여 결론 도출 (V2 조건부) |
| **A-5** | Lazy Generation | 필요할 때만 생성하여 비용 절약 |
| **A-6** | Federated Module | 분산 모듈 네트워크 |
| **A-7** | Remote Executor | 원격 실행 환경 |

# 3장. 사용자가 질문하면 무슨 일이 벌어지나요?

이 장이 VAMOS의 **핵심**입니다. 사용자가 "삼성전자 최근 실적 분석해줘"라고 입력했을 때, VAMOS 내부에서 일어나는 일을 **처음부터 끝까지** 따라가 봅시다.

## 3.1 상태 머신 (State Machine) — 9단계 진행 표시

VAMOS는 모든 요청을 **9단계 상태**로 추적합니다. 택배 추적처럼, 지금 어디까지 진행되었는지 알 수 있습니다.

```
S0_RECEIVED ──→ S1_INTENT_PARSED ──→ S2_EVIDENCE_READY ──→ S3_DECISION_LOCKED
  (접수됨)        (의도 파악 완료)      (근거 수집 완료)       (판단 확정)
                                                                  ↓
S8_DONE ←── S7_MEMORY_COMMITTED ←── S6_SELF_CHECKED ←── S5_OUTPUT_READY ←── S4_EXECUTING
(완료!)      (기억에 저장)           (자체 검증 완료)     (결과 생성 완료)     (실행 중)
```

각 단계를 삼성전자 예시로 따라가 보겠습니다:

---

### S0_RECEIVED (접수됨)

사용자가 "삼성전자 최근 실적 분석해줘"를 입력합니다.

**Front Mini LLM**이 이 입력을 처음 받습니다. Front Mini는 작고 빠른 AI 모델로, 다음을 즉시 판단합니다:
- **안전한 요청인가?** → Yes (위험하지 않음)
- **어떤 분야인가?** → Research (리서치)
- **위험한 내용이 포함되어 있나?** → No

---

### S1_INTENT_PARSED (의도 파악 완료) — I-1 담당

**I-1 Intent Detector**가 사용자의 말을 구조화합니다.

**IntentFrame**(의도 프레임)이라는 구조체를 만듭니다:

```
IntentFrame:
  user_goal: "삼성전자 최근 실적 분석"
  task_type: "research" (리서치)
  domain_hint: "P0_Research" (기본 활성 도메인)
  constraints:
    format: "분석 보고서"
    must_include: ["매출", "영업이익", "전망"]
  risk_flags:
    safety_sensitive: false (안전 이슈 없음)
    cost_sensitive: false (비용 걱정 없음)
  complexity: "moderate" (중간 복잡도)
  confidence: 0.92 (92% 확신)
```

이 과정에서 사용하는 정책 Hooks(훅, 자동 체크 기능):
- **PC_SAFETY_CLASSIFY**: 안전 등급 분류 ("위험한 요청인가?")
- **PC_P2_DOMAIN_HINT**: P2(고위험) 도메인 관련인지 확인

**만약 의도가 모호하다면?**
예를 들어 "그거 분석해줘"처럼 모호하면:
1. 상태가 `I1_S2_AMBIGUOUS`(모호함)로 전환
2. **FB_ASK_CLARIFICATION** 폴백(대안 행동)이 발동
3. VAMOS가 "어떤 기업을 말씀하시는 건가요?" 같은 질문을 1~3개 생성
4. 사용자 답변 후 다시 진행

---

### S2_EVIDENCE_READY (근거 수집 완료) — I-2 담당

**I-2 Context Builder (RAG)**가 관련 정보를 수집합니다.

**RAG**란? **Retrieval-Augmented Generation**의 약자로, "검색해서 가져온 정보를 바탕으로 답변을 생성"하는 기술입니다. AI가 자기 머릿속 지식만으로 답하는 게 아니라, 실제로 정보를 검색해서 가져옵니다.

이 단계에서 일어나는 일:
1. **검색 쿼리 생성**: "삼성전자 2025년 실적", "삼성전자 매출 추이" 등
2. **다중 소스 검색**: 웹 검색, 내부 메모리, 이전 대화 기록
3. **품질 평가**: 찾은 정보마다 **QoD 점수** 매기기

**QoD (Quality of Data)란?** 정보의 품질을 0~1 사이 점수로 매기는 것입니다.

```
QoD = 정확성(0.30) × 관련성(0.25) × 완전성(0.20) × 안전성(0.15) × 효율성(0.10)
```

예시:
- 삼성전자 공식 IR자료 → QoD: 0.92 (매우 높음)
- 3년 전 블로그 글 → QoD: 0.35 (낮음, 사용 안 함)
- StackOverflow 답변 → QoD: 0.50 (보통)

**출처별 기본 가중치:**
| 출처 | 기본 점수 |
|------|----------|
| 공식 문서 (API Doc, RFC) | 0.85 |
| 내부 KB / 팀 문서 | 0.70 |
| 커뮤니티 (StackOverflow) | 0.50 |
| 출처 불명 | 0.30 |

수집 결과는 **EvidencePack**(근거 묶음)이라는 구조체에 담깁니다:

```
EvidencePack:
  items:
    - source: "삼성전자 IR" / qod: 0.92
    - source: "금융감독원 DART" / qod: 0.88
    - source: "이전 대화 메모리" / qod: 0.75
  coverage:
    sufficient: true (충분한 근거 확보)
    gaps: [] (빠진 부분 없음)
```

**만약 근거가 부족하다면?**
- `FB_RAG_RETRY_EXPAND`: 검색 범위를 넓혀서 재검색 (최대 2~3회)
- `FB_RAG_SWITCH_SOURCE`: 다른 출처로 전환
- 그래도 부족하면 → 사용자에게 "정보가 부족합니다" 안내

### RAG가 필요 없는 경우 [REF:영상7]

> 문서가 적으면 RAG 없이 통째로 올리는 것이 최선이다.

| 상황 | 추천 방식 | 이유 |
|------|----------|------|
| 문서 < 5개, 각 10K 토큰 이하 | 전체 컨텍스트 전달 | RAG 오버헤드 > 이점 |
| 문서 5~20개 | 선택적 RAG | 관련 문서만 검색 후 전달 |
| 문서 20개+ 또는 10K+ 토큰 | 필수 RAG | 컨텍스트 윈도우 초과 방지 |
| 정형 데이터 (DB, 스프레드시트) | SQL/API 직접 조회 | RAG보다 정확 |

- **VAMOS 자동 판단**: I-10 Router가 입력 크기에 따라 RAG 사용 여부를 자동 결정
- **사용자 오버라이드**: `/search on|off` 명령으로 강제 전환 가능

### 대용량 파일 처리 패턴 [REF:영상5]

> 저렴한 모델로 먼저 요약 후 메인 AI에 전달하는 "전처리 요약" 패턴

```
[사용자 파일 업로드]
    ↓
[크기 판단]
    ├── < 10K 토큰: 직접 전달 (비용 최소)
    ├── 10K~200K 토큰: 전처리 요약 모델 → 요약본 전달
    └── > 200K 토큰: 분할 + 개별 요약 → 통합 요약 전달

전처리 모델: Gemini Flash ($0.10/1M토큰) 또는 GPT-4o mini
메인 모델: Claude Opus/Sonnet 또는 GPT-4o
비용 절감: 최대 67% (STEP7 F-I 벤치마크 기준)
```

---

### S3_DECISION_LOCKED (판단 확정) — I-5 담당

**I-5 Routing & Decision Kernel**이 최종 판단을 내립니다. 이것이 VAMOS의 **가장 핵심적인 순간**입니다.

이 단계에서 **5개의 Gate(관문)**을 순서대로 통과해야 합니다:

```
┌─────────────────────────────────────────────────────────┐
│                    5개의 Gate (관문)                       │
│                                                           │
│  [1] PolicyGate ──→ "정책상 허용되나?" (최우선)             │
│         ↓                                                 │
│  [2] CostGate ───→ "예산 안에서 가능한가?"                  │
│         ↓                                                 │
│  [3] ApprovalGate → "사용자 승인이 필요한가?"                │
│         ↓                                                 │
│  [4] EvidenceGate → "충분한 근거가 있는가?"                  │
│         ↓                                                 │
│  [5] SelfCheckGate → (나중에 S6에서 실행)                   │
│                                                           │
│  모든 Gate 통과 → Decision 생성 → locked = true            │
└─────────────────────────────────────────────────────────┘
```

**Gate(게이트)란?** 각 단계에서 통과해야 하는 "검문소"입니다. 하나라도 통과 못 하면 작업이 중단됩니다.

각 Gate의 동작:

| Gate | 질문 | 통과 못 하면 |
|------|------|------------|
| **PolicyGate** | 이 요청이 정책에 어긋나지 않는가? | block(차단) / require_approval(승인 요구) / mask(일부 가림) |
| **CostGate** | 예산 안에서 처리 가능한가? | downshift(저비용 모델로 전환) / split(분할) / stop(중지) |
| **ApprovalGate** | 사용자 승인이 필요한 작업인가? | 승인 요청 후 대기 |
| **EvidenceGate** | 근거가 충분한가? | HOLD(보류) / 재검색 루프 |
| **SelfCheckGate** | 결과가 정확한가? (S6에서 실행) | 1회 보정 후 재시도 |

모든 Gate를 통과하면 **Decision**(판단서)이 생성됩니다:

```
Decision:
  decision_id: "dec_abc123"
  conclusion: "ACCEPT" (수락)
  routing:
    selected_blue_node: "research_node_v1" (리서치 블루 노드)
    execution_mode: "main" (메인 모델 사용)
  cost_gate: "normal" (비용 정상)
  locked: true  ← 이후 변경 불가! (단일결정 원칙)
```

**단일결정 원칙 (LOCK)**: Decision이 locked=true가 되면, **이후 결론 변경이 불가**합니다. 이는 AI가 중간에 마음을 바꿔서 혼란을 주는 것을 방지합니다.

---

### S4_EXECUTING (실행 중) — BLUE NODE 담당

Decision에서 지정된 **Blue Node**가 실행됩니다.

이 예시에서는 **Research Blue Node**(리서치 블루 노드)가 활성화됩니다. Blue Node는 EvidencePack과 Decision을 받아서, OTHER BRAINS(도구들)를 활용하여 실제 작업을 수행합니다.

예시 실행 과정:
1. E-2(웹 검색)로 최신 실적 데이터 수집
2. E-3(문서 파서)로 IR 보고서 분석
3. I-2(RAG)로 이전 분석 대비 변화 확인
4. 수집 데이터 종합하여 분석 수행

---

### S5_OUTPUT_READY (결과 생성 완료) — I-4 담당

**I-4 Output Structuring**이 실행 결과를 사용자가 원하는 형태로 정리합니다.

출력 결과는 항상 **3단 구조**로 제공됩니다:

```
[1단] user_response (최종 결과)
  "삼성전자 2025년 실적 분석:
   매출: XX조, 전년 대비 +XX%
   영업이익: XX조, ..."

[2단] evidence_summary (근거 요약)
  "출처1: 삼성전자 IR자료 (QoD: 0.92)
   출처2: DART 공시 (QoD: 0.88)"

[3단] log_report (내부 로그)
  "trace_id: xxx, 소요시간: 4.2초, 사용 비용: ₩15"
```

---

### S6_SELF_CHECKED (자체 검증 완료) — I-6 담당

**I-6 Self-check Engine**이 결과를 스스로 검증합니다.

검증 기준 (**LOCK**, 변경 불가):
- **P0 작업** (기본): self_check_score ≥ 70점이면 PASS
- **P1 작업** (확장): ≥ 75점이면 PASS
- **P2 작업** (고위험): ≥ 80점이면 PASS

```
Self-check 결과:
  score: 0.85 (85점)
  verdict: "PASS" ✅
  reasons: ["출처 3개 교차 확인됨", "최신 데이터(2025Q4) 반영"]
```

**만약 FAIL이면?**
- **Soft Loop**: 자동으로 1회 보정 후 재시도 (자동, 1회만)
- **Hard Loop**: 이전 단계로 되돌아가 재실행 (사용자 승인 필요)
- **위험/정책 위반**: 루프 없이 즉시 거부

---

### S7_MEMORY_COMMITTED (기억에 저장) — I-3 담당

**I-3 Memory System**이 이번 작업 결과를 기억에 저장합니다.

저장 계획:
```
MemoryCommitRequest:
  target_layer: "L1" (프로젝트 메모리)
  commit_mode: "summary" (요약만 저장)
  requires_user_approval: false (L1은 자동 저장 가능)
```

(메모리 시스템 상세는 5장에서 설명합니다)

---

### S8_DONE (완료!)

모든 처리가 끝나고, 사용자에게 최종 결과가 전달됩니다.

전체 과정을 **ResponseEnvelope**(응답 봉투)에 담아 전달합니다:

```
ResponseEnvelope:
  answer:
    summary: "삼성전자 2025년 실적..."
    details: (상세 분석 내용)
    next_actions: ["분기별 추세 비교", "경쟁사 대비 분석"]
  evidence:
    coverage: 0.92
    qod: 0.88
  self_check:
    score: 0.85
    verdict: "PASS"
  decision_ref:
    decision_id: "dec_abc123"
    gates: {policy: "allow", cost: "normal", evidence: "sufficient"}
```

## 3.2 각 단계별 소요 시간 목표

| 구간 | 시간 목표 |
|------|----------|
| 단순 응답 (mini 모델) | ≤ 2초 |
| 복합 응답 (main + 도구 1회) | ≤ 10초 |
| Self-check 소요 | ≤ 1초 |
| 동시 처리 가능 요청 | 5개 |

## 3.3 5단계 Pipeline (워크플로우 관점)

위의 9단계 상태를 **작업 단계** 관점으로 보면 5단계로 정리됩니다:

```
[Intake]  →  [Plan]  →  [Execute]  →  [Verify]  →  [Deliver]
 요청 수집     계획       실행          검증          전달
```

| 단계 | 하는 일 | Gate 검사 |
|------|---------|----------|
| **Intake** | 요청/제약 수집 + 입력 검증 | G0 (입력 검증) |
| **Plan** | 작업 분해 + 라우팅 + 비용 계산 | G1 (정책), G2 (비용) |
| **Execute** | 도구/노드 실행 | (07 Gate 선행) |
| **Verify** | 품질 검사 + Self-check | G3 (품질) |
| **Deliver** | 결과 전달 + 로그 + 메모리 저장 | G4 (최종 승인) |

## 3.4 TEE 루프 (Think-Execute-Evaluate)

각 단계 내부에서는 **TEE 루프**라는 사고 과정이 반복됩니다:

```
Think(생각) → Execute(실행) → Evaluate(평가)
     ↑                              ↓
     └──── 미달 시 다시 생각 ←──────┘
```

1. **Think**: "다음에 뭘 해야 하지?" → Gate 확인, 비용 예측
2. **Execute**: 결정된 행동 실행
3. **Evaluate**: "결과가 괜찮나?" → Self-check

**TEE 최대 반복 횟수** (무한 루프 방지):
- P0 작업: 최대 3회
- P1 작업: 최대 5회
- P2 작업: 최대 10회 (승인 시 확장 가능)

## 3.5 오류가 발생하면? (Failure & Fallback 시스템)

VAMOS는 모든 가능한 오류에 대해 **미리 정해진 대응책**이 있습니다.

### 주요 오류 코드와 대응

| 오류 (Failure) | 의미 | 대응 (Fallback) |
|---------------|------|----------------|
| `OC_I1_PARSE_FAIL` | 의도 파악 실패 | `FB_INTENT_HEURISTIC_PARSE` (경험적 파싱 시도) |
| `OC_I1_AMBIGUOUS` | 의도가 모호함 | `FB_ASK_CLARIFICATION` (사용자에게 질문 1~3개) |
| `OC_I2_RAG_NO_SOURCE` | 검색 결과 없음 | `FB_RAG_RETRY_EXPAND` (검색 범위 확장, 최대 2~3회) |
| `OC_I2_EVIDENCE_QOD_LOW` | 정보 품질 낮음 | `FB_RAG_RETRY_EXPAND` (재검색) |
| `OC_I5_COST_OVER_BUDGET` | 예산 초과 | `FB_COST_DOWNSHIFT` (저비용 모델로 전환) |
| `OC_I5_POLICY_BLOCK` | 정책 위반 | `FB_DENY_WITH_REASON` (사유와 함께 거부) |
| `TOOL_TIMEOUT` | 도구 실행 시간 초과 | `FB_RAG_RETRY_EXPAND` (재시도) |

### Circuit Breaker (회로 차단기)

실패가 반복되면 **Circuit Breaker**가 작동합니다. 이것은 전기 회로의 차단기처럼, 문제가 계속되면 자동으로 "차단"하여 시스템을 보호합니다.

```
CLOSED (정상)          3회 연속 실패         OPEN (차단)
  ● 호출 허용    ──────────────────→    ✗ 호출 차단
                                              ↓ 60초 경과
                  성공하면 CLOSED로      HALF-OPEN (시험)
                 ←─────────────────    ○ 1건만 시도
```

| 상태 | 의미 |
|------|------|
| **CLOSED** | 정상 운영. 호출 허용 |
| **OPEN** | 3회 연속 실패 → 자동 차단. 60초간 호출 금지 |
| **HALF-OPEN** | 60초 후 1건만 시험적으로 호출. 성공 시 CLOSED로 복귀 |

⚠️ P2(고위험) 작업은: OPEN 상태에서 자동 복구 없이, **사용자 승인 후에만** HALF-OPEN 가능

# 4장. ORANGE CORE 상세 — 시스템의 두뇌

ORANGE CORE는 VAMOS의 **중앙 지휘부**입니다. 3장에서 설명한 I-1~I-5가 모두 여기에 속합니다.

## 4.1 핵심 스키마 (데이터 구조) 요약

ORANGE CORE가 사용하는 주요 데이터 구조 3가지:

### IntentFrame (의도 프레임)
사용자의 요청을 구조화한 것. I-1이 생성합니다.
- `user_goal`: 사용자가 원하는 것
- `task_type`: 작업 유형 (explain, plan, code, research 등)
- `domain_hint`: 어떤 분야인지 (P0/P1/P2)
- `confidence`: AI가 "내가 이해한 게 맞나?" 확신도 (0~1). 0.5 미만이면 사람에게 물어봄

### EvidencePack (근거 묶음)
수집된 정보를 담은 것. I-2가 생성합니다.
- `items[]`: 각 근거의 출처, 요약, QoD 점수
- `coverage.sufficient`: 충분한가 여부
- `coverage.gaps[]`: 부족한 부분 목록

### Decision (판단서)
최종 판단 결과. I-5가 생성합니다. **16개 필드, FREEZE(동결)**로 구조 변경 불가.
- `conclusion`: ACCEPT / REJECT / HOLD / ESCALATE
- `locked`: true (변경 불가)
- `routing.selected_blue_node_id`: 어떤 Blue Node가 처리할지
- `cost_gate`: normal / downshift / split / stop

## 4.2 파이프라인 명칭 매핑표

같은 과정을 문서마다 다르게 부르기도 합니다. 혼동을 피하기 위한 매핑표:

| ORANGE CORE 용어 | 워크플로우 용어 | UI 표시 | 상태 코드 |
|-----------------|------------|--------|---------|
| Perception(인지) | Intake(접수) | RECEIVED→INTENT | S0→S1 |
| Reasoning(추론) | Plan(계획) | EVIDENCE→DECISION | S2→S3 |
| Action(실행) | Execute(실행) | EXEC | S4→S5 |
| Memory(기억) | Store(저장) | MEMORY | S7 |
| Reflection(성찰) | Verify(검증) | SELF-CHECK→DONE | S6→S8 |

## 4.3 B-Series ↔ L-Series 메모리 매핑표 (CC-009 — LOCK 변경 불가)

메모리 **B-시리즈**(기억 유형)와 저장 **L-시리즈**(계층)는 번호가 1:1 순서로 대응하지 **않고 비직관적으로 교차**합니다. 혼동을 피하기 위한 정본 매핑(READINESS §6.10):

| B-Series (메모리 유형) | L-Series (저장 계층) |
|----------------------|--------------------|
| B-1 Episodic(일화) | L1 |
| B-2 Procedural(절차) | L3 |
| B-3 Semantic(의미) | L2 |
| B-4 Working(작업) | L0 |

> ⚠️ 이 교차 매핑은 **LOCK(변경 불가)**입니다 — B 번호 ≠ L 번호 순서. (예: B-2 절차기억 → L3, B-4 작업기억 → L0)

# 5장. BLUE NODES 상세 — 분야별 전문가

## 5.1 Blue Node란?

Blue Node는 **특정 분야의 전문가 모듈**입니다. 사람으로 치면 "전문의"에 해당합니다.

핵심 규칙 (**LOCK**):
- ORANGE CORE의 규칙을 반드시 상속 (독립 실행 불가)
- Non-goal/안전/비용/승인 구조 변경 불가

## 5.2 도메인 분류: P0 / P1 / P2

모든 Blue Node는 위험도에 따라 3등급으로 분류됩니다:

### P0 — 기본 활성 (항상 켜져 있음)
안전한 핵심 기능들로, 별도 승인 없이 사용 가능합니다.
- **Dev/System**: 코드 작성, 디버깅, 시스템 설계
- **Research**: 정보 검색, 분석, 보고서 작성
- **Productivity**: 일반 생산성 도구

### P1 — 승인 후 활성 (요청하면 활성화)
약간의 위험이 있어 승인이 필요한 기능들입니다.
- **Content**: 콘텐츠 제작
- **Data & Quant**: 데이터 분석, 정량 분석

승인 방식 2가지 (**LOCK**):
- **A 채널**: CORE가 "이 기능 활성화하면 좋겠습니다"라고 제안
- **B 채널**: 사용자가 직접 "이 기능 켜줘"라고 요청

### P2 — 세션별 승인 (매번 확인, 위험)
높은 위험도가 있어 매 세션마다 승인이 필요합니다.
- **Trading Strategy**: 투자 전략 분석 (실거래는 절대 금지!)

P2의 핵심 규칙:
- **세션 종료 시 자동 OFF** (LOCK) — 다음 세션에서는 다시 승인 필요
- 활성화 시 **모달(팝업) 확인창**으로 재확인 (DEC-011)

## 5.3 MCP (Model Context Protocol)

Blue Node가 외부 도구를 사용할 때 **MCP**라는 프로토콜을 사용합니다.

**MCP란?** AI 모델이 외부 도구(검색엔진, 코드 실행기, DB 등)와 통신하는 표준화된 방법입니다. USB가 다양한 기기를 컴퓨터에 연결하듯, MCP는 다양한 도구를 AI에 연결합니다.

주요 설정 (**LOCK**):
- **전송 방식**: Streamable HTTP (DEC-017 확정)
- **보안**: 모든 연결 TLS(암호화) 필수
- **타임아웃**: 도구 호출 30초, 스트리밍 120초
- **재시도**: 일시적 실패 시 최대 2회

사용 가능한 MCP 서버 목록:

| 서버 | 기능 | 위험도 | 사용 가능 |
|------|------|--------|----------|
| mcp.search.tavily | 웹 검색 | 중 | V1부터 |
| mcp.code.e2b | 코드 샌드박스 | 높음 | V1부터 |
| mcp.doc.pymupdf | PDF 파싱 | 낮음 | V1부터 |
| mcp.browser.playwright | 브라우저 자동 제어 | 높음 | V1부터 |
| mcp.vision.clip | 이미지 분석 | 낮음 | V2부터 |

## 5.4 TemplateSet (작업 템플릿) — 3종

Blue Node가 작업할 때 사용하는 기본 틀입니다:

1. **TS_CORE**: 일반 대화, 요약, 정리
2. **TS_WEB_RESEARCH**: 웹 리서치, 근거 수집, 인용
3. **TS_CODE**: 코딩, 디버깅, 리팩토링

# 6장. INFRA CORE — 보이지 않는 기반 시설

## 6.1 OTHER BRAINS (실행 도구들)

OTHER BRAINS는 BLUE NODE가 사용하는 **실제 도구**들입니다.

| 종류 | 예시 | 필요한 Gate |
|------|------|-----------|
| `llm.text` | GPT-4o, Claude (텍스트 AI) | policy, cost |
| `llm.vision` | 이미지 인식 AI | policy, cost |
| `browser.render` | 웹 브라우저 자동 조작 | policy, cost, approval(고위험) |
| `api.http` | 외부 API 호출 | policy, cost |
| `code.exec` | 코드 실행 (샌드박스 안에서) | policy, cost |
| `data.vector` | 벡터 DB (의미 기반 검색) | cost |
| `storage.obj` | 파일 저장 | policy(PII), cost |

## 6.2 동적 모델 라우팅

VAMOS는 **하나의 AI 모델만 쓰는 게 아닙니다**. 상황에 따라 가장 적합한 모델을 자동 선택합니다.

선택 과정:
1. 입력 분류: 복잡도, 도메인, 예상 토큰 수
2. 후보 선택: 사용 가능한 AI 모델 목록에서 필터링
3. 비용/품질 비교: 가격 대비 품질이 좋은 모델 선택
4. Gate 확인: 정책과 예산 확인
5. 실행 & 실패 시 교체: 최대 2회 교체 가능

**Fallback Chain (대체 모델 순서)**:
```
gpt-4o → claude-sonnet → local-ollama → error_response
```
첫 번째 모델이 실패하면 다음 모델로 자동 전환됩니다 (최대 3단계).

**Multi-Brain Failover**: 연속 3회 타임아웃 시 다음 모델로 전환 (LOCK)

## 6.3 Prompt Cache Manager (프롬프트 캐시)

같은 질문을 반복하면 비용이 낭비되므로, **캐시(임시 저장)**를 사용합니다.

- **Anthropic API**: 캐시 사용 시 **90% 할인**
- **OpenAI API**: 1024 토큰 이상 prefix 자동 캐싱 **50% 할인**
- **응답 캐시**: L0(인메모리, 10분) → L1(Redis, 1시간) → 미스 시 실제 호출

## 6.4 Rate Limit (사용량 제한)

시스템을 보호하기 위한 사용량 제한:

| 항목 | 수치 |
|------|------|
| 일일 토큰 (V1) | 100K 토큰 |
| API 호출 | 분당 60회 |
| Burst Protection | 10초에 10회 초과 → 1분 쿨다운 |
| 병렬 실행 상한 | **3개** (LOCK) |

## 6.5 10계층 아키텍처

VAMOS 시스템 전체를 건물의 층으로 보면:

```
10층  UI/UX ────────────── 사용자가 보는 화면
 9층  Workflow ─────────── 작업 흐름 관리
 8층  Decision Engine ──── 판단/게이트/승인
 7층  Memory/Storage ───── 기억 + 벡터 저장
 6층  RAG Pipeline ─────── 검색 증강 생성
 5층  Brain Routing ────── 모델 선택/라우팅
 4층  Brain Adapter ────── AI 모델 연결 (추상화 계층)
 3층  Tool Execution ───── 외부 도구 실행
 2층  Infra Runtime ────── Docker/DB/로그
 1층  Config/Security ──── 설정 + 보안 정책
```

**규칙**: 호출은 위에서 아래로만 가능합니다 (10층→1층). 아래에서 위로 직접 호출하는 것은 금지이며, 이벤트 통지만 가능합니다.

## 6.6 Backup/Recovery (백업/복구)

- **자동 백업**: 매일 새벽 2시, 로컬 `~/.vamos/backup/`에 저장
- **보존 기간**: 7일분 + 4주분 + 3월분
- **복원 명령**: `vamos restore --date 2026-02-20`

# 7장. STORAGE & MEMORY — 기억 시스템

VAMOS는 사람처럼 **여러 종류의 기억**을 가지고 있습니다.

## 7.1 4계층 메모리 (저장 위치별)

| 계층 | 비유 | 범위 | 유지 기간 |
|------|------|------|----------|
| **L0 Session** | 단기 기억 (오늘 점심 뭐 먹었지?) | 현재 대화 세션 | 세션 종료 (최대 7일) |
| **L1 Project** | 프로젝트 기억 (이 프로젝트에서 지금까지 한 것) | 프로젝트 단위 | 90일 (연장 가능) |
| **L2 Long-term** | 장기 기억 (인생에서 배운 것) | 전역 (모든 프로젝트) | 무기한 |
| **L3 Procedural** | 절차 기억 (자전거 타는 법) | 전역/프로젝트 | 무기한 |

**버전별 활성화**:
- V1: L0 + L1(선택적)
- V2: L0 + L1 + L2(승인 필수)
- V3: 전체 활성화

## 7.2 4종 기억 유형 (내용별)

| 유형 | 비유 | 어디에 저장 |
|------|------|-----------|
| **B-4 Working (작업 기억)** | 계산기에 임시 입력한 숫자 | L0 (세션) |
| **B-1 Episodic (에피소드 기억)** | "지난번에 이런 대화를 했었지" | L1 (프로젝트) |
| **B-3 Semantic (의미 기억)** | "파이썬은 프로그래밍 언어다" | L2 (장기) |
| **B-2 Procedural (절차 기억)** | "이 작업은 이 순서로 해야 해" | L3 (절차) |

## 7.3 Vector DB (벡터 데이터베이스)

**벡터 DB란?** 텍스트를 숫자 배열(벡터)로 변환하여 저장하는 특수한 데이터베이스입니다. 이를 통해 "의미가 비슷한 것"을 빠르게 검색할 수 있습니다.

예: "강아지"를 검색하면 "puppy", "반려견", "개" 등 의미가 비슷한 결과가 나옴

| 버전 | Vector DB |
|------|-----------|
| V1 | **Chroma** (로컬, 무료) |
| V2 | **Qdrant** (서버, 더 빠름) |
| V3 | **Qdrant Cloud** (클라우드) |

## 7.4 Embedding 모델

**Embedding이란?** 텍스트를 숫자 벡터로 변환하는 과정입니다. "사과"라는 단어를 [0.23, 0.87, 0.12, ...]과 같은 숫자 배열로 바꿉니다.

| 환경 | 모델 | 차원 | 비용 |
|------|------|------|------|
| V1 로컬 | **BGE-M3** | 1024차원 | 무료 |
| V1 클라우드 | **text-embedding-3-small** | 1536차원 | 유료 |
| V2+ | 벤치마크 기준 최적 | 1536~3072 | - |

**BGE-M3**: 한국어를 포함한 다국어를 잘 처리하는 무료 embedding 모델. V1에서 비용 절약을 위해 기본 선택됨.

## 7.5 RAG 6단계 파이프라인

**RAG (Retrieval-Augmented Generation)** — "검색해서 가져온 정보를 바탕으로 생성"

```
[1] 수집 ──→ [2] 쪼개기 ──→ [3] 벡터화 ──→ [4] 저장 ──→ [5] 검색 ──→ [6] 생성
```

| 단계 | 하는 일 | 세부 설정 |
|------|---------|----------|
| **수집** | 파일, URL, DB에서 문서 수집 | 문서 15개/프로젝트 |
| **쪼개기** | 긴 문서를 작은 조각으로 나눔 | 300~500 토큰, 오버랩 50~100 |
| **벡터화** | 각 조각을 숫자 배열로 변환 | BGE-M3 사용 |
| **저장** | 벡터 DB에 저장 | 청크 30개/문서 |
| **검색** | 질문과 가장 유사한 조각 찾기 | BM25+Vector 앙상블 + Rerank |
| **생성** | 찾은 정보를 AI에 넣어 답변 생성 | LLM 컨텍스트 주입 |

### GraphRAG (지식 그래프 + RAG)

| 단계 | 버전 | 방식 | 정확도 |
|------|------|------|--------|
| Basic | V1 | 벡터 검색만 | 64%+ |
| Hybrid | V2 | BM25 + 벡터 + Rerank | 83%+ |
| Self-RAG + Graph | V3 | 자체 검증 + 그래프 다중 홉 | 90%+ |

## 7.6 Semantic Cache (의미 기반 캐시)

같은 의미의 질문을 반복하면 캐시된 결과를 재사용합니다.

- **히트 조건**: 코사인 유사도 ≥ **0.95** (95% 이상 유사하면 같은 질문으로 판단)
- **캐시 TTL**: 기본 24시간, 변동 잦은 프로젝트는 6시간
- **무효화**: QoD ≤ 0.4이면 캐시 삭제

## 7.7 저장 정책 (안전 관련)

| 정책 | 의미 | 동작 |
|------|------|------|
| **Allow** | 저장 허용 | 요약/결과만 저장 |
| **Restrict** | 제한적 저장 | 마스킹(***) 처리 후 저장 + TTL |
| **Deny** | 저장 금지 | 모든 장기 저장 금지, 벡터 삽입 금지 |

**PII 마스킹**: 주민번호, 전화번호 같은 민감 정보는 `***`로 가리거나 해시 처리

# 8장. SAFETY, COST & APPROVAL — 안전 시스템

VAMOS에서 가장 중요한 부분 중 하나입니다. "사고"가 나지 않도록 여러 겹의 안전장치가 있습니다.

## 8.1 3-Layer Guardrails (3겹 보호막)

**Guardrails(가드레일)**란? 자동차 도로의 가드레일처럼, AI가 "벗어나지 않도록" 막아주는 안전 시스템입니다.

```
[입력] ──→ [L1 NeMo] ──→ [L2 Guardrails AI] ──→ [L3 LlamaGuard] ──→ [출력]
            입력 필터       처리 중 품질 검증       출력 안전성 검사
```

| 레이어 | 엔진 | 위치 | 역할 |
|--------|------|------|------|
| **L1** | NeMo Guardrails | Intake(접수) | 위험한 입력을 사전 차단 |
| **L2** | Guardrails AI | Plan/Execute | 처리 중 품질/안전성 확인 |
| **L3** | LlamaGuard | Execute/Deliver | 최종 출력의 유해성 검사 |
| **L4** | 비동기 감사 | 전달 이후 | 사후 환각 검증, 편향 탐지 |

**Fail-safe 원칙**: 안전 시스템이 고장나면, **기본적으로 차단**합니다 (열리는 게 아니라 닫힘).

## 8.2 RBAC (역할 기반 접근 제어)

**RBAC**란? **R**ole-**B**ased **A**ccess **C**ontrol. 사용자의 역할에 따라 할 수 있는 것이 다릅니다.

| 역할 | 설명 | 최대 자율 수준 | P2 접근 | 월 비용 한도 |
|------|------|-------------|---------|------------|
| **OWNER** | 시스템 소유자 (최고 권한) | L3 (완전 자동) | O | ₩266,000 |
| **ADMIN** | 위임 관리자 | L2 (반자동) | O | ₩93,000 |
| **OPERATOR** | 일반 운영자 | L1 (감독 하에) | X | ₩40,000 |
| **VIEWER** | 읽기 전용 (보기만) | L0 (수동) | X | ₩0 |

## 8.3 Autonomy Level (자율 수준)

VAMOS가 "혼자서 알아서 하는 정도"를 4단계로 나눕니다:

| 수준 | 이름 | 설명 | 예시 |
|------|------|------|------|
| **L0** | FULL_MANUAL | 모든 행동에 사용자 확인 | "이거 해도 될까요?" 매번 물어봄 |
| **L1** | SUPERVISED | 안전한 것만 자동, 나머지는 확인 | **기본값**. 검색은 자동, 코드 실행은 확인 |
| **L2** | SEMI_AUTO | 읽기/생성은 자동, 쓰기는 확인 | 파일 읽기는 자동, 저장은 확인 |
| **L3** | FULL_AUTO | 거의 다 자동 | 대부분 자동으로 처리 |

⚠️ **L3에서도 절대 불가**: Non-goal 위반, 자기 권한 상승, 비용 한도 변경, 안전 필터 우회

## 8.4 CostBudget (비용 관리)

VAMOS는 AI 모델 사용료를 철저하게 관리합니다.

| 버전 | 일 상한 | **월 상한** | 모델 전략 |
|------|--------|-----------|----------|
| V1 | ₩1,300 | **₩40,000** | Mini 모델 90% 이상 사용 |
| V2 | ₩3,100 | **₩93,000** | Mini 60~70%, Main 30~40% |
| V3 | ₩8,900 | **₩266,000** | Main 중심, 플래그십 적극 활용 |

### Downshift (비용 절약 모드)

예산이 부족해지면 자동으로 **저비용 모델로 전환**됩니다:

```
50% 소진 ─→ 정보 알림 (참고용)
80% 소진 ─→ ⚠️ 경고 (노란색). Mini 모델 강제 전환
95% 소진 ─→ 🔴 긴급. Mini-only 모드
100% 소진 ─→ 🚫 차단. 승인 없이는 사용 불가
```

## 8.5 Approval 2단계 (승인 절차)

위험한 작업은 **2단계 승인**을 거칩니다:

| 단계 | 설명 | 권한 |
|------|------|------|
| **1단계** | "이것을 할 계획입니다" (계획 승인) | OPERATOR 이상 |
| **2단계** | "지금 실행합니다" (실행 승인) | ADMIN/OWNER |

**승인 타임아웃**: 10분 미응답 → 자동 거부

## 8.6 7개 불변 구역 (절대 변경 불가)

1. `safety_rules` — 안전 규칙
2. `cost_ceiling` — 비용 상한
3. `approval_flow` — 승인 흐름
4. `non_goals` — 금지 사항
5. `audit_format` — 감사 형식
6. `data_retention` — 데이터 보존 정책
7. `user_consent` — 사용자 동의

이 7개는 **어떤 경우에도, 심지어 자기진화(Self-evo)에서도** 변경할 수 없습니다.

# 10장. 자기진화 (Self-Evolution) — 스스로 성장하는 AI

## 10.1 핵심 원칙

**가장 중요한 규칙**: Self-evo는 **"제안"만 생성**합니다. **자동 적용은 절대 금지**입니다.

즉, VAMOS가 "이렇게 바꾸면 더 좋겠어요"라고 제안할 수는 있지만, 사용자 승인 없이 스스로 바꾸는 것은 불가능합니다.

<!-- CC-002 DEL: 레거시 §10.2 "동작 과정 (5단계)" 삭제. S-1~S-8 자기진화 모듈 체계가 V1.1에서 재정의되어 기존 5단계 흐름이 정본과 불일치. 상세 동작 과정은 D2.0-05 Agent Workflow §3 및 CLAUDE.md §6 S-Series 참조. -->

## 10.2 바꿀 수 있는 것 (6개)

1. **프롬프트 최적화** — AI에게 보내는 지시문 개선 (단, 핵심 키워드 유지)
2. **도구 조합** — 어떤 도구를 어떤 순서로 사용할지
3. **메모리 관리** — 무엇을 기억하고 잊을지
4. **출력 포맷** — 결과를 어떤 형태로 보여줄지
5. **워크플로우 순서** — 작업 단계의 순서
6. **모델 선택** — 어떤 AI 모델을 쓸지

## 10.3 절대 바꿀 수 없는 것 (7개)

1. 정체성/가치/철학 ("나는 VAMOS다")
2. Non-goal (금지사항)
3. 법/규제/윤리
4. 비용 상한 금액
5. 승인 구조
6. P0 도메인 (기본 기능)
7. P2 도메인 생성/활성화

## 10.4 롤백 (되돌리기)

자기진화가 오히려 나빠졌다면:
1. **자동 스냅샷**: 변경 전 상태를 자동 저장
2. **이상 감지**: QoD < 0.70이 3회 연속, 안전 위반 1회, 사용자 롤백 명령
3. **부분 롤백**: 최소 변경 단위로 되돌리기
4. **전체 롤백**: 스냅샷으로 완전 복구
5. **재적용 잠금**: 롤백된 제안은 **14일간 재적용 금지**

### 효과적인 .vamosrules 작성 가이드 [REF:영상3]

> "잘해줘"보다 "코드가 100줄 넘으면 파일을 분리해라"가 효과적이다.

#### 좋은 규칙 예시 (구체적 행동 지침)
```yaml
# .vamos/rules/code-style.yaml
rules:
  - "함수는 최대 30줄. 초과 시 분리하라"
  - "변수명은 camelCase, 파일명은 kebab-case"
  - "API 호출 시 반드시 try-catch로 감싸라"
  - "커밋 전 lint + 테스트 통과 필수"
  - "새 파일 생성 시 기존 패턴 먼저 확인하라"
```

#### 나쁜 규칙 예시 (모호한 지시)
```yaml
# 비효과적
rules:
  - "깔끔하게 짜줘"          # 기준 없음
  - "좋은 코드 작성"          # 정의 없음
  - "적절히 처리해줘"         # 행동 불명확
```

### 주요 기술 설명

- **Sonnet**: Anthropic의 Claude 모델 중 하나

- **SQLite**: 파일 하나로 동작하는 경량 데이터베이스
- **Postgres**: 본격적인 관계형 데이터베이스
- **Docker Compose**: 여러 서비스를 한 번에 실행하는 도구

## 11.2 프로젝트 구조 (Monorepo)

**Monorepo란?** 하나의 코드 저장소에 프론트엔드, 백엔드, 설정 등 모든 것을 함께 관리하는 방식입니다.

```
vamos/
├── src/                  ← React 프론트엔드 (사용자 화면)
│   ├── components/       ← 화면 구성 요소
│   ├── pages/            ← 페이지 (Dashboard, Chat, Settings 등)
│   ├── hooks/            ← React 커스텀 훅
│   ├── stores/           ← 상태 관리 (Zustand 등)
│   └── types/            ← TypeScript 타입 정의
│
├── src-tauri/            ← Rust 백엔드 (Tauri, 데스크톱 앱 로직)
│   └── src/
│       ├── commands/     ← IPC 커맨드 (프론트↔백 통신)
│       ├── bridge/       ← Python 연결 브릿지
│       └── state/        ← 앱 상태 관리
│
├── backend/              ← Python AI/ML 핵심 로직
│   └── vamos_core/
│       ├── orange_core/  ← 판단 엔진 (I-1~I-5)
│       ├── blue_nodes/   ← 도메인별 노드
│       ├── infra/        ← 인프라 (모델 라우팅, 도구 등)
│       ├── agent/        ← 에이전트 워크플로우
│       ├── storage/      ← 메모리/벡터/캐시
│       ├── safety/       ← 안전/비용/승인
│       ├── schemas/      ← 데이터 스키마 (Pydantic v2)
│       └── mcp/          ← MCP 브릿지/서버/클라이언트
│
├── shared/types/         ← JSON Schema (공유 타입 정의, Golden Source)
├── config/               ← 설정 파일
└── tests/                ← 테스트 (unit, integration, e2e)
```

**3개 언어의 역할**:
- **React/TypeScript**: 사용자가 보는 화면
- **Rust**: 데스크톱 앱 + 프론트↔백 통신 (IPC) + 성능 중심 로직
- **Python**: AI/ML 핵심 로직 (모델 호출, RAG, 메모리 등)

# 15장. AI INVESTING — 투자 분석 보조 시스템

## 15.1 한 줄 정의

**AI INVESTING(AINV)**은 VAMOS 안에 있는 **투자 분석 전용 보조 시스템**입니다. 실제로 주식을 사거나 파는 것이 아니라, "이 전략이 과거에 얼마나 잘 먹혔을까?"를 **시뮬레이션(백테스팅)**하고 분석 결과를 보여주는 시스템입니다.

> ⚠️ **절대 규칙**: AI INVESTING은 **실거래(실제 매매) 절대 금지**입니다. 분석과 시뮬레이션만 가능합니다. 이것은 Non-goal 1번 "실제 주문/계좌 연동 금지"와 동일합니다.

## 15.2 비유로 이해하기

AI INVESTING을 **요리 시뮬레이터**에 비유하면 이해가 쉽습니다:

```
[재료 수집]       → 83개 데이터 소스에서 시장 데이터 모으기
                    (레시피에 필요한 재료를 슈퍼에서 사오기)

[레시피 선택]     → 96개 투자 전략 중 골라서 적용
                    (볶음? 조림? 튀김? 어떤 조리법?)

[시뮬레이션 조리] → 과거 데이터로 "만약 이렇게 투자했다면?" 테스트
                    (실제로 손님에게 내기 전에 맛보기)

[품질 검사]       → 51% Gate: 성공률 51% 이상이어야 합격
                    (맛 테스트 통과해야 메뉴에 올림)

[결과 보고]       → "이 전략은 이런 성과가 나왔습니다" 보고서
                    (요리 평가서 작성)
```

## 15.5 51% Gate — 합격 기준

전략이 실전에서 "쓸 만한지"를 판단하는 **최소 합격 기준**입니다:

```
┌──────────────────────────────────────────┐
│           51% Gate (합격 기준)              │
│                                            │
│  1. 승률(Win Rate) ≥ 51%                   │
│     → 100번 중 51번 이상 이기는가?          │
│                                            │
│  2. 샤프 비율(Sharpe Ratio) ≥ 1.0          │
│     → 위험 대비 수익이 충분한가?            │
│                                            │
│  3. 과적합률(Decay Rate) < 30%              │
│     → 과거에만 잘 먹히고 미래엔 안 되는     │
│       "커닝 전략"이 아닌가?                 │
│                                            │
│  4. 테스트 거래 수 ≥ 30회                   │
│     → 충분한 횟수로 테스트했는가?           │
│                                            │
│  → 하나라도 미달 시: 전략 불합격 ❌          │
└──────────────────────────────────────────┘
```

**왜 51%인가?** 동전 던지기가 50%이므로, 최소한 동전보다는 나아야 합니다. 또한 거래 비용(수수료 등)을 감안하면 51%부터 의미 있는 수준입니다.

# 16장. SDAR — 스스로 고치는 AI (자가진단 & 자동복구)

## 16.1 한 줄 정의

**SDAR(Self-Diagnosis & Auto-Repair)**은 VAMOS가 오류를 **스스로 발견하고, 원인을 분석하고, 고칠 수 있는 범위 내에서 자동으로 수리**하는 시스템입니다.

## 16.2 비유로 이해하기

SDAR을 **스마트 자동차의 자가진단 시스템**에 비유하면:

```
[계기판 경고등]   → L1 Detection: 문제 감지
                    "엔진 온도가 비정상입니다"

[원인 분석]       → L2 Diagnosis: 근본 원인 찾기
                    "냉각수가 부족한 것 같습니다"

[해결책 제안]     → L3 Prescription: 수리 방법 생성
                    "냉각수 보충 또는 정비소 방문 필요"

[수리 실행]       → L4 Repair: 자율 수준에 따라 실행
                    (경미하면 자동 조치, 심각하면 사람에게 알림)

[수리 확인]       → L5 Verification: 수리가 잘 되었는지 검증
                    "수리 후 5분간 관찰 → 정상 확인"
```

**알집(ALZip)이 깨진 압축 파일을 자동 복구하는 것**과 비슷한 개념입니다. 다만 VAMOS는 단순한 파일이 아니라 AI 시스템 전체의 오류를 다룹니다.

## 16.3 왜 필요한가?

기존 VAMOS에도 오류 대응이 있었지만 한계가 있었습니다:

| 기존 방식 | 한계 |
|----------|------|
| I-6 Self-check | 결과의 정확성만 검증. 시스템 오류는 감지 못함 |
| I-20 Failure Manager | 미리 정의된 오류만 대응. 새로운 오류는 대응 못함 |
| Circuit Breaker | 차단만 함. 원인 분석이나 수리는 안 함 |
| 13개 Fallback | 정해진 대안만 실행. 근본 해결이 아님 |

**SDAR은 이 모든 것을 통합**하여: 감지 → 원인 분석 → 수리 방법 생성 → 실행 → 검증까지 한 번에 처리합니다.

## 16.4 5-Layer 파이프라인 (5단계 수리 과정)

```
[L1 Detection]     "무언가 이상하다!" — 30초마다 헬스체크
        ↓
[L2 Diagnosis]     "원인은 이것이다" — 로그 분석, 의존성 추적
        ↓
[L3 Prescription]  "이렇게 고치자" — 수리 후보 1~5개 생성
        ↓
[L4 Repair]        "수리 실행!" — AR-Level에 따라 자동/수동
        ↓
[L5 Verification]  "잘 고쳐졌나?" — 5분간 관찰, 문제 시 롤백
```

## 16.5 단계적 자율수준 (AR-Level) — 핵심 개념

**"어디까지 자동으로 고칠 것인가?"**를 5단계로 나눈 것입니다. 이것이 SDAR의 가장 중요한 설계입니다.

```
AR-L0: 아무것도 자동 안 함    ←── 가장 안전
AR-L1: 진단만, 수리는 사람이
AR-L2: 가벼운 것만 자동 수리  ←── 기본 설정 ⭐
AR-L3: 중간 수준까지 자동
AR-L4: 코드 수정까지 자동     ←── 가장 자율적
```

각 수준을 **자동차 정비**에 비유하면:

| 수준 | 이름 | 비유 | 예시 |
|------|------|------|------|
| **AR-L0** | MANUAL (수동) | "항상 정비소에서만 수리" | 사람이 모든 수리 직접 |
| **AR-L1** | NOTIFY (알림) | "경고등만 켜줌, 사람이 판단" | "이런 문제가 있어요, 이렇게 고치면 될 것 같아요" |
| **AR-L2** | AUTO_SAFE (안전자동) | "와이퍼액 자동 보충 정도" | 서비스 재시작, 캐시 정리, API 재시도 |
| **AR-L3** | AUTO_MODERATE (중간자동) | "타이어 공기압 자동 조절" | 프롬프트 수정, 설정 변경, 인덱스 재구축 |
| **AR-L4** | AUTO_AGGRESSIVE (고급자동) | "엔진 부품 자동 교체" | 코드 핫픽스, 스키마 변경, 의존성 업데이트 |

**기본값은 AR-L2**(안전 자동)입니다. 이는 VAMOS의 L1 SUPERVISED(감독형) 철학과 일치합니다.

### 누가 어디까지 설정할 수 있나?

| 역할 | 최대 설정 가능 수준 |
|------|-------------------|
| OWNER (소유자) | AR-L4까지 가능 |
| ADMIN (관리자) | AR-L3까지 가능 |
| OPERATOR (운영자) | AR-L2까지 가능 |

## 16.6 오류 분류 — 어떤 종류의 문제가 있나?

VAMOS에서 발생할 수 있는 오류를 5가지로 분류합니다:

| 카테고리 | 예시 | 자동 수리 가능? | 비유 |
|---------|------|---------------|------|
| **A. 인프라** | DB 연결 끊김, API 429(과다요청) | AR-L2 (대부분 자동) | "전선 끊어짐 → 다시 연결" |
| **B. 모델/AI** | AI 환각, 품질 저하 | AR-L3 (중위험) | "AI가 헛소리 → 다른 모델로 전환" |
| **C. 로직** | 워크플로우 멈춤, Gate 오설정 | AR-L3~L4 | "길이 막힘 → 우회도로 찾기" |
| **D. 코드** | 버그, 회귀, 라이브러리 깨짐 | AR-L4 (고위험) | "부품 결함 → 교체" |
| **E. 보안** | 해킹 시도, 데이터 유출 | **절대 자동 수리 불가** | "침입자 → 즉시 잠금 + 112 신고" |

> **E(보안)은 절대 자동 수리하지 않습니다.** 즉시 차단하고 사람에게 알립니다.

## 16.7 절대 자동으로 바꿀 수 없는 것 (NEVER_AUTO)

아무리 AR-L4라도 **절대 자동으로 건드릴 수 없는 영역**이 있습니다:

1. **안전 규칙 변경** — 가드레일을 자동으로 끌 수 없음
2. **비용 상한 변경** — 예산을 자동으로 올릴 수 없음
3. **승인 절차 변경** — 승인 프로세스를 건너뛸 수 없음
4. **Non-goal 변경** — "실거래 금지" 같은 금지사항을 해제할 수 없음
5. **가드레일 비활성화** — 보호장치를 끌 수 없음
6. **Gate 우회** — 검문소를 건너뛸 수 없음

이것은 기존 VAMOS의 **RULE 1.3 불변 구역**과 동일합니다.

## 16.8 버전별 SDAR 로드맵

| 버전 | 자율 수준 | 할 수 있는 자동 수리 |
|------|----------|-------------------|
| **V1** | AR-L2 | 서비스 재시작, 캐시 정리, API 재시도, 모델 전환, 속도 조절 (5가지) |
| **V2** | AR-L3 | + 프롬프트 패치, 설정 변경, API 키 교체, 스냅샷 롤백, 로그 압축 (10가지) |
| **V3** | AR-L4 | + 코드 핫픽스, 스키마 변경, 의존성 재설치, 인덱스 재구축 (14가지) |

## 16.9 안전장치

- 같은 문제를 **시간당 최대 3번**만 자동 수리 시도 (무한루프 방지)
- 중간 이상 수리 전에 **반드시 스냅샷(백업)** 생성
- **모든 수리에 사람 알림** 필수 (자동이라도 알림은 감)
- **Emergency Kill Switch**: 어떤 역할이든 **즉시 SDAR 비활성화** 가능

# 17장. Agent Teams — 여러 AI가 팀을 이루어 일하기

## 17.1 한 줄 정의

**Agent Teams**는 VAMOS가 복잡한 작업을 처리할 때 **여러 전문 AI 에이전트가 팀을 구성하여 협력**하는 시스템입니다.

## 17.2 비유로 이해하기

Agent Teams를 **프로젝트 팀**에 비유하면:

```
[PM (프로젝트 매니저)]  → Lead Agent: 작업 분배, 진행 관리
       │
       ├── [리서치 담당]  → Research Agent: 자료 조사
       ├── [개발 담당]    → Coding Agent: 코드 작성
       ├── [분석 담당]    → Quant Agent: 데이터 분석
       ├── [문서 담당]    → Content Agent: 보고서 작성
       ├── [투자 담당]    → Trading Agent: 투자 분석 (AINV 연동)
       └── [수리 담당]    → SDAR Agent: 오류 진단/복구
```

예를 들어 "삼성전자 분석 보고서 만들어줘"라고 요청하면:
1. **Lead Agent**가 작업을 3개로 분해
2. **Research Agent**가 최신 뉴스/실적 수집
3. **Quant Agent**가 재무 데이터 분석
4. **Content Agent**가 결과를 보고서로 정리
5. Lead가 3개 결과를 합쳐서 최종 답변 전달

## 17.3 위임(Delegation) — 누가 무슨 일을 할지 정하기

Lead Agent는 **5가지 점수**를 계산해서 가장 적합한 에이전트에게 일을 맡깁니다:

| 점수 | 의미 | 비유 |
|------|------|------|
| 도메인 매칭 | 이 분야에 맞는 에이전트인가? | "리서치 일은 리서치 담당에게" |
| 스킬 매칭 | 필요한 도구를 갖고 있는가? | "코딩 도구가 있는 사람에게" |
| 가용성 | 지금 바쁘지 않은가? | "한가한 사람에게" |
| 비용 적합도 | 예산 안에서 가능한가? | "예산에 맞는 사람에게" |
| 과거 성과 | 이전에 잘했는가? | "경험 있는 사람에게" |

## 17.4 협업 방식 — 어떻게 함께 일하나?

| 방식 | 설명 | 예시 |
|------|------|------|
| **순차 실행** | A 끝나면 B, B 끝나면 C | "리서치 → 분석 → 보고서" 순서대로 |
| **병렬 실행** | A, B, C 동시에 하고 결과 합치기 | "3팀이 각자 조사 → 합산" |
| **토론(Debate)** | 에이전트끼리 의견 교환 → 합의 | "찬반 토론 후 결론" |
| **감독(Supervisor)** | Lead가 결과 검토 → 부족하면 재지시 | "PM이 검토 후 수정 요청" |
| **인계(Handoff)** | 한 에이전트가 다른 에이전트에게 이어서 | "리서치 완료 → 코딩팀에 넘김" |

## 17.5 안전장치

- **위임 깊이 제한**: Lead → Sub → Sub-sub까지만 (최대 3단계)
- **에이전트 수 제한**: V1은 3개, V2는 5개, V3는 10개까지
- **5 Gates 적용**: 에이전트 생성/실행 전 반드시 PolicyGate + CostGate 통과
- **Delegation Attack 방어**: 에이전트 간 통신은 HMAC 검증 (위조 방지)
- **비용 관리**: 팀 전체 예산 = 작업 예산의 80% (나머지 20%는 예비)

## 17.6 버전별 Agent Teams

| 버전 | 에이전트 수 | 협업 방식 | 수준 |
|------|-----------|----------|------|
| **V1** | 2~3개 | 순차, 감독 | "작은 팀" |
| **V2** | 5~10개 | + 병렬, 토론, 인계 | "본격 팀워크" |
| **V3** | 10~50+개 | + 동적 생성/해체, 자율 팀 | "자율 조직" |

> **상세 스펙**: Master Specification 20장 및 `VAMOS_AGENT_TEAMS_SPEC.md` (2,188줄, 11개 섹션) 참조
