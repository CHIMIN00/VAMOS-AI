# VAMOS BASE RULE — 단계 공통 규칙 (VERSION 1.0)
--------------------------------------------
0. 문서 목적
--------------------------------------------
본 문서는 VAMOS 전체 단계(베이스 → 계획 → 설계 → 구현 → 평가 → 운영)에
공통적으로 적용되는 최상위 규칙(BASE RULE)을 정의한다.
모든 단계 문서는 아래 규칙을 반드시 준수해야 하며,
위배될 경우 해당 문서는 무효로 간주한다.
--------------------------------------------

1. 단계(Phase) 진행 규칙
 1.1 단계는 순차적으로만 진행된다.  
    - 베이스 → 계획 → 설계 → 구현 → 평가 → 운영
 1.2 상위 단계의 승인 없이 하위 단계로 이동할 수 없다.
 1.3 각 단계는 이전 단계 산출물에 100% 의존한다.  
    - 이전 단계에 정의되지 않은 기능/도메인/규칙은 사용할 수 없다.
 1.4 단계 이동은 반드시 문서화 + 승인 절차를 거쳐야 한다.  
    - “구두 승인/대충 작성”은 유효하지 않다.
 1.5 정의되지 않은 것은 존재하지 않는 것으로 간주한다.  
    - 문서에 없으면 기능·규칙·도메인이 존재하지 않는 것과 같다.
--------------------------------------------
2. 문서 구조 규칙
 2.1 모든 단계 문서는 공통 템플릿을 따라야 한다.
    - 제목  
    - 목적  
    - 규칙  
    - 산출물  
    - 세부 항목   
 2.2 자유 형식 문서는 허용되지 않는다.  
    - 템플릿 기반 정형 문서만 허용.
 2.3 문서 변경은 반드시 버전 관리로 기록해야 한다.
    - 예: V1.0 → V1.1 → V1.2
 2.4 문서 간 충돌 시 우선순위는 다음과 같다.  
    1) BASE RULE  
    2) 계획 규칙  
    3) 설계 규칙  
    4) 구현 규칙  
    5) 운영 규칙  
 2.5 BASE RULE에 위배되는 문서는 자동 무효 처리된다.
--------------------------------------------
3. 기능 정의 규칙
3.1 모든 기능은 다음 5요소를 갖춰야 한다.
    - 정의  
    - 목적  
    - 입력(Input)  
    - 출력(Output)  
    - 제약조건(Limits)  
3.2 기능은 두 종류로만 분류한다.
    - I: 내부 기능
    - E: 외부 기능
3.3 기능은 고유 번호를 갖는다.
    - 예: I-1, I-2 … I-24 (I-모듈 전체)
          E-1, E-2 … E-16 (E-모듈 전체)
3.4 단계 내에서 정의되지 않은 기능은 구현할 수 없다.
3.5 기능은 단계별로 세분화된다.
    - 계획 단계: 구조 확정  
    - 설계 단계: 입력/출력/로직 설계  
    - 구현 단계: 코드 작성  
    - 평가 단계: 테스트/검증  
    - 운영 단계: 유지보수/진화  
--------------------------------------------
4. 규칙 및 정책 규칙
 4.1 규칙은 단계별로 누적 적용된다.  
    - 상위 단계 규칙은 하위 모든 단계에 영향을 준다.
 4.2 규칙 충돌 시 우선순위는 다음과 같다.
    1) BASE RULE  
    2) 단계 규칙  
    3) 설계 규칙  
    4) 구현 규칙  
    5) 운영 규칙
 4.3 임의 해석은 금지된다.  
    - 명확한 문서가 없으면 신규 정의 후 진행해야 한다.
 4.4 규칙 변경 시 두 단계 문서를 반드시 수정해야 한다.  
    - 변경이 발생한 단계  
    - 그 이후 단계  
 4.5 위험·안전·비용 정책은 모든 규칙 위에 우선 적용된다.
--------------------------------------------
5. 산출물(Output) 규칙
 5.1 각 단계는 반드시 산출물을 생성해야 한다.  
    - 산출물이 없으면 해당 단계는 완료되지 않은 것.
 5.2 산출물 유형은 다음 중 최소 1개 이상 포함한다.
    - 정의 문서  
    - 기능 도표  
    - 규칙 표  
    - 아키텍처  
    - 체크리스트  
    - 테스트 결과표  
 5.3 각 단계의 산출물은 다음 단계의 입력값이 되어야 한다.
 5.4 평가·운영 단계에서는 이전 단계 산출물이 재활용된다.
--------------------------------------------
6. 변경(Change) 규칙
 6.1 모든 변경은 변경 로그(Change Log)에 기록되어야 한다.
 6.2 설계·구현 단계에서 새로운 기능 추가 시  
    - 반드시 계획 단계 문서에 해당 기능을 등록해야 한다.
 6.3 이전 단계 산출물을 수정하면  
    - 그 이후 단계 문서 모두 재검토해야 한다.
 6.4 변경 없이 구현하는 것은 금지된다.
--------------------------------------------
7. VAMOS AI 자체 규칙
 7.1 VAMOS는 “정의되지 않은 것은 실행하지 않는다” 원칙을 따른다.
 7.2 사용자 요청이라도 규칙/단계에 위배되면 처리하지 않는다.
    - 대신 단계 수정 또는 규칙 갱신을 안내한다.
 7.3 Self-evolution은 문서 규칙 기반으로 작동하며  
    - 단계 승인 절차를 거쳐야 실제 반영된다.
 7.4 모든 기능은 BASE RULE 14개 목표와 연결되어야 한다.
 7.5 AI 자가 지식 확장(I-21) 안전 가드레일
    - 저작권 필터: 수집 콘텐츠는 CC0·퍼블릭도메인·명시적 허용 라이선스만 허용.
    - 개인정보 필터: 수집 데이터에서 PII(성명·연락처·주민번호 등) 자동 탐지 및 제거.
    - 법적 위험 필터: 국내외 저작권법·개인정보보호법 위반 가능성 탐지 시 수집 중단.
    - 필터 우회 시도 탐지: 우회 패턴 감지 시 해당 수집 태스크 즉시 종료 및 로그 기록.
    - 모든 필터 통과 후에만 지식 저장소에 반영된다.
--------------------------------------------
8. 품질(Quality) · 안전(Safety) · 비용(Cost) 공통 규칙
 8.1 안전 규칙은 모든 단계에 최우선 적용된다.  
    - 불법, 위험, 개인정보, 실거래 관련 요소는 금지.
 8.2 비용 상한(V1/V2/V3)은 모든 단계에서 준수해야 한다.
    - **V1 (저비용 모드)**: $20/월 상한 (토큰 환산 약 2,000만 토큰)
    - **V2 (균형 모드)**: $50/월 상한 (토큰 환산 약 5,000만 토큰)
    - **V3 (고정확도 모드)**: $200/월 상한 (토큰 환산 약 2억 토큰)
    - 상한 초과 시 해당 버전의 기능 실행은 자동 중단되며 사용자에게 경고를 전달한다.
 8.3 정확도가 속도보다 우선한다.  
    - 오류 가능성이 있는 상태에서는 단계 이동 금지.
 8.4 Self-check / QA는 모든 단계에 필수 요소이다.  
    - 누락 시 그 단계는 완료되지 않은 것으로 간주.
--------------------------------------------
9. 핵심 규칙 요약 (10개)
 9.1 단계는 순차적이며 역행할 수 없다.  
 9.2 상위 단계 승인 없이는 다음 단계로 이동할 수 없다.  
 9.3 정의되지 않은 기능은 존재하지 않는 것으로 간주한다.  
 9.4 모든 문서는 정형화된 템플릿을 따른다.  
 9.5 규칙 충돌 시 BASE RULE이 최우선이다.  
 9.6 모든 기능은 내부/외부 기능으로 고정 분류한다.  
 9.7 모든 변경은 기록되고 이후 단계에 영향을 준다.  
 9.8 안전/비용/법 준수는 가장 우선된다.  
 9.9 Self-evolution은 문서 규칙 기반으로만 작동한다.  
 9.10 산출물이 없으면 단계는 완료되지 않은 것이다.

1. 멀티모달 기능 버전별 범위
------------------------------------------------------------

1-1. V0 (기초 구축)
 - 텍스트 중심
 - 문서 입력(PDF/텍스트)
 - 로그 시스템
 - 선택적 OCR

1-2. V1 (기본 멀티모달)
 - 이미지 OCR+설명 — **~500 토큰/이미지**
 - 음성 STT — **~200 토큰/분**
 - 영상 프레임 분석 — **~3,000 토큰/60초**
 - 단일 LLM 구조
 - ⚠️ 멀티모달 비용 주의: 대용량 파일은 V2/V3 권장

1-3. V2 (확장 멀티모달)
 - 이미지 패턴·구조 인식
 - 음성 감정 분석/화자 구분
 - 영상 장면 탐지·요약
 - 문서 기반 RAG
 - LLM 2단 구조(미니→메인)

1-4. V3 (AGI 멀티모달)
 - 영상 행동 분석
 - 음성 프로필 분석
 - 이미지→코드·기획 자동 변환
 - LLM + 다른 뇌 + LLM 구조
 - Self-evo 자동 개선



멀티모달 입력 단위별 비용 기준 (추정치, 모델·설정에 따라 변동 가능):
  - 이미지 1장: 약 500 토큰
  - 음성 1분: 약 800 토큰
  - 동영상 1분(프레임 추출 기준): 약 3,000 토큰
  - PDF 1페이지(텍스트+이미지 혼합): 약 1,200 토큰
  - 위 수치는 V1/V2/V3 비용 상한(§8.2) 계산의 기준값으로 사용된다.

------------------------------------------------------------

 - 도메인 판정
 - 위험 분석
 - 비용 추정

7-3. Main LLM Escalate 조건
 - 복합 의도
 - 코드 생성/리팩토링
 - 전략/기획/리포트
 - 멀티모달 포함

 - 비용 상한 80% 도달
 - 짧은 답변 요청
 - 반복 호출

7-5. Tool Model 사용 조건
 - OCR/문서 파싱
 - 코드 실행
 - 임베딩 생성

21. I-21. 데이터 소스·검색 모듈 Self-evo 엔진

    21.1 헤더
        //[✅ I-21. 데이터 소스·검색 모듈 Self-evo 엔진]
        (Data Source & Fetcher Auto-Evolution Engine)

    21.2 설명
        설명

        스스로 탐지·추천하고, 적합한 경우에는 수집 모듈(fetcher/crawler) 템플릿을 자동 생성하는 엔진.

    21.3 이 엔진은 다음을 수행한다:
        정보 부족·지식 공백 감지
            RAG나 메모리에 충분한 근거가 없을 때  
            특정 토픽에서 실패 패턴이 반복될 때  
            여러 답변에서 동일한 논리/데이터 구멍이 생길 때  

        필요한 데이터 소스 후보 생성
            공식 API 추천  
            공식 문서/백서/스펙 페이지  
            학술 아카이브 / 기술 문서  
            신뢰도 높은 사이트/리포트  
            오픈 데이터셋  
            → “책 1권만 보고 답하는 AI → 책 10권을 보고 답하는 AI”로 확장  

        새 fetcher/crawler 모듈 설계 자동 생성
            API 요청 구조 설계  
            페이징/한도 관리  
            응답 구조 파싱 로직  
            HTML/CSS selector/XPath 기반 크롤러 템플릿  
            데이터 필드 구조(JSON schema)  
            → 실제 모듈 코드 초안까지 자동 생성(E-1 같은 코드 생성 모듈과 연결)  

        Self-evo 품질 검증 루프 참여
            I-14 QA 스위트로 새 모듈 테스트  
            성공 여부/품질 결과 기록  
            승인 후 RAG + 메모리 + 그래프에 정식 편입  

        지식 울타리 확장(확장성 AGI 핵심)
            VAMOS의 **“가용 지식 영역”**이 스스로 확장  
            분석 능력 / 설명 품질 / 근거 기반 답변 능력이 지속 강화  

    21.4 연관 내부 기능
        연관 내부 기능  
            I-2. RAG·지식 검색 엔진  
                새 소스/문서 인덱싱  
                품질 가중치 반영  
            I-6. Self-check / Self-evolution 엔진  
                정보 부족/오류 패턴 감지  
                개선 필요 판단  
            I-13. 프롬프트·워크플로우 템플릿 엔진  
                fetcher/crawler 코드 템플릿 생성  
            I-14. QA·벤치마크 스위트  
                새 모듈 검증  
            I-18. 메타학습 Self-evo 힌트 엔진  
                “어떤 소스가 필요한지” 후보 생성  
            I-19. 데이터 품질 평가(QoD)  
                새 소스/데이터의 품질 필터링  
            I-20. 내부 지식 그래프 엔진  
                새 소스에서 얻은 개념·엔티티·관계를 그래프에 통합  

    21.5 7가지 요구 반영
        7가지 요구 반영  
            구현 가능성: 초기엔 “소스 추천 + 수동 구현” 단계부터, 이후 자동 코드 초안 생성으로 확장  
            비용: 새 소스/크롤러 추가는 비용 증가 요소이지만, 중요 영역에 한해 선택적으로 적용 가능

15. E-15. Cloud Library Collector (Cloud Collector)

    15.1 헤더
        //[✅ E-15. Cloud Library Collector (Cloud Collector)]
        (Cloud Library Data Collection Engine — BLUE NODE)
        인터넷 다중 소스(YouTube, Instagram, Website, Blog, Academic 등)에서 지식을 자율 수집하고,
        키워드를 추출하여 S-5 Cloud Evolver에 전달하는 통합 수집 엔진.
        10-Layer 아키텍처 중 Layer 1~6을 담당한다.

    15.2 설명
        설명
        VAMOS Cloud Library 시스템의 “외부 세계에서 데이터를 가져오는” BLUE NODE 실행 모듈.
        시드 키워드 기반 자율 사이트 발견(7-Stage Discovery Pipeline), 사이트 평가(100점 체계),
        플랫폼별 수집(9종 플랫폼), 키워드 추출(템플릿 v1.x 시리즈), VAMOS 4계층 자동 매핑을 수행한다.
        기존 E-1(YouTube), E-2(Instagram) 등 BLUE NODE를 서브 수집기로 재활용하는 오케스트레이터 역할.

    15.3 이 모듈은 다음을 수행한다:

        15.3.1 Layer 1 — INPUT (입력 계층)
            수동 URL 입력, 일괄 URL Import, API Webhook, Seed Keywords(VAMOS DESIGN 추출),
            크론 스케줄 트리거, 이벤트 Webhook(새 콘텐츠 감지) 수신.
            모든 입력을 정규화된 CollectionRequest 객체로 변환.
            기술: Python asyncio, Tauri IPC

        15.3.2 Layer 2 — DISCOVERY (발견 계층) — 7-Stage Discovery Pipeline
            [Stage 1] SEED KEYWORDS: VAMOS DESIGN에서 추출한 시드 키워드(LLM, RAG, Agent, Fine-tuning, Vector DB, Embedding 등). 기존 키워드 풀에서 빈도/트렌드 분석 후 자동 갱신.
            [Stage 2] SEARCH ENGINE QUERY: Google(“{키워드} site:.edu”), Bing(“{키워드} research paper”), DuckDuckGo(“{키워드} open source”) 자동 쿼리 생성. 동의어/관련어 자동 확장. V3: Tavily Search 통합.
            [Stage 3] SITE CANDIDATE POOL: 검색 결과 도메인 추출, site_registry.json 중복 제거, 블랙리스트 대조, 새 사이트 [NEW] 태그 부여.
            [Stage 4] SITE EVALUATION: Trust(25점)+Relevance(30점)+Quality(25점)+Access(20점) = 총 100점 자동 평가.
            [Stage 5] TEMPLATE MATCHING: URL 패턴 1차 분류 → 콘텐츠 분석 2차 분류 → 템플릿 할당(9종). 신규 플랫폼은 동적 템플릿 생성.
            [Stage 6] APPROVAL GATE: AUTO(>=80) / NOTIFY(60-79) / MANUAL(30-59) / BLOCK(<30 또는 블랙리스트).
            [Stage 7] SITE REGISTRY: 승인 사이트 → site_registry.json 등록, 키워드 추출 대본.md 자동 업데이트.

        15.3.3 Layer 3 — EVALUATION (평가 계층)
            소스 평가 알고리즘 (100점 만점):
              신뢰도(Trust, 25점): 도메인 연령(5) + HTTPS(5) + Alexa 순위(5) + 업데이트 빈도(5) + 저자 정보(5)
              관련성(Relevance, 30점): VAMOS 키워드 매칭률 (키워드당 2점, 최대 30점)
              품질(Quality, 25점): 콘텐츠 길이(10) + 코드 예제(5) + 인용(5) + 원본성(5)
              접근성(Access, 20점): robots.txt(5) + RSS(5) + API(5) + 페이월 없음(5)
            V2+ 보완: Semantic Router + Reranker 기반 적응형 우선순위(C7)

        15.3.4 Layer 4 — COLLECTION (수집 계층)
            플랫폼별 수집기 (Platform Collectors):
              YouTube: yt-dlp 메타데이터 + 자막 (템플릿 v1.2/v2.3)
              Instagram: yt-dlp + Whisper 전사 (템플릿 v1.1)
              웹사이트: WebFetch 스크래핑 (템플릿 v1.3)
              GitHub: GitHub API (템플릿 v1.1, V2 예정)
              Twitter/X: API (템플릿 v1.1, V2 예정)
              arXiv: Semantic Scholar API (템플릿 v1.1, V2 예정)
              Substack: RSS (템플릿 v1.1, V2 예정)
              Discord: API (템플릿 v1.1, V2 예정)
              HuggingFace: API (템플릿 v1.1, V2 예정)
            Scheduler & Queue Manager: 크론 기반 정기 실행 + 우선순위 큐(P0 긴급/P1 높음/P2 보통/P3 낮음)
            Rate Limiter: 플랫폼별 API 호출 제한 준수
            Error Handler & Retry: 실패 시 최대 3회 자동 재시도 (지수 백오프 1s→2s→4s)
            수집 주기: 뉴스 1일, YouTube 3일, SNS 1일, 학술 7일, 블로그 주 2회

        15.3.5 Layer 5 — DATA LAKE (데이터 레이크)
            Raw Data Storage: JSON/Markdown 원본 저장
            Normalized Data: 정규화 데이터
            Metadata Index: V0-V1 파일 시스템, V1-V2 SQLite+JSONL, V2+ Postgres
            Deduplication Engine: 글로벌 키워드 인덱스, Levenshtein 유사도(>=0.85 동일), Cosine 유사도, 동의어 그룹화
            TTL 관리: 기술 트렌드 30일, 뉴스/SNS 7일, 학술 365일, 공식 문서 90일

        15.3.6 Layer 6 — EXTRACTION (추출 계층)
            Template Matcher: URL 패턴 1차 → 콘텐츠 2차 → 기본 웹사이트 3차 폴백
            Keyword Extractor: 템플릿 v1.x 시리즈 (9종)
            Dynamic Template Generator: 신규 플랫폼 감지 시 기존 유사 템플릿 참조 → 초안 생성 → Human 검토
            VAMOS 4-Layer Mapper: RULE/PLAN/DESIGN/SCHEMA 자동 매핑
            Priority Classifier: CRITICAL/HIGH/MEDIUM/LOW 자동 분류
            출력: CloudLibraryExtractionOutputSchema (D3 §5.8 참조) → S-5 Layer 7로 전달

    15.4 E-15 담당 기능 (10개)
        A1:  통합 수집 파이프라인 (L1+L4) — 단일 진입점, 자동 라우팅, 병렬 처리
        A4 전반: 소스 품질 평가 (L3) — 수집 전 Trust/Relevance/Quality/Access 평가
        C1:  자율 사이트 발견 (L2) — 7-Stage Discovery Pipeline
        C2:  동적 템플릿 생성 (L6) — 기존 템플릿 기반 변형/복제
        C7:  적응형 우선순위 (L2+L3) — Semantic Router + Reranker
        D1:  외부 API 연동 (L1+L4) — 플랫폼/검색/학술 API
        E1-ALT: 가이드된 자율 발견 (L2) — Seed→Tavily→Agentic RAG→CrewAI
        E4-ALT(수집부): 수집 99% 자동화 (L4) — V3 예외만 인간 개입

    15.5 VAMOS 기존 모듈 연동
        I-1 (Brain Router): I-1이 정보 필요 시 E-15에 수집 요청 (능동적 수집 트리거)
        I-2 (RAG 시스템): E-15 수집 데이터 → I-2 벡터 DB 자동 삽입 (Cloud Library 핵심 가치: 빈 RAG 파이프라인 채움)
        OTHER BRAINS: E-15 수집기들이 OTHER BRAINS(API, 외부 도구) 활용
        E-Series 동료: E-1(YouTube), E-2(Instagram) 등 기존 BLUE NODE를 서브 수집기로 재활용
        S-5 (Cloud Evolver): E-15 Layer 6 출력 → S-5 Layer 7 입력 (JSON Schema 기반 인터페이스 계약)

    15.6 V0→V1→V2→V3 로드맵
        V0: 수동 URL 입력 → 단일 수집기(yt-dlp), Markdown/JSON 저장, 템플릿 v1.2 적용. Cloud Library 미활성.
        V1 (CB1 기초): 통합 수집(A1), 소스 평가(A4 전반), 규칙 기반 우선순위(C7 기초), YouTube Data API(D1 기초). L1-L6 파이프라인. 기술: Python asyncio, yt-dlp, WebFetch, SQLite+JSONL.
        V2 (CB2/CB3): 멀티 플랫폼 수집(A1 완성), 7-Stage Discovery(C1), 동적 템플릿(C2), Semantic Router(C7), 검색/학술 API(D1 확장), Tavily(E1-ALT 기초). 기술: GPT-4o mini+검색 API, Platform Collectors 모듈화.
        V3 (CB4): 완전 자율 Discovery(C1 완성), 동적 템플릿 생성(C2 완성), 5-Layer 가이드된 발견(E1-ALT 완성), 수집 99% 자동화(E4-ALT), Kafka/Redis/WebSocket(D1 고급). 기술: 멀티 에이전트, Agentic RAG+CrewAI+Tavily.

    15.7 연관 내부 기능
        I-1. Brain Router (수집 요청 트리거)
        I-2. RAG·지식 검색 엔진 (수집 데이터 → 벡터 DB 삽입)

        I-13. 프롬프트·워크플로우 템플릿 엔진 (키워드 추출 템플릿 관리)
        I-19. 데이터 품질(QoD) 평가 엔진 (소스 평가 연동)
        I-21. 데이터 소스·검색 모듈 Self-evo 엔진 (소스 확장)
        S-5. Cloud Evolver (Layer 7~10, 분석/검증/버전관리/진화)

    15.8 7가지 요구 반영
        구현 가능성: ✔✔ Phase A(95%) + Phase B(70%) + Phase D(80%) 순차 구현. V1부터 기본 수집 가능.

16. E-16. 외부 답변 품질 평가 도우미

    16.1 헤더
        //[✅ E-16. 외부 답변 품질 평가 도우미]
        (External Answer Quality Evaluator)
        사용자가 받은 답변을 정확도/논리/근거/표현/BASE 준수 기준으로 평가하고, 개선 버전을 제공하는 모드

    16.2 설명
        설명
        VAMOS 말고 다른 AI/사람/문서가 준 답변을
        “이게 괜찮은지, 어디가 문제인지” 평가해주는 모드.

    16.3 이 모드는 다음을 수행한다:
        품질 평가
            정확도/논리 일관성/근거 유무/표현력/안전성 등
            여러 축으로 점수/코멘트 제공

        위험·오류 탐지
            잘못된 정보, 위험 조언, 미확인 주장 등을 감지
            근거 부족/모순 지점 표시

        개선 버전 제안
            같은 질문에 대해
            VAMOS 기준으로 더 안전/정확한 답변 버전 제공

        학습·비교 도구
            사용자가 “좋은 답변 vs 나쁜 답변” 차이를

5. S-5. Cloud Library Evolver (데이터 품질·소스 확장·자가진화 엔진)

    5.1 헤더
        //[✅ S-5. Cloud Library Evolver]
        (CLOUD LIBRARY EVOLVER — DATA QUALITY & SOURCE EXPANSION SELF-EVO ENGINE)
        기존 I-19 + I-21을 묶은 데이터 품질·소스 확장 엔진에,
        Cloud Library 10-Layer 아키텍처의 Layer 7~10 (Analysis → Validation → Version Control → Output)을
        ORANGE CORE 자가진화 엔진으로 통합한 모듈.
        E-15 Cloud Collector가 L1~L6에서 수집한 원본 데이터를 받아,
        분석·검증·버전관리·진화·출력까지 담당하는 VAMOS 지식 진화의 핵심 엔진.

    5.2 설명
        설명
        VAMOS의 지식 풀 자체를 진화시키는 ORANGE CORE 모듈.
        E-15(Cloud Collector)가 L1~L6에서 수집·추출한 데이터를 입력받아,
        Layer 7 (분석/충돌 해결) → Layer 8 (5-Gate 검증) → Layer 9 (버전 관리) → Layer 10 (출력/진화)
        파이프라인을 통해 VAMOS 전체를 자율적으로 업그레이드한다.
        “쓰레기 데이터는 걸러내고, 좋은 데이터/새 소스를 계속 추가”하며,
        S1~S5 자가 진화 사이클을 통해 스스로 패턴을 학습하고 규칙을 개선한다.

        E-15 ↔ S-5 인터페이스 계약:
        E-15의 L6 Extraction 출력 → S-5의 L7 Analysis 입력으로 전달.
        전달 포맷: CloudLibraryExtractionOutputSchema (D3 §5.8 참조).
        전달 내용: source_url, platform, template_used, extracted_keywords[], vamos_mapping{}, raw_text_hash, extraction_confidence.

    5.3 이 엔진은 다음을 수행한다:

        [Layer 7: ANALYSIS — 분석 및 충돌 해결]

        5.3.1 심화 분석 (B2 트렌드 분석)
            E-15에서 전달받은 추출 키워드를 기존 키워드 DB와 대조
            키워드 빈도 변화 추적, 시계열 트렌드 분석, 기초 예측 모델 적용
            기술: 규칙 기반 분석(V1), Postgres 집계 + 시계열(V2+)

        5.3.2 교차 소스 분석 (B3)
            여러 소스에서 추출된 키워드의 일관성 검사
            소스 간 상관관계 분석, 갭(Gap) 분석
            기술: Cross-Source Analyzer(V2+)

        5.3.3 패턴 인식 (B4)
            키워드·소스·VAMOS 매핑 패턴 식별
            반복되는 성공/실패 패턴을 패턴 DB에 축적
            기술: 패턴 DB(Postgres, V2+), GraphRAG(V3)

        5.3.4 충돌 해결 엔진 (B1)
            동일 주제에 대해 소스 간 상반된 정보 발견 시 해결:
            Step 1: 소스 신뢰도 가중치 적용
              - 공식문서(1.0) > 논문(0.9) > 기술문서(0.85) > 기술블로그(0.7) > 뉴스(0.6) > 개인블로그(0.5) > SNS(0.3)
            Step 2: 다수결 원칙 적용
              - 3개 이상 소스 일치 → 자동 채택
              - 2개 소스 일치 → 높은 신뢰도 소스 우선 + 경고 플래그
              - 단일 소스만 존재 → 신뢰도 검토 + 경고 표시
              - 모두 불일치 → CONFLICT_FLAG=true, Human Review 대기열 등록
            Step 3: 결과 기록 (adopted_value, conflict_sources, resolution_method, confidence, alternatives)
            기술: 우선순위 규칙(V1), Postgres + 충돌 해결 알고리즘(V2+)

        [Layer 8: VALIDATION — 5-Gate 검증 체계]

        5.3.5 G0 FORMAT GATE (형식 검증)
            템플릿 섹션 완전성, 필수 필드 존재, 마크다운 문법, 테이블 구조, 코드 블록 닫힘
            통과 기준: 100% 충족 (형식은 타협 불가)
            실패 시: 자동 수정 시도 → 불가 시 거부

        5.3.6 G1 CONTENT GATE (내용 검증)
            키워드 최소 10개 추출, 우선순위 분류(CRITICAL/HIGH/MEDIUM/LOW), 검색 쿼리 예시, VAMOS 매핑, 출처 명시
            통과 기준: 95% 이상 충족
            실패 시: 누락 항목 보완 요청

        5.3.7 G2 CONSISTENCY GATE (일관성 검증)
            기존 키워드 중복 확인, 동의어 그룹 일관성, 우선순위 논리성, VAMOS 계층 매핑 정확성, 이전 버전 호환성
            통과 기준: 90% 이상 충족
            실패 시: 충돌 항목 표시 + 해결 제안

        5.3.8 G3 SECURITY GATE (보안 검증)
            악성 URL 필터링, 민감 정보 누출 방지, 저작권 침해 필터, 스팸/광고 필터, 비윤리적 콘텐츠 필터
            통과 기준: 100% 충족 (보안은 타협 불가)
            실패 시: 즉시 거부 + 알림

        5.3.9 G4 FINAL GATE (최종 검증)
            G0~G3 모두 통과 확인, 변경 영향 범위 분석 완료, 롤백 가능 여부 확인, Human 승인 필요 여부, 배포 준비 상태
            통과 기준: 체크리스트 100% 완료
            통과 시: 버전 업데이트 + 배포

        Gate 실패 피드백 루프:
            Gate 거부 시 S-5 → E-15에 재수집 요청:
            1) 거부 사유 분석 (소스 품질 문제? 데이터 부족? 충돌?)
            2) E-15에 소스 변경 또는 추가 소스 탐색 요청
            3) 재수집 후 다시 Gate 검증 파이프라인 진입

        PolicyGate/CostGate 연동:
            G3 Security Gate → VAMOS RULE 윤리/법규 정책(Section 5) 동기화
            CostGate → V1: E-15만 동작(수집→저장), S-5 수동 트리거 / V2: E-15+S-5 반자동 / V3: 전체 자율
            Brain Router(I-1)가 비용 모드에 따라 Cloud Library 호출 수준 제어

        [Layer 9: VERSION CONTROL — 버전 관리]

        5.3.10 Semantic Versioning 엔진
            MAJOR.MINOR.PATCH 자동 결정:
              - MAJOR: 호환성 깨지는 구조 변경 (breaking changes)
              - MINOR: 키워드 10개 이상 추가 또는 새 섹션 추가
              - PATCH: 오타 수정, 소규모 키워드 추가
            Changelog Generator: Added/Changed/Fixed/Statistics 형식 자동 생성
            Checkpoint Manager: 단계별 체크포인트 저장 (롤백 가능 기간 7일)
            Rollback System: 정확도 10% 이상 하락 시 자동 롤백, 롤백 후 동일 변경 재시도 24시간 금지

        5.3.11 진화 안전장치
            Rate Limiting: 하루 최대 버전 증가 5회, 연속 실패 3회 시 자동 중단, 쿨다운 1시간
            Change Size Limit: 단일 변경 최대 키워드 50개, 최대 파일 5개, 초과 시 분할 또는 Human 승인
            Rollback Policy: 롤백 가능 기간 7일, 정확도 10% 이상 하락 시 자동 롤백
            Audit Trail: 모든 자동 결정 로깅 필수, 결정 근거(reason) 필수 기록, 주간 진화 보고서 자동 생성

        [Layer 10: OUTPUT — 출력 및 진화]

        5.3.12 VAMOS Document Updater
            RULE/PLAN/DESIGN/SCHEMA 4계층 문서 자동 업데이트
            Gate G4 통과 데이터를 해당 계층 문서에 반영

        5.3.13 Dashboard & Reporting (D3)
            실시간 수집 현황, 키워드 통계, 버전 이력 타임라인, 진화 로그 시각화
            수집 ROI 추적: E-15 수집 → I-2 RAG 삽입 → 실제 활용 비율 (N4)

        5.3.14 Multi-Format Export (D2)
            V2: Markdown, JSON, YAML, CSV
            V3: + Excel, PDF

        5.3.15 Notification System (D4)
            Human Review 요청, 에러 알림, Gate 실패 피드백 (이메일/웹훅)
            승인 대기열: MANUAL(30-59점) 사이트, CRITICAL 변경, Gate 3회 연속 실패

        5.3.16 Collaboration Tools (D5)
            외부 협업 도구 연동: V2 파일 공유 수준, V3 API 기반 통합

        5.3.17 Self-Evolution Module (S Feature: S1~S5 자가 진화 사이클)
            S1 PATTERN RECOGNITION: 키워드 추출 결과 분석, 성공/실패 패턴 식별, 새 용어 탐지, 소스별 품질 추적
            S2 RULE GENERATION: 새 키워드 카테고리 제안, 검색 쿼리 패턴 개선, 템플릿 최적화, 소스 우선순위 조정
              V2+ 보완: DSPy + LangGraph + Instructor로 규칙 생성 패턴화
            S3 VALIDATION: 제안된 규칙 시뮬레이션, 기존 데이터 영향 분석, 예상 정확도/커버리지 계산, Gate G0-G4 자동 통과 시험
            S4 DEPLOYMENT: Human-in-the-Loop 승인(CRITICAL 변경), 자동 배포(MINOR), 체크포인트 생성, 변경 이력 기록
            S5 FEEDBACK LOOP: 배포 후 성과 측정(예상 vs 실제), 실패 시 롤백 + 원인 분석, 성공 패턴 강화(패턴 DB)

        5.3.18 데이터 품질(QoD) 평가 (기존 I-19 기능 통합)
            출처 신뢰도, 최신성, 구조화 정도, 중복도, 위험성 등 스코어링
            RAG/메모리에 넣을 값어치가 있는지 판단
            낮은 QoD 데이터는 가중치를 낮추거나 RAG에서 제외
            중복/유사 문서는 대표본만 남기고 정리

        5.3.19 소스 평판 자동 업데이트 (N5)
            수집 데이터의 Gate 통과 비율로 소스 평판 점수 자동 조정
            저품질 소스 자동 감점, 고품질 소스 신뢰도 상향

        5.3.20 데이터 신선도 관리 — TTL (N2)
            기술 트렌드: 30일 TTL → 자동 재수집
            뉴스/SNS: 7일 TTL → 폐기 또는 아카이브
            학술 논문: 365일 TTL → 유효성 재검토
            공식 문서: 90일 TTL → 변경 확인 후 갱신

        5.3.21 점진적 RAG 임베딩 (N8)
            전체 재임베딩 대신 변경 부분만 증분 임베딩하여 I-2 업데이트
            Gate G4 통과 후 자동 임베딩 → I-2 벡터 DB 삽입 트리거

0. 이 문서의 목적

    0.1 개요
        1) 이 문서는 “어떤 파일에 무엇을, 언제, 어떻게 저장할지”를
           파일 단위로 세부적으로 정의한 계획서이다.

    0.2 각 파일 정의 공통 항목
        각 파일(또는 파일 패턴)에 대해 다음을 모두 명시한다.

        1) Path: 실제 저장 경로 형식
        2) Purpose: 이 파일이 존재하는 이유
        3) Schema: 어떤 데이터를 어떤 필드로 저장하는지(예시 스키마 포함)
        4) Write Rules: 언제 생성/갱신/삭제하는지
        5) Retention & Lifecycle: 보존 기간, 아카이브/삭제 규칙
        6) Theme / Tag / Vector: 테마/태그/벡터화 전략
        7) Safety: 민감 정보, 동의, 정책 처리

# 🚀 VAMOS UI·UX PLAN 1.0
(최종 방향성 & 목표 정의 문서 – 미세 튜닝 제외)
1. 전체 UI·UX 철학 (Core Philosophy)
    VAMOS UI·UX는 단순한 채팅형 AI가 아니라
    개인 AGI를 운영·개발·사용하는 이중 구조 시스템이다.

    1.1 핵심 철학 요소

        1.1.1 AGI 구조 시각화 중심
            1) ORANGE CORE와 BLUE NODE를 Sphere 형태로 시각화한다.
            2) CORE ↔ NODE 연결선은 실제 데이터 플로우·정책·라우팅을 반영한다.

        1.1.2 엔지니어링 모드와 사용자 모드 분리
            1) Builder / Lab View: 설계·정책·Self-evo 조정
            2) Hologram / Experience View: 대화·이미지·음성 사용

            1.1.2.1 분리 목적
                1) 정확성 유지
                2) 품질 관리
                3) 개발 효율 증가
                4) 사용자 경험 단순화

        1.1.3 도메인 전용 AI(BLUE NODE) 확장 환경
            1) BLUE NODE는 필요 시 생성하거나 프로젝트 생성 시 자동 생성
            2) BLUE NODE는 독립된 템플릿·정책·라우팅 구조를 가짐

        1.1.4 Self-check → Self-evo 루프 연결
            1) 모든 행동은 Builder에서 품질 평가 가능
            2) Builder 수정 사항은 즉시 사용자 창에 반영
            3) 전체 AGI 품질·정확성·효율 유지

------------------------------------------------------------
2. 전체 UI 구조 (Two Window System)
    VAMOS UI는 두 개의 메인 창으로 구성된다.
        1) Builder / Lab View (엔지니어링 창)
        2) Hologram / Experience View (사용자 창)
------------------------------------------------------------
 2.1 Builder / Lab View (엔지니어링 창)
    AGI 구조·정책·템플릿·Self-evo를 관리하는 개발 공간

    2.1.1 목적
        1) 내부 기능(I) / 외부 기능(E) / Self-evo(S) 관리
        2) 정책·템플릿·라우팅 수정
        3) 오류 추적
        4) 비용 관리
        5) 전체 AI 정확성 유지

    2.1.2 구성 (3패널 레이아웃)

        2.1.2.1 좌측 패널 — 구조/정책/템플릿 트리
            1) spec/
            2) router_config/
            3) storage_policy.json
            4) user_rules.json
            5) templates/
            6) workflows/
            7) memory/global/, memory/project/

            2.1.2.1.1 특징
                1) VS Code 트리 구조
                2) 클릭 시 중앙 패널에서 편집
                3) 스키마 기반 자동 검증 제공

        2.1.2.2 중앙 패널 — 모듈·파이프라인·Self-evo 대시보드
            1) I/E/S 모듈 그래프
            2) 실제 파이프라인 흐름
            3) Self-check 점수
            4) Self-evo 분석 결과
            5) 실패 패턴 분석
            6) 템플릿·정책 개선 추천 목록
            7) 비용 그래프

            2.1.2.2.1 역할
                1) 정확성 관리 핵심 화면
                2) 어떤 모듈이 결과에 영향했는지 추적 가능

        2.1.2.3 우측 패널 — 로그/비용/오류 패널
            1) pipeline logs
            2) trace logs
            3) error logs
            4) 라우팅 결정 이유
            5) 비용 사용량
            6) 경고·리스크 알림

            2.1.2.3.1 역할
                1) 오류 분석
                2) 라우팅 품질 파악
                3) 개선 포인트 확인

    2.1.3 Builder View 결과 반영
        1) 모든 수정 사항은 즉시 ORANGE CORE에 반영
        2) BLUE NODE 행동도 즉시 업데이트됨
 -----------------------------------------------------------
 2.2 Hologram / Experience View (사용자 창)
    사용자가 실제로 사용하는 메인 인터페이스

    2.2.1 목적
        1) CORE + BLUE NODE 시각화
        2) 전문 전용 AI 사용
        3) 대화·음성·이미지 입력 제공
        4) 직관적 AGI 경험 제공

    2.2.2 구성 (3구역)

        2.2.2.1 중앙 — ORANGE CORE + BLUE NODE 패널

            2.2.2.1.1 ORANGE CORE 구성
                1) 전체 지능(메모리, 정책, 라우터, Self-evo)
                2) 지식량·품질·오류 수준에 따라 크기/광도 변화

            2.2.2.1.2 BLUE NODE 구성
                1) 트레이딩 AI
                2) 코딩 AI
                3) 법률 QA
                4) 게임 공략 AI
                5) 사용자 생성 프로젝트 AI

            2.2.2.1.3 BLUE NODE 반응
                1) 활성도에 따라 그래픽 변화
                2) 정확성에 따라 연결선 밝기 변화

        2.2.2.2 하단(혹은 우측) — 대화/입력 패널
            1) 텍스트 입력
            2) 이미지 업로드
            3) 음성 입력
            4) 파이프라인 타임라인
            5) Self-check 표시
            6) 비용 모드 선택
            7) BLUE NODE 선택 후 전용 AI와 대화

        2.2.2.3 좌측(또는 상단) — 프로젝트/BLUE NODE 리스트
            1) 프로젝트 목록
            2) BLUE NODE 목록
            3) NEW BLUE NODE 생성
            4) 최근 사용·즐겨찾기 정렬

            2.2.2.3.1 역할
                1) 프로젝트 전환
                2) 도메인 전용 AI 전환

        2.2.2.4 우측 — 문서/메모리/컨텍스트 패널
            1) 대화 맥락
            2) 프로젝트 메모리
            3) RAG 문서 트리
            4) 인용 문서 강조
            5) 근거 기반 표시

------------------------------------------------------------
3. 전체 Workflow / UX 흐름

    3.1 기본 실행 루프
        1) 사용자 → BLUE NODE 또는 CORE에 요청
        2) 내부 Pipeline 실행(I/E/S)
        3) 결과가 Hologram View에 표시
        4) Self-check 수행
        5) Self-evo 분석
        6) Builder View에 개선 항목 표시
        7) 수정 사항 CORE 반영
        8) BLUE NODE 즉시 강화됨

------------------------------------------------------------
4. 오버레이 & CLI 구조

    4.1 오버레이 패널
        1) 단축키: Ctrl + Shift + Space
        2) 화면 부분 캡처 후 BLUE NODE로 즉시 분석 요청
        3) 실시간 요약/분석/코딩 헬퍼
        4) "자세히 보기" 클릭 시 Hologram View로 전환

    4.2 CLI / Terminal
        4.2.1 명령어
            1) vamos chat
            2) vamos run workflow <이름>
            3) vamos self-evo report
            4) vamos log tail pipeline
        
        4.2.2 역할
            1) 자동화
            2) 배치 실행
            3) 디버깅

    6.1 정확성
        1) 모든 파이프라인 흐름 시각화
        2) 인용 문서 근거 기반 표시

    6.2 품질
        1) Self-check → Self-evo 루프
        2) 오류·부정확성 자동 탐지
        3) BLUE NODE 품질 점수 관리

    6.3 효율성
        1) 두 창 분리로 명확한 작업 흐름

        3) Builder에서 정책·템플릿 즉시 수정

    6.4 확장성
        1) BLUE NODE 무한 확장

    1) UI는 ORANGE CORE + BLUE NODE 시각 구조를 기준으로 한다.
    2) Builder View와 Hologram View의 2창 구조는 정확성·품질·효율성을 극대화한다.
    3) 모든 동작은 Self-evo 루프에 연결되어 AGI가 지속적으로 발전한다.
    4) 오버레이와 CLI는 즉시성·생산성을 높이는 보조 구조다.
    5) 상세 UI 튜닝은 이후 단계에서 수행한다.
