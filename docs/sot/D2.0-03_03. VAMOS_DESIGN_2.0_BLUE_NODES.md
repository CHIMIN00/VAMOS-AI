## 1. BLUE NODE 정의/역할/제약(P0/P1/P2)
### 1.1 정의 (RULE 1.3 근거)
- BLUE NODE는 **도메인 실행 모듈**이다. (RULE 1.3 §4.2)
- BLUE NODE는 ORANGE CORE의 규칙을 **상속**하며, **독립 실행 불가**다. (RULE 1.3 §4.2)

### 1.2 역할
- CORE의 판단/정책/비용/승인 결정을 "실행 단계"로 옮기는 도메인 실행 계층
- 외부 도구/리소스는 NODE 실행 기반(도구 및 외부 자원은 NODE의 실행 기반) (RULE 1.3 §1.3 철학적 전제)

### 1.3 제약 (RULE 1.3 근거)
- **Non-goal/안전/비용 상한/승인 구조는 변경 불가** (RULE 1.3: "비용 상한·정체성·Non-goal·승인 구조는 절대 변경 불가")
- P2 관련:
  - **P2 도메인 자동 생성 금지** (RULE 1.3 §2.6)
  - P2 도메인 활성화는 **명시적 승인 + 세션별 재확인** (RULE 1.3 §3.3)
  - P2 도메인은 **세션 종료 시 자동 OFF** (RULE 1.3 §3.3)

### 1.4 역할 분해(멀티 에이전트) 흡수 규칙(P2 반영 / FREEZE 확정)
- 정의: AutoGen 스타일의 역할 에이전트(Researcher/Coder/Critic 등)는 VAMOS에서는 "BLUE NODE 내부의 Sub-agent 템플릿" 또는 "노드 역할 프리셋"으로만 흡수한다.
- 금지:
  - Sub-agent 간 직접 상호 호출/자유 대화 루프는 금지한다.
  - 모든 호출/실행은 ORANGE CORE의 라우팅/결정 및 07 Gate를 경유한다.
- 연결 참조:
  - 단일결정/라우팅/결정 잠금: 02 참조
  - 실행 파이프라인/루프/로그: 05 참조
  - 승인/정책/비용 게이트: 07 참조

## 2. 도메인 계층 구조(도메인/부도메인/서브에이전트)
### 2.1 용어 (RULE 1.3 근거)
- 도메인: 분야 단위(예: Dev, Research, Productivity 등) (RULE 1.3 §4.3)
- 부도메인: 해당 도메인의 세부 실행 단위 (RULE 1.3 §4.3)
- 우선순위 분류: P0/P1/P2 (RULE 1.3 §4.3)

### 2.2 계층 구조 원칙
- CORE는 "판단/통제", NODE는 "실행" (RULE 1.3 §1.3 철학적 전제)
- 서브에이전트는 "NODE 내부 실행 단위"로 취급하며, CORE 규칙(Non-goal/승인/비용/저장)을 동일하게 상속한다.

---


### 2.3 NODE ↔ OTHER BRAINS / ToolRegistry 편입 규칙 (Design 2.0 / PHASE 3) (NEW)

- 목적: BLUE NODE가 "도메인 두뇌"로서 외부 실행/도구/브라우저/코드실행 등(OTHER BRAINS)을 **직접 호출하지 않고**,
  **ORANGE CORE(I-5 Decision)** 의 단일결정과 **INFRA_CORE(04)의 ToolRegistry**를 통해서만 호출하도록 경로를 고정한다.
- 원칙(LOCK):
  1) Blue Node는 "의도/계획/필요 리소스"를 제안할 수 있으나, 실제 실행은 **Decision.gates.result**가 allow/restrict 범위를 확정한 뒤에만 가능하다. (02 §Decision 확장 참조)
  2) 도구 호출은 반드시 `tool_id`로 표준화하고, 도구의 위험/비용/정책 요구는 04의 ToolRegistry에 의해 고정된다.
  3) Node는 "도구 세부 구현(Playwright/Selenium/API SDK 등)"을 숨기고, ToolRegistry가 선택한 adapter를 통해 실행한다.

- NodeCapabilityProfile (개념 스키마):
  - node_id
  - required_tools: [tool_id...]
  - optional_tools: [tool_id...]
  - risk_class: low|med|high
  - cost_class: v0|v1|v2|v3
  - gates_required: ["policy","cost","approval","evidence","self_check"]

- 연결 참조:
  - 02(ORANGE CORE) — Decision 확장: `optional_signals`, `verify.chain_used`, `gates.result`
  - 04(INFRA CORE) — ToolRegistry / OTHER BRAINS 매핑
  - 07(SAFETY/COST/APPROVAL) — PolicyCheck/Approval/CostBudget 정본 ENUM



#### STEP7-K 보강: MCP 서버 구현 아키텍처 상세 (K-001)

> MCP 서버의 구체적 구현 아키텍처를 §2.3의 ToolRegistry 편입 규칙에 따라 상세화한다.

- MCP 서버 타입:
  - Stdio MCP Server: 로컬 프로세스 (V1 기본)
  - Streamable HTTP: 프로덕션 기본 전송 계층 (DEC-017 LOCK)
  - WebSocket MCP Server: 양방향 실시간 (V3)

- VAMOS MCP 서버 디렉토리 구조:
  ```
  vamos-mcp-server/
  ├─ tools/           # Tool 정의 (20+ 도구)
  ├─ resources/       # Resource 제공 (메모리, KG)
  ├─ prompts/         # Prompt 템플릿
  ├─ sampling/        # LLM 샘플링 요청
  └─ server.py        # MCP 서버 엔트리포인트
  ```

- MCP 도구 목록 (V1 기본 20+):
  1. memory_search: 4계층(L0~L3) 메모리 시맨틱 검색
  2. memory_store: 메모리 저장
  3. kg_query: 지식그래프 쿼리

## 3. NODE 생성/활성화 라이프사이클
### 3.1 P0 기본 활성 범위
- P0는 기본 활성 도메인("기본 안전 범위")로 취급한다.
- 단, P0라도 외부 서비스/API 사용은 약관 준수 체크리스트를 통과해야 한다. (RULE 1.3 §3.1)
- P0 기본 활성 범위는 넓게 가져가되, 도구/저장/비용/민감정보는 07 게이트로 강하게 통제한다. (결정 완료: 7-1 원칙)

### 3.2 P1 후보 생성→승인→활성
- P1은 "도메인 확장" 범주로 취급하며, **사용자 승인 필수**에 포함된다. (RULE 1.3 §9.2: 승인 필수 항목에 "도메인 확장(P1/P2)" 포함)
- 승인 구조는 RULE 1.3의 **2단계 승인**을 따른다:
  - 1단계 계획 승인: 무엇을 할지 설명 + 승인
  - 2단계 실행 승인: 지금 실행할지 최종 확인 (위험 작업 필수) (RULE 1.3 §9.2)

  - 결정 완료: 옵션 A/B "둘 다 허용"
    - 채널 A: CORE 제안형(필요성 감지 → 후보 프로필 생성 → 승인 요청)
    - 채널 B: 사용자 요청형(명시 요청 → 후보 프로필 생성 → 승인 요청)
  - 충돌 방지: 어떤 채널이든 "활성화"는 승인(계획→실행) 없이는 불가하며, 자동 실행으로 넘어가지 않는다.

- P1에서 자동화는 "후보 프로필 생성"까지만 허용하며, 활성화는 사용자 승인 이후에만 가능하다. (체크리스트 3.2 반영)

### 3.2.1 노드 생성/활성/상한/트리거 규칙 확장 (A-2~A-7 + D-6/C-7 편입) (Design 2.0 / PHASE 3) (NEW)

> 목적: "모듈 폭발 방지 + 자동화는 후보 생성까지만 + 실행은 Decision/Gate 확정 후" 원칙을, A/D/C 아이디어에 맞춰 BLUE NODE 라이프사이클에 직접 편입한다.

- 적용 아이디어(근거):
  - A-2 사전 설정 모듈화: NodeCapabilityProfile/TemplateSet을 사전 정의하여 '반복 설정'을 제거
  - A-3 메타 AI(Module Manager): 후보 생성/캐싱/상한 관리/만료 정책을 담당(단, 활성화는 항상 승인)
  - A-4 Debate Mode: "서로 상충하는 실행안/근거"가 존재하면 활성화 전 `verify.chain_used`에 Debate 단계를 포함하도록 제안(결정은 02가 수행)
  - A-5 Lazy Module Generation: **필요 시에만** 후보 생성(자동 생성은 P1까지만), 불필요한 신규 NODE 생성 금지
  - A-6 Federated Module Network: NODE 실행은 '분산된 도메인 런타임/어댑터'로 위임 가능하되, 경로는 ToolRegistry(04)로만 고정
  - A-7 Remote Executor Pool: 원격 실행은 **RE-ADD(기본 OFF)** 로 취급, 활성화 시 07 Approval + CostGate를 필수 통과
  - D-6 GraphRAG: 지식 탐색이 "관계 기반"이어야 할 때 GraphRAG Tool/Adapter를 경유(04), 결과는 optional_signals로 귀결
  - C-7 Domain Simulator: '현실 시뮬레이션'이 필요한 도메인(예: 정책/재무/공학 등)은 Simulator Tool/Adapter를 경유(04)

#### (A) 생성 트리거(후보 생성) — P1 전용
- 트리거(예시 규칙):
  1) 기존 NODE/템플릿으로 목표 달성이 불가능하거나 비용·정책 제약으로 계속 실패하는 경우
  2) 도메인 특화 검증/실행(예: GraphRAG/Simulator)이 필요하다고 02(I-5 Decision)이 판정한 경우
  3) 사용자가 "새 도메인/기능 추가"를 명시 요청한 경우
- 제한:
  - P2는 자동 생성 금지(§1.3) — 후보 생성도 불가
  - 후보 생성은 **프로필 생성 + 승인 요청**까지만 허용(§3.2), 자동 활성화/자동 실행 금지

#### (B) 상한(Cap) — 모듈 폭발 방지(LOCK)
- 목적: Lazy Generation(A-5)의 실효성을 위해 '상한'을 문서 차원에서 명시한다.
- 상한 규칙(LOCK):
  - `active_node_cap`: 세션당 활성 NODE 상한. **V1=3** (Lead+2Sub), V2=10, V3=50 (LOCK-AT-014 정렬, PHASE_B4 §3.13 참조)
  - `candidate_node_cap`: 세션당 후보 NODE 상한. **V1=5**, V2=20, V3=100 (PHASE_B4 §3.13 참조)
  - 상한 도달 시 동작: 신규 후보 생성 금지 → 기존 NODE 재사용 또는 "범용 NODE + ToolRegistry"로 대체 실행

#### (C) 활성 트리거(승인 후 ON)
- 활성 조건(필수):
  1) 07 Approval 결과가 allow
  2) 02 Decision.gates.result가 allow/restrict 범위를 확정
  3) 04 ToolRegistry가 실행 경로(tool_id/adapter/runtime)를 확정

#### (D) 원격 실행(A-7) 편입 규칙 (RE-ADD)
- 원격 실행은 기본 OFF이며, 아래 조건을 모두 만족할 때만 사용 가능:
  - 07 Approval: 원격 실행(외부 런타임/서버) 승인
  - 07 CostGate: 해당 실행의 cost_class가 예산 내
  - 04 ToolRegistry: remote_executor 계열 tool_id가 등록되어 있고 정책 요구를 만족
- 기록: 원격 실행이 사용된 경우 반드시 Decision.optional_signals에 `remote_executor_used=true`를 남긴다.

#### (E) D-6/C-7 실행 결과의 Decision 귀결(필수)
- GraphRAG/Simulator 실행 결과는 "NODE 결과"로 끝내지 않고, 아래 필드에 귀결한다(02 정본):
  - `Decision.optional_signals`: graph_rag_hit / sim_outcome / conflict_flag 등
  - `Decision.verify.chain_used`: 실행 과정에서 실제 사용한 EVX/Verifier 체인
  - `Decision.gates.result`: 최종 policy/approval/cost/evidence 요약

- 연결 참조(요구사항 #D3→D2):
  - D3(본 문서)에서 정의한 **노드 생성/활성/상한/트리거**는, 최종적으로 D2(02 ORANGE CORE)의 Decision 레코드에 기록되어야 한다.
  - 따라서 모든 NODE 실행은 `decision_id`를 입력으로 받고, 실행 후 결과를 Decision 확장 필드로 "반드시" 반영한다. (02 §Decision 확장 참조)

- 추가 문장(LOCK):
  - BLUE NODE의 모든 실행 산출(신호/로그/아티팩트)은 최종적으로 02 Decision 확장 필드(optional_signals / verify.chain_used / gates.result)에 "참조(ref)" 형태로 귀결되어야 한다.

### 3.3 P2 도메인 라이프사이클 (결정 완료/LOCK)

#### 3.3.1 적용 범위(중요)
- 이 P2 라이프사이클 규칙은 **"P2가 승인으로 활성화되는 세션"**에서만 적용된다.
- 기본 정책이 "P2 OFF"인 동안에는, 아래 규칙은 **'미래 확장 대비 규칙'**로만 유지한다.

#### 3.3.2 자동 비활성화 규칙

- 선택(LOCK): **Option A — 세션 종료 시 즉시 OFF**
- 규칙:
  - 세션 종료 시: P2는 **즉시 OFF**로 전환된다.
  - UI/로그에는 "세션 종료로 OFF"가 남아야 한다. (08 UI 규칙 참조)

- 참고(DEFER):
  - "비활성 타임아웃 기반 경고 후 OFF"는 운영 요구(V2+)가 생길 때만 재검토한다.

#### 3.3.3 활성화 조건
1) 사용자 명시적 승인 요청
2) 승인 시 **해당 세션에서만** 활성

#### 3.3.4 자동 비활성화 (LOCK)
- 본 규칙은 3.3.2 "자동 비활성화 규칙(LOCK)"을 따른다.
- 중복 서술을 방지하기 위해 3.3.4에서는 별도 규칙을 추가하지 않는다.

#### 3.3.5 로그/이벤트
- 모든 P2 활성화/비활성화는 `oc.p2.activated` / `oc.p2.deactivated` 이벤트로 기록한다.
- 이벤트 값 목록/정의(SOT)는 02(ORANGE CORE) §6.1(LogEvent Registry)를 정본으로 한다.

#### 3.3.6 메모(향후 DEFER)
- 24시간 운영(상시 실행) 단계에서만 "비활성 타임아웃 후 OFF" 옵션을 재검토한다.

> 근거 연결:
- "P2 활성화: 명시적 승인 + 세션별 재확인", "세션 종료 시 자동 OFF"는 RULE 1.3 §3.3 근거
- "P2 도메인 자동 생성 금지"는 RULE 1.3 §2.6 근거

### 3.4 외부 Agent/Workflow 프레임워크 적용 원칙(P2 반영 / FREEZE 확정)
- 정의: LangGraph / AutoGen / CrewAI / LlamaIndex / ReAct 등은 "워크플로우 실행 엔진(Workflow Engine)" 후보일 뿐이며, VAMOS의 정본(SOT)을 대체하지 않는다.
- 고정: 최종 Decision(단일결정)과 Gate 결과(PolicyCheck/Approval/CostBudget)는 항상 ORANGE CORE(02) → SAFETY/COST/APPROVAL(07) 경로로만 확정된다.
- 금지: 외부 프레임워크가 "승인/정책/비용"을 에이전트 내부 if문으로 임의 처리하여 Gate를 우회하는 구조는 허용하지 않는다. (우회 위험)
- 역할 분리:
  - 외부 프레임워크: 노드/스텝 실행, 상태 전이, 체크포인트/리플레이 같은 "실행 오케스트레이션"만 담당
  - VAMOS CORE(02/07): 결론 확정, 승인 요구/차단, 비용 상한/다운시프트, 정책 판정, 감사 로그 스키마를 담당
- 연결 참조:
  - 실행 워크플로우/루프/재현성(Checkpoint/Replay/Fork) 규칙: 05 참조
  - 승인/정책/비용 Gate 스키마 및 이벤트: 07 참조
  - UI에서 Gate/Trace 표시: 08 참조

#### STEP7-K 보강: A2A 프로토콜 통합 (K-011~K-020)

> A2A(Agent-to-Agent) 프로토콜은 §3.4의 "외부 프레임워크 적용 원칙"을 따르되, 에이전트 간 통신 표준으로서 MCP와 상호보완적 역할을 한다.

**(K-011) A2A 프로토콜 구현 (Google A2A Protocol 기반)**
- A2A 핵심 개념:
  - Agent Card: 에이전트 능력 선언 (JSON-LD)
  - Task: 에이전트 간 작업 단위
  - Message/Part: 멀티모달 메시지 교환
  - Artifact: 작업 결과물
  - Streaming: Streamable HTTP 기반 실시간 통신 (VAMOS 내부. A2A 외부 연동 시 상대 에이전트 프로토콜에 따라 SSE 허용, 단 VAMOS 자체 연결은 DEC-017 LOCK 준수)
- VAMOS A2A Agent Card:
  ```json
  {
    "name": "VAMOS AI",
    "description": "Self-evolving personal AGI assistant",
    "url": "https://vamos.local/a2a",
    "capabilities": {
      "streaming": true,
      "pushNotifications": true,
      "stateTransitionHistory": true
    },
    "skills": [
      {"id": "code-dev", "name": "Software Development"},
      {"id": "investment", "name": "AI-Powered Investment Analysis"},
      {"id": "research", "name": "Deep Research"},
      {"id": "content", "name": "Content Creation"}
    ]
  }
  ```
- VAMOS 위치: A2A(에이전트 간 통신) + MCP(도구 호출) 양쪽 모두 지원하여 최대 호환성 확보
- 구현성: V2 3개월 | V3 풀 A2A

**(K-012) A2A Task Lifecycle 관리**
- Task 상태 머신: `submitted → working → input-required → completed/failed/canceled`
- VAMOS A2A Task 처리 흐름:
  1. 외부 에이전트 → VAMOS: Task 제출
  2. VAMOS: Task 수신 → 적합한 Blue Node 라우팅 (02 Decision 경유)
  3. Blue Node: 작업 수행 → 중간 결과 스트리밍
  4. VAMOS → 외부 에이전트: 결과 반환 (Artifact)
- 장기 실행 Task: 비동기 처리(push notification), 진행률 업데이트, 취소 지원, 타임아웃 설정
- 구현성: V2 3개월

**(K-013) A2A 에이전트 디스커버리**
- 에이전트 검색 메커니즘:
  - `/.well-known/agent.json`: 표준 엔드포인트
  - 에이전트 레지스트리: 중앙 디렉토리 (V3)
  - P2P 디스커버리: 로컬 네트워크 에이전트 탐색
  - 추천: "이 작업에 적합한 에이전트" 자동 제안
- Agent Card 캐싱: 발견된 에이전트 정보 로컬 저장
- 신뢰 평가: 에이전트 신뢰도 점수 관리
- 구현성: V2 2개월

**(K-014) A2A 멀티에이전트 협업 패턴**
- 협업 패턴:
  - Delegation: VAMOS → 외부 에이전트에 작업 위임
  - Collaboration: 여러 에이전트 동시 작업
  - Pipeline: 에이전트 A 결과 → 에이전트 B 입력
  - Competition: 여러 에이전트 경쟁 → 최적 결과 선택
  - Consensus: 다수결/투표로 최종 결정
- VAMOS 내부 에이전트 협업 (Blue Node 간):
  - Dev Node + Research Node: 기술 조사 → 코드 구현

  - Content Node + Dev Node: 기술 문서 작성
  - 전체 Node 협업: 복합 프로젝트
- VAMOS 연동: 모든 협업은 ORANGE CORE(02) 라우팅/결정을 경유하며, 직접 상호 호출 금지 (§1.4 준수)
- 구현성: V2 3개월

**(K-015) A2A 보안 및 신뢰**
- 인증/인가:
  - OAuth 2.0: 에이전트 간 인증

  - 블랙리스트/화이트리스트

## 4. NODE 표준 구성요소
### 4.1 도메인 정책(상속+오버라이드 규칙)
- 기본 원칙: NODE는 CORE 정책을 상속한다. (RULE 1.3 §4.2)
- 오버라이드 허용 범위(안전한 방향만):
  - 허용: 더 엄격한 제한(Restrict/Deny 강화), 더 낮은 비용/리소스 사용, 더 보수적 승인 요구
  - 금지: Non-goal 완화, P2 자동 실행/자동 생성, 비용 상한 완화, 민감정보 저장 허용 등 (RULE 1.3 절대 변경 불가 항목들)

### 4.2 템플릿 세트 소유/주입 방식 — 결정 완료(LOCK) (I-13 연동)

- 정본: 02(ORANGE CORE) §7.13(I-13 Template/Prompt Library) / 05(Agent Workflow) 템플릿 주입·실행 흐름
- 원칙: "템플릿 선택/주입"은 CORE의 단일결정(Decision) 흐름에 포함된다.
- NODE는 CORE가 결정한 템플릿/입력/제약을 받아 실행만 수행하며, 실행 중 임의 변경은 금지한다.
- P2 도메인 템플릿/확장 실행은 07(Approval Gate) 결과를 우선한다.

#### 적용 범위
- V1~V3 공통 원칙(템플릿의 "소유/버전관리"는 CORE)
- 단, V1은 기능 최소화/로컬 실행이므로 템플릿 세트 자체는 "필수 최소 셋"만 운영

#### 규칙
1) BLUE NODE는 템플릿을 "정본으로 소유하지 않는다."
   - NODE 내부에는 템플릿 원문이 아니라, "실행 인터페이스 + 도메인 로직 + 검증 로직"만 둔다.
2) CORE는 NODE 호출 시 **template_set_id(또는 template_bundle_id / template_id)** 를 포함해 "어떤 템플릿으로 실행할지"를 주입한다.
3) 템플릿 변경(개선/교체)은 CORE 쪽에서 버전업하며,
   - 계약(요청/응답 스키마)이 유지되는 한, BLUE NODE 코드는 수정 없이 동작해야 한다.
4) NODE가 임의 템플릿을 "승인/정책/비용 게이트 우회" 용도로 교체하는 행위는 금지한다.
5) 템플릿 정본/네이밍/버전 규칙은 02(ORANGE CORE)의 템플릿/레지스트리(REF 체계)와 연결한다.

- 확장 규칙:
  - 1.0에서는 3종만 "기본 활성(available)"로 두고,
  - 추가 TemplateSet은 P1 단계에서 "승인 후 등록"으로 확장한다(07 승인 정책 준수).

#### 연결 참조
- (LOCK) TemplateSet = 3종
  1) TS_CORE: 일반 대화/요약/정리(기본)
  2) TS_WEB_RESEARCH: 웹 리서치/근거 수집/인용 중심
  3) TS_CODE: 코딩/디버깅/리팩토링 중심
- 템플릿 세트의 구성/ID/네이밍 근거: 02(ORANGE CORE) I-13
- UI 표시: 08의 "Template Injection 표시 규칙(template_id/template_set_id 노출)"을 따른다.

## 5. ORANGE CORE ↔ NODE 인터페이스 계약
### 5.1 계약 원칙
- CORE는 판단/정책/비용/승인을 총괄하고, NODE는 실행을 담당한다. (RULE 1.3 §4.1, §1.3 철학)
- NODE는 독립적으로 "승인 구조/비용 상한/Non-goal"을 변경하거나 우회할 수 없다.

### 5.2 입력/출력/상태/이벤트(최소 명세)
- 입력: CORE가 승인/정책/비용 판단을 완료한 "실행 요청"과 필요한 컨텍스트(프로젝트/세션/도메인)
- 출력: 실행 결과(산출물/요약/근거) + 실패 시 deny 또는 fallback로 귀결
- 상태: P2는 기본 OFF, 승인된 세션에서만 ON (RULE 1.3 §3.3)

#### 5.3 CORE 교체/뇌 교체(예: LLM↔다른 추론 스택/TPU 등) 시 영향 범위 규칙

- 원칙: BLUE NODE가 의존하는 것은 "CORE 내부 구현(모델/하드웨어)"이 아니라 **CORE↔NODE 계약(Contract)** 이다.
- 따라서 아래 2개가 유지되면, CORE를 교체해도 BLUE NODE를 "전부 재코딩"할 필요는 없다.

1) Contract 고정
   - NODE 호출 입력/출력 형태(필수 필드, 에러/폴백 신호, 승인/비용/정책 참조)가 유지된다.
2) Adapter로 차단
   - CORE 내부의 모델/추론 스택 교체는 04(INFRA CORE)의 Adapter/Runtime 계층에서 흡수하고,
   - BLUE NODE는 동일한 실행 계약으로만 호출된다.

- 예외: Contract 자체(필드/의미)가 바뀌는 경우에만, 해당 NODE의 어댑터/검증부 일부 수정이 발생한다.

## 6.5 Claude Tool Use 스키마 및 MCP Bridge Layer (S09-B29-001, S09-B29-003 — CRITICAL)

### 6.5.1 Claude Tool Use 스키마 (S09-B29-001 — CRITICAL)

VAMOS는 Claude API의 Tool Use를 MCP tool_id와 연동하여 사용한다.

```json
{
  "type": "function",
  "name": "vamos_tool_{tool_id}",
  "description": "VAMOS MCP bridge: {tool_id} 실행",
  "parameters": {
    "type": "object",
    "properties": {
      "tool_id": {"type": "string", "description": "실행할 MCP tool_id"},
      "input": {"type": "object", "description": "tool_id의 input_schema에 따른 입력"},
      "decision_id": {"type": "string", "description": "연결된 Decision 레코드 ID"}
    },
    "required": ["tool_id", "input"]
  }
}
```

- Claude가 Tool Use로 `vamos_tool_{tool_id}` 호출 → MCP Bridge Layer에서 수신 → 04 ToolRegistry에서 adapter 선택 → MCP 프로토콜로 실행 → 결과 반환
- 모든 Tool Use 호출은 07 Gate를 통과해야 하며, 차단 시 Claude에 `tool_denied` 오류 반환

### 6.5.2 MCP Bridge Layer (S09-B29-003)

MCP Bridge Layer는 ORANGE CORE(02)와 실제 MCP 서버 사이의 어댑터 계층이다.

**[Bridge Layer 역할]**
- Claude Tool Use 요청 수신 → tool_id 매핑 → MCP 서버로 전달
- 결과 검증 (output_schema 준수 여부) → Decision.optional_signals에 기록
- 실패 시 fallback_tool_id로 자동 전환 또는 deny 반환

**[Bridge Layer 구현 원칙]**
- 상태 비저장(Stateless): Bridge는 요청-응답 단위로만 동작
- 멱등성(Idempotent): 동일 요청은 동일 결과를 보장
- 타임아웃 관리: config.toml의 timeout_tool_s 기준으로 강제 종료

## 6.7 MCP SDK 통합 및 운영 계층 (S07-B22-001~019)

### 6.7.1 MCP SDK 통합 모듈 (S07-B22-001~005)

**MCP Python SDK 통합 구조**

```python
# vamos/mcp/sdk_integration.py
from mcp import ClientSession, StdioServerParameters
from mcp.client.stdio import stdio_client

class MCPClientManager:
    """MCP 클라이언트 중앙 관리자"""
    def __init__(self, server_registry: dict):
        self.sessions: dict[str, ClientSession] = {}
        self.server_registry = server_registry  # §6.4 카탈로그 기반

    async def connect(self, server_id: str) -> ClientSession:
        params = self.server_registry[server_id]
        session = await stdio_client(StdioServerParameters(**params))
        self.sessions[server_id] = session
        return session
```

**Context Providers 통합 (@Docs/@Codebase)**
- `@Docs` 프로바이더: 외부 문서(URL, PDF) → MCP resource로 노출
- `@Codebase` 프로바이더: 로컬 코드베이스 → semantic search MCP 도구로 제공
- 두 프로바이더는 `tool_id: "context.docs@1.0"`, `"context.codebase@1.0"` 으로 tool_call_registry에 등록

**MCP 프로토콜 핸들러**
- Streamable HTTP 기본 핸들러: HTTP/1.1 + chunked encoding
- stdio 핸들러: 개발/테스트 환경 전용
- 공통 인터페이스: `MCPProtocolHandler.send(request) → response`

**서버 레지스트리 동적 로드**
- §6.4 서버 카탈로그 기반으로 `mcp_server_registry.json` 런타임 로드
- 핫 리로드: config 변경 시 재시작 없이 서버 목록 갱신 가능

**자원 관리**
- 동시 MCP 세션 상한: `max_concurrent_mcp: 10` (config.toml)
- 세션 풀(pool): 반복 연결 비용 절감, TTL 300초
- 미사용 세션: 60초 idle 시 자동 종료

### 6.7.2 인증 및 보안 계층 (S07-B22-006~008)

**MCP 인증 시스템**

| 인증 방식 | 사용 서버 | 토큰 관리 |
|---------|---------|---------|

### 6.7.3 캐시 계층 (S07-B22-009~010)

**MCP 응답 캐시 전략**
- 캐시 대상: `risk_class: low`인 읽기 전용 도구 결과
- 캐시 키: `sha256(tool_id + input_hash)`
- TTL: 도구별 설정 (`cache_ttl_s` in tool_call_registry, 기본 60초)
- 캐시 무효화: 외부 상태 변경 이벤트 수신 시 즉시 무효화
- 저장소: Redis (D4 인프라, `D2.0-04 §3.2` 참조)

### 6.7.4 에러 복구 및 재시도 (S07-B22-011~013)

**MCP 에러 복구 정책**

```
에러 유형 → 복구 전략:
  connection_error    → 지수 백오프 재시도 (최대 3회, 1s/2s/4s)
  timeout_error       → 즉시 1회 재시도 → 실패 시 fallback 도구
  tool_not_found      → tool_call_registry 갱신 후 1회 재시도
  permission_denied   → 07 Gate 재인증 → 실패 시 거부 응답
  rate_limit          → 레이트 리미트 헤더 기반 대기 후 재시도
```

- 재시도 실패 시: `LogEvent(type="mcp.error.unrecoverable", tool_id=...)` 기록
- 사용자 알림: risk_class >= med 도구의 에러는 UI 알림 발생

### K-061. 자기진화 에이전트 (Self-Evolving Agent)
- **시중 AI에 없는 핵심 차별화**: 에이전트가 스스로 능력을 확장
  1. 새로운 도구 자동 발견: MCP 레지스트리 탐색
  2. 도구 사용법 자동 학습: 문서 읽기 → 실험
  3. 실패에서 학습: 실패 패턴 → 회피 전략
  4. 프롬프트 자동 최적화: DSPy/OPRO
  5. 새로운 워크플로우 자동 생성: 반복 패턴 감지
- Dream Mode 통합: 비활성 시간에 새 도구 탐색/프롬프트 실험, 사용자 승인 후 자동 통합 (메모리 scoping: K-047 정의 준수 — L1 쓰기 한정, L2/L3 변경은 승인 필수)
- 시중 AI 비교: ChatGPT/Claude/Gemini는 고정 도구+수동 설정, VAMOS는 시간이 갈수록 강력해짐
- 구현성: V2 6개월 | V3 풀 자기진화

### K-062. 예측형 에이전트 (Predictive Agent)
- 사용자 행동 예측 → 사전 준비:
  - 시간 패턴: "매일 오전 9시에 주식 시세 확인" → 8:55에 미리 데이터 수집

  - 컨텍스트 패턴: "이 파일 열면 항상 이 파일도 확인" → 자동 로드
- Proactive Assistance: 미팅 준비 자료 자동 생성, 관심 종목 알림, 프로젝트 마감 알림, 관련 논문 알림

### K-063. 앰비언트 인텔리전스 (Ambient Intelligence)
- 항상 켜져 있는 배경 지능:
  - 시스템 모니터링: CPU/메모리/디스크 이상 감지
  - 뉴스 모니터링: 관심 키워드 실시간 감시
  - 시장 모니터링: 급등/급락/공시 알림
  - 코드 모니터링: 빌드 실패, 테스트 실패 자동 분석
  - 일정 모니터링: 다가오는 일정 알림 + 준비
- 알림 우선순위: P0(긴급), P1(중요), P2(참고), P3(낮음)
- 사용자 방해 최소화: DND 모드, 배치 알림, 스마트 타이밍
- 구현성: V2 기본 모니터링 3개월 | V3 풀 앰비언트 6개월

### K-064. 시간 여행 디버깅 (Time-Travel Debugging)
- 에이전트 실행 히스토리 시간 여행:
  - 모든 에이전트 상태 스냅샷 저장
  - 임의 시점으로 되돌아가 재실행
  - "왜 이렇게 결정했어?" → 해당 시점 컨텍스트 복원
  - 대안 탐색: "이 시점에서 다른 선택을 했다면?"
- 활용: 에이전트 오류 디버깅, 의사결정 과정 학습, 최적 전략 탐색, 투자 판단 복기
- 구현성: V2 4개월

### K-065. 멀티 페르소나 에이전트
- 상황별 VAMOS 페르소나 전환:
  - 개발자 모드: 기술적, 코드 중심, 간결
  - 투자 어드바이저 모드: 분석적, 데이터 중심, 신중
  - 학습 튜터 모드: 친절, 단계적 설명, 격려
  - 크리에이터 모드: 창의적, 영감 제공
  - 비서 모드: 효율적, 실행 중심
- 자동 전환: 대화 컨텍스트 분석 → 페르소나 자동 선택
- 사용자 커스텀 페르소나: 직접 정의 가능
- 구현성: V1 프롬프트 기반 즉시 | V2 자동 전환 2개월

### K-066. 협업형 멀티유저 AI
