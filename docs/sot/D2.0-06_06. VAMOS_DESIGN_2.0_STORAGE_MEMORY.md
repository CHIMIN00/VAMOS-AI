### 1.1 RAG 파이프라인 6단계 (LOCK)

> RAG(Retrieval-Augmented Generation) 파이프라인은 아래 6단계로 정의하며, 각 단계는 독립 모듈로 구현한다.

| 단계 | 명칭 | 설명 |
|------|------|------|
| 1 | 수집(Collect) | 문서/청크 소스 수집 (파일, URL, DB 등) |
| 2 | 쪼개기(Chunk) | 문서를 검색 단위(청크)로 분할 (크기/오버랩 설정 포함) |
| 3 | 벡터화(Embed) | 청크를 임베딩 모델로 벡터 변환 |
| 4 | 저장(Store) | VectorStore 어댑터를 통해 벡터+메타데이터 저장 |
| 5 | 검색(Retrieve) | 쿼리 벡터와 BM25 앙상블로 상위 K개 청크 검색 + Rerank |
| 6 | 생성(Generate) | 검색 결과를 컨텍스트로 LLM에 주입 후 응답 생성 |

- 운영 한계: 문서 15개 / 청크 30개 (§4 인덱싱 제한과 연동)
- 각 단계는 실패 시 레지스트리(§8)에 failure_code를 기록하고 파이프라인을 중단한다.

## 2. 메모리 계층(L0~L3) 정의 + B-Series 연결(LOCK)
(RULE 1.3 §7.1 근거)

> 목적: 저장 레벨(L0~L3)과 “기억 유형(B-Series)”를 분리해 정의하고,
>       실제 저장은 (Lx, By) 조합으로 결정한다.

- L0: Session Memory (세션 단기) — **PLAN 3.0 Session 계층**
  - 범위: 단일 세션(run/session_id)
  - 용도: 현재 대화 컨텍스트, 즉시 작업 보조
  - 구현: PLAN 3.0 메모리 계층 Session (LangChain Memory 방식 미사용, VAMOS 자체 구현)
- L1: Project Memory (프로젝트/작업 단위) — **PLAN 3.0 Project 계층**
  - 범위: project_id 단위
  - 용도: 프로젝트 목표/제약/사용자 선호, 장기 작업 연속성
  - 구현: PLAN 3.0 메모리 계층 Project
- L2: Long-term Knowledge (장기 지식) — **PLAN 3.0 Core 계층**
  - 범위: 전역(tenant/workspace) + 검색 기반 접근
  - 용도: 재사용 가능한 지식, 요약/정리된 사실, QoD 기반 축적
  - 구현: PLAN 3.0 메모리 계층 Core
- L3: Procedural Memory (절차/플레이북/루틴)
  - 범위: 전역 또는 프로젝트(정책에 의해 제한)
  - 용도: “방법(How-to)”을 실행 가능한 형태로 저장(워크플로우/플레이북/템플릿)
  - 특징: 행동에 직접 영향 → 저장/활성은 반드시 D7(07_SAFETY_COST_APPROVAL) 게이트를 거친다. (상세: §2.3, §3.2)

- B-Series(무엇을 기억하나: 내용의 성격)
  - B-4: Working (세션 컨텍스트/작업 중 메모리, 휘발성)  ← L0 핵심 대상
  - B-1: Episodic (사건/대화/상황 기록)  ← L1 핵심 대상
  - B-2: Procedural (절차/플레이북/루틴/템플릿)  ← L3 핵심 대상
  - B-3: Semantic (정리된 사실/지식)  ← L2 핵심 대상

- (Lx, By) 매핑 규칙(LOCK)
  | 저장 조합 | 대표 예시 | 기본 저장 레벨 |
  |---|---|---|
  | (L0, B-4) | 현재 대화 컨텍스트/단기 상태/작업 중 힌트 | L0 |
  | (L1, B-1) | 프로젝트 진행 맥락/결정/요약 | L1 |
  | (L2, B-3) | 검증/요약된 지식(검색 재사용) | L2 |
  | (L3, B-2) | 플레이북/루틴/템플릿/절차 | L3 |

3) 주의사항(감사/운영 관점)  ※체크리스트 요구
- V1(SQLite/파일)은 동시성 제약이 명확하므로 “단일 writer 패턴/락 정책”을 구현/문서에 포함해야 한다.
- V2+(Postgres) 채택 시 “백업/복구/마이그레이션”은 필수 산출물이다.
- Redis는 L0 캐시/세션 가속용으로만 허용하며, L1/L2의 정본 저장소로 사용하지 않는다.

4) 마이그레이션 원칙(V1 → V2)
- V1에서 쌓인 JSONL/SQLite는 “내보내기(Export) → 적재(Load)” 경로로 V2(Postgres)로 이전 가능해야 한다.
- 이전은 스키마 정합(REF/ID/정책결과 링크)을 깨지 않는 방식으로만 수행한다.

> 정합 근거(정본): 01(Overview) 3.3 P1 Technology Combos(V1/V2/V3) 서술을 우선한다.


### 2.2-A VectorStore 어댑터 인터페이스 (LOCK)

- 목적: VectorDB 구현체(Chroma, Qdrant, pgvector 등)를 교체해도 상위 코드를 수정하지 않도록 추상화 계층을 둔다.
- 인터페이스 최소 계약:
  - `upsert(records: VectorRecord[]) → void`
  - `search(query_vector: float[], top_k: int = 10, filters: dict) → VectorRecord[]`  — top_k 기본값: **10** (config.toml `[memory.vector].default_top_k`로 조정 가능, API 레이어 B1은 top_k=5 기본)
  - `delete(ids: str[]) → void`
  - `get_by_id(id: str) → VectorRecord | None`
- 구현 어댑터(버전별):
  - V1: `ChromaAdapter` (로컬 파일 기반)
  - V2: `QdrantAdapter` (서버 기반, 우선) / `PgvectorAdapter` (폴백)
  - V3: 매니지드 Vector DB 어댑터(Qdrant Cloud 또는 동급)
- 제약: 어댑터 교체는 설정(config) 변경만으로 가능해야 하며, 비즈니스 로직 수정을 유발하지 않는다.

### 2.3 P1 Memory Activation by Version

[P1-SCOPE: MUST]

- V1:
  - L0: 세션 메타(요약, 힌트, hash)는 활성화 (SQLite/JSONL 기반).
  - L1: 프로젝트 단위 요약/결과 메타는 선택적으로 활성화.
  - L2: 기본적으로 비활성화 (장기 / 대용량 벡터 저장은 사용하지 않음).
  - QoD 및 TTL은 **SourceQoD+TTL 통합 모듈**로 처리한다. (분리 운용 금지; 상세: §2.5.2~2.5.3)
  - L3: 기본적으로 비활성화 (절차 저장/활성은 D7 게이트 + 비용/승인 전제)
- V2:
  - L0: 동일 (세션 메타).
  - L1: 기본 활성화 (Postgres + Chroma 기반), 프로젝트별 Evidence/RAG 소스 관리.
  - L2: 제한적으로 활성화 (정책/비용 상한 내에서만 장기 문서 저장). **저장 시 승인 필수 (S03-MOD-001 확정)**
  - L3: 제한적으로 활성화 (프로젝트 단위 playbook/template 중심, 승인/상태 기반)
- V3:
  - L0/L1/L2/L3 활성화 가능(정책/비용/승인 게이트 전제), QoD/TTL/PII 정책을 강하게 적용.
  - PII/민감 데이터는 Allow/Restrict/Deny 및 D7 PolicyCheck 결과에 따라 L1/L2 장기 저장을 제한한다.
  - L3(B-2)는 자동 활성 금지, approved/active 전환은 D7 Approval Gate를 따른다.

### 2.4 B-2 Procedural Memory (L3) — 상세(LOCK)

#### 2.4.1 정의
- Procedural Memory는 “실행 가능한 절차”를 저장하는 기억이다.
- 형태:
  - playbook(단계/조건/분기)
  - workflow(파이프라인: Perception→Plan→Execute→Verify→Deliver)  ※05(Agent Workflow) 연결
  - template(prompt/툴 호출 템플릿)
  - routine(반복 작업 자동화 규칙)
- 특징: 답변 품질이 아니라 “행동(실행)”을 바꾸므로 저장/활성에 강한 거버넌스가 필요하다.

#### 2.4.2 최소 필드(저장 레코드 관점)
- 공통: MemoryRecordSchema(정본: D6)를 따른다. (상세: §7.1)
- L3/B-2에서 “추적/감사”를 위해 추가로 확보돼야 하는 정보(스키마에 없으면 D6에서 확장):
  - procedure_id (고유 ID)
  - version (semver 또는 증가형)
  - target_scope (global / project)
  - trigger_conditions (언제 적용되는지)
  - steps (절차 본문: 단계 배열)
  - required_tools (필요 툴/권한)
  - safety_notes (위험/제한/금지)
  - activation_state (draft / approved / active / deprecated)
  - provenance (근거: source_refs + 생성 주체/시각)

#### 2.4.2-A L3 Procedural 진입 시점 및 상세 정의

- **V1**: L3 비활성 (OFF). 절차/템플릿은 L1에 임시 저장만 허용.
- **V2**: L3 제한적 활성. 사용자 승인된 playbook/template만 L3 저장 허용. 자동 활성화 조건: activation_state=approved + 3회 이상 수동 실행 이력.
- **V3**: L3 전면 활성. 자동 생성 routine 포함, DSPy 기반 자동 최적화 연동.
- **벡터 검색 기본값**: L3 검색 시 `top_k=5` (프로시저는 정밀도 우선, L2의 top_k=10 대비 절반). config.toml `[memory.vector].l3_top_k`로 조정 가능.
- **접근 제어**: L3 읽기는 L1 이상 접근 권한 필요, L3 쓰기는 D7 ApprovalGate 필수 (§2.4.3 참조).

#### 2.4.3 생성/승격 파이프라인(게이트 포함)
1) Candidate 생성(draft)
2) PolicyCheck(D7) — result=PASS/BLOCK/NEEDS_APPROVAL/ERROR + reasons 기록
3) (result=NEEDS_APPROVAL 인 경우) Approval(D7) — approval_status=PENDING→APPROVED/DENIED (만료/취소 포함)
4) 저장(write) — L3로 기록 + 버전 부여
5) 활성(active) — 실제 실행에 사용(자동 적용은 policy가 허용한 범위 내)

#### 2.4.4 폐기/롤백
- deprecate: 위험/오류/비용 폭주/정책 위반 발견 시 즉시 비활성화
- rollback: 이전 approved 버전으로 되돌림 (audit log 필수)
- UI 연결: 08(UI/UX)에서 draft/approved/active/deprecated 상태 표시

### 2.5 메모리 시스템 전면 재설계 요소(ADD-053~056)

#### 2.5.1 Memory 재설계 원칙
- 기존 단순 Key-Value 저장에서 벗어나 **계층(L0~L3) + B-Series + QoD + TTL** 을 결합한 통합 메모리 모델로 전환한다.
- 모든 메모리 레코드는 MemoryRecordSchema(정본: D6 §7.1)를 따른다.

#### 2.5.2 SourceQoD 연동
- 저장 시점에 SourceQoD 스코어(0~1)를 산출하여 레코드에 기록한다.
- QoD < 0.4: L2/L3 저장 금지(L0/L1 임시 보관 후 재평가 또는 폐기).
- QoD ≥ 0.7: L2 장기 저장 허용(정책/승인 조건 충족 시).

#### 2.5.3 TTL(Time-To-Live) 정책
- 각 계층별 기본 TTL:
  | 계층 | 기본 TTL |
  |------|----------|
  | L0 | 세션 종료 즉시 만료 |
  | L1 | 90일 (프로젝트 종료 후 30일 연장 가능) |
  | L2 | 무기한 (QoD 재평가 주기 적용) |
  | L3 | 무기한 (activation_state=deprecated 전환으로 폐기) |
- TTL 만료 시 자동 삭제 또는 archived 상태로 전환한다.

#### 2.5.4 Knowledge Base(KB) 연동
- L2 장기 지식은 외부 KB(Confluence, Notion, 내부 문서 저장소 등)와 동기화 가능하다.
- 동기화 방향: KB → L2(읽기/인덱싱), L2 → KB(승인된 요약만 내보내기).
- KB 연동은 project_id 네임스페이스를 유지하며, 혼합 저장 금지 원칙을 준수한다.

#### 메모리 승격/강등 알고리즘 통합 (S7D-040, S7D-041)

#### 메모리 검색 우선순위 (S7D-042 상세)

- 검색 순서: L0(현재 세션) → L1(현재 프로젝트) → L2(글로벌 지식) → L3(절차/템플릿) [V2+: → Archive(아카이브)]
- 스코어링: `relevance * recency * frequency`
- 최대 결과: 레이어당 5개, 총 25개 후보 → reranker → top 5
- 컨텍스트 주입: 검색된 메모리를 시스템 프롬프트에 자동 삽입
- Early Exit: L0에서 충분한 결과 시 하위 계층 탐색 생략 (latency 우선 모드)

## 3. 저장 정책(Allow/Restrict/Deny)과 마스킹 파이프라인
### 3.1 공통 원칙(RULE 1.3 근거)
- 저장 전 사용자 확인(기본) (RULE 1.3 §7.3)
- 민감 데이터 필터링 의무 (RULE 1.3 §7.3)
- 민감 데이터 자동 저장 금지(항상 확인 또는 거부) (RULE 1.3 §7.3)

### 3.2 Allow / Restrict / Deny(계층별 최소 원칙 — L0~L3)
- 표기(LOCK):
  - PASS = Allow에 해당(저장 가능)
  - NEEDS_APPROVAL = Restrict에 해당(승인 필요)
  - BLOCK = Deny에 해당(저장 금지)
  - ERROR = 보수적 처리(기본 BLOCK 또는 사용자 확인)
- 단, 구현/로그/상태값은 D7 PolicyCheckSchema의 enum(PASS/BLOCK/NEEDS_APPROVAL/ERROR)을 SOT로 사용한다.

- Allow:
  - L0/L1/L2: 정책 위반/민감정보 없음 + 프로젝트 범위에 적합한 “요약/결과”만 저장
  - L3(B-2): allow라도 **자동 활성(active) 금지**. (활성은 §2.3.3 게이트를 만족해야 함)

- Restrict:
  - L0/L1/L2: 저장은 가능하나, 마스킹/요약/기간 제한(TTL)/승인 조건이 붙음
  - L3(B-2): 기본값은 restrict 권고(행동 영향 때문)
    - 저장(write) 또는 활성(active) 중 최소 1개는 D7(07_SAFETY_COST_APPROVAL) Approval Gate 요구(정책으로 결정)

- Deny(체크리스트 핵심):
  - 모든 레벨(L0~L3): 정책 위반(PII/금지 도메인/위험 요청 등) 시 저장 불가
  - 어떤 저장소에도 “장기 저장” 불가
  - **벡터 삽입 절대 금지**(RAG/임베딩 금지)
  - 로그/트레이스에도 원문 저장 금지(필요 시 요약/힌트만)

> 결정 완료(§3.2 Restrict 기본 처리): 옵션 B를 정본으로 잠금하되, 안정성을 위해 아래를 “필수 규칙”으로 함께 고정한다.
- 정본: 옵션 B — 마스킹 후 저장 허용 + TTL 기본
  1) 마스킹이 충분하면 “요약본 + TTL”로 저장을 허용할 수 있다.
  2) 마스킹이 불완전하거나 민감도가 높으면 “승인 없이는 저장 금지(deny 또는 보류)”로 처리한다.
  3) Restrict 저장이라도 벡터 삽입은 정책 결과/민감도에 따라 별도 판단한다. Deny면 벡터 삽입은 절대 금지다.
- 주의: TTL “수치”는 PLAN/운영 근거로만 잠금(발명 금지). 현재 문서에서는 “기본 TTL 적용”까지만 고정한다.
- content_summary는 MemoryRecord(§7.2)의 요약 필드명으로 통일한다.
- Approval 연결(고정): restrict 또는 L3 활성 조건에는 D7(07_SAFETY_COST_APPROVAL)의 ApprovalSchema/Approval Gate를 연결한다.

- 한-영 혼용 청크는 언어 경계에서 분할하지 않고, 문장 단위 우선으로 처리한다.

#### 4.3.2 한국어 불용어(Stopword) 처리
- 불용어 목록: 공개 한국어 불용어셋 기반 + 프로젝트별 커스텀 필터.
- BM25 검색 시 불용어 제거를 적용하되, 벡터 임베딩 시에는 원문을 유지한다(의미 손실 방지).

#### 4.3.3 한-영 혼용 문서 처리
- 메타데이터에 `language` 필드(ko/en/mixed) 를 기록한다.
- 검색 시 쿼리 언어와 문서 언어 간 가중치를 조절하여 교차 언어 검색 품질을 확보한다.
- V2 이상: Cross-lingual embedding 모델 후보 평가 (§4.y.3 임베딩 벤치마크와 연동).

#### 4.3.4 청크 크기 권장
- 한국어 기준: 300~500 토큰/청크 (영어 대비 토큰 효율 ~70%).
- 오버랩: 50~100 토큰 (문맥 연속성 보장).

#### Contextual Retrieval 벤치마크 [REF:영상7]

> Anthropic 발표: 청크에 맥락 설명(contextual prefix)을 부착하면 검색 정확도가 **67% 향상**됨.

| 방식 | 검색 정확도 (상대값) | 비용 영향 |
|------|---------------------|-----------|
| 기본 RAG (Vector만) | 1.0x (기준) | 최저 |
| + BM25 하이브리드 | ~1.3x | 낮음 |
| + Contextual Prefix | ~1.67x (+67%) | 중간 (전처리 1회) |
| + Reranker 추가 | ~1.8x | 높음 |

- **VAMOS 적용**: D2.0-06 DEC-004 Hybrid RAG에 Contextual Prefix 단계를 기본 포함
- **비용 주의**: 전처리(프리픽스 생성)는 인덱싱 시 1회만 발생, 검색 시 추가 비용 없음

### 4.x GraphRAG 연동
- 기본 벡터 검색(유사도 기반)에 더해 **GraphRAG**를 지원한다.
- 목적: 개념 간 관계(entity-relation graph)까지 탐색하여 단순 유사도 검색이 놓치는 연결 맥락을 제공한다.
- 적용 범위: L2(글로벌 지식) 저장소에서 활성화 가능(V2 이상).
- 구현 참조: 노드(개념/엔티티) + 엣지(관계) 형태로 인덱싱, 쿼리 시 그래프 순회(traversal)와 벡터 검색을 병합(hybrid)한다.


### 4.y RAG 품질 평가 및 확장 기능

#### 4.y.1 RAGAS 평가 파이프라인
- RAG 응답 품질을 자동 평가하기 위해 **RAGAS(Retrieval-Augmented Generation Assessment)** 프레임워크를 연동한다.
- 평가 지표: Faithfulness(충실도), Answer Relevancy(관련성), Context Precision/Recall.
- 평가 결과는 QoD 스코어 갱신 및 인덱스 재구성 트리거에 활용한다.

#### 4.y.2 다중 검색기(Multi-Retriever)
- 단일 벡터 검색기 외에 BM25(키워드), 그래프 순회, 메타 필터 기반 검색기를 병렬로 운용할 수 있다.
- 결과는 RRF(Reciprocal Rank Fusion) 또는 가중 합산 방식으로 통합한다.

#### 4.y.3 임베딩 벤치마크
- 임베딩 모델 교체/업그레이드 시 기준 데이터셋 대비 Recall@K, MRR 지표로 성능을 비교한다.
- 벤치마크 결과는 D6 변경 이력에 기록하고, 채택 결정은 DEC 항목으로 잠금한다.

#### 4.y.4 Codebase Indexing
- 소스코드 파일(.py/.ts/.md 등)을 청크 단위로 파싱하여 L2 벡터 저장소에 인덱싱한다.
- 청크 단위: 함수/클래스/섹션 경계 우선, 최대 512 토큰.
- 메타: file_path, language, last_modified, chunk_index 포함.


### 4.z Cloud Library(CLib) 출처별 품질 차이 기준
- Cloud Library(외부 API/클라우드 문서 저장소)에서 수집된 콘텐츠는 출처 유형별로 기본 QoD 가중치를 다르게 부여한다.

| 출처 유형 | 기본 QoD 가중치 | 비고 |
|-----------|-----------------|------|
| 공식 문서 (공식 API Doc, RFC 등) | 0.85 | 검증된 1차 소스 |
| 내부 KB / 팀 문서 | 0.70 | 프로젝트 컨텍스트 유효 |
| 커뮤니티/포럼 (StackOverflow 등) | 0.50 | 검증 필요, 단독 인용 지양 |
| 미분류 / 출처 불명 | 0.30 | L2 저장 전 수동 검토 필수 |

- QoD 가중치는 최종 SourceQoD 스코어 산출 시 `base_score × source_weight` 형태로 반영한다.
- QoD < 0.4인 CLib 콘텐츠는 L2 벡터 삽입을 금지하고 `QoD_BELOW_THRESHOLD` 이벤트를 기록한다.

### 4.7 Semantic Cache (ADD-012, MOD-017)

> 동일/유사 질문에 대한 중복 RAG+LLM 호출을 줄이기 위해 의미 기반 캐시를 도입한다.

#### 4.7.1 Semantic Cache 설계
- 캐시 키: 쿼리 임베딩 벡터의 코사인 유사도 기반 매칭.
- 캐시 히트 임계값: cosine_similarity ≥ **0.95** (기본값, 운영 시 조정 가능).
- 캐시 저장소: V1=인메모리(LRU), V2+=Redis 또는 별도 Chroma 컬렉션.
- 캐시 레코드: `{ query_hash, query_embedding, response, source_refs, created_at, ttl, hit_count }`.

#### 4.7.2 캐시 무효화 정책 (MOD-017)
- **TTL 기반**: 캐시 항목 기본 TTL = **24시간** (소스 변동이 잦은 프로젝트는 6시간).
- **소스 변경 연동**: 인덱싱된 문서가 갱신/삭제되면 해당 문서를 source_refs로 참조하는 캐시 항목을 즉시 무효화.
- **QoD 변동 연동**: 소스 QoD가 재평가되어 0.4 이하로 하락하면 관련 캐시 항목 무효화.
- **수동 flush**: 관리자가 프로젝트/전역 단위로 캐시를 강제 초기화할 수 있다.
- 무효화 이벤트는 `cache.invalidated` event_type으로 02 Registry에 기록.

#### 4.7.3 캐시 모니터링
- 히트율(hit ratio), 평균 응답 시간 절감, 캐시 크기를 대시보드(08 UI 연동)에 표시.
- 히트율 < 10% 시 캐시 전략 재검토 알림.

## 8. 레지스트리(Event/Failure/Fallback)

- 규칙(정본):
  - event_type / failure_code / fallback_id의 “값 목록 정본(Source of Truth)”은 02(Registry)이다.
  - 본 문서(06)는 저장 도메인에서의 “의미/발생 조건/연결 흐름”만 정의하며,
    **정본에 없는 코드값을 확정(박제)하지 않는다.**
  - (연결) 실제 값 매핑/잠금은 02(Registry) 및 07(Policy/Cost/Approval) 레지스트리 섹션을 따른다.

- 저장 관련 failure_code (예시 — 값 목록 정본은 02 Registry):
  - ST_PII_DETECTED — PII 탐지 시
  - ST_VECTOR_DENIED — 벡터 삽입 거부 시
  - ST_LONGTERM_DENIED — 장기 저장(L1/L2) 거부 시
  - ST_MASK_INCOMPLETE — 마스킹 불완전 시

> 위 failure_code는 02(ORANGE CORE) §6.2 FailureCode Registry에 등록되어야 한다.

- 저장 관련 event_type (확정 — 02 Registry 연동):
  - (확정) storage.policy.checked — 저장 정책 검사 완료
  - (확정) storage.memory.committed — 메모리 저장 완료
  - (확정) storage.vector.denied — 벡터 삽입 거부
  - (확정) storage.pii.blocked — PII 장기 저장 차단

> 위 event_type은 02(ORANGE CORE) §6.1 LogEvent Registry에 등록되어야 한다.

- UI 연결(08 연동):
  - 저장이 deny 처리되면 08 UI는 “저장 취소 알림”을 표시하고,
    해당 메모리 카드를 Denied 상태로 표시한다.
  - (연결) UI 상태 전환/표시 규칙은 08의 실패 처리 플로우를 참조한다(값 직접 기입 금지).

- 07 PolicyCheck 연동:
  - 저장 관련 deny/restrict/allow 결과는 07의 PolicyCheck 결과와 정합되어야 한다.
  - 즉, 저장 시도는 PolicyCheck(allow/restrict/deny) → 저장 수행/차단 → 로그 기록의 흐름을 따른다.

- 네이밍 규칙:
  - 네이밍 정본은 STEP 1.1에서 잠금한 규칙을 따른다.
  - (정본 미존재/미적용 시 대체 잠금안)
    - event_type: lower.dot
    - failure_code: UPPER_SNAKE
    - fallback_id: FB_UPPER_SNAKE
    - (정본 연결) 02 §6.3 Fallback Strategy Registry 규칙과 동일 패턴을 사용한다.

- 등록 필요(결정 필요, “값”은 D2에 등록 후 사용):
  - event_type 후보(예시):
    - storage.policy.checked
    - storage.memory.write.requested
    - storage.memory.write.completed
    - storage.vector.insert.denied
    - storage.pii.longterm.denied

  - failure_code 후보(예시):
    - PII_LONGTERM_DENIED

  - fallback_id 후보(예시):
    - FB_DENY_STORAGE

- 동작 규칙(의미/조건 정의 — 값은 정본 사용):
  - 민감 개인정보(PII/인증/의료/법률)의 장기 저장(L1/L2) 시도는 즉시 차단(deny)되어야 한다.
  - 동시에 벡터 삽입은 절대 금지하며, vector insert도 deny로 처리한다.
  - 위 이벤트/실패/폴백의 “코드값”은 02(Registry)에 등록/잠금된 값을 사용한다.

## 10. 확장 기능 상세 (Part E Section 8 통합)

> Part E Section 8에서 D2.0-06에 할당된 ~60건의 상세 항목을 통합 정리한다.

### 10.1 Memory Usage Analytics (S03-D20)

- 계층별 저장량 추적: L0/L1/L2/L3 각 레벨의 레코드 수, 총 용량, 평균 QoD 분포를 실시간 집계한다.
- 검색 빈도 분석: RAG 검색 호출 횟수, 히트/미스 비율, 평균 검색 시간을 기록한다.
- I-3 L2 승인 현황: L2 장기 저장 승인 요청/승인/거부 건수를 대시보드(08 연동)에 표시한다.
- 비용 연동: 임베딩 API 호출 비용, 벡터DB 저장 비용을 07 비용 게이트와 연동하여 월간 상한 내 운용 보장.
- Memory Usage Analytics 이벤트: `mem.usage.report` event_type으로 주기적(일 1회) 기록.

### 10.2 임베딩 최적화 / File Search (S05-B01)

#### 10.2.1 Batch Embedding 최적화
- 다수 청크를 한 번의 API 호출로 임베딩 (배치 크기: 기본 100, 최대 2,048).
- Rate limit 대응: 지수 백오프 + 큐 기반 조절.
- 임베딩 캐시: `sha256(chunk_text)` 키로 중복 청크 재임베딩 방지.

#### 10.2.2 File Search 벡터스토어 연동
- OpenAI File Search API(Assistants API)의 벡터스토어를 외부 소스로 활용 가능.
- 연동 방식: File Search 결과를 L2 인덱스에 병합하되, source_type='file_search'로 구분.
- 제한: File Search 벡터스토어는 읽기 전용(VAMOS에서 직접 쓰기 금지), 동기화는 pull 방식.

#### 10.2.3 동적 차원 선택 (Matryoshka)
- BGE-M3의 Matryoshka 지원을 활용하여 저장/성능 트레이드오프 조절.
- V1 기본: 1024차원 (BGE-M3) → 비용 최우선 시 256차원 축소 가능 (§7.3 임베딩 차원 정의 참조).
- 차원 변경 시 기존 인덱스 재빌드 필요 (마이그레이션 스크립트 필수).

### 10.3 벡터 인덱싱 / 데이터 품질 (S05-B14)

#### 10.3.1 인덱스 구조
- Chroma 기본: HNSW (Hierarchical Navigable Small World) 인덱스.
- 파라미터: ef_construction=200, M=16 (기본값, 운영 시 조정).
- V2+ 고부하 시: Qdrant 서버 모드의 추가 인덱스 옵션(quantization 등) 검토.

#### 10.3.2 데이터 품질 필터
- 인덱싱 전 청크 품질 검사:
  - 최소 길이: 20자 미만 청크 제외.
  - 중복 검출: MinHash 기반 near-duplicate 탐지 (Jaccard ≥ 0.85 시 제거).
  - 언어 탐지: 비대상 언어(예: 일본어/중국어 혼입) 청크 필터링.
- 품질 미달 청크는 `chunk.quality.rejected` 이벤트 기록.

#### 10.3.3 인덱스 재구성 트리거
- QoD 재평가 후 L2 소스의 20% 이상이 변경되면 전체 인덱스 재빌드.
- 재빌드는 비동기 백그라운드로 수행하며, 재빌드 중에도 기존 인덱스로 검색 가능(blue-green 방식).

### 10.4 Memory 4계층(L0~L3) 접근 패턴 + Router 쿼리 라우팅 (S06)

#### 10.4.1 Memory 4계층(L0~L3) 접근 패턴
- 검색 순서: L0(세션) → L1(프로젝트) → L2(글로벌) 순차 탐색.
- 각 계층에서 top_k 결과를 수집한 후 통합 랭킹(RRF 또는 스코어 합산)으로 최종 결과 선정.
- L0 히트 시 L1/L2 탐색을 생략하는 "early exit" 옵션 제공 (latency 우선 모드).

#### 10.4.2 Router 기반 쿼리 라우팅
- 쿼리 유형 분류: factual / procedural / creative / meta.
- 라우팅 규칙:
  | 쿼리 유형 | 우선 계층 | 검색기 | 비고 |
  |-----------|----------|--------|------|
  | factual | L2 → L1 | BM25+Vector 하이브리드 | 정확한 사실 검색 |
  | procedural | L3 → L1 | 메타 필터(B-2) | 절차/플레이북 검색 |
  | creative | L1 → L0 | Vector 단독 | 유사 맥락 참조 |
  | meta | L0 | 세션 컨텍스트 | 대화 흐름 참조 |
- Router 구현: 쿼리 임베딩 + 간단 분류기(V1: 규칙 기반, V2: 경량 모델).

### 10.5 RAG 통합 API + Vector DB/KB 통합 (S07)

#### 10.5.1 RAG 통합 API 엔드포인트
- `/rag/retrieve` — 쿼리 기반 검색 (top_k, filters, rerank 옵션).
- `/rag/index` — 문서/청크 인덱싱 요청 (파이프라인 §1.1 6단계 트리거).
- `/rag/delete` — 문서/청크 삭제 (인덱스 + 벡터 동시 제거).
- `/rag/status` — 인덱싱 상태, 저장량, 큐 현황 조회.
- 모든 엔드포인트는 project_id 필수 파라미터.

#### 10.5.2 Vector DB + KB 동기화 파이프라인
- 방향: KB(외부) → L2 벡터 저장소 (읽기/인덱싱).
- 동기화 주기: V1=수동, V2=일 1회 스케줄, V3=실시간 webhook.
- 동기화 시 QoD 재평가(§2.5.2) + 캐시 무효화(§4.7.2) 연동.
- KB → L2 적재 시 source_type='kb_sync' 메타 부여.

#### 10.6.1 EmbeddingConfig 스키마 (S09-B33)

#### 10.6.2 Memory.type 4종 (S09-B33)
- MemoryRecord의 `memory_type` 필드가 취할 수 있는 값:

| memory_type | B-Series | 설명 | 기본 저장 계층 |
|-------------|----------|------|---------------|
| `episodic` | B-1 | 사건/대화/상황 기록 | L1 |
| `procedural` | B-2 | 절차/플레이북/루틴 | L3 |
| `semantic` | B-3 | 정리된 사실/지식 | L2 |
| `working` | B-4 | 세션 컨텍스트/작업 중 메모리 | L0 |

- §2 메모리 계층 정의와 정합.

#### 10.6.3 rag_framework 3단계 (S09-B36)

| 단계 | 버전 | 구성 | 정확도 목표 |
|------|------|------|-----------|
| Basic | V1 | Vector 단독 검색 + 단순 프롬프트 주입 | 64%+ |
| Hybrid+Rerank | V2 | BM25+Vector 앙상블 + Cross-encoder Rerank | 83%+ |
| Self-RAG+Graph | V3 | Self-RAG 자체 검증 + GraphRAG 다중 홉 | 90%+ |

- 각 단계는 이전 단계를 포함(누적 확장).
- V1→V2 전환 조건: L2 활성화 + 검색 품질 평가(RAGAS) 기준 미달 시.
- V2→V3 전환 조건: GraphRAG 인프라(Neo4j) + Self-RAG 검증 모듈 구현 완료 시.

### 10.7 고급 RAG 패턴 (S14-CLIB)

#### 10.7.1 Self-RAG (자체 검증 RAG)
- 생성 후 자체 검증: 생성된 답변이 검색 결과에 의해 **지지(support)** 되는지 자동 판정.
- 판정 결과:
  - `fully_supported`: 검색 결과가 답변을 완전 뒷받침 → 그대로 출력.
  - `partially_supported`: 부분 뒷받침 → 경고 표시 + 추가 검색 1회.
  - `not_supported`: 뒷받침 없음 → 답변 폐기 + 재검색 또는 "모름" 응답.
- 검증 모델: 경량 분류기(V2) 또는 LLM-as-judge(V3).

#### 10.7.2 GraphRAG 고급 패턴
- 커뮤니티 요약: 그래프 클러스터링 후 커뮤니티별 요약 생성 → 전역 질의(global query) 지원.
- Multi-hop 추론: 엔티티→관계→엔티티 체인을 최대 3홉까지 탐색.
- 그래프 구축: V2에서 Neo4j 기반, 엔티티/관계 추출은 LLM 기반 NER+RE.

#### 10.7.3 DSPy + Reflexion 패턴
- **DSPy**: RAG 프롬프트를 자동 최적화하는 프레임워크.
  - 시그니처(Signature): retrieve→generate 파이프라인을 선언적으로 정의.
  - Compiler: 소량 예시로 프롬프트/파라미터 자동 튜닝.
- **Reflexion**: 생성 후 "반성(reflection)" 루프.
  - 답변 평가 → 개선 힌트 생성 → 재생성 (최대 2회 반복).
  - 반성 결과는 L1에 episodic memory로 저장(재사용).
- 적용 범위: V3 (LATER), 연구/실험 단계.

#### 10.7.4 멀티모달 RAG (V3 LATER)
- 이미지/표/차트를 청크로 인덱싱하여 텍스트 검색과 병합.
- 구현: 멀티모달 임베딩 모델 (CLIP 계열) + 텍스트 임베딩 통합 인덱스.
- P1 Scope: LATER (V3 엔터프라이즈).

### 10.8 엔터프라이즈 패턴 (S05-B08, V3)

#### 10.8.1 멀티테넌시
- tenant_id 기반 벡터 저장소 분리 (Chroma collection per tenant 또는 메타 필터).
- 테넌트 간 데이터 혼합 절대 금지 (RULE 1.3 §7.2 확장).

#### 10.8.2 규정 준수 (SOC-2 / GDPR)

- 감사 로깅: 모든 저장/삭제/검색 이벤트를 불변 로그에 기록.
- 암호화: 저장 시 AES-256 (at-rest), 전송 시 TLS 1.3 (in-transit).
