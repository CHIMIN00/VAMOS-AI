## 7. Acceptance Criteria

- **AC-D3-001**: NodeRegistry는 D3 SOT로만 확정된다. (타 문서에서 재정의 금지)

- **AC-D3-002**: D3의 모든 스키마 블록은 D1 Schema Meta Template을 준수한다. (신규 스키마는 ref_legacy 미사용)

- **AC-D3-003**: event_type / failure_code / fallback_id 값 목록은 02 정본 Registry를 REF-only로만 참조한다. (D3에서 코드값 목록 생성 금지)

- **AC-D3-004**: NodeRequestEnvelopeSchema는 03 §5.2의 필수(required) 7개 필드를 모두 포함한다. (request_id, project_id, session_id, node_id, intent_summary, constraints, trace_id)

- **AC-D3-005**: NodeResponseEnvelopeSchema는 03 §5.2.1 최소 필드를 포함한다. (trace_id, node_id, domain, inputs.summary, outputs.result, outputs.evidence_refs, status)

- **AC-D3-006**: 모든 NODE 실행은 decision_id로 02 Decision에 연결 가능해야 한다. (엔벨로프 내 decision_id는 선택이나, 실행 Trace 관점의 연결은 필수; 근거: 03 §6.0)

- **AC-D3-007**: ToolCallRegistrySchema.tool_id는 D4 ToolRegistry와 정합해야 한다. NODE별 도구 사용은 반드시 ToolCallRegistry를 경유한다.

- **AC-D3-008**: MCPBridgeLayerSchema.transport는 "streamable_http"만 허용한다 (DEC-017 LOCK — stdio 전송 제거).

- **AC-D3-009**: CloudLibrarySiteEvaluationSchema의 점수 배분은 Trust(25) + Relevance(30) + Quality(25) + Access(20) = 100점 체계를 준수한다 (LOCK L5).

- **AC-D3-010**: CloudLibraryGateResultSchema는 5단계 Gate(G0 Format 100%, G1 Content 95%, G2 Consistency 90%, G3 Security 100%, G4 Final 100%) 통과 기준을 준수한다 (LOCK L3).

- **AC-D3-011**: CloudLibraryExtractionOutputSchema는 E-15(Layer 6) → S-5(Layer 7) 인터페이스 계약으로, 모든 추출 결과는 이 스키마를 경유해야 한다 (HYBRID_ROADMAP G1 해소).

- **AC-D3-012**: CloudLibraryConflictResolutionSchema의 소스 가중치는 7단계(공식 1.0 > 논문 0.9 > 기술문서 0.85 > 기술블로그 0.7 > 뉴스 0.6 > 개인블로그 0.5 > SNS 0.3)를 준수한다 (LOCK L7).

- **AC-D3-013**: CloudLibraryVersionChangeSchema의 Semantic Versioning은 MAJOR(호환성 변경) / MINOR(키워드 10+/섹션 추가) / PATCH(오타/소수) 기준을 준수한다 (LOCK L4).
