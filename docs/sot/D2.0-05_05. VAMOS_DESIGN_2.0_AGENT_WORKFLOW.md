### 1.1 Agent Mode 열거형 정의 (ADD-009)

> Part E 변경점: 에이전트 모드(자동/반자동/수동) 명세가 없었으므로 Agent Mode 열거형을 정의한다.

- 목적: VAMOS 에이전트의 실행 모드를 표준 열거형으로 정의하여, 워크플로우 엔진과 Gate 시스템이 모드별 분기 처리를 수행할 수 있게 한다.
- Agent Mode 정의(LOCK):

```python
class AgentMode(str, Enum):
    """VAMOS 에이전트 실행 모드 열거형 (ADD-009)"""
    MANUAL = "manual"           # 모든 단계에서 사용자 확인 필수
    SEMI_AUTO = "semi_auto"     # Allowlist 도구는 자동, P2 이상은 승인 필요
    SUPERVISED_AUTO = "supervised_auto"  # 자동 실행 + 감시 로그, P2는 승인 필수
    # AUTONOMOUS = "autonomous"  # V1 금지 (IR-001). V2+에서 제한적 도입 예정
```

- 모드별 Gate 연동:
  | 모드 | P0 실행 | P1 실행 | P2 실행 | Self-check | 비용 초과 |
  |------|---------|---------|---------|-----------|----------|
  | MANUAL | 사용자 확인 | 사용자 확인 | 사용자 확인 | 수동 확인 | 즉시 중단 |
  | SEMI_AUTO | 자동 | 자동 (Allowlist) | 승인 필요 | 자동 1회 | 다운시프트 |
  | SUPERVISED_AUTO | 자동 | 자동 | 승인 필요 | 자동 1회 | 다운시프트 + 알림 |

- V1 기본값: **SEMI_AUTO** (DEC-003 Allowlist 자동승인과 정합)
- V2+에서 SUPERVISED_AUTO 확장, V3에서 제한적 AUTONOMOUS 도입 검토 (IR-001 기준 유지)
- 모드 변경은 세션 단위. 변경 시 LogEvent(`agent.mode.changed`) 기록.

## 2. Agent Pool(Task 분해/우선순위/병렬)
- Task 분해: 사용자의 요청을 실행 가능한 작업 단위로 나누고, 필요 시 BLUE NODE에 위임(03 참조)
- 우선순위: 비용/위험/긴급도에 의해 조정되며, 비용 상한이 최상위 제약으로 작동 (RULE 1.3 §5)
- 병렬: INFRA 제약(예: 최대 3 병렬)을 준수 (04 참조)

> **결정 완료(DEC-003): 승인 모드 = Allowlist 자동승인 확정 (LOCK)**
> - 안전 도구(Allowlist 등록 항목): 자동승인, 승인 절차 생략
> - 위험 도구(Allowlist 미등록 또는 P2 이상): 명시적 사용자 확인 필수
> - 이 결정은 §2.2의 Allowlist 규칙과 연동하여 적용된다.  - V3 확장 옵션: 최대 10개 병렬까지 확장 가능 (04 INFRA 리소스 충분 시, 별도 승인 필요)

### 2.1 P2 작업 Queue 등록 제약(박제)
- **P2 작업은 승인 없이는 Queue 등록 불가** (RULE 1.3: P2 활성화/실행은 승인 기반, 위험 자동 실행 금지)

### 2.2 도구 승인 Allowlist 자동승인 규칙 (IM-004)
- 도구 승인은 **위험 수준에 따라 이원화**한다:
  - **Allowlist 자동승인**: 읽기 전용/조회/로그 조회 등 안전한 도구는 사전 등록 후 승인 없이 자동 실행 허용.
  - **명시적 확인 필요**: 외부 API 호출/파일 쓰기/코드 실행/P2 이상 위험 도구는 반드시 사용자 확인 후 실행.
- Allowlist 등록/변경은 07 Gate 승인 후만 허용하며, 변경 이력은 06(STORAGE)에 기록한다.
- Allowlist에 없는 미등록 도구는 기본값 "확인 필요"로 처리한다.

## 3. Workflow Automation Engine(단계 파이프라인/조건 분기)
- 워크플로우는 항상 07의 PolicyCheck/Approval/Cost 게이트 결과를 선행 입력으로 사용한다.
- 조건 분기:
  - (a) PolicyCheck=deny → 즉시 중단(deny)
  - (b) 비용 상한 초과 → 승인 없이 자동 차단(deny) (RULE 1.3 §5)
  - (c) P2 관련 → 세션별 승인 없으면 중단(deny) (RULE 1.3 §3.3)

> 결정 완료: 표준 5단계 Pipeline/3단 출력은 §7 정본(옵션 A)로 잠금되어 있으며, 워크플로우 엔진은 이를 반드시 준수한다.

## 4. Fallback / Retry / Recovery 설계(Soft loop / Hard loop)
### 4.1 공통 원칙(체크리스트 + RULE 근거)
- 위험/정책 위반은 재시도 불가 → deny로 귀결 (RULE 1.3 Non-goal/정책 우선)
- 타임아웃은 1회 재시도(체크리스트 요구)
- Self-check 임계값/루프 규칙은 02 ORANGE CORE의 정본을 따른다.
  - 정책(정본/LOCK): **02 §7.6(I-6 Self-check 상세)** 를 따른다. (P0/P1/P2 가변 임계값)
  - 1.0 기본값 Freeze: **02 §7.6의 Self-check 임계값 FREEZE 항목**
  - 동작 고정: FAIL 시 자동 Soft loop 최대 1회 → 재차 FAIL이면 (정책/비용/도메인 충돌 없으면) 승인 요청 또는 중단(deny), 충돌 있으면 07 게이트 결과(deny/hold/downshift)를 우선한다.

- "정본: 02 §7.6(I-6 Self-check)"
- "수치표는 02 §7.6 내 임계값 요약표 참고 가능(정본 아님)"

### 4.2 Soft loop vs Hard loop(정의)
- Soft loop: 동일 단계 내에서 입력/프롬프트/검색 범위를 조정해 1회 보정 후 재시도
- Hard loop: 워크플로우를 한 단계 이전으로 되감아(또는 계획 재수립) 재실행
- Hard loop는 자동 실행 금지.
- 승인(Approval)이 승인된 경우에만, Hard loop(외부 큐/재실행)를 “선택적으로” 허용한다.
- 승인 없이는 Soft loop 1회 이후 즉시 종료(deny/hold 포함)로 고정한다.

- 결정 완료(§4.2): **Soft loop 1회 자동 + 승인 기반 Hard loop(선택) 허용** — Self-check 임계값 자체는 02(I-6) 정본으로만 잠금하되, “미달 시 동작”은 다음으로 고정한다:
  1) 자동 1회 Soft loop(입력/프롬프트/검색 범위 보정)
  2) 재차 미달이면 승인 요청(재시도/확장 검색/도구 사용 등) 또는 중단(deny) 중 하나로 종결
  3) 위험/정책 위반 탐지 시에는 루프/재시도 없이 즉시 deny

### 4.3 Walk-Forward Validation(투자 전략 검증 절차) (ADD-044)
- 목적: 투자/예측 관련 워크플로우에서 과거 데이터 과적합을 방지하고, 전략의 실전 유효성을 검증한다.
- 절차(LOCK):
  1) In-Sample 구간으로 전략 파라미터를 최적화한다.
  2) Out-of-Sample 구간(미래 방향 슬라이딩 윈도우)으로 순차 검증을 수행한다.
  3) 검증 통과 기준(수익률/샤프지수/최대낙폭 임계값)은 실행 전 Gate(07 PolicyCheck)에 등록한다.
  4) 검증 실패 시 전략은 자동 차단(deny)되며, 결과는 실패 리포트(§6)에 기록한다.
- 연결 참조:
  - 07(SAFETY/COST/APPROVAL) Gate 등록
  - 06(STORAGE/MEMORY) 검증 이력 저장


### 4.4 Circuit Breaker 패턴(연속 실패 자동 차단) (ADD-072)
- 목적: 특정 에이전트/도구/외부 API에서 연속 실패가 발생할 경우 자동으로 호출을 차단하여 시스템 전체 장애 전파를 방지한다.
- 상태 정의:
  - CLOSED: 정상 동작 상태(호출 허용)
  - OPEN: 차단 상태(연속 실패 임계값 초과 시 자동 전환, 호출 즉시 deny)
  - HALF-OPEN: 복구 탐지 상태(일정 시간 경과 후 제한적 호출 허용, 성공 시 CLOSED 복귀)
- 임계값 규칙(LOCK):
  - 연속 실패 N회(기본값: 3회) 초과 시 OPEN 전환
  - OPEN 유지 시간(기본값: 60초) 경과 후 HALF-OPEN 시도 (단, P2 이상 예외 — 아래 참조)
  - 임계값 변경은 07 Gate 승인 후만 허용
- P2 이상 CB 전이 규칙(LOCK):
  - P2 이상 작업에서 OPEN 상태 진입 시, 60초 자동 HALF-OPEN 전환을 적용하지 않는다.
  - 반드시 사용자 승인 후에만 HALF-OPEN 전환이 허용된다.
  - 승인 이벤트: `agent.cb.p2_halfopen_approved` → LogEvent 기록 필수
- Circuit Breaker 상태 변경은 반드시 LogEvent에 기록한다.
- 9-State(S0~S8) 파이프라인 매핑:
  - CB는 주로 Execute(S3~S4) 단계에서 외부 API/도구 호출 실패 시 작동한다.
  - OPEN 진입 시 현재 파이프라인은 Verify(S5) 단계로 전이하여 실패 처리한다.
  - HALF-OPEN 복구 시도는 Execute(S3) 단계에서 재진입한다.
- 연결 참조:
  - 07(SAFETY/COST/APPROVAL) Gate 임계값 등록
  - 06(STORAGE/MEMORY) 상태 변경 이력 저장
  - 08(HOLOGRAM) §4.4 UI 상태머신 S0~S8 매핑

### 5.1 VAMOS 자체 경량 프레임워크 아키텍처
- VAMOS는 외부 오케스트레이션 프레임워크(OpenAI Swarm, CrewAI 등)에 의존하지 않고, 자체 경량 프레임워크를 사용한다.
- 프레임워크 구성 원칙:
  1) 최소 의존성: 외부 라이브러리 없이 CORE→NODE→Sub-agent 위임 체계만으로 동작 가능
  2) Gate 우선: 모든 실행은 07 Gate(PolicyCheck/Approval/Cost)를 선행 통과해야 함
  3) trace_id 단위 추적: 모든 실행 단위는 VAMOS trace_id로 식별/기록
  4) 확장 인터페이스: 외부 엔진(LangGraph 등) 연결 시 §7.3 어댑터 규칙을 준수
- 결정 완료(LOCK): VAMOS V1은 자체 경량 프레임워크를 기본으로 한다. 외부 엔진은 어댑터를 통해서만 연결.


### 5.2 버전별 에이전트 구조 확장 로드맵 (ADD-075~077, ADD-080, ADD-084)

#### 5.2.1 V2 Autonomous Agent (ADD-075)
- V2에서 추가 예정: 단일 에이전트가 Plan→Execute→Verify 루프를 내부적으로 자율 수행하는 구조.
- 제약: 자율 루프 내에서도 P2 작업은 승인 없이 실행 불가. 모든 Tool 호출은 07 Gate를 경유.
- 설계 원칙: CORE의 단일결정 원칙(02 참조)을 유지하면서 노드 위임 범위를 확장.

#### 5.2.2 V2 GroupChat (ADD-076)
- V2에서 추가 예정: 복수 에이전트가 공유 컨텍스트(채팅 채널)를 통해 협업하는 GroupChat 구조.
- 제약: 에이전트 간 자유 상호 호출(무한 대화/루프) 금지(§7.3 고정 3 준수). 발화 순서는 CORE가 조율.
- 최종 결론 확정은 02의 단일결정 원칙을 따름.

#### 5.2.3 V2 Crew (ADD-077)
- V2에서 추가 예정: 역할 기반 에이전트 팀(Researcher/Coder/Critic 등) Crew 구조.
- 역할 분해는 허용하나, 각 역할의 Tool 호출은 Execute 단계에서만 수행(§7.3 고정 2 준수).
- Crew 전체 결과는 CORE가 최종 검토 후 Decision Locked 처리.

#### 5.2.4 V3 Multi-Agent Mesh (ADD-080)
- V3에서 추가 예정: 다수 에이전트가 메시 네트워크 형태로 분산 협업하는 구조.
- 각 에이전트 노드는 독립적 trace_id를 보유하며, 상위 워크플로우 trace_id와 계층적으로 연결.
- Gate 선행 원칙(§7.3 고정 1)은 메시 내 모든 노드에 동일하게 적용.

#### 5.2.5 V3 Federated Agent (ADD-084)
- V3에서 추가 예정: 외부 시스템/조직의 에이전트와 연합(Federation)하는 구조.
- 외부 에이전트 결과는 반드시 EvidenceBundle로 포장하여 07 Gate 검증 후 Decision에 반영.
- 외부 에이전트에 대한 P2 위임은 별도 연합 승인 정책 적용(세부 정책은 V3 설계 시 확정).

### 멀티 에이전트 vs 싱글 에이전트 판단 기준 [REF:영상1]

> **판단 프레임워크**: 맥락 공유(Context Sharing) 필요도에 따라 결정한다.

| 판단 기준 | 멀티 에이전트 적합 | 싱글 에이전트 적합 |
|-----------|-------------------|-------------------|
| 맥락 연속성 | 낮음 (독립 태스크) | 높음 (연속 추론 필요) |
| 대표 예시 | 여행 계획 (항공+호텔+일정 병렬) | 소설 집필 (일관된 맥락 유지) |
| 정보 공유 | 결과만 공유하면 충분 | 중간 과정 전체 공유 필요 |
| VAMOS 적용 | BLUE NODE 병렬 (최대 3개) | ORANGE CORE 단일 체인 |

- **VAMOS 원칙**: 맥락 연속성이 높은 작업은 ORANGE CORE 싱글 체인으로 처리하고, 독립적 하위 태스크만 BLUE NODE로 분산한다.
- **병렬 상한**: 동시 활성 에이전트 최대 3개 (D2.0-01 LOCK)

### 5.3 Agent 핵심 설계 요소 (ADD-024~027)

> Part E 변경점: Agent 3요소, 통신 패턴, Agentic Coding, 프로파일링이 없었으므로 추가한다.

#### 5.3.1 Agent 3요소 정의 (ADD-024)

- 모든 VAMOS Agent(CORE, Node, Sub-agent)는 다음 3요소를 필수 보유한다:
  1. **Identity (정체성)**: 에이전트 고유 ID, 역할(Role), 도메인, 허용 도구 목록
  2. **Capability (능력)**: 수행 가능한 작업 유형, 사용 가능한 Brain/Tool, 최대 복잡도 등급
  3. **Policy (정책)**: 승인 수준(P0/P1/P2), 비용 한도, Self-check 임계값, 자율 수준(AgentMode)

```python
class AgentProfile:
    """Agent 3요소 표준 프로파일 (ADD-024)"""
    # Identity
    agent_id: str                      # 고유 식별자
    role: Literal["core", "analysis", "coding", "research", "trading"]
    domain: List[str]                  # 담당 도메인
    allowed_tools: List[str]           # Allowlist 기반

    # Capability
    supported_tasks: List[str]         # 수행 가능 작업 유형
    brain_preference: List[str]        # 선호 Brain 순서 (Failover 체인)
    max_complexity: Literal["simple", "moderate", "complex"]

    # Policy
    approval_level: Literal["P0", "P1", "P2"]
    cost_budget: float                 # 세션 내 비용 한도
    selfcheck_threshold: float         # Self-check 임계값 (0~1)
    mode: AgentMode                    # §1.1 Agent Mode 참조
```

#### 5.3.2 Agent 통신 패턴 (ADD-025)

- VAMOS 에이전트 간 통신 방식을 표준화한다:

| 패턴 | 설명 | 사용 시점 | 제약 |
|------|------|----------|------|
| **Request-Response** | CORE→Node 단방향 위임 + 결과 반환 | V1 기본 | 동기 호출, 타임아웃 30초 |
| **Event-Driven** | LogEvent 기반 비동기 알림 | 상태 변경 통지 | §6 LogEvent 스키마 준수 |
| **Pipeline** | 5단계 순차 전달 | 표준 워크플로우 | §7.1 Pipeline 고정 |
| **Broadcast** | CORE→전체 Node 동시 알림 | 긴급 중단/정책 변경 | Gate deny 전파 |

- **직접 Node-to-Node 통신 금지** (LOCK): 모든 에이전트 간 통신은 CORE를 경유한다. 단, Execute 단계 내 Sub-agent 간 데이터 전달은 공유 컨텍스트(trace 단위)를 통해 허용.
- 통신 로그: 모든 에이전트 간 메시지는 `agent.comm.{pattern}` event_type으로 기록.

#### 5.3.3 Agentic Coding 패턴 (ADD-026)

- 목적: AI 에이전트가 코드를 생성/수정/실행하는 워크플로우 패턴을 정의한다.
- Agentic Coding 파이프라인:
  1. **Spec 분석**: 사용자 요구사항을 IntentFrame(02 §11.12.1)으로 분류
  2. **코드 계획**: Plan 단계에서 코드 구조/파일 목록 설계
  3. **코드 생성**: Execute 단계에서 Coding Node(03 Blue)가 코드 생성
  4. **샌드박스 실행**: Docker 샌드박스(02 §1.3-A)에서 코드 실행 및 테스트
  5. **검증**: Self-check(02 I-6) + 코드 품질 게이트(린팅/테스트 통과 확인)
  6. **결과 전달**: Deliver 단계에서 코드 + 실행 결과 + 근거 출력
- 안전 규칙(LOCK):
  - 파일 시스템 쓰기는 샌드박스 내부에서만 허용 (02 §1.3-A Docker 규칙)
  - 외부 패키지 설치는 P1 승인 필요 (07 Gate)
  - 생성된 코드가 기존 코드를 덮어쓰는 경우 P2 승인 필수

#### 5.3.4 Agent 프로파일링 (ADD-027)

- 목적: 에이전트 성능을 지속적으로 프로파일링하여 최적화 데이터를 수집한다.
- 프로파일링 대상:
  | 지표 | 측정 방식 | 저장 | 활용 |
  |------|----------|------|------|
  | 응답 지연 (latency) | 단계별 시간 측정 | trace_spans | 병목 분석 |
  | 작업 성공률 | 완료/실패 비율 | decision_log | Agent 신뢰도 산출 |
  | 비용 효율 | 비용 대비 QoD 점수 | cost_log | 라우팅 최적화 |
  | 도구 호출 효율 | 호출 횟수 대비 유용성 | tool_call_log | Allowlist 개선 |
  | Self-check 통과율 | 1차 통과 vs 재시도 비율 | selfcheck_log | 프롬프트 최적화 |
- 프로파일 데이터는 06(STORAGE)에 저장, 04(INFRA) Observability Dashboard(§10.5)에서 시각화.
- V2+에서 프로파일 데이터를 기반으로 자동 Brain 라우팅 최적화 (I-5 피드백 루프).

## 7. 워크플로우 표준(5단계 Pipeline + 3단 출력)
- 워크플로우는 “표준 5단계 Pipeline + 3단 출력”을 반드시 준수한다. (체크리스트 요구)
- 결정 완료: 표준 파이프라인/출력은 옵션 A를 정본으로 잠금한다.

### 7.1 표준 5단계 Pipeline(정본) [S14-A-013: 최신 패턴으로 업데이트]
1) Intake(요청/제약 수집 + G0 입력 검증 자동 실행)
2) Plan(분해/라우팅 + G1 정책 검사 + G2 비용 검사 + TEE 루프 Think 단계)
3) Execute(도구/노드 실행 + TEE 루프 Execute 단계 + 07 Gate 선행 필수)
4) Verify(G3 품질 검사 + Self-check + EVX 체인 + TEE 루프 Evaluate 단계)
5) Deliver(출력/로그/저장 + G4 최종 승인 + 3단 출력 산출)
- [UI 연결 참조] 본 파이프라인의 각 단계 변경은 **08. Hologram View의 Left Panel (Runtime Pipeline 타임라인)**에 실시간으로 반영되어야 한다.

- 추가 규칙(LOCK):
  - Memory/Reflection은 표준 파이프라인의 Deliver(5) 단계 내부 하위 구간으로 포함된다.
  - EVX 배치에서 사용되는 Memory/Reflection 구간은 위 Deliver 하위 구간을 의미한다.

### 7.2 표준 3단 출력(정본) (LOCK)
| # | 필드명 | 설명 |
|---|--------|------|
| 1 | user_response | 최종 결과 (사용자에게 전달되는 응답) |
| 2 | evidence_summary | 근거/요약 (출처, 신뢰도, QoD 포함) |
| 3 | log_report | 로그/리포트 (trace_id, 저장/승인 이벤트 포함) |

- 규칙(LOCK): 모든 워크플로우 종료 시 위 3개 필드를 반드시 산출한다.
- 스키마 정본: 02 Decision 확장 또는 별도 OutputEnvelope 스키마로 정의한다.

### 7.3 외부 Workflow Engine(그래프/메시지/플로우) 어댑터 규칙(P2 반영 / FREEZE 확정)
- 목적: LangGraph/AutoGen/CrewAI/LlamaIndex/ReAct 등 “외부 실행 엔진”을 쓰더라도, VAMOS의 trace/log/gate 규격을 깨지 않게 고정한다.
- 고정 1) Gate 선행:
  - 어떤 실행 엔진을 쓰더라도 Execute 단계(도구/노드 실행) 전에 07의 PolicyCheck/Approval/Cost 게이트 결과가 선행 입력으로 확정되어야 한다.
  - Gate 결과를 에이전트 내부 if문으로 대체하는 방식은 허용하지 않는다(우회 위험).
- 고정 2) Checkpoint/Replay/Fork 범위:
  - Checkpoint/Replay/Fork(재현/분기)는 “VAMOS trace_id 단위”로만 허용한다.
  - 재현 실행은 동일 trace_id의 실행 이력(LogEvent)과 Gate 판정(07)을 함께 참조해야 한다. (감사/재현성 유지)
- 고정 3) 멀티 에이전트 협업 제약:
  - 역할 분해(Researcher/Coder/Critic 등)는 가능하나,
    - (a) 에이전트끼리 자유 상호 호출(무한 대화/루프)은 금지
    - (b) 모든 Tool 호출은 07 Gate를 통과한 “Execute 단계”에서만 수행
  - 최종 결론 확정(Decision Locked)은 02의 단일결정 원칙을 따른다.
- 연결 참조:
  - 단일결정/충돌 해결/루프 종료 원칙: 02 참조
  - Gate 스키마/이벤트/실패/폴백: 07 참조
  - UI 타임라인/승인 카드/비용 경고: 08 참조

### 7.4 Verify 체인(EVX-*) 배치/기록 규칙 (Design 2.0 / PHASE 3) (NEW)

- 목적: “검증을 했는지/무엇을 썼는지”가 Decision에 남도록 강제한다.
- 원칙(LOCK):
  1) Verify는 “단계별로 선택적으로” 실행될 수 있으나, 실행된 경우 반드시 `Decision.verify.chain_used`에 기록한다.
  2) Gate 요약 결과는 `Decision.gates.result`에 기록한다.
  3) Verify 산출물(근거 번들, QoD 점수, 정책 결과)은 각각 07/06의 정본 스키마로 저장하고 Decision은 그 ref만 가진다.

- 표준 기록 매핑:
  - Self-check(QoD) → Decision.verify.chain_used += ["self_check"]  (EVX-1~6과 별개 “표준 체인 토큰”, 정본 허용)
  - PolicyCheck → Decision.gates.result.policy
  - Approval → Decision.gates.result.approval
  - CostBudget/Downshift → Decision.gates.result.cost
  - EvidenceBundle → Decision.gates.result.evidence + Decision.verify.chain_used

- 추가 규칙(LOCK):
  - Decision.verify.chain_used에는 (a) EVX-1~EVX-6 같은 EVX-ID 또는 (b) self_check/evidence_check 같은 표준 체인 토큰만 기록한다.

- 연결 참조:
  - 02(ORANGE CORE) Decision 확장
  - 07(SAFETY/COST/APPROVAL) Gate 정본 ENUM/스키마
  - 06(STORAGE/MEMORY) Evidence/QoD/MemoryRecord 저장 위치

### 7.4.1 Verify 체인(EVX-1~EVX-6) 배치 위치 + 기록 + 차단 조건(LOCK) (NEW)

- 배치 원칙(LOCK):
  - Verify 체인(EVX-*)은 반드시 아래 5단계 구간 중 하나에 “명시적으로” 배치된다.
    1) Plan 구간: 실행 전 제약/정책/불확실성 신호 생성
    2) Execute 구간: (필요 시) 실행 중 결과 검증/재검증
    3) Verify 구간: 최종 검증(자기검증/적대검증/제약검증)
    4) Memory 구간: 저장 전 검증(저장 정책/민감도/근거 품질)
    5) Reflection 구간: 사후 평가/학습 신호 생성(즉시 적용 금지 → Decision 신호로만 기록)

- 기록 원칙(LOCK):
  - 실행된 EVX는 반드시 `Decision.verify.chain_used`에 EVX-ID로 기록한다.
  - Gate 결과는 반드시 `Decision.gates.result.*`에 기록한다. (Policy/Approval/Cost/Evidence)

- 공통 차단 조건(LOCK):
  - 07 PolicyCheck=deny 이면 모든 EVX 실행/재시도 없이 즉시 중단(deny)한다.
  - 비용 상한 초과/차단 상태면 EVX 확장 실행은 승인 없이는 불가(deny 또는 hold).
  - P2(위험/확장) 도메인 관련 EVX는 승인 없이는 실행 불가(deny 또는 pending→승인요청).
  - Self-check 임계값 미달 시 자동 Soft loop 1회까지만 허용(그 외는 승인요청 또는 중단). (임계값 숫자는 02 정본을 따른다)

| EVX-ID | 목적(요약) | 배치 위치 | Decision 기록 | 막는 조건(요약) |
|-------|------------|----------|--------------|----------------|
| EVX-1 | Code-as-Policy | Plan/Verify | verify.chain_used += [EVX-1], gates.result.policy 참조 | Policy deny / P2 승인 없음 |

### 7.5 프롬프트 관리 체계(라이브러리/최적화/버전관리/A·B테스트) (ADD-061~064)

#### 7.5.1 프롬프트 라이브러리 (ADD-061)
- 워크플로우 내 반복 사용되는 프롬프트는 **중앙 프롬프트 라이브러리**에 등록하여 관리한다.
- 라이브러리 항목 구조: `prompt_id`, `version`, `template`, `tags`, `owner`, `created_at`
- 라이브러리에 등록되지 않은 임시 프롬프트는 P2 이상 작업에서 사용 금지.

#### 7.5.2 프롬프트 최적화 (ADD-062)
- 프롬프트 최적화는 Self-check(QoD) 점수 및 실패 로그를 근거로 수행한다.
- 최적화 결과는 반드시 신규 버전으로 등록하며, 기존 버전을 직접 덮어쓰는 것은 금지한다.

#### 7.5.3 프롬프트 버전관리 (ADD-063)
- 모든 프롬프트는 semantic versioning(MAJOR.MINOR.PATCH)을 따른다.
- 변경 이력(diff, 변경 사유, 승인자)은 06(STORAGE) 프롬프트 버전 레지스트리에 저장한다.
- 롤백은 이전 버전 ref를 지정하는 방식으로만 허용한다.

#### 7.5.4 A/B 테스트 (ADD-064)
- 프롬프트 A/B 테스트는 P1 이하 작업에서만 자동 실행 허용하며, P2 이상은 승인 후 실행한다.
- 테스트 결과(QoD 점수, 성공률, 비용)는 LogEvent에 기록하고, 승인된 버전만 라이브러리 정본으로 확정한다.
- A/B 테스트 중 정책 위반이 탐지되면 즉시 중단(deny)하고 두 버전 모두 롤백한다.

  - 인계 메시지: `{task_id, context, partial_result, handoff_reason}`
  - 인계 시 trace_id 유지 (동일 워크플로우 내)
  - 인계받는 에이전트는 context를 검증한 후 실행 시작
  - 인계 이벤트는 LogEvent에 기록 (event_type: `agent.handoff`)

#### 12.4.6 Function Calling 고급 패턴 (3건)

- 병렬 Function Calling: 독립적 도구 호출을 동시 실행
- 순차 Function Calling: 이전 결과를 다음 호출 입력으로 사용
- 조건부 Function Calling: Gate 결과에 따라 호출 여부 결정
- 모든 FC는 07 Gate 통과 후에만 실행(§7.3 고정1)

### 12.5 Swarm / MCP 패턴 (S06-B20, 31건)

#### 12.5.1 TEE (Think-Execute-Evaluate) 루프 (10건) — CRITICAL

- 목적: 에이전트의 핵심 실행 루프를 정의한다.
- 구조(LOCK):
  1) **Think**: 현재 상태/컨텍스트 분석 → 다음 액션 결정
     - Gate 선행 확인 (07 PolicyCheck)
     - 비용 예측 (Cost 게이트)
  2) **Execute**: 결정된 액션 실행 (도구 호출/서브에이전트 위임)
     - Allowlist 자동승인 또는 명시적 확인 (§2.2)
     - 실행 결과 수집
  3) **Evaluate**: 실행 결과 평가
     - Self-check (QoD 점수, 02 정본)
     - 목표 달성 여부 판단
     - 미달 시: Soft loop 1회 → 재 Think (§4.2)
- TEE 루프 최대 반복: P0=3회, P1=5회, P2=10회 (승인 시 확장)
- 연결 참조: §7.1 표준 5단계 중 Plan/Execute/Verify에 TEE 루프 내장

#### 12.5.2 Routine 패턴 (5건)

- Swarm의 Routine 개념을 VAMOS에 대응:
  - Routine = 사전 정의된 에이전트 동작 시퀀스
  - 시스템 프롬프트 + 도구 목록 + 전환 조건으로 구성
  - 예: 투자 분석 Routine = `{system_prompt, tools: [financial_data, chart], transitions: [{condition: "추가 분석 필요", next: "심층분석_routine"}]}`
- VAMOS 대응: §7.5 프롬프트 라이브러리 + SOP 패턴(§12.4.3)

#### 12.5.3 Context Variables 패턴 (5건)

- 워크플로우 전체에서 공유되는 컨텍스트 변수:
  - 전역 변수: `user_id`, `session_id`, `trace_id`, `budget_remaining`
  - 단계별 변수: `current_stage`, `partial_results`, `gate_results`
  - 에이전트별 변수: `agent_state`, `tool_history`, `error_count`
- 스코프 규칙(LOCK):
  - 전역 변수는 워크플로우 전체에서 읽기/쓰기 가능
  - 단계별 변수는 해당 단계 내에서만 쓰기 가능, 후속 단계에서 읽기 가능
  - 에이전트별 변수는 해당 에이전트만 쓰기 가능

#### 12.5.4 MCP 통합 패턴 (11건)

- MCP 서버를 에이전트 도구로 연결하는 표준 패턴:
  - 도구 등록: MCP 서버 discovery → ToolRegistry(04)에 자동 등록
  - 도구 호출: Agent → MCP Client → MCP Server → 결과 반환
  - 도구 승인: §2.2 Allowlist 규칙 적용
  - 에러 처리: MCP 서버 타임아웃 → Circuit Breaker(§4.4) 적용
  - 멀티 MCP: 동일 기능 복수 MCP 서버 → 라운드로빈 또는 우선순위 기반 선택
- 연결 참조: 03(BLUE NODES) MCP 레이어, 04(INFRA) ToolRegistry

- Skill 선택: 에이전트가 태스크에 맞는 Skill을 자동 선택 (유사도 기반)
- Skill 업데이트: 성공률/QoD 추적 → 개선 후보 생성 → 승인 후 새 버전 등록
- 연결 참조: §7.5 프롬프트 라이브러리, 06(STORAGE) Skill 저장

### 12.7 Trigger/Action / 워크플로우 패턴 (S07-B23, 26건)

#### 12.7.1 Trigger/Action 이벤트 스키마 — CRITICAL (8건)

- 목적: 워크플로우를 자동으로 시작하는 트리거 시스템.
- Trigger 유형:

| 유형 | 설명 | 예시 |
|------|------|------|
| schedule | 시간 기반 | 매일 09:00 시장 리포트 |
| event | 이벤트 기반 | 종목 가격 ±5% 변동 |
| webhook | 외부 호출 | API 요청 |
| manual | 사용자 수동 | UI 버튼 클릭 |
| chain | 워크플로우 완료 후 | 분석 완료 → 리포트 생성 |

- Action 유형: `execute_workflow`, `send_notification`, `create_task`, `update_data`
- Trigger-Action 매핑은 07 Gate에 등록하여 정책 검증 대상에 포함
- 이벤트 스키마: `{trigger_id, trigger_type, condition, action_type, target_workflow, priority_level}`

#### 12.7.2 Prompt Chaining 패턴 (6건)

- 목적: 복수의 프롬프트를 순차/분기로 연결하는 패턴.
- 체인 유형:
  - Sequential: A → B → C (이전 출력이 다음 입력)
  - Branching: A → (조건) → B1 | B2
  - Aggregation: A1, A2, A3 → 종합 B
  - Loop: A → B → (불충분) → A' → B' (최대 N회)
- 모든 체인 단계는 trace_id로 연결

#### 12.7.3 워크플로우 패턴 카탈로그 (12건)

| # | 패턴명 | 구조 | 용도 |
|---|--------|------|------|
| 1 | Sequential | A→B→C | 단순 분석 |
| 2 | Parallel Fan-out | A→[B1,B2,B3]→C | 멀티소스 수집 |
| 3 | Map-Reduce | 데이터 분할→병렬 처리→통합 | 대량 데이터 분석 |
| 4 | Router | 조건별 분기 | 시나리오별 처리 |
| 5 | Retry-with-Backoff | 실패→대기→재시도 | 외부 API 호출 |
| 6 | Saga | 분산 트랜잭션→보상 | 멀티 시스템 연동 |
| 7 | Pipeline | 단계별 변환 | 데이터 정제 |
| 8 | Observer | 이벤트 감지→반응 | 모니터링 |
| 9 | Mediator | 중앙 조율 | 멀티에이전트 |
| 10 | Chain of Responsibility | 핸들러 체인 | 승인 단계 |
| 11 | State Machine | 상태 전이 | 복잡 워크플로우 |
| 12 | Event Sourcing | 이벤트 로그 기반 | 감사/재현 |

### 12.8 Agent Marketplace / 라이프사이클 (S07-B24, 28건)

#### 12.8.1 Agent Marketplace (10건)

- 목적: 에이전트/워크플로우 템플릿을 공유하고 재사용하는 마켓플레이스.
- 마켓플레이스 항목:
  - Agent 프로필: `{agent_id, name, skills, success_rate, cost_per_use}`
  - Workflow 템플릿: `{template_id, name, steps, required_tools, estimated_cost}`
  - Skill Pack: `{skill_ids, description, compatibility}`
- 설치/활성화: 07 Gate 승인 후만 허용
- 품질 보증: 마켓플레이스 등록 시 자동 테스트 + QoD 기준 충족 필수

#### 12.8.2 에이전트 라이프사이클 관리 (10건)

- 에이전트 상태 흐름: `created → initialized → active → (suspended | terminated) → archived`
- 생성(created): 프로필 + 기본 설정
- 초기화(initialized): 도구 연결 + Skill 로드 + Gate 등록
- 활성(active): 태스크 수행 가능
- 일시정지(suspended): 리소스 부족 / 정책 위반 시
- 종료(terminated): 태스크 완료 또는 에러
- 보관(archived): 이력 보존 (06 STORAGE)
- 각 상태 전이는 LogEvent에 기록

#### 12.8.3 배포 자동화 (8건)

- 에이전트/워크플로우 배포 파이프라인:
  1) 개발: Skill + Prompt + Tool 정의
  2) 테스트: 자동 테스트 스위트 실행 (QoD, 안전성, 비용)
  3) 스테이징: 제한된 트래픽으로 A/B 테스트
  4) 프로덕션: 전체 트래픽 전환
  5) 모니터링: 성능/비용/에러율 추적
- 롤백: 이전 버전 즉시 복원 (trace_id 기반)
- 연결 참조: 04(INFRA) 배포 인프라

### 12.9 RAG 통합 / 4레이어 Guardrails 에이전트 연동 (S07-B25, 19건)

#### 12.9.1 RAG 통합 API (10건)

class RAGInterface:

#### 12.9.2 4레이어 Guardrails 에이전트 연동 (9건)

- 에이전트 워크플로우에서 Guardrails 4레이어를 호출하는 시점:
  - L1 NeMo Guardrails: Intake 단계 (입력 필터링)

### 12.11 HITL / DSPy / 오케스트레이션 (S09-NS, 15건)

#### 12.11.1 HITL (Human-in-the-Loop) 패턴 (5건)

- 목적: 에이전트 실행 중 인간 개입이 필요한 시점을 정의한다.
- HITL 개입 시점:
  - **P2 승인**: 위험/확장 작업 실행 전 (§2.1)
  - **불확실성 높음**: QoD < 0.5 (02 정본) 시 에이전트가 인간에게 판단 위임
  - **모호한 의도**: 사용자 질문이 복수 해석 가능 시 명확화 요청
  - **비용 초과 예상**: 예상 비용이 한도의 80%+ 시 사전 확인
  - **결과 검증**: 투자 추천 등 고위험 결과물의 인간 최종 검토
- HITL 이벤트: `agent.hitl_requested`, `agent.hitl_resolved`
- 연결 참조: 08(UI) 승인 카드

#### 12.11.2 DSPy Evaluation 패턴 (5건)

- 목적: 에이전트/프롬프트 성능을 체계적으로 평가하는 프레임워크.
- 평가 항목:
  - Accuracy: 정답률 (벤치마크 데이터셋 기반)
  - Faithfulness: 근거 충실도 (출처 대비 답변 정합성)
  - Relevance: 관련성 (질문 대비 답변 적합도)
  - Toxicity: 유해성 (안전 가이드라인 위반 여부)
  - Cost-efficiency: 비용 효율 (품질 대비 비용)
- 평가 실행: §7.1 Verify 단계에서 자동 실행
- 결과 저장: 06(STORAGE) 평가 이력

#### 12.11.3 오케스트레이션 패턴 (5건)

- 오케스트레이션 유형:

| 유형 | 설명 | 제어 주체 | VAMOS 대응 |
|------|------|----------|-----------|
| Centralized | 단일 오케스트레이터가 전체 제어 | CORE | V1 기본 |
| Hierarchical | 계층적 위임 | CORE→NODE→Sub | §5 구조 |
| Market-based | 경매/입찰 방식 태스크 할당 | Marketplace | V3+ |
| Stigmergic | 환경 기반 간접 협업 | 공유 메모리 | V3+ |

- V1은 Centralized 전용, V2부터 Hierarchical 지원, V3에서 고급 패턴 도입
- planning_strategy: ReAct(기본) / LLM+P(복잡 작업) / Monte Carlo(불확실 환경)

### 12.12 StateGraph / Agent Loop (S10-YS, 8건)

#### 12.12.1 HITL Node 구현 (2건)

- StateGraph 내 HITL 노드 정의:
  - 상태: `pending_human → human_reviewing → human_decided`
  - 입력: 에이전트 결과 + 승인 요청 사유
  - 출력: `approve/deny/modify` + 인간 코멘트
  - 타임아웃: 설정 가능 (기본 30분, P2는 무제한)

#### 12.12.2 StateGraph 기반 워크플로우 (2건)

- VAMOS 5단계 파이프라인을 StateGraph로 표현:

```
Intake → Plan → [Gate Check] → Execute → Verify → Deliver
                    ↓(deny)              ↑(retry)
                  Abort ←──────── Soft Loop
```

- 각 노드는 상태(state)를 가지며, 전이(transition) 조건으로 분기
- 상태 저장: 06(STORAGE) 체크포인트

#### 12.12.3 Multi-Agent 코드 예제 (2건)

- V2+ 멀티에이전트 워크플로우 코드 패턴 참조:

```python
# V2+ 멀티에이전트 패턴 참조 (VAMOS 자체 프레임워크)
class MultiAgentWorkflow:
    def __init__(self):
        self.orchestrator = CoreAgent(role="orchestrator")
        self.agents = {
            "researcher": NodeAgent(role="researcher", skills=["web_search", "rag"]),
            "analyst": NodeAgent(role="analyst", skills=["financial_analysis"]),
            "critic": NodeAgent(role="critic", skills=["verify", "adversarial"])
        }

    async def execute(self, task):
        plan = await self.orchestrator.plan(task)
        results = await asyncio.gather(*[
            self.agents[a].execute(step) for a, step in plan.assignments
        ])
        verified = await self.agents["critic"].verify(results)
        return self.orchestrator.deliver(verified)
```

#### 12.12.4 Agent Loop 패턴 (2건)

- TEE 루프(§12.5.1)의 구현 레벨 패턴:
  - 루프 진입: 태스크 수신 → 컨텍스트 로드
  - Think: LLM 호출 → 다음 액션 결정
  - Execute: 도구 호출 / 서브에이전트 위임
  - Evaluate: 결과 평가 → 목표 달성 여부
  - 루프 종료: 목표 달성 OR 최대 반복 OR Gate deny

### 12.16 G0~G4 워크플로우 / Gate 피드백 (S14-CLIB, 5건)

- 목적: §7.1 5단계 Pipeline 각 단계에 Gate를 통합하는 상세 규칙.
- Gate-Pipeline 매핑(LOCK):

| Pipeline 단계 | Gate | 역할 | 실패 시 |
|-------------|------|------|--------|
| 1) Intake | G0 | 입력 검증 (형식/범위/안전) | 즉시 거부 |
| 2) Plan | G1 | 정책 검사 (PolicyCheck) | deny |
| 2) Plan | G2 | 비용 검사 (CostBudget) | deny/downshift |
| 3) Execute | — | 실행 (07 Gate 선행 완료 상태) | — |
| 4) Verify | G3 | 품질 검사 (QoD + EVX) | Soft loop/deny |
| 5) Deliver | G4 | 최종 승인 (P1+: Approval) | hold/deny |

- Gate 피드백 루프:
  - G3 실패 → Soft loop 1회 (§4.2) → Execute 재실행
  - G3 재실패 → 승인 요청 또는 deny
  - G2 downshift → 저비용 모델로 자동 전환 후 재실행
  - G4 hold → 사용자 승인 대기 큐 등록
- 모든 Gate 결과는 Decision.gates.result에 기록 (§7.4)
- 연결 참조: 07(SAFETY/COST/APPROVAL) Gate 정본, 02(CORE) Decision 스키마

  - Browser Node: 웹 자동화 (Playwright)
  - File Node: 파일 읽기/쓰기/변환
  - Decision Node: 조건 분기
  - Loop Node: 반복 처리
  - Human Node: 사용자 입력 대기 (HITL, §12.11.1 연동)
  - Schedule Node: 시간 기반 트리거
  - Notification Node: 알림 발송
- **엣지(Edge)**: 데이터 흐름 + 조건 (§3 조건 분기와 통합)
- **변수(Variable)**: 워크플로우 간 데이터 전달 (§12.5.3 Context Variables 패턴 준수)
- **에러 핸들러**: 실패 시 재시도/폴백/알림 (§4 Fallback/Retry/Recovery 설계 준수)
- **상태 관리**: `WorkflowState`: pending → running → paused → completed → failed
- **실행 모드**: sequential (기본), parallel (fan-out/fan-in, §14.1 병렬 분기 패턴 연동)
- **기술 스택**: Python asyncio, NetworkX (DAG), Pydantic v2 (스키마)
- **VAMOS 연동**:
  - §7.1 표준 5단계 Pipeline과 통합 (엔진이 Pipeline을 실행)
  - §7.3 고정1 Gate 선행 원칙 준수 — 모든 Execute 전 07 Gate 통과
  - §12.5.3 Context Variables로 워크플로우 상태 공유
  - 06(STORAGE) 워크플로우 정의 JSON 저장
- **시중 도구 비교**:
  - n8n: 오픈소스, 셀프호스팅, 노드 기반
  - Make.com: 비주얼, 클라우드, 유료
  - Zapier: 가장 많은 연동, 유료
  - VAMOS 차별화: AI 네이티브 + 대화형 생성 + 개인화 + 로컬
- **V1/V2/V3 로드맵**: V1: 기본 DAG 엔진 즉시 | V2: 비주얼 에디터(React Flow) 3개월 | V3: 마켓플레이스
