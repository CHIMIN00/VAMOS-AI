## 6. Acceptance Criteria

- **AC-D7-001**: D7의 모든 스키마 블록은 D1 Schema Meta Template의 `_meta` 필수 속성(`schema_name`, `schema_version`, `owner_doc`, `sec_id_slug`, `ref`, `ref_locked`, `ref_legacy`)을 포함한다.

- **AC-D7-002**: §4의 각 스키마 필드는 07의 derived view(§6.2~§6.4) 및 비용/다운시프트 LOCK(§4.1~§4.2) 범위 내에서만 정의되며, 근거 없는 필드/enum/코드값을 추가하지 않는다.

- **AC-D7-003**: §5에 스키마별 Example JSON이 최소 1개 이상 존재한다(총 4개).

- **AC-D7-004**: event_type/failure_code/fallback_id 코드값 목록은 02 정본 레지스트리에서만 참조하며, D7에서 신규 생성하지 않는다(REF-only 원칙).

- **AC-D7-005**: GuardrailsCheckSchema.overall_decision은 3-Layer 모두 통과 시에만 "allow"이다. 어느 Layer든 실패 시 "restrict" 또는 "deny".

- **AC-D7-006**: RBACRoleSchema.role enum은 OWNER | ADMIN | OPERATOR | VIEWER 4개만 허용한다 (07 §3.6 LOCK).

- **AC-D7-007**: AutonomyLevelSchema.level은 L0~L3만 허용하며, L2/L3에서만 auto_execute=true가 허용된다 (07 §3.2.1 LOCK).

## 7. Failure / Fallback (REF-only 연결)

### 7.1 원칙
- event_type / failure_code / fallback_id의 **값 목록**은 02 정본 레지스트리에서만 확정된다.
- D7에서는 목록을 재정의하지 않으며, 시나리오/의미/연결만 정의한다(REF-only).

### 7.2 PolicyCheckSchema.decision 처리 연결

| decision | 의미 | 관련 코드값 참조 |
|----------|------|------------------|
| deny | 즉시 중단 | failure_code/fallback_id/event_type → 02 정본 |
| restrict | 안전한 축소 실행(제한 안내/일부 차단) | failure_code/fallback_id/event_type → 02 정본 |
| allow | 실행 허용 | - |

### 7.3 ApprovalSchema 연결
- PolicyCheck 결과가 "승인 필요"로 분기되는 경우(`required_approval_id` 사용), ApprovalSchema를 생성/참조하여 plan/execute 승인 게이트를 거친다.

### 7.4 CostBudget/Downshift 연결
- `used_today`/`used_month` 기준 임계(80%/100%) 도달 시 DownshiftSchema 정책이 적용된다.
  - 80% 경고 → 근접 구간 Mini 우선 강제
  - 100% 초과 → 승인 없이 자동 차단(deny)
- 해당 이벤트/실패/폴백 코드값은 02 정본 레지스트리에서만 선택/기록한다.
