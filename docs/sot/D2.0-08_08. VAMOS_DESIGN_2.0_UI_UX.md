## 1. UI/UX 철학(구조 가시성/이중모드/근거투명성/오케스트레이션 고정)

### 1.1 구조 가시성(Observability First)
- 사용자는 “결과”를 보되, 필요 시 “왜 그렇게 됐는지(근거/상태/비용/승인)”를 UI에서 즉시 확인 가능해야 한다.
- Builder는 전체 시스템 관측(Trace/Graph/Logs)을 최우선으로 하고, Hologram은 관측 정보를 ‘필요할 때만’ 노출한다.

### 1.2 이중 모드(Two Views, One System)
- **Builder View**는 시스템을 “만드는 창(설정/정책/템플릿/레지스트리/디버그)”이다.
- **Hologram View**는 사용자가 “쓰는 창(대화/산출물/근거/상태/승인)”이다.
- 두 뷰는 동일한 run/trace/decision을 공유하고, 표현 방식만 다르다.

### 1.3 근거 투명성(Evidence Transparency)
- Hologram은 복잡한 내부 구조를 숨기되, 인용/출처/근거(QoD 포함)는 항상 접근 가능해야 한다.
- Evidence(문서/웹/메모리)는 “출처 리스트 + 인용 점프 + QoD(상/중/하)”를 기본 제공한다.

### 1.4 오케스트레이션 고정(Decision Lock 제약의 UI 반영)
- ORANGE CORE가 Decision Lock 이후 확정한 “결론/라우팅”은 UI에서 임의 수정 불가.
- 재시도/재생성은 “근거/실행/포맷” 축의 수정에 한정하며, 결론 변경 버튼/조작은 제공하지 않는다.

### 1.5 승인/비용/정책을 UI에서 ‘행위 제한’으로 구현
- 승인 필요(P2/정책 변경/비용 임계)는 실행을 HOLD로 전환하고, UI에서 명시적으로 승인/거절을 받는다.
- 비용 임계/초과는 다운시프트 또는 차단을 UI 피드백(경고/모드 전환/실행 중단)으로 확정한다.
- 민감정보는 저장/출력 전 단계에서 마스킹 상태를 시각적으로 표시하고, 승인 없는 정책 변경을 금지한다.

### 1.6 “프레임워크 도입 여부와 무관하게” 동일하게 보이는 중앙 게이트(UI 고정)
- 원칙(P2 반영): LangGraph(LOCK)/AutoGen/CrewAI/LlamaIndex 등 어떤 실행 엔진을 쓰더라도 (정본: LangGraph LOCK, 타 프레임워크는 비교·참조용), UI에서 다음 3가지는 동일하게 유지되어야 한다.
  1) Trace 타임라인: run/trace/decision 단위로 단계(Intake/Plan/Execute/Verify/Deliver) 및 이벤트(LogEvent)를 추적 가능
  2) Approval 카드: 승인 필요 시 HOLD 전환 + 승인/거절 기록이 trace에 귀속
  3) 비용 경고/차단 토스트: 상한 근접/초과 시 다운시프트 또는 차단이 즉시 표면화
- 연결 참조:
  - trace_id / Decision 객체: 02 참조
  - 워크플로우 단계 및 로그 출력: 05 참조
  - Gate 스키마/이벤트/실패/폴백: 07 참조

### 1.7 P2 / Hard loop / Template Injection UI 표시 규칙

- P2 기본 상태는 OFF로 표시한다.
- P2가 켜지는 경우(승인된 세션):
  - “승인 카드(Approval)”에서 P2 활성 상태가 명시되어야 한다.
  - 세션 종료 시 P2는 자동 OFF로 돌아가며, UI에도 “세션 종료로 OFF”가 남아야 한다.

- 재시도 루프 표시:
  - 기본은 Soft loop 1회만 자동 수행(진행 로그에 1회 표기)
  - Hard loop는 “승인 버튼/카드” 없이는 UI에서 선택 불가(비활성)

- Template Injection 표시:
  - 활성 노드/컨텍스트 로드 시 template_id(template_set_id)를 UI에서 확인 가능해야 한다.

## 2. 화면 구조

### 2.1 Builder View(설정/정책/템플릿/레지스트리/디버그) — The Cockpit

**목적**: ORANGE CORE ↔ BLUE NODE 내부 흐름을 시각화하고, 실행을 제어하며, 디버깅/감사(Audit) 가능한 엔지니어링 대시보드 제공  
**타겟**: 시스템 관리자, 고급 사용자, Debug Mode 사용자

#### 2.1.1 레이아웃(좌/중/우) — 3-Column Fluid Layout
- **Left Panel (Navigation & Context)**  
  - 폭: 고정(약 250~300px)  
  - 역할: 리소스 탐색 + 현재 작업 컨텍스트 설정  
  - 구성: 상단(프로젝트/세션 선택), 중단(트리 뷰), 하단(시스템 상태 요약/정책 스냅샷)
- **Center Panel (Main Canvas)**  
  - 폭: 유동(Flex-grow)  
  - 역할: 실행 흐름(Graph) 시각화 + 노드/정책/템플릿 편집  
  - 구성: 상단(툴바: 실행/중지/스텝/시뮬레이트), 중단(Graph/Editor), 하단(Node Inspector: I/O 데이터)
- **Right Panel (Control & Observability)**  
  - 폭: 고정/접기 가능(약 350~400px)  
  - 역할: 실행 중 이벤트/승인/비용/메모리 제어  
  - 구성: 탭(Logs | Approval | Cost | Memory) + Decision/Cost/Memory 드릴다운

#### 2.1.2 패널 구성(필수)
- **Resource / Project Tree (Left)**
  - `Projects > Sessions > Runs(trace_id)`
  - `System > Prompts > Templates`
  - `Registry > EventTypes / FailureCodes / Fallbacks / NodeRegistry(P0/P1/P2)`
  - `Storage > L0 / L1 / L2 (Explorer)`
  - 필터: trace_id 검색, 날짜/기간, 노드/도메인(P0/P1/P2)
  - 하단 요약: Safety 정책 스냅샷, Cost 모드(V0~V3), Approval 규칙(hold 조건)
- **Workflow Graph Canvas (Center)**
  - ORANGE CORE: 중앙 육각형(⬢) 노드, active 시 pulsing
  - BLUE NODE: 주변 원형(●) 노드, 상태별 테두리 색상 변화
  - Flow Edge: 실행 중 경로 하이라이트/애니메이션
  - 오버레이: `Last Execution Time(ms)`, `Token Usage` 배지
  - **Module Manager (Tab 확장)**:
    - 역할: 외부 모듈(.py/.js) 및 커스텀 노드 패키지 관리 + S/E-Module 운영 현황 단일 뷰
    - 구성: 
      - Installed Modules 리스트(버전/상태), Import Wizard(Git/Local), Hot-Reload 트리거
      - **S/E-Module 운영 뷰 (확장)**:
        - 표시 컬럼: module_id, module_type(S/E), state(ON/OFF), approval_status, cost_state, verification_badge, uncertainty_alert
        - 인터랙션: 행 클릭 시 Right Panel에 Evidence Card + Approval Card 동기 표시
  
- **Logs (Right / Tab)**
  - 시간순 스트림(trace_id 중심), severity 필터(INFO/WARN/ERROR/CRITICAL)
  - 로그 클릭 시 JSON payload 전체 드릴다운
- **Approval (Right / Tab)**
  - 승인 대기 항목 카드 스택(사유/컨텍스트/대안/조치)
  - 액션: Approve / Deny / Edit & Retry
- **Cost (Right / Tab)**
  - Budget Gauge(V1/V2/V3 상한 대비 %), Live Token Counter
  - Downshift Control(Mini/Summary 등) + 적용 시 상태/경로 색상 반영
- **Memory (Right / Tab)**
  - 저장 후보 리스트(턴 기준), Policy Check 상태(Sensitive/Approved/Denied)
  - 액션: Commit Selected / Discard All
  - L0/L1/L2 포함/제외 토글

**I-10 핵심 보장**
- UI는 항상 Core의 현재 state를 "1:1"로 반영

## 5. UI 이벤트 명세 (UI Event Registry)

모든 UI 이벤트는 LogEvent(event_type)를 따른다. UI는 이벤트 스트림을 구독하여 화면(상태/토스트/HUD/패널)을 갱신한다.  
원칙: **(1) 추적 가능(trace_id 중심), (2) 결정 잠금(Decision Lock) 이후 결론 조작 불가, (3) 실패는 failure_code로 식별 + fallback_id로 반드시 처리**.

### 5.1 네이밍 규칙 / 네임스페이스

- Event Type 규칙: `ui.{layer}.{subject}.{action}` (소문자, 점(.) 구분)
  - 예: `ui.frontmini.pii.detected`, `ui.core.decision.locked`, `ui.tool.error.timeout`
- Layer 네임스페이스(권장):
  - Front Mini: `ui.frontmini.*`
  - Orange Core / Gate: `ui.core.*`, `ui.gate.*`
  - Blue Node: `ui.node.*`
  - Main Orchestrator: `ui.main.*`
  - Tool / Infra: `ui.tool.*`
  - Memory: `ui.memory.*`
  - (기존 뷰 기반 이벤트는 유지 가능) Builder/Hologram 공통: `ui.builder.*`, `ui.hologram.*`, `ui.common.*`

### 5.2 EventType 네임스페이스 연결 규칙

1. **네임스페이스 분리 원칙**
   - `oc.*`: ORANGE CORE 내부 이벤트 (02 정본)
   - `ui.*`: UI 레이어 이벤트 (08 정본)
   - `bn.*`: BLUE NODE 이벤트 (03 정본)
   - `if.*`: INFRA 이벤트 (04 정본)

2. **연결 규칙**
   - UI 이벤트는 CORE 이벤트를 "트리거"할 수 있다
   - 예: `ui.hologram.input.sent` → `oc.request.received` → `oc.i1.parse.started`
   - 로그 기록 시 `parent_event_id`로 연결 추적

3. **trace_id 공유**
   - 동일 요청에 대한 UI/CORE/NODE 이벤트는 동일 trace_id를 공유한다

4. **네임스페이스 확장(워크플로우/메모리/정책)**
   - `wf.*`: WORKFLOW 이벤트 (05 정본)
   - `mem.*`: STORAGE/MEMORY 이벤트 (06 정본)
   - `sc.*`: SAFETY/COST/APPROVAL 이벤트 (07 정본)

5. **권장 이벤트 체인(기본 순서 고정)**
   - `ui.*` → `oc.*` → `bn.*`/`if.*` → `wf.*` → `mem.*`/`sc.*`

6. **연결 필드 규칙(추적 고정)**
   - 모든 이벤트는 동일 요청 단위로 `trace_id`를 공유한다.
   - UI 이벤트가 CORE/기타 이벤트를 트리거하면, 트리거된 이벤트는 `parent_event_id`로 UI 이벤트를 연결한다.

## 6. 멀티모달 입력 플로우(UI 관점)

VAMOS의 멀티모달 입력은 **5-Phase 파이프라인(Intake → Plan → Execute → Verify → Deliver)**을 따르며, 이를 **4계층 아키텍처(Front Mini / ORANGE CORE / BLUE NODE+Main / Tool)**가 실행한다.
- **5-Phase Pipeline** (논리적 실행 단계): Intake → Plan → Execute → Verify → Deliver
- **4계층 아키텍처** (물리적 컴포넌트): Front Mini → ORANGE CORE → BLUE NODE+Main → Tool
> 주의: 5-Phase는 “실행 흐름”이고, 4계층은 “시스템 구조”이다. 혼용하지 않는다.

전 구간은 LogEvent로 추적되며 UI는 “프리뷰 → 진행 상태 → 근거/승인/비용 → 산출물”을 일관된 카드/패널로 노출한다.

### 6.1 전체 아키텍처 플로우 (Pipeline Flow)

[공통 파이프라인]
1) User Input (Hologram View)
- 파일/이미지/음성/영상 업로드 또는 캡처/녹음
- UI: 입력 프리뷰 카드 + “분석 중…” 상태바 시작

2) Front Mini (Pre-process)
- 메타데이터 추출(MIME, Size, Duration)
- 1차 안전 검사: PII 탐지, 악성코드 스캔
- 초기 요약: 텍스트 추출(OCR/STT), 캡션 생성
- UI: 프리뷰 카드 업데이트(요약/태그/민감 배지)

3) ORANGE CORE (Decision)
- I-1(Intent): 의도 파악(예: 요약/변환/분석/추출)
- Gate Check: Policy(Non-goal), Cost(예산), Approval(P2/위험)
- Routing: 최적 BLUE NODE 선정 + Decision Lock

4) BLUE NODE (Context)
- 도메인 컨텍스트 로딩(P0/P1/P2)
- Workflow/Step 계획 생성(툴/근거/검증 포함)

5) Main Orchestrator & Tool (Execution)
- 실제 도구 호출(분석/변환/검색)
- 중간 결과 스트리밍 + Evidence 링크
- Self-check 후 최종 출력 렌더링 + 저장 후보 생성

[모달리티별 특화 분기]
- 이미지: OCR(텍스트 추출) → 객체 인식 → 캡션 생성 → (필요 시) 구조화/코드 변환
- 음성: 노이즈 제거 → (옵션) 화자 분리 → STT → 뉘앙스/키워드 태깅
- 영상: 키프레임 추출 + 오디오 추출 → (이미지+음성 파이프라인 병렬) → 타임라인 요약
- 문서(PDF 등): 구조 파싱(표/차트 분리) → 섹션 청크 → (필요 시) 임베딩 준비 → 요약/변환

### 6.3 02(I-4)와의 연결 참조

- 전처리 위임: MINI_ANALYZING는 02 문서의 I-4 전처리(구조화/OCR/STT) 호출로 수행된다.
- 게이트 근거: CORE_GATING에서 I-4 메타데이터가 Safety/Cost/Approval 판단 근거로 주입된다.

- Evidence 생성: I-4의 OCR/STT/구조화 결과는 I-2(RAG) 인덱싱 또는 I-3(메모리) 저장 후보로 연결된다.

- VAMOS 차별화: 문서 → 자동 지식그래프 노드 생성, 반복 문서 패턴 학습 → 변경점 자동 하이라이트, 5-Gate(Policy/Cost/Evidence/Approval/SelfCheck) 연동 숫자/통계 데이터 자동 Evidence Gate 검증

## 7. 실패/폴백의 UI 표현(안내 문구/재시도/승인요청)

모든 실패는 `failure_code`로 식별되며, 기본 처리로 **fallback_id 또는 deny** 중 하나로 반드시 수렴한다.  
UI 원칙: (1) 중앙 흐름은 가능한 유지(몰입 방해 최소화), (2) 우측 패널/HUD/토스트로 명확히 안내, (3) 재시도/대안 버튼 제공.

### 7.1 Front Mini (입력 단계) Failure & Fallback

| Failure Code | Trigger | Default | Fallback ID | UI Message (KO) |
|---|---|---|---|---|
| FM_ERR_FMT | 지원하지 않는 형식 | Deny | FB_REJECT_INPUT | 지원하지 않는 파일 형식입니다. (PDF, JPG, MP3 등만 가능) |
| FM_ERR_SIZE | 용량 초과(> Limit) | Deny | FB_REJECT_INPUT | 파일 크기가 제한을 초과했습니다. (최대 25MB) |
| FM_ERR_PII | PII 다수 감지 | Retry | FB_MASK_AND_CONFIRM | 문서에 민감한 정보가 포함되어 있습니다. 마스킹 후 진행할까요? |
| FM_ERR_ZERO | 빈 파일/내용 없음 | Deny | FB_REQ_REUPLOAD | 파일 내용이 비어 있거나 읽을 수 없습니다. 다시 확인해 주세요. |

### 7.2 ORANGE CORE (판단 단계) Failure & Fallback — UI 매핑표

> - **정본 Failure Code (02)**: 런타임/로그에서 사용하는 공식 코드 (02 §6.2 정본)

- 업로드에 실패했습니다. 네트워크 상태를 확인하거나 잠시 후 다시 시도해 주세요.
- 음성 인식에 실패했습니다. 잡음이 적은 곳에서 다시 시도해 주세요.

[정책/안전]
- 이 요청은 VAMOS 안전 정책(Non-goal)에 위배되어 처리할 수 없습니다.
- 개인정보(PII)가 감지되어 해당 부분은 마스킹 처리 후 저장됩니다.
- 이 도메인(P2)은 위험도가 높아 세션별 승인 후에만 사용할 수 있습니다.
- 실거래/주문/계좌/API 연동 요청은 VAMOS 안전 정책(Non-goal)상 수행할 수 없습니다. (거절)
- VAMOS는 법률에 대한 단정적 판단을 제공하지 않습니다. 일반적인 정보만 안내하며, 구체적인 상담은 변호사에게 문의해 주세요.
- VAMOS는 의료에 대한 단정적 판단(진단/처방)을 제공하지 않습니다. 일반적인 정보만 안내하며, 구체적인 상담은 의사에게 문의해 주세요.
- 민감 정보는 장기 저장(L1/L2)할 수 없습니다. 해당 요청은 저장되지 않으며 필요한 경우 마스킹/요약 형태로만 처리됩니다.

[비용/자원]
- 현재 세션의 비용 상한에 도달했습니다. 비용 절약을 위해 일부 기능이 제한될 수 있습니다.
- 비용 효율을 위해 고성능 모델 대신 일반 모델로 전환하여 답변합니다.

[실행/결과]
- 도구 실행 시간이 초과되었습니다. 부분 결과만 표시하거나 다시 시도합니다.
- 결과의 신뢰도가 낮아(Self-check), 자동으로 근거를 재검색하고 있습니다.
- 외부 사이트 접근이 차단되었습니다(403). 대체 경로로 정보를 탐색합니다.

## 9. 결정 필요(옵션 2개) — FREEZE 반영

### 9.0 FREEZE 결정
- **채택(Freeze): 옵션 A (고정 Dock 중심)**
- **보존(삭제 금지): 옵션 B (오버레이/접기 중심)**
- 의미:
  - 구현/기본 UI 동작은 **옵션 A**를 “기본값(Default)”으로 확정한다.
  - **옵션 B**는 “대안(Alternative)”으로 문서에 남기되, 현재 버전(Design 1.0)에서는 기본값으로 쓰지 않는다.
  - 향후 버전(예: 1.1/2.0)에서 실험/전환 후보로 유지한다.

### 9.1 옵션 A (고정 Dock 중심) — ✅ DEFAULT / FREEZE
- **상태**: DEFAULT(기본값) + FREEZE(1.0 확정)
- **정의**:
  - Builder: 우측 패널을 **고정 탭(Logs/Approval/Cost/Memory)** 구조로 운영
  - Hologram: 우측 HUD를 **기본 노출(고정 또는 반고정)**로 운영하되, 중앙 화면 과점유를 피하는 수준에서 유지
- **적용 규칙(필수)**:
  - 기본 레이아웃은 A를 따른다.
  - 관련 UI 토글/설정이 없는 상태에서도 A가 동작해야 한다.
  - Approval/Cost/Evidence는 접근 경로가 항상 1-click 이내여야 한다.

### 9.2 옵션 B (오버레이/접기 중심) — 🧷 KEEP (Alternative, Not Default)
- **상태**: KEEP(보존) / Alternative(대안) / Not Default(기본값 아님)
- **정의**:
  - Builder: 우측 패널을 기본 접기(필요 시 확장)
  - Hologram: Glass HUD를 기본 최소(필요 시 슬라이드 인)
- **보존 규칙(필수)**:
  - 문서에서 삭제하지 않는다.
  - “차기 실험/전환 후보”로만 명시하고, Design 1.0의 기본 구현 요구사항에는 포함하지 않는다.
  - 단, P0/P1 알림(차단/승인)은 B에서도 반드시 사용자에게 도달해야 한다(중앙 모달/슬라이드 인 등).

### 9.3 문서 표기 규칙(중요)
- 옵션 A 섹션 제목에는 반드시 `DEFAULT / FREEZE` 표기를 유지한다.
- 옵션 B 섹션 제목에는 반드시 `KEEP / Alternative / Not Default` 표기를 유지한다.
- 두 옵션 모두 남기되, **구현 기준은 옵션 A**로 해석되도록 한다.

**목적**: AI가 생성한 콘텐츠를 사용자에게 명확히 표시 (투명성 원칙)

**워터마킹 UI 규칙**
- AI 생성 텍스트: 우측 하단에 `[AI 생성]` 배지 표시

> **감사추적 분리 원칙**: 워터마킹 이벤트는 `ui.watermark.*` 전용 네임스페이스를 사용하여, `ui.hologram.evidence.*` 이벤트와의 감사추적 모호성을 제거한다. 워터마킹 비활성화 시에도 `ui.watermark.audit.logged`는 항상 발행된다.
