## 1. 개요

### 1.1 문서 성격

6-13_Operations는 14-섹션 종합계획서가 아닌 **운영 매뉴얼** 형식으로 작성된다. Part2 §6.12의 12개 서브섹션(6.12.1~6.12.12) 구조를 그대로 유지하며, 각 서브섹션에 대해:

### 5.2 모니터링 메트릭 계산식

> P-1 해결: 모니터링 메트릭의 구체적 계산식, 단위, 알림 임계값을 정의한다.

| 메트릭 | 계산식 | 단위 | WARNING 임계값 | CRITICAL 임계값 | 참조 LOCK |
|--------|--------|------|---------------|----------------|----------|
| 파이프라인 성공률 | (성공 건수 / 전체 건수) x 100 | % | <95% | <90% | — |
| 평균 응답 시간 (p95) | percentile(response_times, 0.95) | ms | >3000ms | >5000ms | — |
| 에러율 | (에러 건수 / 전체 요청) x 100 | % | >1% | >5% | — |
| CPU 사용률 | avg(cpu_usage, 5min) | % | >70% | >85% | LOCK-OP-13 (CX31 2vCPU) |
| 메모리 사용률 | (used_memory / total_memory) x 100 | % | >75% | >90% | LOCK-OP-13 (CX31 8GB) |
| 디스크 사용률 | (used_disk / total_disk) x 100 | % | >80% | >90% | LOCK-OP-13 (CX31 80GB) |
| QoD 이동평균 | SMA(qod_scores, window=24h) | 점수 | — | <0.60 | LOCK-ML-07 |
| 비용 소진율 | (당월 누적 비용 / LOCK-OP-08 상한) x 100 | % | >70% (1단계) | >95% (3단계), 100% (4단계) | LOCK-OP-07, LOCK-OP-08 ($200) |
| 헬스체크 실패율 | (실패 건수 / 전체 체크) x 100 | % | >5% | >20% | LOCK-OP-09 |
| 로그 적재 지연 | max(log_ingestion_delay, 5min) | 초 | >30s | >120s | LOCK-OP-10 |

**알림 발행 규칙**:
- WARNING: 해당 메트릭 대시보드 표시 + Slack 채널 알림 (§6.12.4 참조)
- CRITICAL: 즉시 당번 호출 + 인시던트 생성 (§6.12.3 프로세스 진입)
- 비용 소진율은 LOCK-OP-07 4단계 체계 적용: 70%(경고) → 85%(주의) → 95%(위험) → 100%(초과, 즉시 throttle)

### 5.3 인시던트 에스컬레이션 경로

> P-2 해결: P0~P3 분류에 따른 구체적 에스컬레이션 경로, 대응 시간, 담당, 조치를 정의한다.

| 단계 | 등급 | 대응 시간 | 담당 | 조치 |
|------|------|----------|------|------|
| L0 자동 복구 | P3 (Low) | 즉시 | 시스템 자동 | 자동 재시작, 헬스체크 기반 셀프힐링 (LOCK-OP-09 주기), 로그 자동 기록 |
| L1 당번 대응 | P2 (Medium) | 15분 이내 | 당번 운영자 | 로그 확인 (08_log-retention/ 런북), 수동 재시작, 서비스 격리, 영향 범위 1차 파악 |
| L2 팀 대응 | P1 (High) | 30분 이내 | 운영팀 전체 | 롤백 결정 (06_rollback/ 런북), 장애 범위 파악, 고객 알림, 포스트모템 준비 |
| L3 비상 대응 | P0 (Critical) | 15분 이내 | 전체 팀 + 의사결정권자 | 즉시 롤백 (LOCK-OP-12), Kill Switch 검토, 전사 알림, 비용 긴급 차단 |

**에스컬레이션 조건**:

| 전환 | 조건 | 자동/수동 |
|------|------|----------|
| P3 → P2 | 자동 복구 실패 2회 이상 또는 10분 이내 미복구 | 자동 |
| P2 → P1 | 30분 이내 미해결 또는 영향 서비스 2개 이상 | 수동 (당번 판단) |
| P1 → P0 | 전체 서비스 중단 또는 데이터 손실 위험 또는 비용 상한 초과 (LOCK-OP-08) | 수동 (팀 리드 판단) |

**에스컬레이션 후 절차**:
1. 인시던트 ID 발급 (04_incident-response/ 런북 참조)
2. 영향 범위 + 타임라인 기록
3. 해결 후 포스트모템 작성 (원인/영향/재발 방지 포함)
4. LOCK-OP-11 (P0 15분 복구 목표) 충족 여부 사후 검증
