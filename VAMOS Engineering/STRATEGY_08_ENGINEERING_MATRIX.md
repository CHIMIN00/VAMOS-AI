# 1. 프레임워크 도출 근거

## 1.1 왜 기준점 매트릭스가 필요한가

VAMOS 하네스 엔지니어링 계획서(`STRATEGY_09_HARNESS_ENGINEERING.md`)는 **"AI가 생성하는 코드의 일관성/멱등성을 보장하는 틀"**에 집중한다. 그러나 VAMOS 프로젝트 전체를 보면 하네스 엔지니어링이 다루지 않는 별도 엔지니어링 영역이 상당히 존재한다.

**식별된 하네스 외 엔지니어링 8개 영역:**

| # | 영역 | 하네스 계획서 커버 여부 |
|---|------|---------------------|
| 1 | SOT 정합성 엔지니어링 | §4에서 인식만 (독립 엔지니어링 미수립) |
| 2 | 컨텍스트 윈도우 엔지니어링 | 미언급 |
| 3 | 다중 언어 스택 통합 엔지니어링 | "ESLint, Clippy 미생성"으로만 언급 |
| 4 | 데이터/메모리 엔지니어링 | 미언급 |
| 5 | 보안/거버넌스 엔지니어링 | 미언급 |
| 6 | 비용 최적화 엔지니어링 | 미언급 |
| 7 | 에이전트 오케스트레이션 엔지니어링 | 미언급 |
| 8 | UX/프론트엔드 엔지니어링 | 미언급 |

이 8개 영역 + 하네스 엔지니어링을 **하나도 빠짐없이** 수용하는 기준 프레임워크가 필요하다.

## 1.2 VAMOS의 본질적 특성

VAMOS는 **"AI(Claude Code)가 만드는 AI 제품"**이다. 따라서 두 가지 관점이 동시에 존재한다:

```
관점 1: VAMOS를 "설계"하는 일  — 무엇을 만들 것인가 (What)
관점 2: VAMOS를 "구축"하는 일  — 어떻게 만들 것인가 (How)  ← AI가 코드를 생성
관점 3: VAMOS를 "작동"시키는 일 — 어떻게 돌릴 것인가 (Run) ← 완성된 제품
```

# 2. 2축 정의 — 목적(Purpose) × 시점(Phase)

## 축 1: 목적 (Purpose) — "무엇을 위한 엔지니어링인가"

| 코드 | 목적 | 정의 | 핵심 질문 |
|------|------|------|----------|
| **D** | **설계 (Define)** | VAMOS가 "무엇인지" 정의하는 자산의 정확성 확보 | "설계 문서가 정확하고 일관적인가?" |
| **B** | **구축 (Build)** | AI가 올바른 코드를 일관되게 생산하는 프로세스 확보 | "AI가 매번 같은 품질의 코드를 생산하는가?" |
| **R** | **작동 (Run)** | 완성된 VAMOS 제품이 올바르게 동작하는 것을 확보 | "제품이 설계대로 안전하게 작동하는가?" |
| **X** | **횡단 (Cross)** | D/B/R 3행 이상을 관통하는 관심사 통합 관리 | "전체를 관통하는 품질이 유지되는가?" |

## 축 2: 시점 (Phase) — "언제 수행하는 엔지니어링인가"

| 코드 | 시점 | 정의 | 진입 조건 → 완료 조건 |
|------|------|------|---------------------|
| **1** | **구현 전 (Pre)** | 코드 생산 착수 전 기반 확보 | 문서 존재 → 기반이 검증됨 |
| **2** | **구현 중 (Mid)** | 코드 생산 중 품질 강제 | 기반 확보 → 코드 산출물 생산됨 |
| **3** | **구현 후 (Post)** | 산출물 검증 및 운영 | 코드 존재 → 검증됨/운영 중 |
| **F** | **피드백 (Feedback)** | 후행 셀에서 선행 셀 문제 발견 시 역류 | 문제 발견 → 수정 완료 + 복귀 |

## 분류 원칙 — 경계 판단 기준 3가지 질문

어떤 엔지니어링 활동이 어느 셀에 속하는지 판단하는 질문:

| # | 질문 | 판별 |
|---|------|------|
| 1 | **대상이 문서/스키마인가, 코드 생산 프로세스인가, 제품 동작인가?** | D vs B vs R |
| 2 | **코드 생산 전에 완료되어야 하는가, 생산 중에 작동하는가, 생산 후에 수행하는가?** | 1 vs 2 vs 3 |
| 3 | **이 활동이 없으면 다음 단계가 불가능한가, 품질이 떨어지는가, 운영이 안 되는가?** | 우선순위 판단 |

# 8. X행 경계 판별 기준

## 3대 판별 원칙

| 원칙 | 설명 | X행 여부 |
|------|------|---------|
| **3행 이상 관통** | D/B/R 중 3개 이상 행에 걸치면 X | ✅ X행 |
| **독립 수명주기** | D/B/R과 별도 주기로 관리되면 X | ✅ X행 |
| **제거 시 전체 붕괴** | 특정 행이 아니라 매트릭스 전체가 영향받으면 X | ✅ X행 |

**반대**: 특정 셀에 명확히 귀속되면 X에 넣지 않음

## 적용 예시

| 활동 | 판별 | 배치 |
|------|------|------|
| 보안/거버넌스 | D1+B2+R2+R3 = 4행 관통 | ✅ X행 |
| 테스트 전략 | 없으면 B3+R3 모두 불가 | ✅ X행 |
| Git 브랜치 전략 | D/B/R과 별도로 관리 | ✅ X행 |
| 문서화 전략 | 없으면 D2+D3 붕괴 | ✅ X행 |
| Gate 구현 | R2a에서만 구현 | ❌ R2a |
| 린터 설정 | B1에서만 세팅 | ❌ B1 |
| RAG 파이프라인 | R2b에서만 구현 | ❌ R2b |
| Hologram 렌더링 | R2c에서만 구현 | ❌ R2c |
| 멱등성 테스트 | B2a에서만 실행 | ❌ B2a |
| CPS 로깅 | B2a에서만 사용 | ❌ B2a |

# 9. 하네스 엔지니어링 5계층과 매트릭스 통합 맵

기존 하네스 계획서의 L1~L5 계층이 매트릭스에 위치하는 곳:

```
하네스 계층             매트릭스 셀        근거
─────────────────────  ────────────────  ─────────────────────────
L1 (SOT 정합)          D1               설계 정합성 확보
L2 (SOT↔SOT2 일관)     D1 + D2          설계 정합성 + 변경 추적
L3 (CLAUDE.md 정확)     B1 + B2b         환경 세팅 + 컨텍스트 관리
L4 (코드 린트/테스트)   B1 + B2a         환경 세팅 + 하네스 실행
L5 (Eval 품질)          B3               산출물 품질 평가
```

```
                    하네스 5계층
                  ┌───────────────┐
                  │  L5 (Eval)    │──→ B3
                  ├───────────────┤
                  │  L4 (Code)    │──→ B1 + B2a
                  ├───────────────┤
                  │  L3 (Context) │──→ B1 + B2b
                  ├───────────────┤
                  │  L2 (SOT2)    │──→ D1 + D2
                  ├───────────────┤
                  │  L1 (SOT)     │──→ D1
                  └───────────────┘

결론: 하네스 엔지니어링 = D행 + B행의 부분집합
      R행과 X행은 하네스 계획서 범위 밖 → 별도 엔지니어링 필요
```
