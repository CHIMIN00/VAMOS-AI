# VAMOS CPS 템플릿 (Context / Problem / Solution)

> **생성**: Phase 2-6 (2026-06-11) · **근거**: STRATEGY_09 하네스 6단계 프로세스 2단계 "플랜 수립 — CPS 프레임워크" (GAP-2 해소)
> **용도**: AI에게 작업을 지시하기 전, 맥락→문제→해결책 순서로 구조화하면 AI가 제멋대로 하지 않는다. Phase 3(R1 결정), Phase 4(V0 모듈 구현) 세션 프롬프트의 표준 골격.

---

## 템플릿

```markdown
# [작업 ID] — [작업명 1줄]

## C. Context (맥락)
- **무엇의 일부인가**: (Phase/트랙/모듈 — 예: Phase 4 R2a, I-1 Intent Detector)
- **정본 근거**: (파일 경로 + 섹션 — 예: D2.0-02 §7.1, D2.1-D2 §4.1)
- **선행 산출물**: (이 작업이 의존하는 기존 산출물 경로)
- **불변 제약**: (관련 LOCK/FREEZE — 예: Decision 18필드 FREEZE, config LOCK 20키[D13])
- **컨텍스트 로딩**: (CONTEXT_LOADING_MAP.md 해당 Phase 행 참조)

## P. Problem (문제)
- **해결할 것**: (1~3문장. 측정 가능하게)
- **완료 기준 (DoD)**: (검증 가능한 조건 목록 — ruff/vamos_lint/pytest PASS 포함)
- **하지 말 것**: (Non-goal, 범위 밖, 임의 창작 금지 항목)
- **실패 시 (A1)**: (막히면 어떻게 — 중단·보고·기록 기준)

## S. Solution (해결책)
- **접근**: (단계 1→2→3, 각 단계의 산출물)
- **검증**: (단계별 검증 방법 — 결정론 우선, 멱등성: 같은 입력=같은 출력)
- **산출물 경로**: (디스크 실존 검증 대상 — R16: 보고서 ✅ ≠ 산출물 존재)
- **기록**: (PROGRESS.md 갱신 + 커밋 메시지 + ADR 필요 여부[A6])
```

## 사용 규칙

1. **C 없이 P를 쓰지 않는다** — 맥락(정본 근거) 없는 문제 정의는 AI 임의 해석을 유발.
2. **P의 완료 기준은 자동 검증 가능해야 한다** — 하네스 게이트(ruff·vamos_lint·pytest)에 연결.
3. **S는 멱등이어야 한다** — 어떤 모델/세션이 실행해도 동일 결과 (STRATEGY_09 핵심 원리).
4. 기존 적용례: ROADMAP_SESSION_EXECUTION_PROMPTS.md의 STEP 1~5 구조(프롬프트 검증→실행→산출물 검증→문서 갱신→갱신 검증)는 본 CPS의 절차적 전개형 — 신규 세션 프롬프트는 CPS 골격 + STEP 5단계로 작성.
