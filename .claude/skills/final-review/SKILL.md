# VAMOS 최종 완료 판정 스킬

> `/final-review [대상]` — "이것이 최종본인가?"를 증명하는 종합 검증

## 이 스킬이 해결하는 문제

```
문제 1: "최종본이야?" → "네" → 수정사항 발견 → 무한루프
원인 1: 검증이 부분적으로만 수행됨

문제 2: 기준 자체가 틀려서, 기준을 통과해도 실제 오류가 남음
원인 2: "기준이 맞는지" 검증하는 단계가 없었음

해결: Step 0(기준점 메타 검증) → Step 1~4(실제 검증) → 판정
```

## 핵심 원칙

**모든 판정은 반드시 Step 0(기준점 검증)부터 시작한다.**

```
기존 (문제): 기준 생성 → 검증 → 판정 (기준이 틀리면 판정도 틀림)
개선 (현재): 기준 생성 → 기준 자체 검증 → 검증 → 판정 (기준부터 맞춤)
```

## Step -1: 대용량 컨텍스트 사전 처리 (모든 모드 공통)

**모든 모드(A/B/C/D/E/F)에서 Step 0 전에 반드시 실행됩니다.**
검증 대상 파일이 컨텍스트 윈도우를 초과하면, 검증 자체가 불완전해집니다.
Step -1은 이 문제를 사전에 해결합니다.

```
Step -1-1: 파일 크기 + 유형 분류
  → 검증 대상 파일의 줄 수 확인 (wc -l 또는 Read offset 탐색)
  → 분류:
    소형 (≤ 2000줄): 직접 읽기 → Step 0 진행
    대형 (> 2000줄): Step -1-2로 진행
  → 유형 분류:
    SOT 파일: /sot-cache, /sot-rag, /sot-search 활용 가능
    Non-SOT 파일: 체크포인트 패턴 적용 (v13_plan, PART2 등)

Step -1-2: 인프라 가용성 점검
  → SOT 파일인 경우:
    [1] /sot-cache 가용? → python -c "import json; ..." 캐시 파일 존재 확인
    [2] /sot-search 가용? → python -c "import whoosh" 또는 벡터 인덱스 존재 확인
    [3] /sot-rag 가용? → 임베딩 인덱스 존재 확인
    [4] /docling 가용? → 구조 파싱 캐시 존재 확인
  → 가용성 결과에 따른 전략 결정:
    FULL: 4개 도구 모두 가용 → 최적 경로
    PARTIAL: 일부 가용 → 가용 도구만 사용 + Grep fallback 병행
    NONE: 전부 불가 → offset 분할 + 체크포인트 패턴 (순수 fallback)
  → stamp에 infrastructure_availability 기록

Step -1-3: 캐시 신선도 검증 (SOT 파일 + /sot-cache 가용 시)
  → 검증 대상 파일의 lastModified 타임스탬프 확인
  → /sot-cache 캐시 파일의 생성 타임스탬프 확인
  → 비교: cache_time + 1초 여유 마진 < file_modified_time → STALE
    FRESH: 캐시 사용 가능
    STALE: /sot-cache --refresh 동기 실행 후 완료 대기 → 재확인
  → 갱신 중 검증 진행 방지: 갱신 완료까지 Step 0 진입 차단

Step -1-4: 대용량 파일 처리 전략 실행
  → [전략 A] SOT + FULL 인프라:
    /sot-cache → 캐시된 구조 데이터 로드
    /sot-rag → 검증 항목별 다중 쿼리 (쿼리당 2~3 변형)
      → 쿼리 변형 예: "session 15 output" + "세션15 산출물" + "S15 deliverable"
      → top-3 × 3쿼리 = 최대 9 passage → 중복 제거
      → RAG 결과 간 충돌 시: 원본 offset 읽기로 fallback
    /sot-search → 키워드 기반 보조 검색
    /docling → 문서 구조 사전 파싱

  → [전략 B] SOT + PARTIAL 인프라:
    가용 도구 우선 사용 + 불가 도구는 Grep fallback
    Grep fallback 시 키워드 동의어 확장:
      "산출물" → "산출물|output|결과물|deliverable"
      "사전조건" → "사전조건|prerequisite|선행조건|입력"

  → [전략 C] Non-SOT 또는 NONE 인프라 (체크포인트 패턴):
    1. 파일을 논리적 섹션으로 분할 (마크다운 ## 헤딩 기준)
       → 분할 결과를 사용자에게 표시 (섹션 경계 오류 방지)
    2. 각 섹션을 offset 분할 읽기 (CLAUDE.md Rule 1 준수: 2000줄 초과 시 분할)
    3. 각 chunk 읽기 직후: 모드별 추출 스키마에 따라 핵심값 추출
    4. 추출 결과를 체크포인트 파일에 저장:
       → 경로: .tmp/fr_checkpoint_{filename}_{section}.json
       → 구조:
         {
           "source_file": "파일명",
           "section": "섹션명",
           "source_lines": [시작행, 끝행],
           "extracted_at": "타임스탬프",
           "schema_version": "모드별 스키마 버전",
           "key_facts": [ {"field": "값", "confidence": 0.95} ],
           "relations": [ {"from": "A", "to": "B", "type": "depends_on"} ],
           "ambiguous_items": [ {"item": "...", "reason": "동의어 가능성"} ]
         }
    5. 다음 chunk 전에: 이전 체크포인트와 모순 검사
       → 모순 발견 시: 해당 행 원본 재읽기로 spot-check
    6. 모든 chunk 완료 후: 체크포인트 요약본 생성
       → 2단 구조: chunk별 상세 + 전체 요약
       → 요약본이 검증의 "작업 메모리" 역할

  → [모든 전략 공통] 교차 참조 처리:
    파일 A에서 비교 대상 값 추출 → 중간 JSON 저장
    파일 B에서 비교 대상 값 추출 → 중간 JSON 저장
    추출 시 정규화: "세션번호 + 키워드" 형태로 통일
      예: {"id": "S15", "topic": "UI컴포넌트", "output": "경로"}
    두 JSON만 로드하여 비교 (원본 대 원본 비교 회피)
    의미적 의심 항목 → 원본 행으로 spot-check fallback

Step -1-5: 실행 전 .tmp 정리
  → .tmp/fr_checkpoint_* 파일이 이전 실행에서 잔존하는지 확인
  → 잔존 시: 타임스탬프 확인 → 현재 실행과 무관하면 삭제
  → 정리 후 Step 0 진입

Step -1 판정:
  → CONTEXT-READY: 소형 파일 또는 대용량 처리 전략 실행 완료
  → CONTEXT-DEGRADED: fallback 전략 사용 (Grep 기반, 인프라 미가용)
    → stamp에 verification_depth: "DEGRADED" 기록
    → 사용자에게 "인프라 미가용으로 검증 깊이 제한" 알림
  → CONTEXT-FAILED: 파일 읽기 자체 실패 → 검증 중단

Step -1 결과는 stamp의 context_handling 섹션에 기록됩니다.
```

### 모드별 추출 스키마 (Step -1-4 전략 C에서 사용)

```
Mode A (EA/CM):
  추출 필드: {ea_id, source_text, sot_file, sot_line, category, confidence}

Mode B (도구/스킬):
  추출 필드: {tool_name, group, description, cost_tier, dependencies}

Mode C (Phase):
  추출 필드: {phase_id, ea_list, gate_criteria, pc_numbers}

Mode D (일반):
  추출 필드: {task_item, status, reference_file, verification_point}

Mode E (계획서):
  추출 필드: {session_id, precondition, output_path, gate, mode,
             agent_list, skill_list, methodology, pc_number}

Mode F (부재/초과):
  추출 필드: {expected_item, source_ref, presence_status, severity}
```

### 교차 파일 비교 정규화 규칙

```
정규화 레이어 (추출 후 자동 적용):
  1. 세션 번호 통일: "세션 15" = "session 15" = "S15" = "세션15" → "S15"
  2. Phase 번호 통일: "Phase 0" = "phase0" = "P0" → "P0"
  3. 파일 경로 통일: 역슬래시 → 정슬래시, 대소문자 무시
  4. 수치 통일: "68개" = "68" (단위 분리)
  5. 동의어 매핑: {"산출물": "output", "사전조건": "prerequisite", ...}

비교 시 불일치 판정 기준:
  EXACT_MATCH: 정규화 후 완전 일치
  SEMANTIC_MATCH: 정규화 후 동의어 매핑으로 일치 → PASS (단, 원본 표현 통일 권장)
  MISMATCH: 정규화 + 동의어로도 불일치 → FAIL (원본 spot-check 필수)
  UNCERTAIN: confidence < 0.7 → 원본 재읽기 후 재판정
```

## Step 0: 기준점 메타 검증 (모든 모드 공통)

**모든 모드(A/B/C/D/E/F)에서 Step 1 전에 반드시 실행됩니다.**

```
Step 0-1: 파이프라인 역분해 (이론적 — 빠진 기준 찾기)
  → 대상 작업의 전체 프로세스를 단계별로 분해
  → 각 단계에서 발생 가능한 모든 실패 유형 나열
  → 나열된 실패 유형이 이번 체크리스트에 전부 매핑되는지 확인
  → 매핑 안 되는 실패 유형 발견 → 기준 항목 자동 추가

  예시 (모드 A: EA 검증):
    파이프라인: SOT읽기 → 추출 → JSON생성 → 검증 → 판정
    실패 유형 30개 나열 → FR-A01~A14와 매핑
    → "SOT 간 모순"이 매핑 안 됨 → FR-A15 자동 추가

Step 0-2: 반대 관점 질문 (이론적 — 느슨한 기준 찾기)
  → [항목 단위] 각 기준 항목에 대해 역질문:
    "이 기준이 PASS여도 실제로 문제가 있을 수 있는 경우는?"
    "이 기준에 잡히지 않는 오류 유형은?"
  → 해당 경우 발견 시 기준 강화 또는 항목 추가

  → [전체 체계 단위] 모든 기준을 한꺼번에 통과한 시나리오에 대해 역질문:
    "전체 기준이 PASS인데 실제로는 실행 불가능/부정확한 시나리오 3개를 구성하라"
    → 각 시나리오가 기존 기준 중 하나라도 탐지하는지 확인
    → 전부 탐지 불가 → 해당 실패 유형에 대한 기준 추가
    → 1개 이상 탐지 → OK (기존 기준으로 커버)
    ※ 항목 단위 역질문은 개별 기준의 느슨함을 잡지만,
       기준 간 빈틈(개별로는 PASS이나 조합하면 FAIL)은 놓침.
       전체 체계 단위 역질문이 이 조합적 빈틈을 탐지함.

  예시 (항목 단위):
    FR-A07 "source_text 물리적 존재" PASS여도 문제인 경우?
    → source_text가 존재하지만 문맥이 다른 경우 (동음이의어)
    → FR-A03 적대적 감사에서 커버됨 → OK (추가 불필요)

  예시 (전체 체계 단위):
    시나리오: "문서 내 모든 경로가 존재하고(FR-D19 PASS), 숫자가 일치하고(FR-D15 PASS),
              선언-실행이 일치하지만(FR-D17 PASS), 환경 설치 상태 기술이 실측과 다름"
    → FR-D19 기존 범위(파일 경로만)로는 탐지 불가 → FR-D19 범위 확장 필요
    → 실제 v13 사례에서 이 시나리오가 발생했음 (Step 0-4d로 보완됨)

Step 0-3: 완전성 매트릭스 (이론적 — 구조적 빈 곳 찾기)
  → 실패 유형(행) × 검증 계층(열) 매트릭스 생성
  → 빈 셀 확인:
    - 특정 실패 유형이 어떤 계층에서도 커버 안 됨 → 기준 추가
    - 모든 셀에 최소 1개 커버 → 구조적 완전

Step 0-4: 경험적 검증 (실제 데이터 — 무한 재귀를 끊는 닻)
  → Step 0-1~0-3은 이론적 검증 (AI 지식 한계 내에서만 작동)
  → Step 0-4는 실제 데이터로 "이론에서 놓친 오류"를 잡음

  실행 방법 (대상에 따라 자동 선택):
    [EA/CM 대상 — 모드 A/C]:
      → golden-set이 있으면: 정답 대비 실제 정확도 측정 (A-8)
      → golden-set이 없으면: 샘플 3건 실제 추출 → 원본 대조 확인
      → 이론 체크리스트에 없던 오류 발견 시 → 기준 추가

    [도구 체계 — 모드 B]:
      → 최근 Phase 결과에서 실제 발생한 오류 목록 수집
      → 각 오류가 현재 도구로 사전 탐지 가능했는지 확인
      → 탐지 불가했던 오류 → 빈 도구 영역으로 판정

    [일반 작업 — 모드 D]:
      → 변경된 파일을 실제로 읽어서 결과 확인
      → 기준에 없던 문제 발견 시 → 기준 추가

Step 0-5: 자기참조 교차검증 (target = /final-review 자체일 때만 활성화)
  → 자기 기준으로 자기를 판단하는 순환 참조를 보완하는 추가 단계
  → 일반 검증(target ≠ 자기 자신)에서는 실행하지 않음 (기존 로직 변경 없음)

  실행 조건: target이 final-review SKILL.md 자체이거나,
            "/final-review 자체가 최종이야?" 유형의 질문일 때

  0-5a: 실행 이력 기반 유효성 검증 (외부 증거)
    → 과거 /final-review 실행 이력에서:
      - 실제 오류를 발견한 건수 (true positive)
      - 오류가 아닌데 지적한 건수 (false positive)
      - 놓친 오류 건수 (false negative — 이후 발견된 것)
    → 최소 3건의 실행 이력이 없으면 → INSUFFICIENT_DATA 판정
    → 이력 있으면: 발견율 > 0이면 도구가 작동하는 외부 증거

  0-5b: 다른 스킬과의 교차 일치 검증
    → /validate, /audit, /sot-check 등 다른 검증 스킬이
      같은 대상에 대해 내린 판정과 /final-review 판정 비교
    → 판정 불일치 시: 불일치 원인 분석 → 기준 보완
    → 판정 일치 시: 독립 검증으로서의 교차 확인 성립

  0-5c: 자기참조 모순 탐지
    → Step 0-1~0-3의 결론과 Step 0-4/0-5a의 결론이 모순되면:
      - 이론(0-1~0-3): "기준 충분" + 경험(0-4/0-5a): "실제 오류 있음"
        → 경험적 결과 우선 (이론 기준 보완)
      - 이론(0-1~0-3): "기준 부족" + 경험(0-4/0-5a): "실제 문제 없음"
        → 이론 기준 추가 후 경험적으로 재확인
    → 모순이 없으면: 자기검증 신뢰도 확보

  자기검증 판정 기준:
    SELF-VERIFIED: 0-5a 실행 이력 ≥ 3건 + 발견율 > 0 + 0-5b 교차 일치 + 0-5c 모순 없음
    SELF-PARTIAL:  실행 이력 < 3건 또는 교차 일치 미확인 (기능은 작동하나 증거 부족)
    SELF-FAILED:   0-5c에서 모순 발견 + 해소 불가

Step 0 깊이 예산 (수확 체감 중단점):
  → Step 0-1~0-4에서 기준을 추가할 때, 다음을 확인:
    1. 추가 기준이 Step 0-4 경험적 검증에서 실제 오류를 잡는가?
       → 잡으면: 유효한 추가 (진행)
       → 못 잡으면: 이론적 보완일 뿐 (추가하되 MINOR로 분류)
    2. 직전 3개 추가 기준이 모두 경험적 검증에서 오류를 못 잡았으면:
       → 수확 체감 도달 → 기준 추가 중단, 현재까지 확정
    3. 총 추가 기준이 기존 기준 수의 50%를 초과하면:
       → WARNING: "기준 과잉 의심 — 대상 자체의 품질 문제 확인 필요"
       → 기준 추가보다 대상 수정이 우선일 수 있음
  → stamp에 depth_budget 기록:
    {
      "criteria_added": 숫자,
      "criteria_effective": 숫자,   // 경험적으로 오류를 잡은 수
      "diminishing_returns_reached": true|false
    }

Step 0 판정:
  → 기준 추가/수정 없음 → "기준 확정" → Step 1 진행
  → 기준 추가/수정 있음 → 기준 보완 후 → Step 1 진행
  → Step 0 결과도 stamp에 기록 (기준 변경 시 stamp 무효화)
  → 자기검증 시: Step 0-5 결과도 stamp에 추가 기록

Step 0 실행 증명 (절차적 강제):
  → Step 0 각 하위 단계(0-1~0-4)는 반드시 실행 결과를 stamp에 기록해야 함
  → stamp의 step0_meta_validation 섹션에 각 단계별 결과가 비어있으면:
    - 해당 stamp는 "STEP0_INCOMPLETE"로 무효 처리
    - Step 1 판정 결과와 무관하게 최종 판정 불가
  → Step 0 생략 탐지: stamp에 step0_meta_validation이 없거나
    pipeline_decomposition.failure_types_found = 0이면 생략으로 간주
  → Step 0에서 기준이 추가/강화되면 기존 PASS 판정은 자동 무효화
    (새 기준으로 재검증 필요)
  → Step 0-4d 실행 증명 (문서/계획서 대상 시 필수):
    - stamp의 step0_meta_validation에 factual_claim_verification 필드 필수
    - 필드 구조:
      {
        "claims_extracted": 숫자,      // 추출된 외부 사실 주장 수
        "claims_verified": 숫자,       // 검증 실행된 주장 수
        "claims_skipped": 숫자,        // SKIP 처리된 주장 수 (사유 포함)
        "mismatches": [                // 불일치 목록
          {"claim": "...", "expected": "...", "actual": "...", "severity": "CRITICAL|MAJOR|MINOR"}
        ]
      }
    - claims_extracted = 0이고 대상이 환경/설치/상태를 기술하면 → 추출 누락으로 간주
    - factual_claim_verification 필드 자체가 없으면 → STEP0_INCOMPLETE
```

## 전체 실행 흐름 (모든 모드 공통)

```
Step -1: 대용량 컨텍스트 사전 처리
  -1-1: 파일 크기 + 유형 분류 (소형/대형, SOT/Non-SOT)
  -1-2: 인프라 가용성 점검 (sot-cache/rag/search/docling)
  -1-3: 캐시 신선도 검증 (STALE → 갱신 후 진행)
  -1-4: 대용량 처리 전략 실행 (전략 A/B/C 자동 선택)
  -1-5: 이전 실행 .tmp 정리
  → CONTEXT-READY / CONTEXT-DEGRADED / CONTEXT-FAILED
    ↓
Step 0: 기준점 메타 검증
  0-1: 파이프라인 역분해 → 빠진 기준 찾기
  0-2: 반대 관점 질문 → 느슨한 기준 찾기 (항목 단위 + 전체 체계 단위)
  0-3: 완전성 매트릭스 → 구조적 빈 곳 찾기
  0-4: 경험적 검증 → 이론에서 놓친 오류 잡기 (무한 재귀 차단)
    0-4a~c: 숫자/선언/경로 내부 정합성
    0-4d: 외부 사실 주장 → 현실 대조 (0-4a~c와 독립/병렬 실행)
  0-5: 자기참조 교차검증 → 자기 자신을 검증할 때만 활성화 (순환 참조 보완)
  깊이 예산: 수확 체감 도달 시 기준 추가 중단
  → 기준 확정
    ↓
Step 1: 확정된 기준으로 검증 실행
  → 체크포인트 기반 검증 시: 각 항목에 confidence 점수 부여
  → confidence < 0.7인 항목은 원본 spot-check 자동 트리거
    ↓
Step 2: 결과 집계
  → 섹션별 독립 검증 시: 커버리지 매트릭스 생성
  → 빈 셀 = 검증 누락 → 해당 섹션 재검증
  → 교차 참조 항목: 별도 교차 참조 전용 패스로 검증
    ↓
Step 3: 판정 (FINAL / NEAR-FINAL / NOT_FINAL)
  → CONTEXT-DEGRADED인 경우: 판정에 "검증 깊이 제한" 부기
    ↓
Step 4: stamp 생성 (기준 + 결과 + 컨텍스트 처리 메타 모두 기록)
  → .tmp/fr_checkpoint_* 정리 (검증 완료 후)
  → 사후 발견 시: skill_update_recommended 트리거 → 다음 실행에서 SKILL.md 갱신 알림
```

```
/final-review plan
/final-review cross-verify
```

계획서/실행가이드처럼 "정의+실행+요약+외부참조" 4레이어 구조를 가진 문서의

### 모드 F: 부재 탐지 + 초과 탐지 (Absence & Excess Detection)

```
/final-review absence
/final-review completeness
/final-review missing
```

"있는 것이 맞는지"가 아닌 "있어야 하는데 없는 것"과 "없어야 하는데 있는 것"을 찾습니다.
외부 기준 대조 + 이전 버전 비교 + 도메인 모델 생성 + 이해관계자 관점의 4단 접근.
모든 모드의 공통 맹점(문서 내부에서 기준을 도출하므로 미언급 항목은 탐지 불가)을 해결합니다.

```
              → FR-D05: 파싱 실패 = 파일 손상, 이후 검증 전체 무의미
    MAJOR:    FR-D06~D07, FR-D17 (불일치 시 수정 필요)
              FR-D19 MAJOR급 (상태 불일치 — OK↔미설치, FAIL↔해결됨)
    MINOR:    FR-D08~D14, FR-D18, FR-D20 (WARNING 허용)
              FR-D19 MINOR급 (날짜 경과, 버전 마이너 차이, 카운트 ±1)

  판정 기준:
  | 판정 | 조건 |
  |------|------|
  | FINAL | CRITICAL 0건 FAIL + MAJOR 0건 FAIL + MINOR WARNING 허용 |
  | NEAR-FINAL | CRITICAL 0건 FAIL + MAJOR 1~2건 WARNING + Step 0 기준 확정 |
  | NOT_FINAL | CRITICAL 1건 이상 FAIL 또는 MAJOR 3건 이상 FAIL |

Step 4: stamp 생성
```

## 모드 E: 계획서 다층 교차검증 (Plan Cross-Verification)

### 이 모드가 해결하는 문제

```
문제: 계획서 내부의 "정의 섹션"과 "실행 가이드"가 서로 다른 내용을 기술
원인: 정의 작성 → 실행 작성 과정에서 변환 오류, 또는 한쪽만 수정 후 다른 쪽 미반영
기존 한계: 모드 D의 FR-D15~D20은 숫자/경로 수준만 교차검산하며,
          방법론/에이전트/스킬/체크포인트의 의미적 정합성은 미검증
```

### 발동 조건

```
/final-review plan
/final-review cross-verify

자동 발동: 대상이 계획서/실행가이드이고 다음 구조를 가질 때:
  - "정의/방법론" 섹션 + "세션별/Step별 실행 가이드" 섹션이 동시 존재
  - "요약표/인덱스" 섹션이 존재
  - "외부 참조 문서" 경로가 포함
```

### Step 0: 기준점 메타 검증 (계획서 특화)

```
0-1: 문서 레이어 식별
  → 대상 문서에서 4개 레이어 자동 탐지:
    L1(정의): 방법론/구조/체크포인트를 선언하는 섹션
    L2(실행): Step별로 실제 수행 내용을 기술하는 섹션
    L3(요약): 표/인덱스/체크리스트로 압축한 섹션
    L4(외부): 참조 문서/이전 산출물/외부 파일
  → 각 레이어의 시작-종료 행 범위 기록
  → 레이어 미탐지 시: 해당 Pass 건너뜀 (최소 L1+L2 필수)

0-2: 교차 대조 축 정의
  → 문서 내에서 "같은 대상을 두 곳 이상에서 기술하는" 항목 추출:
    - 에이전트 수/이름/범위
    - 스킬/도구 배정
    - 체크포인트(PC) 번호와 의미
    - 산출물 파일 경로
    - Phase/세션 구조
    - 수치(세션수, 항목수, 파일수 등)
  → 추출된 축이 체크리스트 FR-E에 전부 매핑되는지 확인
  → 매핑 안 되는 축 → FR-E 항목 자동 추가

0-3: 역질문
  [축 단위] 각 교차 대조 축에 대해:
  → "L1과 L2가 일치해도 문제인 경우?"
    예: 둘 다 틀린 값으로 일치 (원본과 불일치)
    → FR-E12 (L4 외부 참조 대조)로 커버
  → "L2와 L3가 일치해도 문제인 경우?"
    예: 요약표가 상세보다 먼저 작성되어 둘 다 구버전
    → FR-E08 (최신성 확인)로 커버
  [전체 교차 구조 단위] "FR-E01~E19 전부 PASS인데 계획이 실행 불가능한 시나리오 3개 구성"
  → 예: 모든 레이어 정합하지만 환경 설치 상태가 현실과 다름 → FR-E19로 커버
  → 예: 모든 교차 축 일치하지만 의존성 순서가 현실 제약과 충돌 → FR-E14로 커버
  → 전부 탐지 불가 시 → FR-E 항목 추가

0-4: 경험적 검증
  → 실제 문서에서 임의 3개 축 선정 → L1↔L2 값 대조
  → 불일치 발견 시 기준 강화
  → 이론에 없던 불일치 유형 발견 시 FR-E 추가
  → 깊이 예산: 직전 3개 추가 FR-E가 경험적 오류 0건이면 중단
```

### Step 1: 확정된 체크리스트로 검증 (4-Pass)

```
═══════════════════════════════════════════════
Pass 1: L2 내부 정합성 (기존 final review 수준)
═══════════════════════════════════════════════

[FR-E01] 항목 전수 존재
         → L2(실행가이드) 내 선언된 모든 항목(세션/Step/에이전트)이 빠짐없이 기술됨
         → 방법: 번호 연속성 확인 (예: 세션 1~53 전수)

[FR-E02] 번호/라벨 정합성
         → 게이트의 PC 번호가 정의된 PC 범위 내에 있음
         → PC 번호 중복 할당 없음
         → 모든 정의된 PC가 최소 1개 세션 게이트에 할당됨

[FR-E03] 산출물 경로 일관성
         → L2 내 모든 산출물 경로가 동일 기준(상대/절대) 사용
         → 산출물 루트 경로 정의가 명시됨

═══════════════════════════════════════════════
Pass 2: L1 ↔ L2 교차 정합성 (핵심 — 이번에 누락된 부분)
═══════════════════════════════════════════════

[FR-E04] 에이전트 구성 일치
         → L1(정의)의 에이전트 수/이름/범위가 L2(실행)와 동일
         → 방법: L1에서 에이전트 목록 추출 → L2의 해당 세션에서 동일 에이전트 확인
         → 불일치 유형: 수 불일치, 범위 중복, 범위 누락

[FR-E05] 스킬/도구 배정 일치
         → L1(정의)에서 특정 작업에 배정한 스킬이 L2(실행)의 해당 Step과 동일
         → 방법: L1의 스킬 배정 테이블 → L2의 Step별 스킬 1:1 대조
         → 정의에만 있고 실행에 없는 스킬 → FAIL
         → 실행에만 있고 정의에 없는 스킬 → WARNING

[FR-E06] 체크포인트 의미적 일치
         → L1(정의)의 PC-N 설명과 L2(실행) 게이트의 PC-N이 같은 의미
         → 방법: PC 번호로 매칭 → 설명 텍스트 의미 비교
         → 번호는 같지만 의미가 다른 경우(예: PC3-5 vs PC3-7) → CRITICAL

[FR-E07] 방법론 구조 일치
         → L1(정의)에서 선언한 구조(STEP A/B, Dimension, Wave 등)가
           L2(실행)에서 실제로 반영됨
         → 방법: L1의 구조 키워드 추출 → L2에서 해당 키워드 사용 여부 확인
         → 정의에 "STEP A/B 순방향/역방향" 선언 → 실행에 미반영 → FAIL

[FR-E08] 핸드오프 입출력 일치
         → L1(정의)의 "Phase N → Phase N+1 핸드오프" 출력이
           L2(실행)의 Phase N+1 사전조건/입력과 동일
         → 방법: L1 핸드오프 테이블의 출력 → L2 다음 Phase 첫 세션의 입력 매칭

═══════════════════════════════════════════════
Pass 3: L2 ↔ L3 요약 정합성
═══════════════════════════════════════════════

[FR-E09] 요약표 ↔ 상세 내용 일치
         → L3(요약표)의 각 행이 L2(실행가이드)의 해당 세션과 동일한 내용
         → 방법: 요약표의 (세션, 작업내용, 모드, 게이트) 4개 열을
           L2 해당 세션의 제목/실행모드/게이트와 1:1 대조
         → 불일치 시 L2가 정본 (요약표를 L2에 맞춰 수정)

[FR-E10] 수치 합산 교차검산
         → L3(요약표)에서 집계 가능한 수치(세션 수, Phase 수)가
           L2(실행가이드) 실제 카운트와 일치
         → L1(정의) "소계" 섹션의 수치와도 교차 확인

═══════════════════════════════════════════════
Pass 4: L4 외부 참조 정합성
═══════════════════════════════════════════════

[FR-E11] 외부 참조 문서 방법론 일치
         → L2(실행가이드)가 참조하는 외부 문서(프롬프트, 가이드 등)의
           방법론/구조가 L2 내용과 일치
         → 방법: 참조 문서 핵심 구조 추출 → L2 해당 세션과 대조
         → 예: v6 프롬프트의 "5개 Agent" 구조 ↔ 세션 9~10 "Agent 1~5" 일치 여부

[FR-E12] 이전 산출물 디렉토리 매칭
         → L2에서 참조하는 이전 산출물 경로(v8_results/ 등)가
           실제 파일시스템에 존재하고, 해당 Phase의 산출물과 대응
         → 방법: 경로 추출 → 존재 확인 + 내용물 구조 매칭

[FR-E13] 파일 수/항목 수 실측 검증
         → L1/L2에서 언급하는 수치("68개 SOT", "33개 세션 파일" 등)가
           실제 파일시스템 카운트와 일치
         → 방법: 수치 추출 → 해당 경로 ls | wc -l 비교

═══════════════════════════════════════════════
Pass 5: 실행 가능성 + 표현 품질 검증 (Feasibility & Clarity)
═══════════════════════════════════════════════

[FR-E14] 의존성 DAG 무순환 검증
         → L2(실행가이드)의 세션/Step 간 의존성(사전조건 → 산출물)을
           방향 그래프(DAG)로 구성했을 때 순환(cycle)이 없어야 함
         → 방법: 각 세션의 "사전조건"에서 참조하는 이전 세션 산출물 추출
           → 의존 관계 그래프 구성 → 위상 정렬 가능 여부 확인
         → 순환 발견 시: CRITICAL (실행 불가)

[FR-E15] 병렬 실행 충돌 검증
         → "병렬"로 선언된 에이전트/세션이 실제로 동시 실행 가능한지 확인
         → 방법: 병렬 선언된 항목들의 입력 파일 목록 비교
           → 같은 파일을 쓰기(write)로 접근하는 항목이 2개 이상이면 충돌
           → 같은 파일을 읽기(read)만 하면 병렬 가능
         → 충돌 발견 시: MAJOR (병렬 불가, 순차로 변경 필요)

[FR-E16] 사전조건 충족 가능성 검증
         → 각 세션의 "사전조건"이 직전 세션(들)의 "산출물"로 실제 충족 가능한지
         → 방법: 세션 N의 사전조건 목록 → 세션 1~(N-1)의 산출물 합집합에 존재?
           → 사전조건에 있지만 어떤 이전 세션 산출물에도 없는 항목 = FAIL
         → 미충족 사전조건 발견 시: CRITICAL (실행 진입 불가)

[FR-E17] 리소스 제약 현실성 검증
         → 계획서가 전제하는 리소스(컨텍스트 윈도우, API 비용, 실행 시간 등)가
           현실적 범위 내인지 확인
         → 방법: 세션당 예상 토큰/파일 수 추정 → 컨텍스트 한계 초과 여부
           → 비용 추정치가 명시되어 있으면 합산 검증
         → 비현실적 전제 발견 시: WARNING

[FR-E18] 모호성 표현 탐지
         → L1(정의)/L2(실행)의 체크포인트, 완료조건, 게이트에
           검증 불가능한 모호 표현이 포함되어 있는지 확인
         → 탐지 대상 패턴:
           "~등", "필요 시", "적절히", "충분히", "가능한", "해당 시"
           "기타", "etc", "필요에 따라", "상황에 맞게"
         → 방법: 체크포인트/게이트 텍스트에서 모호 패턴 regex 매칭
         → 모호 표현이 완료 기준에 사용된 경우: MAJOR (검증 불가)
         → 모호 표현이 설명/참고에만 사용된 경우: MINOR (허용)

[FR-E19] 외부 사실 주장 검증 (모드 D FR-D19 자동 포함)
         → 계획서가 환경/설치/상태/버전을 사실로 기술하는 부분에 대해
           모드 D의 FR-D19 검증을 자동 실행
         → 발동 조건 (예시 — 이 목록에 없어도 "외부 사실 주장"이면 활성화):
           L1 또는 L2에 다음 패턴이 존재할 때 자동 활성화:
           - 설치 상태 기술 (OK/SKIP/FAIL/설치 완료/미설치)
           - pip install / npm install 등 패키지 설치 명령어
           - 버전 번호 (vX.Y.Z)
           - 실행 결과 참조 (PASS/FAIL/N건)
           - 환경 변수/API 키 상태
         → 발동 조건에 해당하지 않으면: SKIP (stamp에 "no_factual_claims_in_plan" 기록)
         → 검증 방법: Step 0-4d와 동일 (외부 사실 주장 추출 → 실측 대조)
         → 심각도: 모드 D FR-D19와 동일 (CRITICAL/MAJOR/MINOR)
         → 이유: 모드 E는 L1↔L2↔L3↔L4 정합성에 집중하므로,
                "문서 vs 현실" 검증이 구조적으로 누락됨.
                FR-E19가 이 간극을 자동으로 메움.
         → v13 실제 사례: 모드 E에서 PLAN-VERIFIED 판정 후,
           환경 설치 상태 3건 불일치가 사후 발견됨 → 이 항목으로 예방
```

### Step 2~3: 판정

```
판정 기준:
  CRITICAL: FR-E04, FR-E05, FR-E06, FR-E07, FR-E08 (L1↔L2 교차 — 방법론 불일치)
            FR-E14, FR-E16 (실행 가능성 — 순환/사전조건 미충족 시 실행 불가)
            FR-E19 CRITICAL급 (존재하지 않는 패키지/명령어 — 실행 불가)
  MAJOR:    FR-E01, FR-E02, FR-E03, FR-E09, FR-E11 (구조/요약/외부 불일치)
            FR-E15, FR-E18(완료기준 내 모호성) (병렬 충돌/검증불가 표현)
            FR-E19 MAJOR급 (상태 불일치 — OK↔미설치, FAIL↔해결됨)
  MINOR:    FR-E10, FR-E12, FR-E13 (수치/경로 — 실행 시 확인 가능)
            FR-E17, FR-E18(설명 내 모호성) (리소스/참고 수준 모호성)
            FR-E19 MINOR급 (날짜 경과, 버전 마이너 차이)

| 판정 | 조건 |
|------|------|
| PLAN-VERIFIED | Step 0 기준 확정 + CRITICAL 0건 + MAJOR 0건 + MINOR WARNING 허용 |
```

## 모드 F: 부재 탐지 + 초과 탐지 (Absence & Excess Detection)

### 이 모드가 해결하는 문제

```
문제: 모든 모드(A~E)는 "있는 것이 맞는지" 검증하지만,
     "있어야 하는데 없는 것"은 체계적으로 찾지 못함
원인: 모든 모드가 문서 내부에서 기준을 도출하므로,
     문서에 아예 언급되지 않은 항목은 기준 자체가 생성되지 않음
추가: "없어야 하는데 있는 것" (초과/잉여/deprecated) 역시 미탐지

검증 사분면:
              │ 문서에 있는 것        │ 문서에 없는 것
  ────────────┼──────────────────────┼──────────────────────
  있어야 하는 것│ Mode A~E (정확성)    │ Mode F (부재 탐지) ← 신규
  ────────────┼──────────────────────┼──────────────────────
  없어야 하는 것│ Mode F (초과 탐지) ← │ 정상 (검증 불필요)
              │           신규       │
```

### 발동 조건

```
/final-review absence
/final-review completeness
/final-review missing

자동 발동: "빠진 거 없어?", "누락 확인", "없는 부분 찾아줘"
또는 Mode D/E 실행 후 추가 완전성 확인이 필요할 때
```

### Step 0: 기준점 메타 검증 (부재 탐지 특화)

```
0-1: 외부 기준 소스 식별
  → 사용자 지정 기준 문서가 있는 경우:
    예: "SOT 기준으로 검증해줘" → docs/sot/* 를 기준으로 사용
    예: "v12 계획 대비 확인해줘" → v12 계획서를 기준으로 사용
  → 미지정 시 자동 탐색 (우선순위):
    1순위: 대상 문서가 참조하는 외부 문서 (import/include/참조 경로)
    2순위: 같은 디렉토리의 관련 문서 (이전 버전, 상위 문서)
    3순위: 프로젝트 내 SOT/요구사항 문서
    4순위: 없으면 → AI 도메인 생성 (접근 3, 한계 명시)
  → 사용된 기준 소스와 신뢰도 기록

0-2: 기대 항목 유형 정의
  → 기준 소스에서 추출할 항목 유형 결정:
    - 필수 항목 (MUST): 기준 소스에서 명시적으로 요구하는 것
    - 관례 항목 (SHOULD): 이전 버전/유사 문서에 존재하던 것
    - 권장 항목 (MAY): 도메인 지식 기반 제안
  → 유형별 심각도 매핑 사전 정의

0-3: 역질문
  [항목 단위]
  → "외부 기준이 불완전하면?" → 기준 자체의 완전성도 확인 필요
    → 기준 소스가 단일이면 → WARNING (단일 소스 의존)
    → 기준 소스가 2개 이상이면 → 교차 확인으로 보완
  → "의도적 제외를 오탐으로 잡으면?"
    → Step F-2에서 EXCLUDED 분류로 처리 (사용자 확인)
  [전체 체계 단위] "FR-F01~F08 전부 PASS인데 실제로 부재/초과가 남는 시나리오 3개 구성"
  → 예: 기준 소스 자체가 현실과 다르면 부재 탐지 결과 전체 오염
  → 예: 기준 소스에 없는 항목이 대상에 필요하지만 FR-F03(AI 도메인)으로만 커버
  → 전부 탐지 불가 시 → FR-F 항목 추가

0-4: 경험적 검증
  → 대상 문서에서 임의 섹션 3개 선정
  → 기준 소스에서 해당 섹션이 다뤄야 할 항목 추출
  → 실제 대상 문서에 존재하는지 확인
  → 이론적 기대 목록에 없던 부재 항목 발견 시 → 기준 보완
  → 깊이 예산: 직전 3개 추가 FR-F가 경험적 오류 0건이면 중단

0-4d: 기준 소스 신뢰성 검증 (부재 탐지 전제 조건)
  → 기준 소스로 사용되는 외부 파일/문서의 실존 확인:
    - 파일 경로 → ls/glob 실존 확인
    - 파일 수 주장 ("68개 SOT") → 실제 카운트 대조
  → 기준 소스가 환경/설치/상태를 기술하는 경우:
    - 해당 기술이 현실과 일치하는지 Step 0-4d(모드 D) 방식으로 검증
    - 기준 소스 자체의 사실 불일치 발견 시:
      → WARNING: "기준 소스 신뢰도 감소 — 해당 기준에서 도출된 기대 항목의 심각도 1단계 하향"
      → 예: MUST → SHOULD로 하향 (기준 자체가 틀렸을 가능성)
  → 기준 소스가 AI 도메인 생성(접근 4)인 경우: SKIP (검증 대상 없음)
  → stamp에 기록:
    {
      "reference_sources_verified": 숫자,
      "reference_sources_mismatched": 숫자,
      "reliability_adjusted": true|false
    }
```

### Step 1: 확정된 체크리스트로 검증

```
═══════════════════════════════════════════════
Phase A: 부재 탐지 (Absence Detection)
═══════════════════════════════════════════════

[FR-F01] 외부 기준 대조 (External Reference Checklist)
         → 기준 소스(SOT, 요구사항, 상위 문서)에서 "대상이 다뤄야 할 항목" 추출
         → 대상 문서에서 각 항목의 존재 여부 확인
         → 방법:
           1. 기준 소스에서 핵심 항목/토픽/요구사항 목록 추출
           2. 대상 문서 전체를 검색하여 각 항목의 커버 여부 판정
           3. 커버 = COVERED, 미커버 + 미언급 = MISSING
         → 예: SOT 68개에서 "PART2가 다뤄야 할 항목" 추출 → v13 계획에 매핑
           → 매핑 안 되는 항목 = 부재

[FR-F02] 이전 버전 비교 (Comparative Absence)
         → 이전 버전/유사 문서에서 존재했던 항목이 현재에도 존재하는지
         → 방법:
           1. 이전 버전의 섹션/항목/키워드 목록 추출
           2. 현재 문서에서 각 항목 존재 확인
           3. 부재 시: "의도적 제거" 명시 여부 확인
              → 명시됨 = EXCLUDED (정상)
              → 미명시 = REGRESSION_MISSING (회귀 누락 의심)
         → 예: v12 계획에 있던 "§6 참조 57건 검증"이 v13에 없음
           → 의도적 제거 표시 없음 → REGRESSION_MISSING

[FR-F03] 도메인 모델 생성 (Generative Completeness)
         → AI 도메인 지식으로 "이 유형의 문서에 통상 포함되는 항목" 생성
         → 방법:
           1. 대상 문서의 유형/목적 식별 (계획서, 검증보고서, 가이드 등)
           2. 해당 유형에 통상적으로 포함되는 섹션/항목 목록 생성
           3. 대상 문서에서 각 항목 존재 확인
           4. 부재 시: DOMAIN_SUGGESTED (AI 제안, 보장 못함 명시)
         → ※ 이 접근은 AI 지식에 의존하므로 신뢰도가 접근 1/2보다 낮음
         → 결과에 반드시 "AI 도메인 추론 — 확실성 제한" 표기

[FR-F04] 이해관계자 관점 (Stakeholder Coverage)
         → 문서의 각 이해관계자가 필요로 하는 정보가 포함되어 있는지
         → 방법:
           1. 대상 문서의 이해관계자 식별 (개발자, 감사자, 실행자, 유지보수자)
           2. 각 이해관계자 관점에서 "반드시 필요한 정보" 목록 생성
              개발자: 기술 상세, API, 구현 방법
              감사자: 체크포인트, 완료 기준, 추적성
              실행자: Step별 가이드, 사전조건, 산출물
              유지보수자: 명명 규칙, 문서화 수준, 의존성 명시
           3. 대상 문서에서 각 관점별 정보 존재 확인
         → 관점별 커버리지 비율 산출

═══════════════════════════════════════════════
Phase B: 초과 탐지 (Excess Detection)
═══════════════════════════════════════════════

[FR-F05] 폐기 항목 잔존 탐지 (Deprecated Retention)
         → 이전 버전에서 명시적으로 폐기/대체된 항목이 현재 문서에 남아있는지
         → 방법:
           1. 이전 버전의 변경이력/수정사항에서 "삭제", "대체", "폐기" 항목 추출
           2. 현재 문서에서 해당 항목 존재 여부 확인
           3. 존재 시: DEPRECATED_RETAINED (제거 필요)
         → 이전 버전이 없으면: SKIP (탐지 불가, 접근 3으로 보완)

[FR-F06] 범위 초과 탐지 (Scope Creep)
         → 문서의 선언된 범위/목적 밖의 내용이 포함되어 있는지
         → 방법:
           1. 문서의 목적/범위 선언 추출 (서문, 개요, 제목 등)
           2. 각 섹션이 선언된 범위 내에 있는지 판정
           3. 범위 밖 내용: SCOPE_EXCEEDED (제거 또는 범위 확장 필요)
         → 범위 선언이 없으면: WARNING (범위 미정의)

[FR-F07] 중복 콘텐츠 잔존 탐지 (Duplicate Retention)
         → 동일하거나 거의 동일한 내용이 문서 내 2곳 이상에 존재하는지
         → 방법:
           1. 유사 텍스트 블록 탐지 (문장/단락 수준)
           2. 의도적 반복(요약, 참조 목적)과 비의도적 중복 구분
           3. 비의도적 중복: DUPLICATE_FOUND (통합 필요)
         → 의도적 반복은 "참조: 원본 위치" 표시 여부로 구분

[FR-F08] 임시/미완성 콘텐츠 잔존 탐지 (Draft Retention)
         → TODO, FIXME, TBD, WIP, "임시", "나중에", "확인 필요" 등
           임시 마커가 최종 문서에 남아있는지
         → 방법: regex 패턴 매칭
           "TODO|FIXME|TBD|WIP|HACK|XXX|임시|나중에|확인.필요|미정|미확인"
         → 발견 시: DRAFT_MARKER_FOUND (해소 필요)
```

```
    MINOR:    FR-F06 SCOPE_EXCEEDED (범위 초과 — 유해하지 않은 경우)

              FR-F08 DRAFT_MARKER_FOUND (참고 영역 내 임시 마커)

|------|------|

    → 사용자에게 기준 소스 제공 요청
```

### stamp 무효화 조건

```
1. SOT 원본 파일이 변경됨 (source_file_hash 불일치)
2. EA/CM 파일이 수정됨 (target_hash 불일치)
3. 검증 스크립트가 업데이트됨 (tool_versions 불일치)
4. 기준 자체가 변경됨 (criteria_hash 불일치)     ← 신규
5. 사용자가 명시적으로 /final-review --force 실행

→ 무효화 시 stamp 삭제 → Step 0부터 재실행 필요
→ 특히 조건 4: 기준이 바뀌면 이전 판정은 자동 무효

stamp 무결성 보호:
→ stamp 생성 시 전체 JSON의 SHA-256 해시를 "integrity_hash" 필드에 기록
→ stamp 읽기 시 integrity_hash 재계산하여 일치 확인
→ 불일치 시: "STAMP_TAMPERED" 경고 + stamp 무효 처리
→ 계산 방법: integrity_hash 필드를 제외한 나머지 JSON을 정렬 직렬화 후 SHA-256

stamp 이력 보존:
→ 새 stamp 생성 시 기존 stamp를 final_review_stamp_history.json에 append
→ 이력 파일 구조: [{"archived_at": "...", "stamp": {...}}, ...]

→ 이력을 통해 판정 변화 추적 가능 (FINAL→NOT_FINAL→FINAL 등)
```

## Step 0이 무한 재귀를 끊는 이유

```
Q: "기준이 맞아?"
A: Step 0-1~0-3 (이론적 검증)

Q: "그 이론적 검증이 맞아?"
A: Step 0-4 (경험적 검증 — 실제 데이터로 확인)

Q: "경험적 검증도 놓칠 수 있지 않아?"
A: 경험적 검증은 "현실에서 틀린 것"을 찾으므로
   AI 지식과 무관하게 작동함.
   → 이론에서 놓친 오류도 현실에서 드러남
   → 이론(0-1~0-3) + 경험(0-4) = 무한 재귀의 닻

Q: "/final-review 자체가 최종이야?"
A: Step 0-5 자기참조 교차검증으로 확인:
   → 0-5a: 실제 실행해서 오류를 잡았는가? (외부 증거)
   → 0-5b: 다른 스킬 판정과 일치하는가? (교차 확인)
   → 0-5c: 이론과 경험이 모순 없는가? (자기모순 탐지)
   → 자기 기준만으로 판단하지 않고, 외부 증거 + 교차 검증으로 보완
   → 일반 검증 로직은 변경 없이 유지 (치우침 없음)

Q: "있는 건 확인했는데, 없는 건 어떻게 찾아?"
A: Mode F(부재 탐지)로 확인:
   → FR-F01: 외부 기준 문서에서 "있어야 할 것" 목록 추출 → 대상 대조
   → FR-F02: 이전 버전과 비교 → 빠진 항목 식별
   → FR-F03: AI 도메인 지식으로 "통상 필요한 것" 생성 (보장 제한)
   → FR-F04: 이해관계자별 필요 정보 확인
   → 외부 기준이 있으면 결정론적 부재 탐지 가능
   → 외부 기준이 없으면 AI 추론에 의존 (한계 명시)

Q: "Mode A~E는 PASS인데 Mode F에서 MISSING이 나오면?"
A: Cross-mode reconciliation으로 처리:
   → Mode A~E: "있는 것이 맞다" (정확성 확인됨)
   → Mode F: "있어야 할 것이 빠져있다" (완전성 미달)
   → 두 판정은 모순이 아님 — 정확하지만 불완전한 상태
   → 보수적 판정 채택: "정확하지만 불완전" → NOT_FINAL 유지

Q: "그래도 100%는 아니지?"
A: 맞음. 남은 0.5~2%는:
   → LLM 본질적 한계 (확률적 생성)
   → SOT 원본의 모호성

   → 이것은 "기준 부족"이 아니라 "기술 한계"이므로
```

## 검증의 수렴 한계와 완화 전략

대용량 컨텍스트 처리 + 다층 검증 과정에서 발생하는 파생 오류는
궁극적으로 3가지 근본 한계로 수렴합니다. 이들은 해결이 아닌 완화만 가능합니다.

### 수렴 한계 1: AI 추출의 비결정성

```
문제: AI가 chunk에서 "핵심값"을 추출할 때, 실행마다 다른 값을 추출할 수 있음
     추출 스키마를 고정해도 "어떤 값을 넣을지"는 확률적
     → 체크포인트 기반 검증의 재현성이 보장되지 않음

영향 범위:
  - Step -1-4 체크포인트 추출 (오류 2-A, 4-A)
  - 교차 파일 비교 시 표현 불일치 (오류 7-A)

완화 전략 (SKILL.md 내장):
  [W1-1] 모드별 추출 스키마 고정 (위 "모드별 추출 스키마" 참조)
    → 추출 필드를 사전 정의하여 자유도 제한
  [W1-2] confidence 점수 부여 (각 추출값에 0.0~1.0)
    → confidence < 0.7인 항목은 자동 spot-check 트리거
  [W1-3] 정규화 레이어 (위 "교차 파일 비교 정규화 규칙" 참조)
    → 동의어/표현 차이를 정규화로 흡수
  [W1-4] 이전 체크포인트와 모순 검사
    → 같은 파일의 다른 chunk에서 모순되는 값 추출 시 경고

잔여 위험: ~2% (스키마 외 암묵적 정보 누락 가능)
  → stamp에 "extraction_confidence_avg" 기록
  → 평균 confidence < 0.8이면 stamp에 WARNING 부기
```

### 수렴 한계 2: 정보 손실의 불가피성

```
문제: 대용량 파일 → 체크포인트 요약 변환 시 정보 손실은 물리적으로 불가피
     개별 사실은 보존되지만, 사실 간의 관계/맥락이 손실됨
     → "구조적으로 맞지만 의미적으로 틀린" 판정 가능

영향 범위:
  - chunk 간 컨텍스트 단절 (오류 4-B)
  - 섹션 간 교차 참조 누락 (오류 6-B)
  - 추출본 비교 시 맥락 손실 (오류 7-B)

완화 전략 (SKILL.md 내장):
  [W2-1] relations 필드에 관계 정보 명시적 기록
    → {"from": "S15", "to": "S14", "type": "depends_on"} 형태
  [W2-2] 2단 체크포인트 구조 (chunk별 상세 + 전체 요약)
    → 의심 시 상세 레벨로 drill-down 가능
  [W2-3] 교차 참조 전용 별도 패스
    → 섹션 독립 검증 후, 교차 참조만 별도로 검증
    → 이때 관련 chunk 2개를 동시 로드하여 비교
  [W2-4] 손실 영역 명시적 기록
    → 체크포인트에 "not_extracted" 필드: 추출하지 않은 정보 유형 기록
    → 후속 검증에서 해당 영역 spot-check 가능

잔여 위험: ~3% (암묵적 관계, 문맥 의존적 의미)
  → stamp에 "information_coverage_estimate" 기록 (추정 커버리지 %)
  → 커버리지 < 85%이면 사용자에게 수동 확인 권장
```

### 수렴 한계 3: 검증의 자기참조 한계

```
문제: 커버리지 매트릭스가 완전한지 확인하려면 또 다른 매트릭스가 필요 → 무한 회귀
     RAG 결과가 맞는지 확인하려면 원본을 읽어야 하는데, 원본이 크면 또 RAG 필요
     → 검증 도구 자체의 완전성을 검증 도구로 증명할 수 없음

영향 범위:
  - 커버리지 매트릭스 완전성 (오류 6-C)
  - RAG false confidence (오류 3-C)
  - 검증 기준 자체의 검증 (Step 0의 메타 검증)

완화 전략 (SKILL.md 내장):
  [W3-1] Mode F(외부 기준 대조)로 자기참조 끊기
    → 내부 생성 기준이 아닌 외부 문서에서 기대 항목 도출
  [W3-2] 다중 소스 교차 확인
    → 단일 소스 의존 시 WARNING
    → 2개 이상 독립 소스에서 같은 결론 → 신뢰도 상승
  [W3-3] 경험적 검증(Step 0-4)으로 이론적 한계 보완
    → 실제 데이터 3건 이상 spot-check
  [W3-4] 사람의 최종 확인 경계 설정
    → CONTEXT-DEGRADED + confidence_avg < 0.8인 경우:
      "AI 검증 완료, 사람의 최종 확인 권장" 명시

잔여 위험: ~1% (자기참조 한계는 기술적으로 해소 불가)
  → 이것은 괴델의 불완전성 정리와 유사한 구조적 한계
  → 기준을 더 추가해도 해결되지 않음 → 자연 종료점
```
