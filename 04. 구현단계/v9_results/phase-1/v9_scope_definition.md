## 3. v9 SCOPE DEFINITION

```
v9 SCOPE DEFINITION
====================================================================

IN SCOPE (v9 검증 대상):
  - PART2 문서 내부의 정합성 (섹션 간 수치/명칭/번호 일치)
  - 완전성 (누락 항목, 미참조 산출물)
  - 구현 가능성 (모호/불완전 명세, 모순 지시)
  - 수량 일관성 (모듈 수, 항목 수, 파일 수 교차 검증)
  - 파일 경로 정합성 (PART2 경로 ↔ PHASE_B2 모노레포 구조)
  - 의존성 순서 (Stage/Phase 간 전제조건 체인)
  - 외부 의존성 실현성 (SOT 문서 참조 유효성)
  - §7.5 중 "문서 검증 가능" 태그 항목 (10개):
    · I-모듈 번호 일관성
    · E-모듈 역할 일관성
    · B/C/D-Series 존재 확인
    · 스키마 버전 통일
    · LOCK 값 반영 (문서 간 대조)
    · RBAC 역할 통일
    · 비용 임계값 단일화
    · SDAR NEVER_AUTO 완전성
    · 정본 우선순위 준수
    · SOURCE_CONFLICT 전수 해결 상태

OUT OF SCOPE (v9 범위 외):
  - 코드 레벨 동작 검증 (Gate bypass, PII 마스킹, Non-goal 차단 등)
  - 성능 벤치마크 (응답 시간, 동시 처리, 토큰 카운팅)
  - 보안 침투 테스트 (Docker 샌드박스, 네트워크 격리)
  - 런타임 행위 검증 (P2 자동 OFF, 비용 상한 동작)

BOUNDARY RULE:
  - §7.5 항목 중 "문서 검증 가능" 태그만 v9 범위 내
  - "구현 후 검증" 태그 항목은 v9에서 SKIP 처리 (RULE-14 적용)
  - SKIP된 항목이 프롬프트 결과에 출현 시 → FALSE_NEGATIVE가 아닌 OUT_OF_SCOPE로 분류

====================================================================
```

## 5. RULE-14 연동

| 규칙 | 내용 | 약점 번호 |
|------|------|----------|
| RULE-14 | v9 범위 = PART2 문서 내부의 정합성, 완전성, 구현 가능성. §7.5의 "코드 동작 검증" 항목(PII 마스킹 동작, Gate 우회 불가 등)은 v9 범위 외 → SKIP 처리 | #22 |

**적용 방법**: Phase 1 Wave 실행 시, 결과 판정에서 §7.5.2(보안 검토), §7.5.3(성능 검토)의 구현 후 검증 항목이 결함으로 보고될 경우:
1. 해당 항목을 `OUT_OF_SCOPE`로 재분류
2. FN(False Negative) 또는 REAL_ERROR 집계에서 제외
3. 별도 `out_of_scope_log`에 기록하여 구현 단계 검증 시 참조
