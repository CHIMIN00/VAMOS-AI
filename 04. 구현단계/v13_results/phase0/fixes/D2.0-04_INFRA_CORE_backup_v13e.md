## 0.1 개발 환경 셋업

### 전제 조건
- Python 3.11 이상
- Docker Desktop (V2/V3) 또는 로컬 Python 환경 (V1)
- WSL2 (Windows 사용자, V1 이상 권장)

### V1 로컬 셋업 절차
1. 저장소 클론 및 `venv` 생성: `python -m venv .venv && source .venv/bin/activate`
2. 의존성 설치: `pip install -r requirements.txt`
3. 환경 변수 설정: `.env.example`을 `.env`로 복사 후 API 키/경로 입력
4. SQLite DB 초기화: `python scripts/init_db.py`
5. Chroma 로컬 인스턴스 기동: `python scripts/start_chroma.py`
6. ORANGE CORE 실행: `python -m vamos.core.main`

### V2 Docker Compose 셋업 절차
1. `docker-compose.yml` 환경 변수 확인 (`.env` 파일 참조)
2. `docker compose up -d` 로 전체 스택 기동
3. Postgres 마이그레이션: `docker compose exec core python scripts/migrate.py`
4. Qdrant 헬스체크: `curl http://localhost:6333/healthz`
5. ORANGE CORE API 헬스체크: `curl http://localhost:8000/health`

> 주의: API 키 등 민감 정보는 `.env`에만 보관하고 VCS에 커밋하지 않는다.

## 1. INFRA-CORE 목표/원칙(뇌 교체 가능, 하드웨어 독립)
- Brain Engine(모델/HW/런타임)은 교체 가능해야 한다. (문서 체크리스트 요구)
- CORE는 “판단/통제”에 집중하고, 실행 리소스는 INFRA 계층에서 추상화한다. (RULE 1.3 철학: CORE 판단, NODE 실행)
- 비용 상한(V1/V2/V3)과 승인 구조는 INFRA 동작의 최상위 제약으로 반영되어야 한다. (RULE 1.3 §5, §9)

## 2. Brain Engine Modularization (CORE vs Brain)
- CORE(ORANGE CORE):
  - 정책/승인/비용/도메인 라우팅의 “결정”을 담당 (RULE 1.3 §4.1)
- Brain Engine(OTHER BRAINS):
  - 모델 실행/계산 리소스 제공
  - CORE는 Brain Engine의 내부 구현(모델명/HW)을 직접 의존하지 않는다.

- 결정 완료(§2): 옵션 A — Brain Engine은 “모델(텍스트/멀티모달)” 실행 계층까지만 포함한다.
- 도구 런타임(코드 실행기/브라우저/크롤러/API)은 §7 Tool 실행 계층에서 관리하며, 07의 PolicyCheck/Approval/Cost 게이트를 선행 조건으로 한다. (역할 충돌 방지)

## 3. Brain Adapter Layer(표준 인터페이스)
- Adapter의 목적: CORE↔Brain Engine 사이의 표준 인터페이스 제공(구현 교체 가능)
- 계약:
  - CORE는 “요청 유형/제약/예산/정책 결과”만 전달
  - Adapter는 해당 요청을 특정 Brain Engine 호출로 변환하고 결과를 표준 형태로 반환

> 결정 완료(LOCK): 표준 응답 envelope 최소 필드는 “추적/감사 가능 수준”을 따른다.

- 최소 필드(LOCK):
  - output_text: str  # 모델 출력 텍스트
  - evidence_summary: str  # 근거 요약
  - cost_used_estimate: float  # 추정 비용(토큰 기준)
  - warnings: list[str]  # 경고 목록
  - tool_calls: list[ToolCallRecord] | None  # 도구 호출 기록(선택)
  - trace_id: str  # 추적/감사용 UUID
  - qod_hint: float | None  # 품질 추정(0.0~1.0, DEC-010 LOCK)(선택)
- 선택(optional):
  - tool_calls: list[ToolCallRecord] | None
  - qod_hint: float | None  # 0.0~1.0(DEC-010 LOCK)
- 규칙:
  - tool_calls/qod_hint는 “가능할 때만” 포함(도구 미사용/미측정이면 생략 가능).
  - trace_id는 추적/감사 목적상 가능한 한 항상 포함한다.

### 3.2 OHLCV 조건부 Required 스키마 (Pydantic)

```python
from pydantic import BaseModel, Field, model_validator
from typing import Optional
from datetime import datetime

class OHLCVRecord(BaseModel):
    timestamp: datetime = Field(..., description="캔들 시각 (UTC)")
    open: Optional[float] = Field(None, description="시가")
    high: Optional[float] = Field(None, description="고가")
    low: Optional[float] = Field(None, description="저가")
    close: float = Field(..., description="종가 (항상 required)")
    volume: Optional[float] = Field(None, description="거래량")
    source: str = Field(..., description="데이터 소스 식별자")

    @model_validator(mode="after")
    def ohlv_all_or_none(self) -> "OHLCVRecord":
        """open/high/low/volume은 모두 있거나 모두 없어야 한다."""
        partial = [self.open, self.high, self.low, self.volume]
        filled = [v for v in partial if v is not None]
        if 0 < len(filled) < 4:
            raise ValueError("open/high/low/volume은 모두 제공되거나 모두 생략해야 합니다.")
        return self

    class Config:
        extra = "forbid"
```

> 결정(LOCK): ConnectorResponse는 Brain Adapter가 CORE에 반환하는 표준 봉투(envelope)이다. OHLCVRecord는 외부 데이터 커넥터가 INFRA로 전달하는 시계열 단위 레코드이다.

## 4. Hardware Abstraction Layer(HAL) & 실행자원(“OTHER BRAINS”)
- HAL은 “어떤 HW/런타임을 쓰든 동일 계약으로 호출”되게 한다.
- 실행자원 예:
  - Front Mini / Main / Tool 계층(모델 계층) (RULE 1.3 §4.6)
  - 로컬/서버/외부 API 등은 07 정책 게이트를 거쳐 사용

### 4.1 OTHER BRAINS Layer 매핑 표 (Design 2.0 / PHASE 3) (LOCK)

| 인프라 유형 | 역할 | 호출 주체 | Gate 통과(정본=07) |
|------------|------|----------|--------------------|
| Search Engine | 웹/문서 검색 | I-2 | 07: policy, cost |
| Vector DB | 임베딩 저장/검색 | I-2, I-3 | 07: cost (+06: storage policy if write) |
| Code Executor | 샌드박스 실행 | BLUE NODE | 07: policy, cost (+approval 조건부: risk_class=med/high) |
| External API | 외부 연동 | I-11 | 07: policy, cost (+approval 조건부: P2/고위험) |
| Solver | 논리/수학 | I-5 | 07: cost (+policy 조건부: P2/제약요청) |

- 주의(LOCK): 위 모든 호출은 7.2 ToolRegistry의 `tool_id` 경유 규칙을 우선 적용한다.
- 기록(LOCK): 각 호출의 gate 결과는 02 Decision 확장 필드의 `gates.result.*`에 귀결된다. (07의 PolicyCheck/Approval/Cost 결과를 그대로 연결)

#### 4.1.1 04→07 Gate 연결 규칙(LOCK) — OTHER BRAINS 실행 공통 계약

- 원칙(LOCK):
  - OTHER BRAINS 실행은 항상 ToolRegistry(7.2)를 경유한다. (직접 URL/SDK/코드 호출 금지)
  - ToolRegistryEntry.required_gates에 따라 07 Gate를 선행 통과해야 한다.
  - Gate 결과는 02 Decision 확장 필드에 그대로 기록된다.
    - Decision.gates.result.policy  ← 07 PolicyCheck(deny/restrict/allow)
    - Decision.gates.result.approval ← 07 Approval(approved/denied/pending)
    - Decision.gates.result.cost     ← 07 CostBudget/Downshift/Block 결과

- 보조 규칙:
  - 저장/메모리 계층 정책(06)은 “저장 행위”에 대한 제약이며,
    OTHER BRAINS 실행 자체의 Gate(정책/승인/비용)는 07을 정본으로 한다.


#### 4.1.2 외부 데이터 소스 외부화 규칙 (ADD-073)

> **배경**: Goldman Sachs 등 특정 데이터 소스 식별자가 코드에 하드코딩된 것이 확인되었다. 이는 소스 변경 시 코드 수정이 필요하고, 정책/비용 게이트 우회 위험을 초래한다.

#### 외부화 원칙 (LOCK)
- 모든 외부 데이터 소스 식별자(provider name, endpoint, credential ref)는 **설정 파일**(`config/data_sources.yaml`)에서 관리한다.
- 코드 내 소스 문자열 리터럴(`"goldman_sachs"`, `"bloomberg"` 등) 직접 참조를 금지한다.
- 소스 참조는 반드시 `SourceRegistry`를 통해 `source_id`로만 접근한다.

#### 설정 파일 구조 예시 (`config/data_sources.yaml`)

```yaml
data_sources:
  - source_id: gs_equity_us
    display_name: "Goldman Sachs US Equity"
    provider: goldman_sachs
    endpoint_ref: "${GS_API_ENDPOINT}"   # 환경 변수 참조
    credential_ref: "${GS_API_KEY}"
    risk_class: medium
    required_gates: [policy, cost]
    enabled: true

  - source_id: bloomberg_fx
    display_name: "Bloomberg FX"
    provider: bloomberg
    endpoint_ref: "${BLOOMBERG_ENDPOINT}"
    credential_ref: "${BLOOMBERG_API_KEY}"
    risk_class: medium
    required_gates: [policy, cost]
    enabled: false   # 비활성화 시 코드 변경 없이 차단
```

#### SourceRegistry 인터페이스

```python
class SourceRegistry:
    def get(self, source_id: str) -> DataSourceConfig: ...
    def list_enabled(self) -> list[DataSourceConfig]: ...
    def is_allowed(self, source_id: str, gate_result: GateResult) -> bool: ...
```

> 결정(LOCK): `config/data_sources.yaml`이 부재하거나 `source_id`가 미등록인 경우, ToolRegistry 경유 없이 호출을 시도하면 즉시 `PolicyViolationError`를 발생시킨다.

##### 4.3.6.2 TCO(총소유비용) 프레임워크

- TCO 구성 요소:
  - **CapEx**: HW 구매/임대 초기 비용
  - **OpEx**: 전력, 냉각, 유지보수, 인건비
  - **API 비용**: 클라우드 LLM/벡터 API 사용료 (상대 비용으로 표기, 실제 금액은 07 CostBudget 계측 후 확정)
  - **기회비용**: 운영 장애/다운타임에 따른 비용
- 평가 주기: V2 이상에서는 월 단위 TCO 리뷰를 수행하고, 임계값 초과 시 다운시프트 전략을 07과 연동하여 자동 적용한다.

##### 4.3.6.3 냉각/전력 고려사항

- V1: 일반 데스크톱 전력 범위(65W~150W TDP) 내 유지. 별도 냉각 불필요.
- V2: 서버 또는 워크스테이션 기준 UPS 연결 권장. 공냉 또는 수냉 중 환경에 맞게 선택.
- V3: 데이터센터 수준 전력 밀도(랙당 kW) 및 냉각 용량(PUE 목표 1.4 이하) 계획 수립 필요.

##### 4.3.6.6 NVLink / InfiniBand (멀티 GPU 연결)

- NVLink: 동일 노드 내 GPU 간 고대역폭 연결. V3에서 다중 GPU 추론 시 필요.
- InfiniBand: 노드 간 저지연 통신. V3 분산 학습/추론 클러스터에서 필수.
- V1/V2: 해당 없음. 단일 GPU 또는 CPU 기반 운영.
- 원칙(LOCK): NVLink/InfiniBand 구성은 V3 운영 계획 수립 시 인프라 설계 단계에서 별도 검토한다.

##### 4.3.6.7 분산 학습 정책

- VAMOS P1(V1/V2/V3) 범위에서 분산 학습은 **Non-goal**로 분류한다. (RULE 1.3 BASE Non-goal 준수)
- 분산 추론(멀티 GPU 서빙)은 V3에서 선택적으로 검토 가능하나, 도입 전 07 Approval 게이트를 통과해야 한다.
- 분산 학습이 필요한 경우: CORE가 아닌 외부 Fine-tuning 파이프라인으로 처리하고, 결과 모델을 Brain Engine에 등록하는 방식을 원칙으로 한다.

### 9.2b 개발 환경 셋업 절차 (MOD-026)

- 파일/SQLite 동시성:
  - SQLite는 단일 writer 모델이라, 로그/메모리 등 다수 쓰기 작업에서는 lock 경합이 발생하므로, 배치 인서트·전용 writer 프로세스를 사용한다.
  - JSONL 파일은 프로세스 간 동시 append 시 손상 위험이 있으므로 파일 락 또는 로깅 전용 프로세스를 둔다.

- 로그 롤링/압축/보관:
  - JSONL/DB 로그는 일 단위 롤링 + gzip 압축 + 90일 이상 보관 정책을 기본으로 두어, 디스크 사용량·쿼리 성능 저하를 방지한다.
  - Postgres 로그/LogEvent 테이블은 파티셔닝 또는 TTL 정책으로 오래된 데이터는 아카이브/삭제한다.

- 백업/스냅샷/복구 리허설:
  - Postgres/Qdrant/Redis 등 핵심 스토리지는 주기적 스냅샷·백업을 설정하고, 실제 복구 리허설을 통해 RTO/RPO를 검증한다.
  - 로컬 V1 구성을 쓰더라도 Git 등으로 스키마/설정/샘플 데이터를 버전 관리해 머신 교체 시 복원이 쉽도록 한다.

- 비용 폭주 방지:
  - OpenAI/Anthropic 등 LLM API는 계정/프로젝트별 월 한도·알림을 설정하고,
    07의 CostBudget 게이트(상한 초과 시 차단/승인요청/HOLD 등) 운용과 일치시키도록 한다.
  - 다운시프트 전략(cheap 모델로 강제 라우팅, 기능 비활성화, HOLD→승인 요청)을 구현해 API 사용량이 예산을 초과하지 않도록 한다.

- PII/민감 데이터 처리:
  - 장기 저장(L1/L2) 전에 PII 감지와 마스킹을 수행하고, 정책상 Deny인 경우에는 contentsummary 수준만 L0에 남긴다.
  - 클라우드 LLM API로 민감 데이터를 전송할 때는 비식별화/요약본을 우선 사용하고, 원본은 로컬/내부용으로만 처리한다.

- 스키마/버전 호환성 및 **데이터 마이그레이션 전략 (MOD-016)**:
  - V1(로컬) → V2/V3(서버)로 갈 때 LogEvent/MemoryRecord/CostBudget 스키마는 컬럼 추가 방식으로만 확장하고, 제거/변경은 마이그레이션 스크립트를 사용한다.
  - 이벤트 네이밍 규칙(event_type, failure_code, fallback_id)은 ORANGE CORE/02의 정본을 기준으로 재사용한다.
  - **V1→V2 마이그레이션 절차**:
    1. SQLite DB 덤프: `sqlite3 vamos.db .dump > v1_backup.sql`
    2. Postgres 스키마 생성: `python -m vamos.migrate --target v2`
    3. 데이터 이관: `python -m vamos.migrate --from sqlite --to postgres`
    4. 벡터 데이터 이관: Chroma → Qdrant (임베딩 재생성 또는 직접 이관)
    5. 검증: 레코드 수 비교 + 샘플 쿼리 검증
  - **V2→V3 마이그레이션 절차**:
    1. Postgres 스냅샷 백업 (RDS 또는 pg_dump)
    2. K8s 매니지드 Postgres로 pg_restore
    3. Qdrant 볼륨 마이그레이션 또는 Qdrant Cloud 이관
    4. 설정 환경변수 `VAMOS_ENV=v3`으로 전환
    5. 롤링 배포 후 헬스체크 확인
  - 마이그레이션 중 다운타임: V2→V3는 최대 30분 유지보수 창 허용

- 관측성/트레이스:
  - 모든 요청에 trace_id를 부여하고, ui.* → oc.* → bn.* → if.* → wf.* → mem./sc.* 체인을 따라 LogEvent를 남겨 추후 디버깅·감사 가능하도록 한다.
  - 최소한 요청 수, 실패율, 토큰 사용량, 평균 지연을 메트릭으로 수집해, 비용·성능 최적화에 활용한다.

- 단일 서버 SPOF 관리 및 **장애 복구 절차 (MOD-019)**:
  - V2 조합에서는 단일 VM 장애가 전체 서비스 중단으로 이어지므로, 주기적 스냅샷·재시작 스크립트·헬스 체크를 준비한다.
  - 점진적 확장을 고려해, 코드 레벨에서 DB/벡터/로그를 인터페이스(Repository/LogSink)로 추상화해두면 K8s/V3로 옮기기 쉽다.
  - **장애 시나리오별 복구 절차**:
    | 시나리오 | 감지 방법 | 복구 절차 | 목표 RTO |
    |---------|----------|----------|---------|
    | DB 커넥션 끊김 | 헬스체크 실패 | `docker compose restart postgres` → 커넥션 풀 재시작 | 5분 |
    | LLM API 오류(429) | 오류율 급증 | 자동 큐잉 + 지수 백오프 재시도 (max 3회) | 2분 |
    | 디스크 용량 부족 | 로그 롤링 실패 | 오래된 JSONL 압축/삭제 → 알림 발송 | 10분 |
    | Core 프로세스 크래시 | PID 없음 | `docker compose up -d core` → 로그 확인 | 3분 |
    | Qdrant 데이터 손상 | 쿼리 오류 | 최근 스냅샷 복원 → 재인덱싱 | 30분 |
  - 복구 후 필수 확인: trace_id 연속성, 미처리 큐 항목 처리, 비용 상한 초과 여부

- 로그 스택 선택 시 주의:
  - ELK는 강력하지만 리소스/운영 비용이 크므로, 로그 볼륨·팀 역량을 고려해 Loki+Grafana 등 더 가벼운 대안을 우선 검토한다.

#### 10.4.2 자체 트레이싱 (S06-B19-003, CRITICAL)

### 10.6 Fallback Chain + Circuit Breaker (S13-AINV, S09-B34)

#### 10.6.1 Fallback Chain
- Brain Engine 또는 외부 API 실패 시 대체 경로를 순차 시도.
- 체인 구성: `config/fallback_chains.yaml`.
- 예시: `gpt-4o → claude-sonnet → local-ollama → error_response`.
- 체인 최대 깊이: 3단계 (무한 재시도 방지).

#### 10.6.2 Circuit Breaker
- 연속 실패 감지 시 해당 엔진/API를 일시 차단하여 리소스 낭비 방지.
- 상태 머신: CLOSED → OPEN (연속 3회 실패) → HALF-OPEN (60초 후 1건 시도) → CLOSED. (D2.0-05 §4.4 LOCK 기준)
- Circuit 상태 이벤트: `infra.circuit.opened`, `infra.circuit.closed`.
- 차단 중 요청은 즉시 Fallback Chain 다음 단계로 이동.

  - 환경변수/Vault 관리 (HashiCorp Vault 또는 .env + encryption)
  - 로그 마스킹: API Key, 토큰, PII 자동 마스킹
  - 90일 키 로테이션 자동화

### P7-MIG: 배포·환경 분리 (3건)
- **정의 (LOCK)**: V0: 수동 배포. V1: Blue-Green (단일 서버). V2: Rolling Update (ECS). V3: Canary Deployment (K8s)
- **확장**: 배포 파이프라인
  - Blue-Green 배포: 무중단 전환, 즉시 롤백 가능
  - 환경 분리: dev → staging → prod (프로모션 규칙)
  - 데이터 마이그레이션 자동화: Alembic/Flyway 스타일 스크립트

### P7-RTG: 라우팅 인프라 상세 (3건)
- **정의 (LOCK)**: V0: 하드코딩 규칙. V1: 규칙 기반 라우터 (I-10 Router). V2: 가중치 기반 동적 라우팅. V3: ML 기반 적응형 라우팅 (S-5 Router Evolution 연동)
- **확장**: 모델 라우팅 엔진
  - 4축 결정 트리: 복잡도 × 비용 × 속도 × 도메인
  - 모델 풀 동적 관리: 등록/해제/헬스체크/자동 교체
  - A/B 테스트 프레임워크: 트래픽 분배 → 메트릭 비교 → 자동 승격

### P7-LOG: 로그 저장 인프라 (1건)
- **정의 (LOCK)**: V0: stdout + 파일 로테이션. V1: 구조화 로그 (JSON) + SQLite. V2: ELK Stack (Elasticsearch). V3: 분산 트레이싱 (Jaeger + OpenTelemetry)
- **확장**: 로그 스토리지
  - SQLite + 해시 체인 (변조 방지)
  - 일간 파일 분리, 월간 아카이브
  - 보존 정책: 일간 90일, 월간 2년, 연간 영구
